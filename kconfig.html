    VERSION 3.8 & 4.x · KRaft

    Overview

    This guide covers production configuration of Apache Kafka across the 3.8 and 4.x lines, organized by component (broker/controller, producer, consumer) and by use-case scenario. Both run on KRaft (the built-in Raft metadata quorum). In 3.8, ZooKeeper is deprecated but still selectable; the metadata version is 3.8-IV0. In 4.0+, ZooKeeper is completely removed — KRaft is the only mode. Parameters that differ between versions are tagged inline with a version badge.

    How to read this guide: Each parameter lists its default, what it does, pros/cons of changing it, and a recommendation. Scenario sections give copy-paste config blocks tuned for a specific goal (throughput, latency, durability, etc.).

    1. Version Landscape: 3.8 → 4.x

    Kafka 4.0 (released March 18, 2025) is the first major release to run without ZooKeeper. Kafka 4.1 (Sept 2025) is the latest stable line. There is no direct ZooKeeper → 4.x upgrade: you must migrate to KRaft on a 3.x build (3.9 is the recommended bridge), finalize, then roll to 4.x.

    AreaKafka 3.8Kafka 4.0 / 4.1+
    Upgrade path: ZooKeeper cluster → migrate to KRaft on 3.7+/3.9 (dual-write phase, rollback possible) → finalize metadata with kafka-features.sh → rolling upgrade brokers to 4.x → finalize with --release-version 4.0. After consumer groups adopt the new (KIP-848) protocol, the cluster can only be downgraded to 3.4.1+.

    2. Architecture & Roles

    In KRaft, each process plays one or more process.roles: broker, controller, or both. Controllers manage cluster metadata via a Raft quorum (stored in the internal __cluster_metadata topic); brokers handle client produce/fetch traffic. For production, run an odd number (3 or 5) of dedicated controllers and separate brokers. This layout is identical in 3.8 and 4.x — the difference is that 4.x has no ZooKeeper fallback at all.

    # Dedicated controller node
process.roles=controller
node.id=1
controller.quorum.voters=1@c1:9093,2@c2:9093,3@c3:9093
listeners=CONTROLLER://:9093

# Dedicated broker node
process.roles=broker
node.id=11
controller.quorum.voters=1@c1:9093,2@c2:9093,3@c3:9093
listeners=PLAINTEXT://:9092
inter.broker.listener.name=PLAINTEXT

    Combined mode (broker,controller) is acceptable for dev or small clusters but not recommended for large production deployments where you want fault isolation.
