    VERSION 3.8 · KRaft

    Overview

    This guide covers production configuration of Apache Kafka 3.8, organized by component (broker/controller, producer, consumer) and by use-case scenario. Kafka 3.8 runs on KRaft (the built-in Raft metadata quorum); ZooKeeper is deprecated and removed in 4.x, so all examples use KRaft. The metadata version for 3.8 is 3.8-IV0.

    How to read this guide: Each parameter lists its default, what it does, pros/cons of changing it, and a recommendation. Scenario sections give copy-paste config blocks tuned for a specific goal (throughput, latency, durability, etc.).

    1. Architecture & Roles

    In KRaft, each process plays one or more process.roles: broker, controller, or both. Controllers manage cluster metadata via a Raft quorum; brokers handle client produce/fetch traffic. For production, run an odd number (3 or 5) of dedicated controllers and separate brokers.

    # Dedicated controller node
process.roles=controller
node.id=1
controller.quorum.voters=1@c1:9093,2@c2:9093,3@c3:9093
listeners=CONTROLLER://:9093

# Dedicated broker node
process.roles=broker
node.id=11
controller.quorum.voters=1@c1:9093,2@c2:9093,3@c3:9093
listeners=PLAINTEXT://:9092
inter.broker.listener.name=PLAINTEXT

    Combined mode (broker,controller) is acceptable for dev or small clusters but not recommended for large production deployments where you want fault isolation.
