Domain Vocabulary

A structured glossary of terms used in data extraction, transformation, loading pipelines, and NiFi dataflow architectures. Covers 6 domains · 57 terms.

Source System Originating system where data is extracted from — a database, REST API, flat file, or event stream.
Ingestion The process of importing raw data into a pipeline from source systems — the entry point of any ETL flow.
Extract The E in ETL. Pulling data out of source systems, including full snapshots or incremental deltas.
Connector A configured interface to a specific source or destination (e.g. JDBC, S3, Kafka connector).
Reader Component that parses and reads data in a specific format: CSV, JSON, Avro, Parquet, XML, etc.
Schema The structural definition of data — field names, data types, nullability, and constraints. Shared contract between producer and consumer.
Change Data Capture (CDC) Tracks row-level inserts, updates, and deletes in source databases to enable incremental and near-real-time ingestion.
Polling Periodically querying a source system on a schedule to check for new or changed data since the last run.
Event-Driven Ingestion Triggering data ingestion based on events — file arrival, a message on a queue, or a webhook notification.
Pipeline An end-to-end sequence of processing steps connecting a source to a destination, covering extract, transform, and load.
Dataflow NiFi's programming model — routing data as FlowFiles through a directed graph of processors and connections.
FlowFile NiFi's core data unit — binary content plus a map of string attributes (filename, uuid, size, mime.type) that travels through the flow.
Processor A NiFi component performing one specific operation on FlowFiles — e.g. GetFile, RouteOnAttribute, ConvertRecord, or PutDatabaseRecord.
Connection A queue between two NiFi processors that buffers FlowFiles in transit and applies back-pressure when thresholds are met.
Process Group A logical container grouping processors and connections into a reusable, version-controllable sub-flow unit in NiFi.
Controller Service A shared NiFi service (e.g. DBCPConnectionPool, AvroSchemaRegistry) referenced by multiple processors to avoid duplicating configuration.
Back Pressure Flow control mechanism that pauses upstream processors when a connection queue exceeds its object count or data size threshold.
Funnel A NiFi component that merges multiple incoming connection streams into a single output for simplified routing.
Remote Process Group (RPG) Enables NiFi site-to-site data transfer between separate NiFi instances or clusters over HTTP or RAW socket protocol.
TransformThe T in ETL — converting, restructuring, or calculating data values and shapes to meet target requirements.
MappingDefining how source fields correspond to target fields, including type coercions, renames, and derived calculations.
EnrichmentAugmenting records with additional information from lookup tables, reference datasets, or external APIs.
NormalizationRestructuring relational data to reduce redundancy — splitting across tables to eliminate duplication.
DenormalizationCombining tables and repeating data intentionally to optimize read performance at the cost of storage efficiency.
Expression Language (NiFi EL)NiFi's built-in DSL using ${"{attribute}"} syntax to dynamically evaluate and manipulate FlowFile attributes.
Scripting ProcessorNiFi processors (ExecuteScript, ExecuteGroovy) that run custom Python, Groovy, or JS logic inline within the flow.
Record ProcessorNiFi processors operating on structured records within a FlowFile — UpdateRecord, QueryRecord, ConvertRecord, etc.
Lookup ServiceA NiFi controller service enriching records by resolving field values against a database, file, or in-memory cache.
Schema RegistryCentralized store for data schemas and their versions (Confluent, NiFi Avro), enabling schema evolution and compatibility checks.
ValidationChecking that data conforms to expected types, value ranges, formats, and business rules before further processing.
CleansingCorrecting or removing inaccurate, incomplete, inconsistent, or malformed records from the dataset.
DeduplicationIdentifying and removing duplicate records using key matching, fuzzy matching, or hash-based comparison.
Null HandlingStrategy for managing missing or null field values — reject, substitute defaults, propagate nulls, or flag for review.
Data ProfilingStatistical analysis of a dataset to understand structure, completeness, value distributions, and anomalies.
QuarantineIsolating invalid or suspect records in a separate queue or table for review without halting the main pipeline.
Dead Letter Queue (DLQ)A dedicated holding queue for FlowFiles or messages that fail processing after all retries, awaiting inspection or replay.
LineageA record of a data element's origin, movements, transformations, and usage throughout its lifecycle across systems.
Data ContractA formal agreement on the structure, semantics, SLAs, and ownership of data exchanged between producing and consuming teams.
LoadThe L in ETL — writing transformed data into a target storage system.
Target / SinkThe destination system receiving loaded data — a data warehouse, lake, relational DB, message queue, or API endpoint.
Full LoadLoading the complete dataset on every run, replacing or truncating existing data. Simple but costly for large volumes.
Incremental LoadLoading only new or changed records since the last pipeline run, using a watermark, CDC, or timestamp filter.
UpsertInsert new records and update existing ones in a single operation on a primary key — prevents duplicates on reruns.
PartitioningDividing stored data into logical segments by a column (date, region, category) for parallelism and query pruning.
WriterComponent that serializes and writes records to a target in a specific format — Parquet, JSON, ORC, CSV.
Staging AreaIntermediate storage where raw or semi-processed data lands before final transformation and loading into the target.
Data LakeA large-scale repository holding raw data in native format (S3, ADLS, GCS) — schema-on-read, high flexibility.
Data WarehouseStructured, query-optimized storage for historical, curated data (Snowflake, Redshift, BigQuery) — schema-on-write.
OrchestrationCoordinating the sequence, scheduling, dependencies, and retry behavior of pipeline tasks (Airflow, Prefect, NiFi).
SchedulingRunning pipelines on a CRON expression, at a fixed interval, or on-demand in response to an external trigger.
DAG (Directed Acyclic Graph)A graph of pipeline tasks with defined dependency edges and no cycles — the execution model of Apache Airflow and similar tools.
Retry LogicAutomatically re-executing failed tasks a configured number of times with optional exponential backoff between attempts.
IdempotencyA pipeline operation that can be safely re-run multiple times without producing duplicate or unintended side effects.
WatermarkA high-water mark timestamp or offset tracking the furthest point successfully processed — used to resume incremental loads.
ThroughputData volume processed per unit of time (records/sec, MB/s) — a key metric for capacity planning.
LatencyEnd-to-end time delay between data arriving at source and being available at destination — critical for real-time pipelines.
Monitoring & AlertingObserving pipeline health, error rates, SLA breaches, and queue depths; notifying engineers when thresholds are exceeded.
Provenance (NiFi)NiFi's detailed audit event history — every action (RECEIVE, SEND, FORK, JOIN, DROP, MODIFY) on every FlowFile, with timestamps and chain of custody.
