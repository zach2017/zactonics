Building an AWS AMI Factory

An automated pipeline that bakes a custom Amazon Machine Image every time AWS publishes a new Amazon Linux base. The image ships with Java 21 and Python pre-installed, runs validation tests, and fans out a notification through SNS and SQS so downstream teams can rebuild their applications.

01 · Background

What we’re building, and why

Before any code, the concept in plain language. Skip ahead if you already know what an AMI is.

Picture baking cookies. Instead of measuring flour and sugar every morning, you make one perfect batch of dough and freeze portions of it. Each morning you thaw a portion and bake. Same recipe every time, no measuring.

An Amazon Machine Image (AMI) is that frozen dough — for a server. It captures a fully configured operating system plus any software you pre-installed. When you launch an EC2 instance from an AMI, it boots already set up.

In this tutorial we build a golden base AMI: a clean Amazon Linux server with Java 21 and Python already installed and tested. Other teams build their application images on top of our golden image, so everyone starts from the same trusted foundation.

02 · The plan

Seven steps, in order

The entire workflow before any code. Each step here maps directly to a code block below.

  1. 01

    A trigger fires when a new Amazon Linux AMI is published

    AWS releases new base AMIs regularly with security patches. We listen for that event so our build runs automatically. Two trigger styles are common: an SNS notification from AWS’s public AMI topic, or an EventBridge rule watching SSM Parameter Store.

  2. 02

    Look up the latest Amazon Linux AMI ID

    The trigger tells us “something changed.” We then query SSM Parameter Store for the exact newest AMI ID. AWS publishes these as public parameters.

  3. 03

    Launch a temporary build instance from that base

    Packer (or our shell script) spins up a disposable EC2 instance from the latest base image. This is where we install software.

  4. 04

    Install Java 21 + Python and configure them

    A provisioning script runs on the temp instance: install java-21-amazon-corretto and python3, set JAVA_HOME, update packages.

  5. 05

    Run validation tests

    Before saving anything, verify: does java --version report 21? Does python3 --version succeed? If any test fails the build halts and no AMI is published.

  6. 06

    Bake the new AMI

    Tests passed. Packer snapshots the instance into a new AMI and tags it with build date, source AMI ID, Java version, and a build identifier.

  7. 07

    Publish a notification to SNS → SQS

    The new AMI ID goes to an SNS topic. The topic fans the message out to an SQS queue that downstream teams poll. They read it and rebuild their application images on top of the new base.

04 · Implementation

Three tools, one job

The same pipeline implemented three ways. Packer is recommended for the bake itself; Terraform manages the surrounding AWS infrastructure; the AWS CLI version shows what’s happening underneath.

Why Packer

Packer is a free HashiCorp tool whose only job is building machine images. One declarative file describes the build; one command runs it. Packer launches the temp instance, provisions, tests, snapshots, and cleans up.

STEPS 02 — 06

amibuild.pkr.hcl — the Packer template

hcl
# Describes the full image build in one declarative file
packer {
  required_plugins {
    amazon = { version = ">= 1.3.0", source = "github.com/hashicorp/amazon" }
  }
}

# STEP 02: look up the LATEST Amazon Linux 2023 AMI automatically
source "amazon-ebs" "base" {
  region        = "us-east-1"
  instance_type = "t3.micro"
  ssh_username  = "ec2-user"

  source_ami_filter {
    filters = {
      name                = "al2023-ami-*-x86_64"
      virtualization-type = "hvm"
      root-device-type    = "ebs"
    }
    most_recent = true          # always grab the newest
    owners      = ["amazon"]
  }

  # STEP 06: how the baked AMI will be named & tagged
  ami_name = "golden-java21-python-{{timestamp}}"
  tags = {
    BaseOS    = "AmazonLinux2023"
    Java      = "21-Corretto"
    BuiltBy   = "AMIFactory"
    SourceAMI = "{{ .SourceAMI }}"
  }
}

build {
  sources = ["source.amazon-ebs.base"]

  # STEP 04: install our software
  provisioner "shell" {
    script = "provision.sh"
  }

  # STEP 05: validate — build FAILS if any command returns non-zero
  provisioner "shell" {
    inline = [
      "echo '--- Running tests ---'",
      "java --version | grep '21' || (echo 'JAVA TEST FAILED' && exit 1)",
      "python3 --version || (echo 'PYTHON TEST FAILED' && exit 1)",
      "echo 'ALL TESTS PASSED'"
    ]
  }

  # STEP 07: write the new AMI ID to a file for the next stage
  post-processor "manifest" {
    output = "manifest.json"
  }
}
STEP 04

provision.sh — the install script

bash
#!/bin/bash
set -euo pipefail   # stop instantly on any error

echo "Updating base packages..."
sudo dnf update -y

echo "Installing Java 21 (Amazon Corretto)..."
sudo dnf install -y java-21-amazon-corretto-devel

echo "Installing Python 3 + pip..."
sudo dnf install -y python3 python3-pip

# make Java visible to all future shells
echo 'export JAVA_HOME=/usr/lib/jvm/java-21-amazon-corretto' \
  | sudo tee /etc/profile.d/java.sh

echo "Provisioning complete."
RUN IT

Build the image

bash
packer init  amibuild.pkr.hcl
packer build amibuild.pkr.hcl

# on success, manifest.json now holds the new AMI ID
STEP 07

Publish the new AMI ID

bash
# pull the AMI id from Packer's manifest and announce it
NEW_AMI=$(jq -r '.builds[-1].artifact_id' manifest.json | cut -d':' -f2)

aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:111122223333:new-base-ami \
  --subject  "New golden AMI ready" \
  --message  "{\"ami_id\":\"$NEW_AMI\",\"java\":\"21\",\"status\":\"READY\"}"
Downstream teams have an SQS queue subscribed to that SNS topic, so this single publish reaches every consumer at once.

Where Terraform fits

Packer bakes the image. Terraform builds the surrounding infrastructure: the SNS topic, the SQS queue, the subscription wiring, the EventBridge trigger, IAM roles, and optionally EC2 Image Builder if you prefer AWS’s native baking service. Use them together, each doing what it’s best at.

STEPS 01 & 07

main.tf — messaging + trigger infrastructure

terraform
provider "aws" { region = "us-east-1" }

# STEP 07a: the topic our build publishes to
resource "aws_sns_topic" "new_ami" {
  name = "new-base-ami"
}

# STEP 07b: the queue downstream apps poll from
resource "aws_sqs_queue" "app_rebuild" {
  name                       = "app-rebuild-queue"
  visibility_timeout_seconds = 300
}

# STEP 07c: connect topic -> queue (fan-out)
resource "aws_sns_topic_subscription" "to_sqs" {
  topic_arn = aws_sns_topic.new_ami.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.app_rebuild.arn
}

# allow SNS to drop messages into the queue
resource "aws_sqs_queue_policy" "allow_sns" {
  queue_url = aws_sqs_queue.app_rebuild.id
  policy = jsonencode({
    Version = "2012-10-17",
    Statement = [{
      Effect    = "Allow",
      Principal = { Service = "sns.amazonaws.com" },
      Action    = "sqs:SendMessage",
      Resource  = aws_sqs_queue.app_rebuild.arn,
      Condition = {
        ArnEquals = { "aws:SourceArn" = aws_sns_topic.new_ami.arn }
      }
    }]
  })
}

# STEP 01: EventBridge rule — fires when the SSM parameter
# holding the latest AMI id changes value
resource "aws_cloudwatch_event_rule" "ami_changed" {
  name        = "al2023-ami-updated"
  description = "Latest Amazon Linux AMI changed"
  event_pattern = jsonencode({
    source      = ["aws.ssm"],
    detail-type = ["Parameter Store Change"],
    detail = {
      name      = [{ prefix = "/aws/service/ami-amazon-linux-latest/" }],
      operation = ["Update"]
    }
  })
}

# point the rule at whatever runs Packer (Lambda, CodeBuild, etc.)
resource "aws_cloudwatch_event_target" "run_builder" {
  rule = aws_cloudwatch_event_rule.ami_changed.name
  arn  = aws_lambda_function.start_packer.arn
}

output "sns_topic_arn" { value = aws_sns_topic.new_ami.arn }
output "sqs_queue_url" { value = aws_sqs_queue.app_rebuild.id }

The no-special-tools path

With only the AWS CLI and bash, you can do every step by hand. This is the most transparent way to learn what Packer abstracts — but you manage cleanup yourself.

STEP 02

Find the latest AMI from SSM

bash
BASE_AMI=$(aws ssm get-parameter \
  --name /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64 \
  --query 'Parameter.Value' --output text)
echo "Latest base AMI: $BASE_AMI"
STEP 03

Launch the temp build server

bash
INSTANCE_ID=$(aws ec2 run-instances \
  --image-id "$BASE_AMI" \
  --instance-type t3.micro \
  --key-name my-build-key \
  --security-group-ids sg-0123456789 \
  --query 'Instances[0].InstanceId' --output text)

aws ec2 wait instance-running --instance-ids "$INSTANCE_ID"

IP=$(aws ec2 describe-instances --instance-ids "$INSTANCE_ID" \
  --query 'Reservations[0].Instances[0].PublicIpAddress' --output text)
STEPS 04 & 05

Provision and test over SSH

bash
ssh -o StrictHostKeyChecking=no ec2-user@"$IP" 'bash -s' << 'EOF'
set -euo pipefail
sudo dnf update -y
sudo dnf install -y java-21-amazon-corretto-devel python3 python3-pip

# --- tests ---
java --version | grep 21 || { echo "JAVA FAIL"; exit 1; }
python3 --version          || { echo "PY FAIL";   exit 1; }
echo "TESTS PASSED"
EOF
STEP 06

Bake the AMI — only runs if the SSH block above succeeded

bash
NEW_AMI=$(aws ec2 create-image \
  --instance-id "$INSTANCE_ID" \
  --name "golden-java21-python-$(date +%s)" \
  --tag-specifications 'ResourceType=image,Tags=[{Key=Java,Value=21},{Key=BuiltBy,Value=AMIFactory}]' \
  --query 'ImageId' --output text)

aws ec2 wait image-available --image-ids "$NEW_AMI"

# clean up the temporary build server
aws ec2 terminate-instances --instance-ids "$INSTANCE_ID"
STEP 07

Announce via SNS

bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:111122223333:new-base-ami \
  --message "{\"ami_id\":\"$NEW_AMI\",\"status\":\"READY\"}"
05 · Downstream

What consumers do

The other side of step 07. Each consuming team polls their SQS queue. When a READY message arrives, they kick off their own Packer build using our new AMI as the base.

bash · downstream poller
# runs on a schedule (cron, EventBridge, etc.)
MSG=$(aws sqs receive-message --queue-url "$QUEUE_URL" --query 'Messages[0]')
NEW_BASE=$(echo "$MSG" | jq -r '.Body | fromjson | .Message | fromjson | .ami_id')

# feed it into their own packer build as a variable
packer build -var "base_ami=$NEW_BASE" app-image.pkr.hcl

# delete the message so it isn't processed twice
RH=$(echo "$MSG" | jq -r '.ReceiptHandle')
aws sqs delete-message --queue-url "$QUEUE_URL" --receipt-handle "$RH"

What is an AMI?

AMI stands for Amazon Machine Image. It’s a complete template of a server’s disk: the operating system, system settings, and any software you installed.

When you launch an EC2 instance (a virtual server), you must tell AWS which AMI to start from. Think of the AMI as the “starting save file” for a server.

Custom AMIs let you boot servers that are already configured — no setup time, and every server is identical, which prevents the classic “it works on my machine” problem.

Read the official AMI documentation →

Why “golden image”?

A golden image is your one official, approved, tested base that everyone builds from. The name comes from manufacturing: the “golden master” copy that everything else is stamped from.

Because security patches and approved software live in the golden image, every team automatically inherits them. Update the golden image once, and the fix flows everywhere.

AWS services glossary

  • EC2 — virtual servers in the cloud.
  • AMI — the saved image an EC2 server boots from.
  • SSM Parameter Store — where AWS stores values, including the ID of the latest AMI.
  • EventBridge — listens for events and triggers actions.
  • SNS — sends one message out to many subscribers (a megaphone).
  • SQS — a queue that holds messages until a worker reads them (a mailbox).
  • Lambda — runs small bits of code with no server to manage.
  • IAM — controls who and what is allowed to do what.

How the trigger works

AWS stores the ID of the newest Amazon Linux AMI in SSM Parameter Store. When AWS releases a new one, that stored value changes.

EventBridge watches for that exact change and automatically starts your build — no human needed.

AWS also publishes notifications to public SNS topics for AMI updates, which you can subscribe to as an alternative trigger.

What is Amazon Corretto?

Corretto is Amazon’s free, production-ready build of Java (OpenJDK). Corretto 21 gives you Java 21 with long-term security updates from AWS.

One command installs it: dnf install java-21-amazon-corretto-devel.

Corretto 21 documentation →

SNS vs SQS

SNS — megaphone (push). You publish once; SNS instantly pushes a copy to every subscriber. Great for “tell everyone something happened.”

SQS — mailbox (pull). Messages sit safely in a queue until a worker is ready to pick them up and process them one at a time.

The pattern used here is SNS → SQS fan-out: SNS broadcasts the event, and each consuming team has its own SQS queue subscribed to it, so no one misses or blocks anyone else.

EC2 Image Builder

EC2 Image Builder is AWS’s own managed service for baking AMIs. Instead of running Packer yourself, AWS runs the build, tests, and distribution on a schedule or trigger.

Trade-off: less portable than Packer (AWS-only), but zero build infrastructure to maintain. You can define an Image Builder pipeline entirely in Terraform.
