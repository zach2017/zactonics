00 · Prerequisites & setup

Before you begin: tools, roles, network, gotchas

Don’t skip this section. Most failed AMI builds trace back to a missing IAM permission, the wrong subnet, or a security group that won’t accept SSH. Walk this list end to end before running anything.

Heads up: Every code sample uses the placeholder account ID 111122223333 and region us-east-1. Replace both with your real values before running. ARNs are region- and account-specific.

Tools to install on your workstation

# 3. Terraform — for the infrastructure-as-code path
brew install terraform   # or use the same hashicorp.repo above

brew install jq          # or: sudo dnf install -y jq

00.2

AWS account & CLI configuration

  • An AWS account with billing active and at least one IAM user or SSO identity you can sign in as. Root user is never used for daily work.
  • Multi-factor authentication (MFA) enabled on every IAM identity. Required for production credentials.
  • A default region picked (we use us-east-1). All resources in this tutorial must live in the same region unless you explicitly copy AMIs across regions.
  • Local AWS credentials configured. Run aws configure or set up ~/.aws/config with an SSO profile.
  • Sufficient service quotas — specifically EC2 instance count, EBS snapshots, and AMIs per region. Default quotas are usually fine for one factory, but check Service Quotas console in regulated environments.

00.3

The four IAM roles you need

This pipeline involves several principals, each needing the minimum permissions for its job. Don’t reuse one mega-role across all of them — that violates least privilege and makes audits painful.

① PackerBuildRole

arn:aws:iam::111122223333:role/PackerBuildRole

Assumed by whoever runs packer build (CI runner, Lambda, your laptop). Allows Packer to launch a temporary EC2 instance, snapshot it, and tag the AMI.

ec2:RunInstances · ec2:TerminateInstances
ec2:CreateImage · ec2:DescribeImages
ec2:CreateTags · ec2:CreateKeyPair
iam:PassRole (for the instance profile)

② PackerInstanceProfile

arn:aws:iam::111122223333:instance-profile/PackerInstanceProfile

Attached to the temporary build instance itself. Lets the in-progress build pull from S3, write CloudWatch logs, or connect via SSM Session Manager (so you don’t need SSH).

AmazonSSMManagedInstanceCore
s3:GetObject (artifact bucket only)
logs:CreateLogStream · logs:PutLogEvents

③ AmiPublishRole

arn:aws:iam::111122223333:role/AmiPublishRole

Used by the step that publishes the new AMI ID to SNS. Separate from the build role so the build can fail without ever holding publish credentials.

sns:Publish (specific topic ARN only)
ssm:PutParameter (latest-ami pointer)

④ GitLabOidcRole

arn:aws:iam::111122223333:role/GitLabOidcRole

Federated role your GitLab pipeline assumes through OIDC. No long-lived AWS keys ever leave the platform. Trust policy locked to a specific project & branch.

sts:AssumeRole into ① and ③
Trust policy: gitlab.com OIDC provider

00.5

Network: VPC, subnet, security group

Packer needs to talk to the temporary build instance over SSH (or SSM). The instance also needs internet access to install packages from dnf repos. Three components are required:

  • A VPC — the default VPC is fine for tutorials, but production should use a dedicated VPC with no inbound rules from the internet.
  • A subnet with outbound internet access — either a public subnet with an Internet Gateway, or a private subnet routed through a NAT Gateway. Without this, dnf install hangs forever.
  • A security group allowing SSH from your runner. Best practice: use SSM Session Manager (covered below) and require zero inbound rules.
terraform · network.tf
# A minimal security group: allow SSH from the runner's known CIDR only
resource "aws_security_group" "packer_build" {
  name        = "packer-build"
  description = "Allow SSH inbound from CI runners only"
  vpc_id      = var.vpc_id

  ingress {
    description = "SSH from runners"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = [var.runner_cidr]   # NEVER 0.0.0.0/0
  }

  egress {
    description = "All outbound (to reach dnf repos)"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
Better: skip SSH entirely. Set Packer’s ssh_interface = "session_manager" and remove the inbound rule. Builds become 100% private and there’s nothing to scan from the internet.

00.7

Gotchas you’ll hit if you don’t read this

These are the failures we’ve seen most often. Knowing them up front saves hours of debugging.

Gotcha 1 — iam:PassRole denied. When Packer launches the build instance, it must pass the instance profile role to EC2. If PackerBuildRole lacks iam:PassRole on the instance profile ARN, you get a cryptic UnauthorizedOperation. The Terraform above includes it — don’t remove it.
Gotcha 2 — wrong AMI owner. owners = ["amazon"] is right for Amazon Linux. For Ubuntu it’s 099720109477; for RHEL it’s 309956199498. Empty filter results are the #1 cause of “Packer just hangs.”
Gotcha 3 — region drift. Your local AWS_REGION, the Packer template region, the Terraform provider region, and the SNS topic ARN region must all match. AMIs are region-scoped; an AMI built in us-east-1 is not visible in us-west-2 until you copy it.
Gotcha 4 — SSH timeout. If the build hangs at “Waiting for SSH to become available,” check (in order): security group allows inbound 22 from the runner, subnet route table has 0.0.0.0/0 → IGW or NAT, the instance has a public IP if you’re using IGW, and your key pair name matches.
Gotcha 5 — KMS denied on EBS. If your org uses a customer-managed KMS key for EBS encryption, the Packer role needs kms:Encrypt, kms:Decrypt, kms:ReEncrypt*, kms:GenerateDataKey* on that key. Default AWS-managed keys don’t need this.
Gotcha 6 — dnf hanging on update. If dnf update -y takes more than a few minutes, the build instance has no outbound internet. Check route table and NAT gateway.
01 · Background

What we’re building, and why

Before any code, the concept in plain language. Skip ahead if you already know what an AMI is.

Picture baking cookies. Instead of measuring flour and sugar every morning, you make one perfect batch of dough and freeze portions of it. Each morning you thaw a portion and bake. Same recipe every time, no measuring.

An Amazon Machine Image (AMI) is that frozen dough — for a server. It captures a fully configured operating system plus any software you pre-installed. When you launch an EC2 instance from an AMI, it boots already set up.

In this tutorial we build a golden base AMI: a clean Amazon Linux server with Java 21 and Python already installed and tested. Other teams build their application images on top of our golden image, so everyone starts from the same trusted foundation.

02 · The plan

Seven steps, in order

The entire workflow before any code. Each step maps directly to a code block below.

  1. 01

    A trigger fires when a new Amazon Linux AMI is published

    AWS releases new base AMIs regularly with security patches. Two trigger styles are common: an SNS notification from AWS’s public AMI topic, or an EventBridge rule watching SSM Parameter Store.

  2. 02

    Look up the latest Amazon Linux AMI ID

    The trigger says “something changed.” We then query SSM Parameter Store for the exact newest AMI ID.

  3. 03

    Launch a temporary build instance from that base

    Packer (or our shell script) spins up a disposable EC2 instance. This is where we install software.

  4. 04

    Install Java 21 + Python and configure them

    A provisioning script runs on the temp instance: install java-21-amazon-corretto and python3, set JAVA_HOME, update packages.

  5. 05

    Run validation tests

    Verify: does java --version report 21? Does python3 --version succeed? If any test fails the build halts and no AMI is published.

  6. 06

    Bake the new AMI

    Packer snapshots the instance into a new AMI and tags it with build date, source AMI ID, Java version, and a build identifier.

  7. 07

    Publish a notification to SNS → SQS

    The new AMI ID goes to an SNS topic. The topic fans out to an SQS queue that downstream teams poll. They read it and rebuild their application images on top of the new base.

04 · Implementation

Three tools, one job

The same pipeline implemented three ways. Packer is recommended for the bake; Terraform manages the surrounding infrastructure; the AWS CLI version shows what’s happening underneath.

Why Packer

A free HashiCorp tool whose only job is building machine images. One declarative file describes the build; one command runs it.

STEPS 02 — 06

amibuild.pkr.hcl — the Packer template

hcl
packer {
  required_plugins {
    amazon = { version = ">= 1.3.0", source = "github.com/hashicorp/amazon" }
  }
}

# STEP 02: look up the LATEST Amazon Linux 2023 AMI automatically
source "amazon-ebs" "base" {
  region        = "us-east-1"
  instance_type = "t3.micro"
  ssh_username  = "ec2-user"
  iam_instance_profile = "PackerInstanceProfile"

  source_ami_filter {
    filters = {
      name                = "al2023-ami-*-x86_64"
      virtualization-type = "hvm"
      root-device-type    = "ebs"
    }
    most_recent = true
    owners      = ["amazon"]
  }

  ami_name = "golden-java21-python-{{timestamp}}"
  tags = {
    BaseOS    = "AmazonLinux2023"
    Java      = "21-Corretto"
    BuiltBy   = "AMIFactory"
    SourceAMI = "{{ .SourceAMI }}"
  }
}

build {
  sources = ["source.amazon-ebs.base"]

  provisioner "shell" { script = "provision.sh" }

  provisioner "shell" {
    inline = [
      "java --version | grep '21' || (echo 'JAVA TEST FAILED' && exit 1)",
      "python3 --version || (echo 'PYTHON TEST FAILED' && exit 1)",
      "echo 'ALL TESTS PASSED'"
    ]
  }

  post-processor "manifest" { output = "manifest.json" }
}

provision.sh

bash
#!/bin/bash
set -euo pipefail

sudo dnf update -y

Build the image

Publish the new AMI ID

Where Terraform fits

Packer bakes the image. Terraform builds the surrounding infrastructure: SNS, SQS, EventBridge, IAM, and optionally EC2 Image Builder if you prefer AWS’s native baking.

STEPS 01 & 07

main.tf — messaging + trigger infrastructure

terraform
provider "aws" { region = "us-east-1" }

resource "aws_sns_topic" "new_ami" { name = "new-base-ami" }

resource "aws_sqs_queue" "app_rebuild" {
  name                       = "app-rebuild-queue"
  visibility_timeout_seconds = 300
}

resource "aws_sns_topic_subscription" "to_sqs" {
  topic_arn = aws_sns_topic.new_ami.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.app_rebuild.arn
}

resource "aws_sqs_queue_policy" "allow_sns" {
  queue_url = aws_sqs_queue.app_rebuild.id
  policy = jsonencode({
    Version = "2012-10-17",
    Statement = [{
      Effect    = "Allow",
      Principal = { Service = "sns.amazonaws.com" },
      Action    = "sqs:SendMessage",
      Resource  = aws_sqs_queue.app_rebuild.arn,
      Condition = { ArnEquals = { "aws:SourceArn" = aws_sns_topic.new_ami.arn } }
    }]
  })
}

resource "aws_cloudwatch_event_rule" "ami_changed" {
  name = "al2023-ami-updated"
  event_pattern = jsonencode({
    source      = ["aws.ssm"],
    detail-type = ["Parameter Store Change"],
    detail = {
      name      = [{ prefix = "/aws/service/ami-amazon-linux-latest/" }],
      operation = ["Update"]
    }
  })
}

  rule = aws_cloudwatch_event_rule.ami_changed.name
  arn  = aws_lambda_function.start_packer.arn

The no-special-tools path

With only the AWS CLI and bash, you can do every step by hand — transparent but you manage cleanup yourself.

STEPS 02 — 07

Full pipeline in one shell script

bash · build.sh
#!/bin/bash
set -euo pipefail

# STEP 02
BASE_AMI=$(aws ssm get-parameter \
  --name /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64 \
  --query 'Parameter.Value' --output text)

# STEP 03
INSTANCE_ID=$(aws ec2 run-instances \
  --image-id "$BASE_AMI" --instance-type t3.micro \
  --iam-instance-profile Name=PackerInstanceProfile \
  --security-group-ids sg-0123456789 \
  --query 'Instances[0].InstanceId' --output text)
aws ec2 wait instance-running --instance-ids "$INSTANCE_ID"

# STEPS 04 + 05 — via SSM Run Command, no SSH needed
aws ssm send-command \
  --instance-ids "$INSTANCE_ID" \
  --document-name "AWS-RunShellScript" \
  --parameters 'commands=[
    "sudo dnf update -y",
    "sudo dnf install -y java-21-amazon-corretto-devel python3 python3-pip",
    "java --version | grep 21 || exit 1",
    "python3 --version || exit 1"
  ]'

# STEP 06
NEW_AMI=$(aws ec2 create-image \
  --instance-id "$INSTANCE_ID" \
  --name "golden-java21-python-$(date +%s)" \
  --query 'ImageId' --output text)
aws ec2 wait image-available --image-ids "$NEW_AMI"
aws ec2 terminate-instances --instance-ids "$INSTANCE_ID"

# STEP 07
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:111122223333:new-base-ami \
  --message "{\"ami_id\":\"$NEW_AMI\",\"status\":\"READY\"}"
05 · Downstream

What consumers do

Each consuming team polls their SQS queue. When a READY message arrives, they kick off their own Packer build using our new AMI as the base.

bash · downstream poller
MSG=$(aws sqs receive-message --queue-url "$QUEUE_URL" --query 'Messages[0]')
NEW_BASE=$(echo "$MSG" | jq -r '.Body | fromjson | .Message | fromjson | .ami_id')

packer build -var "base_ami=$NEW_BASE" app-image.pkr.hcl

RH=$(echo "$MSG" | jq -r '.ReceiptHandle')
aws sqs delete-message --queue-url "$QUEUE_URL" --receipt-handle "$RH"
06 · CI/CD

Running the factory in a GitLab pipeline

Manual builds are fine for learning. In production the pipeline runs in CI, authenticates to AWS without long-lived secrets, gates publish behind manual approval, and surfaces every artifact. Here’s the full GitLab setup.

06.1

Federate GitLab to AWS with OIDC (no static keys)

Storing AWS access keys as GitLab variables is the old way. Today, GitLab signs every job with a short-lived JWT and AWS exchanges it for temporary credentials. Set this up once.

terraform · oidc.tf
# Register GitLab as an OIDC identity provider in AWS
resource "aws_iam_openid_connect_provider" "gitlab" {
  url             = "https://gitlab.com"
  client_id_list  = ["https://gitlab.com"]
  thumbprint_list = ["b3dd7606d2b5a8b4a13771dbecc9ee1cecafa38a"]
  # Use `openssl s_client -connect gitlab.com:443` to recompute current thumbprint
}

# ④ The role GitLab assumes — trust policy locks to a specific project & branch
data "aws_iam_policy_document" "gitlab_trust" {
  statement {
    actions = ["sts:AssumeRoleWithWebIdentity"]
    principals {
      type        = "Federated"
      identifiers = [aws_iam_openid_connect_provider.gitlab.arn]
    }
    condition {
      test     = "StringEquals"
      variable = "gitlab.com:aud"
      values   = ["https://gitlab.com"]
    }
    condition {
      test     = "StringLike"
      variable = "gitlab.com:sub"
      values   = ["project_path:my-org/ami-factory:ref_type:branch:ref:main"]
    }
  }
}

resource "aws_iam_role" "gitlab_oidc" {
  name               = "GitLabOidcRole"
  assume_role_policy = data.aws_iam_policy_document.gitlab_trust.json
}

# Allow the OIDC role to assume the Packer build & publish roles
resource "aws_iam_role_policy" "gitlab_assume" {
  role = aws_iam_role.gitlab_oidc.id
  policy = jsonencode({
    Version = "2012-10-17",
    Statement = [{
      Effect   = "Allow",
      Action   = "sts:AssumeRole",
      Resource = [
        "arn:aws:iam::${var.account_id}:role/PackerBuildRole",
        "arn:aws:iam::${var.account_id}:role/AmiPublishRole"
      ]
    }]
  })
}
06.2

.gitlab-ci.yml — the pipeline

Four stages: validate (syntax check), plan (Terraform dry-run for infra changes), build (run Packer), publish (announce via SNS, gated by manual approval).

yaml · .gitlab-ci.yml
stages:
  - validate
  - plan
  - build
  - publish

variables:
  AWS_REGION:          "us-east-1"
  AWS_DEFAULT_REGION:  "us-east-1"
  PACKER_ROLE_ARN:     "arn:aws:iam::111122223333:role/PackerBuildRole"
  PUBLISH_ROLE_ARN:    "arn:aws:iam::111122223333:role/AmiPublishRole"
  SNS_TOPIC_ARN:       "arn:aws:sns:us-east-1:111122223333:new-base-ami"

# Inject a GitLab-issued OIDC token into every job
default:
  id_tokens:
    GITLAB_OIDC_TOKEN:
      aud: https://gitlab.com

# Reusable: exchange OIDC token for AWS creds
.aws_oidc_login: &aws_oidc_login |
  CREDS=$(aws sts assume-role-with-web-identity \
    --role-arn "arn:aws:iam::111122223333:role/GitLabOidcRole" \
    --role-session-name "gitlab-$CI_JOB_ID" \
    --web-identity-token "$GITLAB_OIDC_TOKEN" \
    --duration-seconds 3600 \
    --query 'Credentials' --output json)
  export AWS_ACCESS_KEY_ID=$(echo "$CREDS" | jq -r .AccessKeyId)
  export AWS_SECRET_ACCESS_KEY=$(echo "$CREDS" | jq -r .SecretAccessKey)
  export AWS_SESSION_TOKEN=$(echo "$CREDS" | jq -r .SessionToken)

# ---------- VALIDATE ----------
validate:
  stage: validate
  image: hashicorp/packer:1.10
  script:
    - *aws_oidc_login
    - packer init  amibuild.pkr.hcl
    - packer validate amibuild.pkr.hcl

    - *aws_oidc_login

    paths: [manifest.json, packer.log]

  stage: publish
  image: amazon/aws-cli:latest
  needs: [packer-build]
  script:
    - *aws_oidc_login
    - PUBLISH_CREDS=$(aws sts assume-role --role-arn "$PUBLISH_ROLE_ARN" --role-session-name publish)
    - export AWS_ACCESS_KEY_ID=$(echo "$PUBLISH_CREDS" | jq -r .Credentials.AccessKeyId)
    - export AWS_SECRET_ACCESS_KEY=$(echo "$PUBLISH_CREDS" | jq -r .Credentials.SecretAccessKey)
    - export AWS_SESSION_TOKEN=$(echo "$PUBLISH_CREDS" | jq -r .Credentials.SessionToken)
    - NEW_AMI=$(jq -r '.builds[-1].artifact_id' manifest.json | cut -d':' -f2)
    - echo "Publishing AMI $NEW_AMI"
    - aws sns publish --topic-arn "$SNS_TOPIC_ARN" --message "{\"ami_id\":\"$NEW_AMI\",\"status\":\"READY\"}"

    - if: $CI_COMMIT_BRANCH == "main"
Why the manual gate? Auto-publishing a fresh AMI immediately makes every downstream team rebuild. A 5-second human check on the build log prevents bad images from cascading through the org. Move to automatic after you trust the test suite.
06.3

Triggering on a schedule or AWS event

The pipeline runs on merge to main by default. To also rebuild when AWS publishes a new base AMI, choose one:

  • GitLab schedule — configure a daily pipeline at Settings → CI/CD → Schedules. Simplest, but rebuilds even when nothing changed.
  • EventBridge → Lambda → GitLab API — the AWS-side trigger from section 02. Lambda calls GitLab’s POST /projects/:id/trigger/pipeline endpoint with a trigger token. Rebuilds only when something actually changed.
  • SNS subscription → webhook — subscribe an HTTPS endpoint (GitLab webhook) directly to the public Amazon Linux AMI SNS topic. No Lambda required.
07 · Alternative paths

Ansible, Systems Manager & Image Builder

Packer’s shell provisioner works, but it isn’t the only choice. Here are three production-grade alternatives for the provisioning step itself.

Why use Ansible inside Packer

Shell scripts are linear and don’t track state. Ansible playbooks are idempotent (re-running produces the same result), describe configuration declaratively, and the same playbook can configure a baked AMI and an existing running server. If you already use Ansible for config management, reuse the same playbooks here.

Swap the shell provisioner for Ansible

  provisioner "ansible" {
    playbook_file   = "./playbooks/base.yml"
    user            = "ec2-user"
    use_proxy       = false
    extra_arguments = ["--extra-vars", "java_version=21"]
  }
}

  become: true

    java_version: 21

Or write a custom Automation document

Full control over each step. The document below mirrors the seven-step plan exactly.

yaml · golden-ami.yml
schemaVersion: '0.3'
description: Build golden AMI with Java 21 + Python
assumeRole: "{{ AutomationAssumeRole }}"
parameters:
  SourceAmiId:          { type: String }
  SubnetId:             { type: String }
  AutomationAssumeRole: { type: String }

mainSteps:
  - name: launchInstance
    action: aws:runInstances
    inputs:
      ImageId: "{{ SourceAmiId }}"
      InstanceType: t3.micro
      SubnetId: "{{ SubnetId }}"
      IamInstanceProfileName: PackerInstanceProfile
      MinInstanceCount: 1
      MaxInstanceCount: 1

  - name: installSoftware
    action: aws:runCommand
    inputs:
      DocumentName: AWS-RunShellScript
      InstanceIds: ["{{ launchInstance.InstanceIds }}"]
      Parameters:
        commands:
          - sudo dnf update -y
          - sudo dnf install -y java-21-amazon-corretto-devel python3 python3-pip

  - name: runTests
    action: aws:runCommand
    inputs:
      DocumentName: AWS-RunShellScript
      InstanceIds: ["{{ launchInstance.InstanceIds }}"]
      Parameters:
        commands:
          - java --version | grep 21
          - python3 --version

  - name: createImage
    action: aws:createImage
    inputs:
      InstanceId: "{{ launchInstance.InstanceIds }}"
      ImageName: "golden-java21-{{ global:DATE_TIME }}"
      NoReboot: false

  - name: terminateInstance
    action: aws:changeInstanceState
    inputs:
      InstanceIds: ["{{ launchInstance.InstanceIds }}"]
      DesiredState: terminated
Bonus SSM capability: pair this with SSM State Manager to continuously verify that already-running instances still match your golden config — drift detection for free.

08 · Best practices

Production-readiness checklist

A factory that works in dev breaks in prod without these. Treat this as a checklist before you let the pipeline run unattended.

  1. Separate AWS accounts for build vs production

    Bake AMIs in a dedicated build account, then share them with workload accounts via Resource Access Manager. Blast radius of a compromised build is contained.

  2. Always use least-privilege IAM

    The Packer role from section 00.4 lists every action by name. Avoid ec2:* or Resource: "*" in production. Use IAM Access Analyzer to refine policies based on actual usage.

  3. OIDC instead of long-lived access keys

    GitLab, GitHub, CircleCI, Buildkite, Jenkins on EKS all support OIDC to AWS. Rotate nothing, store nothing. If you still have static access keys in CI variables, that’s the next thing to fix.

  4. Encrypt EBS volumes with a customer-managed KMS key

    Default AWS-managed encryption is fine for many cases, but a CMK lets you audit who decrypts what, rotate the key on your schedule, and revoke access surgically. Set encrypt_boot = true and kms_key_id in Packer.

  5. Tag every resource consistently

    At minimum: BuiltBy, SourceAMI, BuildDate, Java/Python version, GitCommit, Pipeline, Owner. Tags drive cost allocation, lifecycle policies, and incident forensics.

  6. Version AMIs semantically

    Don’t just use timestamps. Use names like golden-base-v1.4.2-20260301 so consumers can pin to a known-good version and roll back deterministically.

  7. Scan images for vulnerabilities before publishing

    Run Amazon Inspector against the freshly built AMI, or call out to Trivy / Grype as a Packer post-processor. Fail the build on critical CVEs.

  8. Test by booting before announcing

    Add a stage that launches an instance from the new AMI, runs an integration smoke test, then terminates. Only on success does SNS publish. The validation inside Packer catches install errors; this catches boot-time issues.

  9. Multi-region distribution via copy

    Use Packer’s ami_regions list or Image Builder’s distribution config to copy to every region you operate in. Each region gets its own AMI ID; surface a map of region → AMI to consumers.

  10. Implement an AMI lifecycle policy

    Keep the last 5-10 versions. Deprecate older AMIs with aws ec2 enable-image-deprecation so consumers see warnings. Eventually deregister and delete underlying snapshots — orphaned snapshots are the #1 sneaky AWS cost.

  11. Never bake secrets into AMIs

    Anyone who can launch the AMI can read its disk. Pull secrets at boot from Secrets Manager or SSM Parameter Store via instance role, never embed them.

  12. Make provisioning idempotent

    Whether shell or Ansible, the script must be re-runnable. dnf install already is; echo >> /etc/profile is not (use tee or template files). Idempotence makes debugging dramatically easier.

  13. Audit everything via CloudTrail

    Enable CloudTrail with a multi-region trail to a write-only S3 bucket. Every RunInstances, CreateImage, AssumeRole is logged. Invaluable for “who built this AMI?” and for post-incident review.

  14. Pin every tool version

    packer 1.10.0, terraform 1.7.5, amazon plugin = "= 1.3.0". Floating versions cause non-reproducible builds. Treat your build environment itself as immutable infrastructure — pin it in a Docker image used by CI.

  15. Cost: use Spot for builds, on-demand for tests

    The build instance lives for 5-15 minutes; it’s a perfect Spot candidate (70-90% cheaper). If Spot capacity is interrupted, fall back to on-demand automatically. Don’t use Spot for the post-bake smoke test — you want predictability there.

09 · Troubleshooting

Common errors & how to fix them

Real errors from real builds. Find your error message on the left; the cause & fix are on the right.

UnauthorizedOperation: You are not authorized to perform: ec2:RunInstances
Cause: The role running Packer is missing ec2:RunInstances, or (more often) missing iam:PassRole on the instance profile. Fix: use the policy in section 00.4 verbatim; double-check the instance profile ARN in the iam:PassRole resource matches what Packer is trying to pass.
InvalidAMIID.NotFound: The image id '[ami-xxx]' does not exist
Cause: Region mismatch — an AMI from us-east-1 doesn’t exist in us-west-2. Fix: verify AWS_REGION, Packer’s region field, and the source AMI’s region all agree. If you need the AMI in multiple regions, use aws ec2 copy-image or Packer’s ami_regions.
Timeout waiting for SSH
Cause: one of: security group has no inbound 22 from runner IP, subnet has no route to internet (so no public IP gets assigned), wrong key pair, or instance is in a private subnet and you set associate_public_ip_address = false. Fix: first try ssh_interface = "session_manager" in Packer — it bypasses SSH entirely and just needs AmazonSSMManagedInstanceCore on the instance profile.
source_ami_filter returned no results
Cause: wrong owner ID, wrong AMI name pattern, or no matching AMI in this region. Fix: Test the filter with aws ec2 describe-images --owners amazon --filters "Name=name,Values=al2023-ami-*-x86_64" --query 'Images[*].Name' --output text | tr '\t' '\n' | sort | tail. Confirm at least one row comes back before retrying Packer.
Packer hangs at "Provisioning with shell script"
Cause: the provisioning script is waiting for user input. dnf install java (without -y) prompts for confirmation, hangs forever. Fix: always pass -y to dnf/yum/apt; always use set -euo pipefail at the top of every shell script so failures stop immediately rather than hanging.
KMS: not authorized to perform: kms:CreateGrant
Cause: EBS encryption uses a customer-managed KMS key, but the Packer role lacks permission on that key. Fix: add the Packer role as a key user in the KMS key’s key policy (not just IAM). Required actions: Encrypt, Decrypt, ReEncrypt*, GenerateDataKey*, DescribeKey, CreateGrant.
SNS topic does not exist (or you do not have access to perform sns:Publish)
Cause: the topic ARN is wrong (region or account portion), or the publish role lacks sns:Publish on that specific ARN. Fix: verify with aws sns list-topics --region us-east-1 and recheck the topic ARN region matches your CLI region.
SQS queue receives no messages, even though SNS publish succeeded
Cause: the subscription isn’t confirmed, or the queue policy doesn’t allow SNS to send to it. Fix: verify with aws sns list-subscriptions-by-topic --topic-arn $ARN — status must be Confirmed (not PendingConfirmation). Confirm the SQS queue policy has the aws:SourceArn condition from section 04.
Build instance terminated unexpectedly during build
Cause: you used Spot capacity and AWS reclaimed it, or someone set InstanceInitiatedShutdownBehavior=terminate with a shutdown in the script. Fix: for Spot, use Packer’s spot_price = "auto" with on-demand fallback. Never call shutdown inside provisioning — Packer handles termination.
Image creation timed out after 1800 seconds
Cause: snapshotting a large root volume (especially with lots of small files) can exceed Packer’s default timeout. Fix: set aws_polling { delay_seconds = 30, max_attempts = 200 } in the source block. Also: clean up before baking (dnf clean all, remove /tmp/*, drop bash history) to shrink the snapshot.
Could not assume role with web identity (GitLab CI)
Cause: trust policy on the OIDC role doesn’t match the actual subject in the JWT. Fix: in a GitLab job, run echo $GITLAB_OIDC_TOKEN | cut -d. -f2 | base64 -d | jq and compare the sub claim to the trust policy condition. Common gotcha: trust policy uses StringEquals but the sub uses a different format (branch vs MR vs tag).
Packer plugin "amazon" not found
Cause: ran packer build without first running packer init. Fix: always packer init template.pkr.hcl in CI before packer build. Cache the ~/.packer.d/plugins directory to skip re-downloads.
Still stuck? Run Packer with PACKER_LOG=1 packer build ... for a verbose trace, and pipe to a file: 2>&1 | tee packer-debug.log. The relevant error is usually 50-100 lines above the final failure message.

What is an AMI?

AMI stands for Amazon Machine Image. It’s a complete template of a server’s disk: the operating system, system settings, and any software you installed.

When you launch an EC2 instance (a virtual server), you must tell AWS which AMI to start from. Think of the AMI as the “starting save file” for a server.

Custom AMIs let you boot servers that are already configured — no setup time, and every server is identical, which prevents the classic “it works on my machine” problem.

Read the official AMI documentation →

Why “golden image”?

A golden image is your one official, approved, tested base that everyone builds from. The name comes from manufacturing: the “golden master” copy that everything else is stamped from.

Because security patches and approved software live in the golden image, every team automatically inherits them. Update the golden image once, and the fix flows everywhere.

AWS services glossary

  • EC2 — virtual servers in the cloud.
  • AMI — the saved image an EC2 server boots from.
  • SSM Parameter Store — where AWS stores values, including the ID of the latest AMI.
  • EventBridge — listens for events and triggers actions.
  • SNS — sends one message out to many subscribers (megaphone).
  • SQS — a queue that holds messages until a worker reads them (mailbox).
  • Lambda — runs small bits of code with no server to manage.
  • IAM — controls who and what is allowed to do what.
  • OIDC — OpenID Connect, lets external systems (GitLab, GitHub) authenticate to AWS without static credentials.

How the trigger works

AWS stores the ID of the newest Amazon Linux AMI in SSM Parameter Store. When AWS releases a new one, that stored value changes.

EventBridge watches for that exact change and automatically starts your build — no human needed.

AWS also publishes notifications to public SNS topics for AMI updates.

What is Amazon Corretto?

Corretto is Amazon’s free, production-ready build of Java (OpenJDK). Corretto 21 gives you Java 21 with long-term security updates from AWS.

One command installs it: dnf install java-21-amazon-corretto-devel.

SNS vs SQS

SNS — megaphone (push). You publish once; SNS instantly pushes a copy to every subscriber.

SQS — mailbox (pull). Messages sit safely in a queue until a worker is ready to pick them up and process them one at a time.

The pattern used here is SNS → SQS fan-out: SNS broadcasts the event; each consuming team has its own SQS queue subscribed to it.
