Level 0 — Anatomy of a Play

How vars, gather_facts, pre_tasks, roles, tasks, post_tasks and handlers fit together — and the order they run.

Execution order & structure

docs ↗

A play runs sections in a fixed order. Handlers notified anywhere in a section flush at the end of that section.

Run order: gather_facts → pre_tasks (+ their handlers) → roles → tasks → post_tasks (+ their handlers). Handlers from pre_tasks flush before roles start, so a config change there can restart a service before the app role runs. Use pre_tasks for prep/validation (cache update, drain from LB) and post_tasks for verification (smoke tests, re-add to LB).
Line by line — full play skeleton
  1. - name: / hosts: web — start one play targeting the web inventory group.
  2. become: true — run tasks with privilege escalation (sudo) for the whole play.
  3. gather_facts: true — run the setup module first so ansible_facts.* are available.
  4. vars: — static play-scoped variables (app_version, listen_port).
  5. vars_files: — load extra vars from files; the second path is templated with {{ env }} so it resolves per-environment.
  6. pre_tasks: — runs before roles; here it updates the dnf cache and drains the host from the LB. Their notified handlers flush before roles begin.
  7. changed_when: false — the drain command reports nothing as 'changed' (it's an action, not a state change).
  8. roles: — runs after pre_tasks; webapp receives a role-scoped var webapp_port built from listen_port.
  9. tasks: — runs after roles; templating the config and notify-ing a handler when it changes.
  10. post_tasks: — runs last; smoke-tests the health endpoint (with retries) then re-adds the host to the LB.
  11. handlers: — only run when notified; flush at the end of each section. restart app bounces the unit.
🟡 Full play skeleton 
- name: Deploy web tier
  hosts: web
  become: true
  gather_facts: true            # collect ansible_facts before anything

  vars:                          # static play vars
    app_version: "2.4.1"
    listen_port: 8443
  vars_files:                    # load vars from files
    - vars/common.yml
    - "vars/{{ env }}.yml"

  pre_tasks:                     # runs FIRST (e.g. prep / drain)
    - name: Refresh package cache
      ansible.builtin.dnf:
        update_cache: true
    - name: Drain host from load balancer
      ansible.builtin.command: /usr/local/bin/lb-drain {{ inventory_hostname }}
      changed_when: false

  roles:                         # runs after pre_tasks (+ their handlers)
    - role: common
    - role: webapp
      vars:
        webapp_port: "{{ listen_port }}"

  tasks:                         # runs after roles
    - name: Ensure app config
      ansible.builtin.template:
        src: app.conf.j2
        dest: /etc/app/app.conf
      notify: restart app

  post_tasks:                    # runs LAST (e.g. verify / re-enable)
    - name: Smoke test health endpoint
      ansible.builtin.uri:
        url: "https://localhost:{{ listen_port }}/healthz"
        validate_certs: false
        status_code: 200
      retries: 5
      delay: 3
    - name: Re-add host to load balancer
      ansible.builtin.command: /usr/local/bin/lb-enable {{ inventory_hostname }}
      changed_when: false

  handlers:                      # triggered by notify; flush at section end
    - name: restart app
      ansible.builtin.systemd_service:
        name: app
        state: restarted

vars · vars_files · set_fact · precedence

docs ↗

Where to declare variables and which source wins when two collide.

Precedence (low → high, abbreviated): role defaults → inventory group_vars → inventory host_vars → play vars → vars_files → role vars/ → block/task vars → set_fact/registered → --extra-vars (-e, always wins). Put overridable knobs in role defaults/main.yml; put non-negotiables in role vars/main.yml. -e on the CLI beats everything.
🟢 Simple 
vars:
  pkg_state: present
  app_user: appuser

# use anywhere with Jinja
- ansible.builtin.debug:
    msg: "Installing as {{ app_user }}"
Line by line — variable sources & precedence
  1. group_vars/prod.yml — vars auto-loaded for every host in group prod; here a nested dict feature_flags.
  2. vars_files: with {{ env | default('dev') }} — Jinja substitution + the default filter, so a missing env falls back to dev.
  3. set_fact: — computes api_endpoint at runtime from a fact (ansible_facts.fqdn) and a play var; set facts sit near the top of precedence.
  4. The commented -e line — extra-vars passed on the CLI override every other source, so reserve them for true runtime overrides.
🟡 Complete 
# group_vars/prod.yml
db_pool: 50
feature_flags: { beta: false }

# play overrides per-env file
  vars_files:
    - "vars/{{ env | default('dev') }}.yml"

# computed at runtime, highest of the "normal" sources
- name: Derive endpoint
  ansible.builtin.set_fact:
    api_endpoint: "https://{{ ansible_facts.fqdn }}:{{ listen_port }}"

# override at launch (wins over all):
# ansible-playbook site.yml -e "app_version=2.5.0 env=prod"

Variable substitution — keywords & syntax

docs ↗

Ansible substitutes variables with Jinja2. Reference for the operators, filters, tests, and magic variables you'll use constantly.

Context: {{ }} = print/expression, {% %} = statement (if/for), {# #} = comment. A value that starts with {{ must be quoted in YAML (key: "{{ x }}") or YAML treats it as a dict. Never nest braces ({{ a_{{b}} }} is invalid) — use the vars/lookup('vars', ...) indirection instead. Undefined vars raise errors unless you pipe through | default(...).
🟢 Core syntax 
{{ var }}                 # substitute a variable
{{ dict.key }}  {{ dict['key'] }}   # dict access (dot or bracket)
{{ list[0] }}             # index a list
{{ a }}-{{ b }}           # concatenate by adjacency
"{{ x }}"                 # MUST quote if value starts with {{
{# this is a Jinja comment #}
{% if env == 'prod' %}...{% endif %}   # statement block
{% for h in groups['web'] %}{{ h }} {% endfor %}
Filters (| transforms the value)
  1. | default('dev') — fallback if undefined.
  2. | default(omit) — drop the param entirely if unset.
  3. | mandatory — fail if undefined.
  4. | upper | lower | trim | capitalize — string ops.
  5. | int | float | bool | string — type casts.
  6. | round(1) — numeric rounding.
  7. | length | min | max | sum — aggregates.
  8. | join(',') | split(',') — list ⇄ string.
  9. | unique | sort | reverse | flatten — list ops.
  10. | map('upper') | select | reject — iterate/filter.
  11. | to_json | from_json | to_nice_yaml — serialize.
  12. | regex_replace('a','b') — regex substitution.
  13. | b64encode | b64decode — base64.
  14. | password_hash('sha512') — hash a secret.
  15. | combine(other) — merge two dicts.
  16. | ansible.builtin.basename | dirname — path parts.
 🟡 Filters & tests 
name: "{{ raw_name | trim | lower }}"
pool: "{{ db_pool | default(5) | int }}"
opts: "{{ extra | default(omit) }}"
csv:  "{{ items | join(',') }}"
big:  "{{ nums | max }}"
merged: "{{ base_cfg | combine(env_cfg) }}"

# tests use 'is' / 'is not'
when: myvar is defined
when: result is succeeded
when: path is exists
when: item is not none
when: version is version('2.0','>=')
Magic vars, lookups & conditionals
  1. {{ inventory_hostname }} — current host's inventory name.
  2. {{ hostvars['db1'].ansible_host }} — read another host's vars.
  3. {{ groups['web'] }} — list of hosts in a group.
  4. {{ group_names }} — groups the current host belongs to.
  5. {{ ansible_facts.* }} — gathered facts (e.g. os_family).
  6. {{ item }} / {{ ansible_loop.index }} — loop value / counter.
  7. {{ lookup('env','HOME') }} — controller env var.
  8. {{ lookup('file','/p/x') }} — file contents.
  9. {{ lookup('vars', dyn_name) }} — indirect (dynamic) var name.
  10. {{ query('items', a, b) }} — lookup returning a list (for loops).
  11. {{ 50 if env=='prod' else 5 }} — inline ternary.
  12. {{ a ~ b }} — explicit string concatenation operator.
 🟡 Magic vars & lookups 
url: "https://{{ inventory_hostname }}:{{ port }}"
peer: "{{ hostvars['db1']['ansible_default_ipv4']['address'] }}"
home: "{{ lookup('env','HOME') }}"
key:  "{{ lookup('file', pubkey_path) }}"
dyn:  "{{ lookup('vars', 'cfg_' ~ env) }}"  # cfg_prod / cfg_dev
tier: "{{ 'big' if ansible_facts.memtotal_mb > 8000 else 'small' }}"

# loop with index + label
- ansible.builtin.debug:
    msg: "#{{ ansible_loop.index }} = {{ item }}"
  loop: "{{ groups['web'] }}"
  loop_control:
    extended: true        # enables ansible_loop.*

gather_facts / ansible.builtin.setup

docs ↗

Collect system facts (OS, network, CPU, memory) into ansible_facts. The setup module is what gather_facts: true calls.

Context: Fact gathering adds time per host. Disable with gather_facts: false when a play needs no facts, or narrow scope with gather_subset to speed up. Re-gather mid-play by calling setup explicitly (e.g. after installing a package that adds an interface). Enable the fact cache in ansible.cfg for large fleets. Custom facts: drop scripts/INI in /etc/ansible/facts.d/*.fact → exposed under ansible_local.
🟢 Simple 
- name: Show the OS family
  ansible.builtin.debug:
    var: ansible_facts.os_family
# common facts: ansible_facts.distribution,
# .distribution_version, .default_ipv4.address,
# .processor_vcpus, .memtotal_mb, .fqdn
Line by line — scoped & on-demand fact gathering
  1. gather_subset: with '!all' then network/hardware — collect only those subsets to cut gathering time on large fleets.
  2. ansible.builtin.setup: — explicitly re-run fact gathering mid-play (e.g. after a NIC appears).
  3. filter: "ansible_eth*" — return only facts matching the glob, keeping output small.
  4. when: ansible_facts.os_family == "RedHat" — branch logic off a gathered fact.
🟡 Complete 
# speed up: gather only what you need
- hosts: all
  gather_facts: true
  gather_subset:
    - "!all"
    - network
    - hardware

# re-gather subset on demand
- name: Refresh network facts after NIC change
  ansible.builtin.setup:
    gather_subset:
      - network
    filter: "ansible_eth*"

- name: Branch on distro
  ansible.builtin.debug:
    msg: "RHEL-like"
  when: ansible_facts.os_family == "RedHat"

roles · import_role · include_role

docs ↗

Package reusable automation. Standard dirs: tasks/ defaults/ vars/ handlers/ templates/ files/ meta/ tests/.

Context: import_role is static (parsed at parse time — tags/handlers known up front); include_role is dynamic (resolved at runtime — needed inside loops or behind when). Declare ordering/deps in meta/main.yml. Keep tunables in defaults/main.yml (lowest precedence, easy to override); put constants in vars/main.yml. Put role verification under tests/ (a small inventory + test.yml) or in the play's post_tasks.
🟢 Simple — layout 
roles/
  webapp/
    defaults/main.yml   # overridable vars
    vars/main.yml       # fixed vars
    tasks/main.yml      # entry point
    handlers/main.yml
    templates/app.conf.j2
    files/
    meta/main.yml       # deps, metadata
    tests/
      inventory
      test.yml          # role smoke test
Line by line — dependencies, includes & tests
  1. meta/main.yml dependencies: — declare that common runs before this role, passing it a var.
  2. roles: static entry — import_role-style; webapp gets a role-scoped var and a tags list for selective runs (--tags web).
  3. include_role: — dynamic include, required because it's inside a loop and guarded by when; tenant_id changes per iteration.
  4. loop: "{{ active_tenants }}" — iterate the list var; {{ item }} is the current element.
  5. tests/test.yml — a tiny standalone play that hits the health endpoint to verify the role post-deploy.
🟡 Complete — usage & deps 
# meta/main.yml  (run 'common' before this role)
dependencies:
  - role: common
    vars: { common_hardening: true }

# static include with role-scoped vars + tags
  roles:
    - role: webapp
      vars: { webapp_port: 8443 }
      tags: [web]

# dynamic include inside a loop / conditional
- name: Apply tenant roles
  ansible.builtin.include_role:
    name: tenant
  vars:
    tenant_id: "{{ item }}"
  loop: "{{ active_tenants }}"
  when: enable_multitenancy | default(false)

# roles/webapp/tests/test.yml — post-deploy check
- hosts: localhost
  tasks:
    - ansible.builtin.uri:
        url: "http://127.0.0.1:8443/healthz"
        status_code: 200

Level 1 — The Basics

The five modules you'll use every single day.

ansible.builtin.debug

docs ↗

Print messages or variable values during a run — your print() for playbooks.

Context: Controlled by -v verbosity via the verbosity param. Use var: to dump a variable (no Jinja braces) and msg: for formatted strings. Never put secrets in debug output — it lands in logs.
🟢 Simple 
- name: Say hello
  ansible.builtin.debug:
    msg: "Hello from {{ inventory_hostname }}"
Line by line
  1. var: ansible_facts.network_interfaces — dump a variable by name (no {{ }} needed with var:).
  2. verbosity: 2 — only print when run with -vv or higher, keeping normal output clean.
  3. msg: >- — YAML folded block scalar; newlines become spaces, the - strips the trailing newline.
  4. {{ ... | round(1) }} — Jinja expression doing math (MB→GB) piped through the round filter.
  5. when: ... is defined — only run if the fact exists, avoiding undefined-variable errors.
🟡 Complete 
- name: Dump structured facts only when very verbose
  ansible.builtin.debug:
    var: ansible_facts.network_interfaces
    verbosity: 2          # shows only with -vv

- name: Conditional formatted message
  ansible.builtin.debug:
    msg: >-
      Host {{ inventory_hostname }} has
      {{ ansible_facts.processor_vcpus }} vCPUs and
      {{ (ansible_facts.memtotal_mb / 1024) | round(1) }} GB RAM
  when: ansible_facts.memtotal_mb is defined

ansible.builtin.command

docs ↗

Run a binary directly (no shell). Safer than shell — no pipes, redirects, or env expansion.

Context: Not idempotent by itself — guard it with creates:/removes: or when:. If you need |, >, && or $VARS, use ansible.builtin.shell instead.
🟢 Simple 
- name: Check version
  ansible.builtin.command: cat /etc/os-release
Line by line
  1. cmd: — the executable and args (run directly, no shell).
  2. chdir: — change into this directory before running.
  3. creates: — idempotency guard: skip the task entirely if this path already exists.
  4. register: migrate_out — capture rc/stdout/stderr into a variable for later use.
  5. changed_when: — define 'changed' yourself: only when stdout contains applied.
  6. failed_when: — treat rc 0 and 2 as success; anything else fails the task.
🟡 Complete 
- name: Run migration once (idempotent guard)
  ansible.builtin.command:
    cmd: /opt/app/bin/migrate --apply
    chdir: /opt/app
    creates: /opt/app/.migrated      # skips if file exists
  register: migrate_out
  changed_when: "'applied' in migrate_out.stdout"
  failed_when: migrate_out.rc not in [0, 2]

ansible.builtin.copy

docs ↗

Copy files to remote hosts, set ownership/permissions, optionally back up.

Context: For dynamic content use template. Use content: to write inline strings. backup: true keeps a timestamped copy before overwriting — great for configs.
🟢 Simple 
- name: Drop a config file
  ansible.builtin.copy:
    src: app.conf
    dest: /etc/app/app.conf
Line by line
  1. src/dest — source file (controller) → destination path (remote).
  2. owner/group/mode: "0640" — set ownership and octal perms (quote octal to keep leading zero).
  3. backup: true — save a timestamped copy of the old file before overwriting.
  4. validate: "nginx -t -c %s" — run a check on the temp file first; %s is the temp path. Task fails if invalid, so a bad config never lands.
  5. notify: reload nginx — trigger the handler only if the file actually changed.
🟡 Complete 
- name: Deploy secured config with backup & validation
  ansible.builtin.copy:
    src: nginx.conf
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    mode: "0640"
    backup: true
    validate: "nginx -t -c %s"   # %s = temp path, fails task if invalid
  notify: reload nginx

Level 2 — Files, Packages & Services

Configuration management core.

ansible.builtin.file

docs ↗

Manage paths: create dirs, symlinks, set perms, or delete with state: absent.

Context: state options: file, directory, link, hard, touch, absent. To create parent dirs recursively, use state: directory + recurse: true for perms.
🟢 Simple 
- name: Ensure log dir exists
  ansible.builtin.file:
    path: /var/log/app
    state: directory
Line by line
  1. First task — state: directory with recurse: true applies owner/group/mode down the whole tree.
  2. owner: appuser / group: appgrp — set ownership of the created path(s).
  3. Second task — state: link creates a symlink at dest pointing to src.
  4. force: true — replace an existing file/link at dest if needed (common for blue/green 'current' switches).
🟡 Complete 
- name: Create owned tree + symlink
  ansible.builtin.file:
    path: /opt/app/releases
    state: directory
    owner: appuser
    group: appgrp
    mode: "0755"
    recurse: true

- name: Point 'current' at latest release
  ansible.builtin.file:
    src: /opt/app/releases/v2.4.1
    dest: /opt/app/current
    state: link
    force: true

ansible.builtin.package

docs ↗

OS-agnostic package install — auto-picks apt/yum/dnf/zypper.

Context: Great for cross-distro roles. When you need distro-specific options (e.g. apt's update_cache), use ansible.builtin.apt/dnf directly.
🟢 Simple 
- name: Install git
  ansible.builtin.package:
    name: git
    state: present
Line by line
  1. name: as a list — install several packages in one task.
  2. state: latest — ensure newest available (use pinned versions in prod instead).
  3. register: pkg + retries/delay/until — retry up to 3 times, 5s apart, until the task succeeds (resilient to transient mirror failures).
🟡 Complete 
- name: Install a tool set across distros
  ansible.builtin.package:
    name:
      - curl
      - jq
      - python3-pip
    state: latest
  register: pkg
  retries: 3
  delay: 5
  until: pkg is succeeded

ansible.builtin.service

docs ↗

Start/stop/enable services across init systems.

Context: For systemd-only features (daemon-reload, masking) prefer ansible.builtin.systemd_service. Pair with handlers so restarts only fire on change.
🟢 Simple 
- name: Ensure nginx running
  ansible.builtin.service:
    name: nginx
    state: started
    enabled: true
Line by line — handler form
  1. Defined in handlers/main.yml so it only runs when notify-ed.
  2. systemd_service — used instead of service because we need systemd features.
  3. state: reloaded — graceful reload (no dropped connections) rather than a full restart.
  4. daemon_reload: true — re-read unit files first, in case the unit changed.
🟡 Complete 
# handlers/main.yml
- name: reload nginx
  ansible.builtin.systemd_service:
    name: nginx
    state: reloaded
    daemon_reload: true

Level 2ᵇ — systemd & dnf in depth

The two builtins you reach for most on RHEL/Fedora/Rocky hosts.

ansible.builtin.systemd_service

docs ↗

Full systemd control: start/stop/restart/reload, enable at boot, daemon_reload, masking, and scopes (system/user/global).

Context: systemd_service (alias systemd) is preferred over the generic service module when you need daemon_reload: true (after dropping a new unit file), masked:, scope: user/global, or to verify the unit is actually active. Always daemon_reload after writing/editing a .service file, or systemd won't see your changes. Use state: reloaded for graceful config reloads vs restarted for a full bounce.
🟢 Simple 
- name: Enable & start service
  ansible.builtin.systemd_service:
    name: nginx
    state: started
    enabled: true
Line by line
  1. First task — template drops a unit file into /etc/systemd/system and notifies a reload handler.
  2. Second task — state: started + enabled: true ensures it runs now and at boot.
  3. daemon_reload: true — mandatory after adding/editing a .service file so systemd sees it.
  4. Third task — masked: true fully disables a unit (symlinks it to /dev/null) so it can't be started.
  5. Handler — state: reloaded for a graceful config reload of myapp.
🟡 Complete 
- name: Install a custom unit file
  ansible.builtin.template:
    src: myapp.service.j2
    dest: /etc/systemd/system/myapp.service
    mode: "0644"
  notify: reload myapp

- name: Ensure unit is loaded, enabled, running
  ansible.builtin.systemd_service:
    name: myapp
    state: started
    enabled: true
    daemon_reload: true        # pick up the new unit file

- name: Mask a service so it can't be started
  ansible.builtin.systemd_service:
    name: telnet.socket
    masked: true

# handlers/main.yml
- name: reload myapp
  ansible.builtin.systemd_service:
    name: myapp
    state: reloaded
    daemon_reload: true

ansible.builtin.dnf

docs ↗

Manage packages on dnf-based distros (RHEL 8+, Fedora, Rocky, Alma): install, remove, groups, modules, GPG keys, cache.

Context: Use ansible.builtin.dnf directly (not the generic package) when you need dnf-specific features: update_cache, enablerepo/disablerepo, group installs (@group), install_weak_deps, security-only updates, or appstream module streams. For RHEL 7 use the yum module. Prefer pinned versions (name: nginx-1.24.*) in prod over state: latest to keep runs deterministic. Always import the repo GPG key first and keep gpgcheck on.
🟢 Simple 
- name: Install packages
  ansible.builtin.dnf:
    name:
      - httpd
      - mod_ssl
    state: present
    update_cache: true
Line by line
  1. rpm_key — import the vendor's GPG key into the RPM keyring so signed packages are trusted.
  2. yum_repository — define the repo; $releasever is a dnf variable, gpgcheck: true enforces signature checks.
  3. name: "vendor-agent-3.2.*" — version-pinned glob for deterministic installs.
  4. enablerepo: vendor + disablerepo: "*" — install only from that repo.
  5. install_weak_deps: false — skip optional recommended packages.
  6. name: "@Development Tools" — the @ prefix installs a package group.
  7. security: true with state: latest — apply security errata only; register: patched captures whether anything changed.
  8. Final task — reboot conditionally, only if patches landed and the flag allows it.
🟡 Complete 
- name: Trust a vendor repo GPG key
  ansible.builtin.rpm_key:
    key: https://repo.vendor.com/RPM-GPG-KEY-vendor
    state: present

- name: Add the repo
  ansible.builtin.yum_repository:
    name: vendor
    description: Vendor Repo
    baseurl: https://repo.vendor.com/el/$releasever/
    gpgcheck: true
    gpgkey: https://repo.vendor.com/RPM-GPG-KEY-vendor

- name: Install pinned version from that repo only
  ansible.builtin.dnf:
    name: "vendor-agent-3.2.*"
    state: present
    enablerepo: vendor
    disablerepo: "*"
    install_weak_deps: false

- name: Install a package group
  ansible.builtin.dnf:
    name: "@Development Tools"
    state: present

- name: Apply security updates only
  ansible.builtin.dnf:
    name: "*"
    state: latest
    security: true
  register: patched

- name: Reboot if a kernel/security update needs it
  ansible.builtin.reboot:
  when: patched.changed and reboot_if_needed | default(false)

Level 3 — Control Flow & Templates

Loops, conditionals, dynamic content.

ansible.builtin.template

docs ↗

Render a Jinja2 .j2 file with variables and copy it to the host.

Context: Templates run on the controller. Add validate: for config safety and {{ ansible_managed }} headers so humans know not to hand-edit.
🟢 Simple 
- name: Render config
  ansible.builtin.template:
    src: app.conf.j2
    dest: /etc/app/app.conf
Line by line
  1. src: "app.{{ env }}.conf.j2" — the template filename itself is built via substitution, selecting an env-specific template.
  2. mode/backup — secure perms and keep the previous version.
  3. validate: — run appctl check on the rendered temp file before installing.
  4. vars: with {{ 50 if env == 'prod' else 5 }} — Jinja inline conditional (ternary) sets a bigger pool in prod.
  5. notify: restart app — only restart when the rendered file changes.
🟡 Complete 
- name: Render env-specific config
  ansible.builtin.template:
    src: "app.{{ env }}.conf.j2"
    dest: /etc/app/app.conf
    mode: "0640"
    backup: true
    validate: "/usr/bin/appctl check %s"
  vars:
    db_pool: "{{ 50 if env == 'prod' else 5 }}"
  notify: restart app

ansible.builtin.set_fact + loops

docs ↗

Compute and store variables mid-play; iterate with loop.

Context: set_fact values persist for the host across the play. Prefer loop: over the legacy with_*. Use loop_control: label: to keep output readable.
🟢 Simple 
- name: Build a fact
  ansible.builtin.set_fact:

End-to-end ETL with builtins

docs ↗

Extract via get_url/uri, transform via command, load via cloud modules — with idempotent guards.

Context: Keep each stage idempotent. Use register + changed_when so reruns are safe. Stamp a run_id with set_fact for traceable artifact paths.
Line by line — Extract → Transform → Load
  1. set_fact: run_id — stamp a unique id from ansible_date_time.iso8601_basic_short used in every artifact path for traceability.
  2. EXTRACT — get_url downloads the source; checksum: "sha256:{{ expected_sha }}" verifies integrity.
  3. TRANSFORM — command runs the transform; creates: makes it idempotent and changed_when: tf.rc == 0 reports change only on success.
  4. LOAD — s3_object mode: put with encrypt: true ships the result; the key uses {{ run_id }}.
  5. VALIDATE — uri POST tells the warehouse to refresh, sending run_id in a JSON body, expecting 202.
🟡 Complete 
- name: Stamp run id

- name: Pull source dataset

    dest: "/data/raw/{{ run_id }}.csv"

    checksum: "sha256:{{ expected_sha }}"

# TRANSFORM
- name: Run transform job
  ansible.builtin.command:
    cmd: "python3 transform.py /data/raw/{{ run_id }}.csv /data/out/{{ run_id }}.parquet"
    creates: "/data/out/{{ run_id }}.parquet"
  register: tf
  changed_when: tf.rc == 0

# LOAD (AWS)
- name: Load to data lake
  amazon.aws.s3_object:
    bucket: my-data-lake
    object: "/curated/{{ run_id }}.parquet"
    src: "/data/out/{{ run_id }}.parquet"
    mode: put
    encrypt: true

# VALIDATE
- name: Notify warehouse to refresh

    method: POST
    body_format: json
    body: { run_id: "{{ run_id }}" }

Certificates, Keystores & Trusted CAs

Managing TLS material with builtins + the community.crypto collection.

Install & trust a CA certificate

docs ↗

Drop a CA into the OS trust store so all system tools trust it — builtins only.

Context: Trust-store path differs by distro: Debian/Ubuntu use /usr/local/share/ca-certificates/*.crt + update-ca-certificates; RHEL/Fedora use /etc/pki/ca-trust/source/anchors/*.pem + update-ca-trust. Trigger the rebuild via a handler so it runs once after any cert change. Set mode: "0644" — CA public certs are world-readable; never place private keys here.
Line by line — cross-distro CA trust
  1. RedHat task — copy the CA into /etc/pki/ca-trust/source/anchors/, guarded by when: os_family == "RedHat".
  2. Debian task — copy into /usr/local/share/ca-certificates/; the file must end in .crt for Debian's tooling.
  3. mode: "0644" — CA public certs are world-readable (never put private keys here).
  4. notify: — each branch triggers its own handler.
  5. Handlers — update-ca-trust extract (RedHat) / update-ca-certificates (Debian) rebuild the system trust bundle once.
🟡 Complete — cross-distro 
- name: Install internal root CA (RedHat)
  ansible.builtin.copy:
    src: internal-root-ca.pem
    dest: /etc/pki/ca-trust/source/anchors/internal-root-ca.pem
    owner: root
    group: root
    mode: "0644"
  when: ansible_facts.os_family == "RedHat"
  notify: update ca trust (redhat)

- name: Install internal root CA (Debian)
  ansible.builtin.copy:
    src: internal-root-ca.crt          # MUST end in .crt for Debian
    dest: /usr/local/share/ca-certificates/internal-root-ca.crt
    mode: "0644"
  when: ansible_facts.os_family == "Debian"
  notify: update ca trust (debian)

# handlers/main.yml
- name: update ca trust (redhat)
  ansible.builtin.command: update-ca-trust extract
- name: update ca trust (debian)
  ansible.builtin.command: update-ca-certificates

Issue keys & certs (community.crypto)

docs ↗

Generate a private key, CSR, and a (self-signed or CA-signed) certificate idempotently.

Context: Install with ansible-galaxy collection install community.crypto (needs the cryptography python lib). These modules are idempotent — they only regenerate when inputs change. Keep private keys mode: "0600", owned by the service user, and never commit them: store passphrases in Ansible Vault and mark tasks no_log: true. For public PKI, the acme_certificate module automates Let's Encrypt.
🟢 Simple — self-signed 
- community.crypto.openssl_privatekey:
    path: /etc/ssl/private/app.key
    mode: "0600"

- community.crypto.x509_certificate:
    path: /etc/ssl/certs/app.crt
    privatekey_path: /etc/ssl/private/app.key
    provider: selfsigned
Line by line — key + CSR + CA-signed cert
  1. openssl_privatekey — 4096-bit key owned by the app user, mode: "0600", task marked no_log: true so key material never hits logs.
  2. openssl_csr — CSR referencing that key; common_name + subject_alt_name list the names the cert is valid for.
  3. x509_certificate provider: ownca — sign the CSR with your internal CA.
  4. ownca_path/ownca_privatekey_path — the CA cert and key doing the signing.
  5. ownca_not_after: "+365d" — validity of one year.
  6. notify: reload nginx — reload the server when the cert changes. All modules here are idempotent — they regenerate only when inputs change.
🟡 Complete — key + CSR + signed cert 
- name: Private key (secured)

    size: 4096
    owner: appuser
    mode: "0600"

- name: CSR with SANs
  community.crypto.openssl_csr:
    path: /etc/ssl/csr/app.csr
    privatekey_path: /etc/ssl/private/app.key
    common_name: app.internal.example.com
    subject_alt_name:
      - "DNS:app.internal.example.com"
      - "DNS:app"

- name: Sign with our internal CA
  community.crypto.x509_certificate:
    path: /etc/ssl/certs/app.crt
    csr_path: /etc/ssl/csr/app.csr

    ownca_privatekey_path: /etc/pki/ca/internal-root-ca.key

docs ↗

Build JKS/PKCS12 keystores for the JVM and bundle cert+key for app servers.

Best Practices for Builtins

1. Always FQCN

Use ansible.builtin.copy, not copy. Prevents collection collisions and is lint-clean.

2. Idempotency first

Guard command/shell with creates, removes, when, or changed_when. Reruns must be no-ops.

3. command > shell

Only use shell when you truly need pipes/redirects/expansion. command avoids injection risk.

4. Name every task

Readable output and audit logs. Names double as documentation.

5. Handlers for restarts

notify + handlers means services restart only when config actually changes.

6. Secrets in Vault

Never inline keys/passwords. Use ansible-vault or cloud secret stores; pass via vars.

7. loop, not with_*

Modern loop: + loop_control: label: keeps logs clean and behavior predictable.

8. Validate before apply

Use validate: on copy/template to fail safely on bad config.

9. check_mode & --diff

Dry-run with --check --diff in CI before real applies; mark read-only tasks check_mode: false where needed.

10. Retry transient ops

Network/cloud calls: retries + delay + until for resilience.

11. block/rescue for safety

Wrap risky deploys with rollback in rescue and cleanup in always.

12. Don't log secrets

Add no_log: true on tasks handling sensitive data.

13. Separate prep / verify

Use pre_tasks for prep (drain, cache) and post_tasks for verification (smoke tests, re-enable). Roles stay focused on the "what."

14. defaults vs vars

Overridable role knobs → defaults/main.yml (lowest precedence). Non-negotiable constants → vars/main.yml. -e always wins, so reserve it for runtime overrides.

15. Trim fact gathering

Disable gather_facts when unused, or scope with gather_subset. Enable a fact cache for large fleets to cut run time.

16. daemon_reload after units

Always daemon_reload: true after writing a .service file, and prefer systemd_service over service for systemd hosts.

17. Pin packages in prod

Prefer name: pkg-1.2.* over state: latest for deterministic runs; keep gpgcheck on and import keys with rpm_key first.

18. Private keys 0600 + Vault

Keys mode:"0600" owned by the service user; passphrases in ansible-vault; key/keystore tasks no_log: true. Public/CA certs are 0644.

19. Trust store, not per-app

Install internal CAs into the OS trust store (update-ca-trust / update-ca-certificates via handler) so every tool trusts them, instead of disabling validate_certs.

20. Idempotent crypto

Use community.crypto modules (not shell openssl) so keys/certs regenerate only when inputs change; automate renewals before expiry.
