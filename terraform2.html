Section 1

What is EKS? (And what is Kubernetes?)

Let's start with a story. Imagine you wrote a small app — maybe a website. To run it, you need a computer that's always on. That's a server. Easy enough: one app, one server.

But what happens when your app gets popular? One server isn't enough. Now you have ten servers, and you need someone to decide which server runs which copy of your app, to restart copies that crash, to add more servers when traffic spikes, and to take servers offline for repairs without anyone noticing. Doing all of that by hand would be a full-time job — actually, many full-time jobs.

Kubernetes: the robot manager for your apps

Kubernetes (often shortened to K8s — there are 8 letters between the "K" and the "s") is software that does that managing for you. You tell it "I always want 5 copies of my app running," and it makes that true. A copy crashes? Kubernetes starts a new one. Need to update your app? Kubernetes swaps copies out one at a time so the app never goes fully dark. Think of it as an extremely reliable robot manager standing over a room full of computers, making sure your apps are always running the way you asked.

The little packaged-up copies of your app that Kubernetes moves around are called containers, and the individual app units Kubernetes schedules are called pods.

Why "managed"? Enter EKS

Here's the catch: Kubernetes itself is complicated to run. The "brain" of Kubernetes — the part that makes all those decisions — is a cluster of computers that needs to be set up, secured, backed up, and patched constantly. Running that brain yourself is hard and risky.

Amazon EKS (Elastic Kubernetes Service) solves this. EKS is Amazon's managed Kubernetes. The word "managed" is the whole point: Amazon runs and babysits the Kubernetes brain for you, around the clock, spread across multiple data centers so it doesn't go down. You just say "give me a cluster," and Amazon hands you a working, secure Kubernetes brain. You bring the worker computers that run your actual apps.

The two halves of any Kubernetes cluster
┌───────────────────────────────────────────────────────────┐ │ CONTROL PLANE ("the brain") ← Amazon runs this │ │ • API server (you send commands here) │ │ • Scheduler (decides where apps run) │ │ • etcd database (remembers everything) │ └───────────────────────────────────────────────────────────┘ ▲ │ gives orders to ▼ ┌───────────────────────────────────────────────────────────┐ │ DATA PLANE ("the muscle") ← YOU run this │ │ • Worker Node 1 [ your app ] [ your app ] │ │ • Worker Node 2 [ your app ] [ database ] │ │ • Worker Node 3 [ ...scales up and down... ] │ │ (these are EC2 virtual computers in a "node group") │ └───────────────────────────────────────────────────────────┘

The EKS pieces you'll hear about

A few EKS-specific words show up everywhere. Here they are in plain English:

  • Managed node group — a batch of worker computers (Amazon EC2 machines) that EKS sets up and looks after for you. You say "I want between 1 and 3 of these," and EKS keeps that true.
  • Add-ons — small helper programs every cluster needs: networking for pods (vpc-cni), an in-cluster phone book (coredns), traffic routing (kube-proxy), and disk storage (aws-ebs-csi-driver). EKS can install and update these for you.
  • IRSA (IAM Roles for Service Accounts) — a way to give one specific app permission to use one specific AWS thing, instead of handing out master keys. This is the safe, grown-up way to do permissions, and it's essential when many teams share one cluster.
  • Access entries — the modern way to say "this person/role is allowed to control the cluster." It replaced an older, fiddlier method called the aws-auth ConfigMap.
💡 One-sentence summary Kubernetes is a robot that runs your apps across many computers; EKS is Amazon doing the hardest, most dangerous part of running that robot for you.
Section 2

What is Terraform?

Now you can ask Amazon for a cluster. But how do you ask? You could log into the Amazon website and click buttons — choose a region, click "create VPC," click "create cluster," and so on. That works once. But it has big problems:

  • You'll forget exactly which buttons you clicked. Next month, can you rebuild it identically? Probably not.
  • A teammate can't see what you did or review it before it happens.
  • If you need the same setup in a second region, you click everything all over again — and make mistakes.

Terraform fixes this with an idea called Infrastructure as Code, usually shortened to IaC. Instead of clicking buttons, you write down what you want in text files. Things like: "I want a network. Inside it, I want a Kubernetes cluster. Attached to it, I want 3 worker computers." You save those files, and Terraform reads them and builds exactly that in Amazon for you.

The magic part: Terraform remembers what it built. If you later change the file to say "5 worker computers instead of 3," Terraform compares what you want against what already exists, figures out the difference, and makes only that one change. You describe the destination; Terraform figures out the driving directions.

How Terraform thinks
YOUR FILES TERRAFORM AMAZON CLOUD (what you want) (what exists) "3 worker nodes" ───▶ compares want vs. have ───▶ currently: 0 nodes │ ▼ "I need to CREATE 3" │ ▼ makes the API calls ───────▶ now: 3 nodes ✓

The 5 Terraform words you need

WordPlain meaningLearn more
ResourceOne single thing you want built (one bucket, one cluster, one network).
ModuleA reusable bundle of many resources someone already figured out, so you don't reinvent it. We use Amazon's official ones.
ProviderA plugin that teaches Terraform how to talk to a specific service (the aws provider, the kubernetes provider).
StateTerraform's memory file. It records everything Terraform has built. Protect this file — it's the source of truth.
Plan / ApplyPlan = a preview ("here's what I'm about to change"). Apply = actually do it. Always read the plan first.
💡 One-sentence summary Terraform turns "click buttons in the cloud" into "write a text file and save it" — so your whole setup is written down, repeatable, and reviewable.
Section 3

Why run Terraform in a pipeline (GitHub Actions)?

You could run Terraform from your own laptop. Lots of people start that way. But on a real team, that gets dangerous fast: two people run it at once and corrupt the memory file; someone applies a change nobody reviewed; the "official" version of your infrastructure lives on one person's computer.

A pipeline is an automated assistant that runs your commands for you, the same way every time, on a shared server in the cloud. GitHub Actions is the pipeline built into GitHub (where your code lives).

Here's the workflow it gives you, which mirrors how software teams already review code:

You change a file │ ▼ Open a "Pull Request" ──▶ Pipeline runs `terraform plan` (a proposal to merge) and posts the preview as a comment │ │ │ ▼ │ A human reads the preview and │ approves — BEFORE anything changes ▼ Merge to the main branch ──▶ Pipeline runs `terraform apply` (optionally waits for a final OK) │ ▼ Change is live in AWS ✓

Why this is so much better:

  • Review before reality. Every change is previewed and approved before it touches the cloud.
  • One source of truth. Whatever is in your main branch is exactly what's deployed. No mystery.
  • No stored passwords. The pipeline logs into AWS using a temporary, auto-expiring pass called OIDC — there are no long-lived AWS keys sitting in GitHub to be stolen.
  • No collisions. A lock (using a small Amazon database) stops two runs from happening at the same time.
  • A paper trail. Git history shows who changed what and when, forever.
💡 One-sentence summary A pipeline makes infrastructure changes safe and boring: propose → preview → review → apply, all automatically and on the record.
Section 4

How to structure your GitHub repository

A repository (or "repo") is just the folder that holds all your project's files in GitHub. How you arrange the folders inside it matters a lot. Here's the layout this guide uses, and below it, why it looks this way.

# Your repository, folder by folder . ├── .github/ │ └── workflows/ │ ├── cluster.yml # Pipeline: preview on proposal, build on merge │ └── lint.yml # Pipeline: safety + security checks on every proposal ├── terraform/ │ ├── bootstrap/ # Run ONCE: creates the "memory file" storage │ │ ├── main.tf │ │ ├── variables.tf │ │ ├── outputs.tf │ │ └── terraform.tfvars.example │ ├── cluster/ # ★ THE CLUSTER — what this guide focuses on │ │ ├── backend.tf # Where the memory file is kept │ │ ├── providers.tf # Which clouds/tools Terraform talks to │ │ ├── variables.tf # All the knobs you can turn │ │ ├── vpc.tf # The private network │ │ ├── eks.tf # The cluster brain + worker nodes + add-ons │ │ ├── outputs.tf # Facts other folders will need later │ │ └── terraform.tfvars.example │ ├── node-groups/ # FUTURE: extra worker computers (empty for now) │ │ └── README.md │ └── tenants/ # FUTURE: per-team isolation (empty for now) │ └── README.md ├── .gitignore # Files Git should never save (secrets, memory files) ├── .tflint.hcl # Rules for the code-checker └── README.md # The short version of this page

Why are cluster, node-groups, and tenants separate folders?

This is the single most important decision in the whole setup, so here's the reasoning slowly.

Each of those folders is its own independent Terraform project with its own separate memory file. Picture three separate notebooks instead of one shared notebook. Because they're separate:

  • A risky change to one team's setup (in tenants/) physically cannot accidentally damage the cluster's brain (in cluster/) — they don't share a memory file.
  • The pipeline only runs the folder that actually changed. Edit a worker pool? Only the node-group pipeline runs. The cluster is left untouched.
  • Different people can own different folders, with different approval rules.

This idea — keeping the dangerous, rarely-changed "foundation" apart from the frequently-changed "stuff on top" — is a core best practice. The official AWS guidance on Terraform structure says the same thing.

🔮 About the empty folders node-groups/ and tenants/ start as placeholders with just a README inside. That's on purpose — this guide builds the cluster foundation only. Section 11 shows exactly how those folders get filled in later without ever touching the cluster.
Section 5

What actually gets built

When the pipeline runs the cluster/ folder, here's the picture it creates inside your AWS account. Don't worry about every box yet — we'll explain each file that makes these in Section 8.

Your cluster, top to bottom
┌─────────────────────────────────────────────────────────────────┐ │ VPC — your own private network in AWS, spanning 3 data centers │ │ (these data centers are called "Availability Zones") │ │ │ │ ┌─────────────── PUBLIC subnets (one per zone) ───────────────┐ │ │ │ Internet-facing doors. Load balancers live here. │ │ │ │ A "NAT Gateway" sits here so private nodes can reach OUT │ │ │ │ to the internet — but the internet can't reach IN to them. │ │ │ └──────────────────────────────────────────────────────────────┘ │ │ │ │ ┌─────────────── PRIVATE subnets (one per zone) ──────────────┐ │ │ │ EKS Control Plane (the brain) — managed by Amazon │ │ │ │ │ │ │ │ "system" node group — a couple of small worker computers, │ │ │ │ reserved (via a "taint") for core add-ons only. │ │ │ │ │ │ │ │ Add-ons running here: vpc-cni · coredns · kube-proxy · │ │ │ │ aws-ebs-csi-driver │ │ │ └──────────────────────────────────────────────────────────────┘ │ │ │ │ Plus: an OIDC provider (enables IRSA / safe per-app permissions) │ │ and access entries (who is allowed to control the cluster) │ └─────────────────────────────────────────────────────────────────┘

Separately, the bootstrap folder (which you run just once at the very start) creates two small things that store Terraform's memory file safely: an S3 bucket (cloud file storage, with version history and encryption turned on) and a DynamoDB table (a tiny database used only as a lock so two runs can't collide).

🪧 Why only ONE small "system" node group? The cluster ships with just enough worker computers to run the core add-ons, and those nodes are "tainted" — marked as off-limits to normal apps. Real application capacity and per-team capacity are added later in the node-groups/ and tenants/ folders. This keeps the foundation tiny, cheap, and stable.

Section 7 · The main event

Step-by-step setup

Follow these in order. Each step says what you do and why it matters.

1

Create the "memory file" storage (bootstrap)

Remember Terraform's memory file (the state)? It needs a safe home before the pipeline can use it. There's a classic chicken-and-egg here: you can't store the storage's own state in storage that doesn't exist yet. So this one folder runs from your laptop and keeps its state locally. You run it once.

cd terraform/bootstrap # Copy the example settings file, then edit it with your real values cp terraform.tfvars.example terraform.tfvars terraform init # downloads the AWS plugin; uses LOCAL state on purpose terraform apply # type "yes" to create the S3 bucket + lock table

When it finishes, it prints two names — state_bucket_name and lock_table_name. Write them down. You'll paste them into GitHub in step 3.

2

Let GitHub log into AWS safely (OIDC)

The pipeline needs permission to build things in AWS — but we refuse to store a permanent AWS password in GitHub. Instead we set up OIDC: a trust handshake where GitHub proves who it is and AWS hands back a temporary pass that expires in minutes.

In AWS, you create an IAM OIDC identity provider for GitHub, then an IAM role that trusts only your repository. The trust rule looks like this:

// IAM role "trust policy" — who is allowed to assume this role { "Version": "2012-10-17", "Statement": [{ "Effect": "Allow", "Principal": { "Federated": "arn:aws:iam::<ACCOUNT_ID>:oidc-provider/token.actions.githubusercontent.com" }, "Action": "sts:AssumeRoleWithWebIdentity", "Condition": { "StringEquals": { "token.actions.githubusercontent.com:aud": "sts.amazonaws.com" }, // This line is the lock: only THIS repo can use the role "StringLike": { "token.actions.githubusercontent.com:sub": "repo:<ORG>/<REPO>:*" } } }] }

Then attach a permissions policy to that role allowing the AWS actions Terraform needs (EKS, VPC, IAM, S3, DynamoDB). Start a little broad to get going, then tighten over time toward "least privilege" — only what's actually used.

⚠️ Keep the lock tight The :sub condition controls which repo (and optionally which branch) may borrow this role. Keep it as specific as you can — this is your main security boundary.
3

Tell GitHub your settings (repository variables)

In your GitHub repo, go to Settings → Secrets and variables → Actions → Variables and add these. They're not secret (no passwords here), so plain "Variables" is correct.

Variable nameExample value
AWS_ROLE_ARNarn:aws:iam::123456789012:role/gha-eks
AWS_REGIONus-east-1
TF_STATE_BUCKETeks-platform-tfstate-123456789012
TF_LOCK_TABLEeks-platform-tflock
PROJECT_NAMEeks-platform
ENVIRONMENTdev
4

Add an approval gate (recommended)

In GitHub, create an Environment named production and add yourself as a required reviewer. Now, before the pipeline applies any real change, it pauses and waits for you to click approve. This is your seatbelt.

5

Open a pull request and watch it work

Push the code to a branch and open a pull request (a proposal to merge your changes). The pipeline springs to life: it checks formatting, sets things up, and runs terraform plan, then posts the full preview as a comment right on your pull request. Read it carefully. When you're happy, merge to main. The apply job runs (pausing for your approval if you did step 4) and builds the cluster.

Finally, point kubectl at your brand-new cluster with the command Terraform printed for you:

aws eks update-kubeconfig --region us-east-1 --name eks-platform-dev kubectl get nodes # you should see your system node(s) listed!
🎉 That's the whole setup Bootstrap once, wire up OIDC + variables once, then it's just pull requests forever after.
Section 8

The Terraform files, explained

Every .tf file in the project is heavily commented inside. Here's the plain-English tour of what each one does and why.

📁 terraform/bootstrap/ — the memory-file storage

FileWhat it does, in plain words
main.tfCreates the S3 bucket that holds Terraform's state, with three protections: versioning (keep a history so you can roll back a bad state), encryption (state can contain secrets, so scramble it at rest), and a public-access block (never let it be seen publicly). It also sets prevent_destroy so a careless command can't delete the bucket holding all your memory. Finally it makes the DynamoDB lock table so two pipeline runs can't collide.
variables.tfThe adjustable inputs: region, project name, and your 12-digit account ID. Each has a validation rule that rejects obviously-wrong input before any cloud call is made.
outputs.tfPrints the bucket and table names so you can paste them into GitHub.
Why does this folder use local state?It's the chicken-and-egg again: you can't keep the storage's state inside storage that doesn't exist yet. So bootstrap keeps its small state on your laptop. It almost never changes after the first run.

📁 terraform/cluster/ — the cluster itself

backend.tf — where the memory lives

Tells Terraform to store this folder's state in your S3 bucket at the path cluster/terraform.tfstate, and to lock using your DynamoDB table. Because backend settings can't use variables, the bucket and table names are passed in by the pipeline at "init" time — keeping account-specific details out of the file. It also pins the Terraform version and the plugins it needs (aws, kubernetes, tls).

providers.tf — who Terraform talks to

Sets up the AWS provider (with your region and a set of tags stamped onto every resource so you can track cost and ownership). It also sets up the Kubernetes provider, which authenticates using a short-lived token from aws eks get-token — so no cluster password is ever written to disk.

variables.tf — all the knobs

Every setting you might change: region, Kubernetes version, network size, how many availability zones, the baseline node group's size and machine type, who gets admin, and whether the cluster's control door is open to the public internet (and from which addresses). Several knobs have validation rules.

vpc.tf — the private network

Uses Amazon's official VPC module to build the network: public and private subnets spread across availability zones, a NAT gateway so private nodes can reach out to the internet, and — critically — special tags on the subnets that EKS and load balancers require to find the right place to put things. The function cidrsubnet() automatically slices your big network range into neat per-zone pieces, so you don't do the math by hand.

eks.tf — the heart of everything

Uses Amazon's official EKS module — the community standard maintained by experts — to create, in one place:

  • the control plane (the brain) at your chosen Kubernetes version, with the public door optionally open and restricted to specific addresses;
  • IRSA turned on (the OIDC provider), so individual apps can get narrow, safe permissions;
  • the four core add-ons, with pod-networking installed before nodes join so networking is ready instantly, and the storage driver wired to its own narrow IRSA permission;
  • the single "system" node group, labeled and tainted so only core components run there;
  • access entries granting admin to the pipeline (and anyone you list).

outputs.tf — facts for later folders

This is the contract the future node-groups/ and tenants/ folders read. It publishes the cluster name, its address, the network ID, the private subnet IDs, the node security group, and the all-important OIDC provider ARN. Future folders read these automatically (via a terraform_remote_state lookup) instead of hardcoding anything. It also prints the ready-to-run kubectl setup command.

💡 Why use Amazon's official modules instead of writing it all yourself? Setting up EKS correctly involves hundreds of small, easy-to-get-wrong details (IAM roles, security groups, the OIDC provider, add-on wiring). The official modules encode all of that, are battle-tested, and are kept up to date by the community. You get best practices for free.
Section 9

The GitHub Actions pipeline, explained

.github/workflows/cluster.yml — preview & build

This is the main pipeline for the cluster. Reading it top to bottom in plain words:

Part of the fileWhat it means
on:When does this run? On pull requests that touch the cluster folder (preview only), on merges to main that touch it (build), and via a manual "Run" button. A path filter means unrelated edits don't needlessly run Terraform.
permissions:What can the run do? Exactly three things: request an OIDC token (required to log into AWS), read the repo, and write comments on pull requests.
concurrency:No collisions. Runs on the same branch are queued, and an in-progress build is never cut off halfway.
env:Settings. Pulls your non-secret values (region, state bucket, lock table) from the repository variables, so the file has no account details baked in.
environment:The approval gate. On main, ties the run to the production environment so it waits for your approval.

Then the steps, in order — this is the actual recipe:

1 Checkout # copy the repo onto the runner machine 2 Configure AWS (OIDC) # trade the OIDC token for temporary AWS credentials 3 Setup Terraform # install the exact pinned Terraform version 4 Format Check # fail if the code isn't tidily formatted 5 Init # connect to the S3 backend (bucket/table passed in here) 6 Validate # check the config makes internal sense 7 Plan # build the preview; save it to a file 8 Comment on PR # post that preview onto the pull request 9 Fail if plan errored # stop on a real error (but "has changes" is fine) 10 Apply # ONLY on main: apply the EXACT preview you reviewed
Why save the plan to a file and apply that exact file? It guarantees that what a human reviewed is precisely what gets built — nothing can sneak in between the preview and the build.

.github/workflows/lint.yml — safety & security checks

This second pipeline runs on every pull request that touches any Terraform, acting as guardrails before the main pipeline even runs:

  • TFLint — catches mistakes like deprecated machine types or broken references, and enforces a consistent style.
  • Trivy — scans your infrastructure code for security problems (like an accidentally wide-open firewall) and fails the pull request on serious findings. While you fix the first batch, you can set it to "advisory" so it warns without blocking.
💡 The big picture Two pipelines, working together: lint.yml says "is this code safe and clean?" and cluster.yml says "preview the change, let a human approve, then build it."
Section 10

How to make changes safely (day-to-day)

The golden rule: never run terraform apply from your laptop against the shared cluster. Let the pipeline do it. Here's the everyday routine:

1. Make a branch # git checkout -b change/bigger-node-pool 2. Edit a .tf file # then run: terraform fmt -recursive (tidies formatting) 3. Open a pull request # pipeline previews the change + posts it as a comment 4. READ the preview # confirm it's ONLY what you intended 5. Merge to main # pipeline builds it (waits for your approval if set up) 6. Verify # check the Actions log; kubectl get nodes

How to read a plan (this matters)

The preview uses symbols. Learn these four — they're the difference between a safe change and an outage:

SymbolMeansWorry level
+ createSomething new will be added.Usually safe
~ updateSomething existing will be changed in place.Usually safe
-/+ replaceSomething will be destroyed and rebuilt. Can cause downtime or data loss.⚠️ Read carefully
- destroySomething will be deleted.⚠️ Read carefully
🚨 Always pause on red If you see replace or destroy on anything important (the cluster, a database, storage), stop and make sure that's truly what you want before merging. Reverting code can't always undo deleted data.

Drift, upgrades, and rollbacks

  • Drift (someone changed things by hand in the AWS console): the next preview will show the difference. Either undo the manual change or write it into your files, then apply so reality and your files match again.
  • Upgrading Kubernetes: bump the version number by one step at a time (e.g. 1.33 → 1.34), open a pull request, review, merge. Skipping versions is not allowed.
  • Rolling back: revert the commit that caused the problem and merge — the pipeline applies the previous good state. (But remember: deleted data may not come back.)
Section 11 · The future folders

Adding node groups & multi-tenant projects

These are designed to be additive — you never edit the cluster/ folder to add capacity or teams. Here's how each future folder gets filled in.

📁 Adding more worker computers (node-groups/)

Say you need beefier machines for a heavy app, or GPU machines for AI work. You add a new node group in its own folder, which reads the cluster's published facts and attaches new workers to the existing cluster.

Step one: read the cluster's outputs using a remote state lookup (this is how separate folders share facts safely):

data "terraform_remote_state" "cluster" { backend = "s3" config = { bucket = "eks-platform-tfstate-123456789012" key = "cluster/terraform.tfstate" # reads the cluster's memory file region = "us-east-1" } }

Step two: describe your node groups as data, not copy-pasted code — a simple list you loop over. Adding a new pool becomes a few lines:

locals { node_groups = { general = { instance_types = ["m5.large"], min = 2, max = 6 } memory = { instance_types = ["r5.xlarge"], min = 0, max = 4 } gpu = { instance_types = ["g4dn.xlarge"], min = 0, max = 2 } # ← new pool = one line } }

Each pool attaches to the cluster using the looked-up facts (cluster name, private subnets, node security group). Then copy cluster.yml to a node-groups.yml pipeline (change the folder and the state key) so this folder gets its own preview/build pipeline.

📁 Adding multi-tenant teams (tenants/)

Multi-tenancy means many independent teams (or customers, or projects) safely sharing one cluster without stepping on each other. You build one reusable "tenant" recipe that, for each team, creates layered isolation:

LayerWhat it protects
NamespaceA labeled area of the cluster — the basic boundary between teams.
ResourceQuota + LimitRangeCaps how much CPU and memory a team can use, so one team can't starve the others.
NetworkPolicyA firewall between teams — by default, one team's apps can't reach another's.
RBAC Role/RoleBindingLimits each team's people to only their own area.
Dedicated node group (optional)Gives a team its own physical worker computers for hard isolation.
IRSA role per teamScoped AWS permissions for that team's apps, trusting the cluster's OIDC provider.

With that recipe (a Terraform module) built once, onboarding a new team is a single block:

module "tenant_acme" { source = "../modules/tenant" tenant_name = "acme" cpu_quota = "20" memory_quota = "40Gi" oidc_provider_arn = data.terraform_remote_state.cluster.outputs.oidc_provider_arn } module "tenant_globex" { # a second team = another small block source = "../modules/tenant" tenant_name = "globex" cpu_quota = "10" memory_quota = "20Gi" oidc_provider_arn = data.terraform_remote_state.cluster.outputs.oidc_provider_arn }
Strongest isolation tip For maximum safety, give each team its own memory file (a state key like tenants/acme/terraform.tfstate), so a mistake on one team can't affect another's state.
Section 12

Kubernetes versions & what this costs

The version clock (don't get caught off guard)

Kubernetes releases a new version roughly every few months, and EKS follows along. Each version has a support window you need to respect:

  • Standard support lasts about 14 months from a version's release on EKS.
  • After that, extended support runs another 12 months — but it costs significantly more (see the table).
  • If you never upgrade, EKS will eventually auto-upgrade you, which you don't want to be surprised by.

The lesson: upgrade one minor version at a time, on a schedule, while you're still in standard support.

⏱️ Keep version numbers current Kubernetes versions move fast. Before your first build, check the official page for the newest version in standard support and use that, rather than copying a number from any guide (including this one).

💸 Turn it off when experimenting An idle cluster still bills every hour. When you're just learning, run a teardown (or terraform destroy on the cluster folder) when you're done — but note the bootstrap bucket is deliberately protected from deletion. Always check the current EKS pricing page, since prices change.
Section 13

Troubleshooting common problems

Error you seeWhat's likely wrong & how to fix it
Error acquiring the state lockA previous run crashed while holding the lock. Make sure no run is active, then carefully run terraform force-unlock <LOCK_ID>.

Kubernetes was originally built at Google (based on a system they used internally called Borg) and is now open-source, governed by the Cloud Native Computing Foundation. It groups your computers into a "cluster" and presents them as one big pool of resources. You describe the desired state ("5 copies of my app, each needing this much memory"), and Kubernetes continuously works to make reality match that description — a model called "declarative" management.

Kubernetes — Overview

A container is your app plus everything it needs to run, packaged into one portable unit (commonly built with Docker). It runs the same on any machine. A pod is the smallest thing Kubernetes schedules — usually one container, sometimes a few that must live together. Kubernetes creates, moves, and destroys pods to keep your desired state true.

Kubernetes — Pods

EKS gives you a certified, conformant Kubernetes control plane that AWS operates across multiple Availability Zones for high availability, and patches for you. It integrates natively with AWS identity (IAM), networking (VPC), load balancers, and storage. You manage the worker nodes and your apps; AWS manages the hardest part.

EKS — What is Amazon EKS?

A managed node group is a set of EC2 instances whose lifecycle EKS handles: provisioning, updates, and graceful draining during changes. You set the instance type and the min / desired / max counts, and EKS keeps the group within those bounds. Since EKS 1.30, new groups default to the Amazon Linux 2023 operating system.

EKS — Managed node groups

Add-ons are the small but essential programs every cluster needs. EKS can install and update them for you as "managed add-ons": vpc-cni (gives pods network addresses), coredns (in-cluster DNS / phone book), kube-proxy (routes traffic to the right pods), and the aws-ebs-csi-driver (lets pods request persistent EBS disks).

EKS — Cluster add-ons

Normally, anything running on a node could use the node's AWS permissions — too broad. IRSA lets a specific Kubernetes "service account" (the identity attached to a pod) assume a specific, narrow IAM role instead. This is least-privilege done right, and it's the foundation for safe multi-tenancy and for add-ons like the EBS CSI driver. It works via an OIDC provider EKS creates for your cluster.

EKS — IRSA

Access entries are the modern, API-driven way to grant IAM users and roles permissions inside the cluster. They replaced the older aws-auth ConfigMap, which was easy to misedit and could lock you out. Our setup grants the pipeline (and any roles you list) cluster-admin via access entries.

EKS — Access entries

Terraform, by HashiCorp, is the most widely used Infrastructure-as-Code tool. You write configuration in a readable language called HCL, and Terraform's providers translate it into API calls against AWS (or hundreds of other platforms). Its key trick is comparing your desired configuration against recorded state to compute a minimal, safe set of changes.

Terraform — Introduction

A resource is the basic building block — one real object Terraform manages, like aws_s3_bucket or aws_eks_cluster. You declare its desired settings; Terraform creates, updates, or deletes it to match.

Terraform — Resources

A module is a reusable package of resources. Instead of hand-writing every subnet, route, and gateway for a network, you call the official VPC module and pass a few inputs. Modules encode best practices and save enormous amounts of error-prone boilerplate. This project uses the official AWS VPC and EKS modules.

Terraform — Modules The official EKS module

A provider is a plugin that teaches Terraform how to talk to a particular API. The aws provider knows AWS; the kubernetes provider knows the Kubernetes API. You pin provider versions so upgrades never surprise you.

Terraform — Providers

State is the file that records everything Terraform has built and maps your config to real resource IDs. It's the source of truth Terraform diffs against. It can contain secrets, so we store it in an encrypted, versioned, private S3 bucket — never in Git, never just on a laptop.

Terraform — State

terraform plan previews changes without making them — it shows exactly what will be created, changed, or destroyed. terraform apply carries out those changes. Our pipeline runs plan on pull requests (for review) and apply only after merge to main.

GitHub Actions is GitHub's built-in automation. You describe "workflows" in YAML files under .github/workflows/, and GitHub runs them on fresh cloud machines in response to events like pull requests and merges. It's where our plan/apply and lint/security checks run.

OpenID Connect (OIDC) lets GitHub prove its identity to AWS for a single job. AWS then hands back temporary credentials that expire in minutes. There's no long-lived AWS access key stored in GitHub to leak or rotate. The IAM role's trust policy restricts which repository (and optionally branch) may use it.

When Terraform runs, it writes a lock record to a small DynamoDB table. If a second run starts while the first holds the lock, it waits or fails instead of writing at the same time and corrupting state. This is why the bootstrap step creates a DynamoDB table alongside the S3 bucket.

Multi-tenancy means multiple teams or workloads safely sharing one cluster. AWS's best-practices guide covers the isolation layers — namespaces, quotas, network policies, RBAC, and node isolation — and the trade-offs between "soft" and "hard" isolation. Our tenant module bundles these per team.

EKS Best Practices — Multi-tenancy

A namespace is a named partition inside a cluster. It's the first boundary between teams: names don't collide across namespaces, and quotas, network policies, and RBAC all attach to a namespace.

Kubernetes — Namespaces

A ResourceQuota caps the total CPU, memory, and object counts a namespace may use. A LimitRange sets sensible per-pod defaults and ceilings. Together they ensure one team can't consume the whole cluster.

Kubernetes — Resource Quotas

A NetworkPolicy is a firewall for pod traffic. A default-deny policy means a team's pods can't be reached by other teams' pods unless explicitly allowed — crucial for isolation. (Your cluster needs a CNI that enforces them; the AWS VPC CNI supports network policies.)

Kubernetes — Network Policies

RBAC controls who can do what inside the cluster. A Role lists allowed actions within a namespace; a RoleBinding grants that Role to specific users or groups. This scopes each team's people to just their own namespace.

Kubernetes — RBAC
