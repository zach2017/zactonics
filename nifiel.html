ℹ️ What is NiFi Expression Language?
NiFi EL is a powerful scripting language used within Apache NiFi processor properties to dynamically reference and manipulate FlowFile attributes. It is embedded in ${ } syntax and enables routing decisions, transformations, and dynamic configuration without custom code.

Syntax Basics

Basic Attribute Reference

// Reference a FlowFile attribute ${product.name} ${stock.quantity} ${unit.price} ${shelf.aisle} ${expiry.date} // Call a function on an attribute ${product.name}:toUpperCase() ${unit.price}:multiply(1.08) // Nested function calls (chaining) ${product.name}:trim():toLower():replaceAll(' ','-')

Conditionals — ifElse & Decision Logic

Core Pattern: ${condition:ifElse('value-if-true','value-if-false')}
The condition expression must evaluate to true or false. Nested ifElse creates multi-branch logic.

🟡 Simple: Stock Status

// Label stock level as OK / LOW / CRITICAL ${quantity}:lt(5) :ifElse( 'CRITICAL', ${quantity}:lt(20) :ifElse('LOW','OK') ) // qty=3 → "CRITICAL" // qty=14 → "LOW" // qty=50 → "OK"

🟡 Category-based Routing

// Assign storage zone by category ${product.category}:equals('frozen') :ifElse( 'FREEZER_ZONE', ${product.category}:equals('dairy') :ifElse( 'REFRIGERATED_ZONE', 'DRY_GOODS_ZONE' ) )

🔴 Complex: Pricing Tier with Expiry Check

// Multi-condition pricing: expired → $0, near-expiry → 50% off, clearance → 25% off, normal // Inputs: days.to.expiry, unit.price, on.clearance ${days.to.expiry}:le(0) :ifElse( '0.00', ${days.to.expiry}:le(2) :ifElse( ${unit.price}:multiply(0.50):format('%.2f'), ${on.clearance}:equals('true') :ifElse( ${unit.price}:multiply(0.75):format('%.2f'), ${unit.price}:format('%.2f') ) ) )

Boolean Logic — and, or, not

AND — both must be true

// In stock AND not expired ${quantity}:gt(0) :and( ${days.to.expiry}:gt(0) )

OR — either is true

// Organic OR fair-trade ${label.organic}:equals('true') :or( ${label.fairtrade} :equals('true') )

NOT — invert boolean

// Not recalled ${product.recalled} :equals('true') :not()

🔐 Complex: Eligibility for Premium Display

// A product qualifies for premium display if: // - margin > 30% AND stock > 10 AND (organic OR premium brand) // - AND not on recall list AND not near-expiry ${margin.pct}:gt(30) :and(${quantity}:gt(10)) :and( ${label.organic}:equals('true') :or(${brand.tier}:equals('premium')) ) :and(${product.recalled}:equals('true'):not()) :and(${days.to.expiry}:gt(14))

Function Chaining

💡 Chaining Best Practice: Each : passes the result of the previous function as the subject of the next. Chains are evaluated left-to-right. Use line breaks for readability in complex chains.

📁 Dynamic File Naming

// Generate inventory report filename // "inventory_produce_20250814.csv" 'inventory_' ${product.category}:toLower():trim() '_' ${now()}:format('yyyyMMdd') '.csv' // In UpdateAttribute filename property: inventory_${product.category:toLower():trim()}_${now():format('yyyyMMdd')}.csv

🔄 Transform + Validate + Route

// Barcode: strip whitespace, validate 13 digits ${barcode} :trim() :replaceAll('[^0-9]','') :length() :equals(13) :ifElse('VALID_EAN13','INVALID_BARCODE')

Scenario: Inventory Control

SCENARIO 1 Reorder Alert Generation

Route FlowFiles to reorder queue when stock falls below reorder threshold, factoring in lead time demand.

❌ BAD — fragile, no null safety
${quantity}:lt(${reorder.point}) // Fails silently if reorder.point is missing // No consideration of lead time demand
✅ GOOD — null-safe, lead time aware
// Safety stock = lead.days * daily.sales ${quantity}:lt( ${reorder.point:isNull():ifElse(20,${reorder.point})} :plus( ${lead.days}:multiply(${daily.sales}) ) ) // Result: true when stock ≤ reorder + safety

SCENARIO 2 Receiving Dock Validation

Validate incoming delivery records: check quantity, temperature compliance, and supplier code format.

// Valid delivery: qty > 0, temp in range, supplier code 6 chars ${received.qty}:gt(0) :and( ${delivery.temp.celsius}:ge(1) :and(${delivery.temp.celsius}:le(5)) ) :and( ${supplier.code}:matches('[A-Z]{2}[0-9]{4}') ) :ifElse('ACCEPT','REJECT')

Bad vs Good Examples

1. Null Safety

❌ BAD — Will produce empty/broken result
${supplier.phone}:substring(0, 3) // If supplier.phone is missing → empty string → error
✅ GOOD — Null-safe with fallback
${supplier.phone:isNull():ifElse('UNKNOWN',${supplier.phone})} :substring(0,3) // Safe: returns "UNK" or area code

2. Division by Zero Guard

❌ BAD — Crashes on zero daily sales
${quantity}:divide(${daily.sales}) // daily.sales=0 → ArithmeticException
✅ GOOD — Guard against zero divisor
${daily.sales}:gt(0):ifElse( ${quantity}:divide(${daily.sales}):toWholeNumber(), 'N/A' ) // Returns days of stock or "N/A" when no sales data

3. Empty String vs Null

❌ BAD — isNull() won't catch empty strings
${product.notes}:isNull():ifElse('No notes', ${product.notes}) // product.notes="" → returns "" not "No notes"
✅ GOOD — Check both null AND empty
${product.notes}:isNull():or( ${product.notes}:isEmpty() ):ifElse('No notes',${product.notes}) // Handles null, "", and whitespace-only

4. Case-Insensitive Comparison

❌ BAD — Case-sensitive, brittle
${product.category}:equals('Dairy') // Misses "dairy", "DAIRY", "dAIRY"
✅ GOOD — Normalize before comparing
${product.category}:trim():toLower():equals('dairy') // Catches all variants reliably

5. Readability — Intermediate Attributes

❌ BAD — One-liner, impossible to debug
${unit.price:multiply(${order.quantity:ge(100):ifElse(0.80,${order.quantity:ge(50):ifElse(0.90,1.00)})}):multiply(${customer.loyalty:equals('true'):ifElse(0.95,1.00)}):format('%.2f')}
✅ GOOD — Use UpdateAttribute for intermediate steps
// Step 1 (UpdateAttribute): bulk_factor ${order.quantity}:ge(100):ifElse(0.80, ${order.quantity}:ge(50):ifElse(0.90,1.00) ) // Step 2 (UpdateAttribute): loyalty_factor ${customer.loyalty}:equals('true'):ifElse(0.95,1.00) // Step 3: final_price (readable) ${unit.price}:multiply(${bulk_factor}) :multiply(${loyalty_factor}) :format('%.2f')
✦

Common Patterns & Best Practices

✅ Always use :trim():toLower() before :equals()
Attribute values from POS systems, suppliers, and manual entry are inconsistently cased and padded.
✅ Use intermediate UpdateAttribute steps
Break complex logic into named intermediate attributes. Improves debuggability and reuse.
⚠️ isNull() ≠ isEmpty()
A missing attribute is null; an attribute set to "" is empty. Check both with :isNull():or(${attr:isEmpty()}).
⚠️ Guard all divisions
Always check the divisor is non-zero before dividing, especially with velocity and sales rate attributes.
❌ Never nest more than 3 ifElse levels
Beyond 3 levels, move to a RouteOnAttribute with separate expressions per route, or use a scripting processor.
ℹ️ Use format() before output
Always call :format('%.2f') on price values before writing to downstream systems to avoid floating-point string issues.
✅ Attribute naming conventions
Use dot-notation consistently: product.sku, shelf.aisle, delivery.temp.celsius. Avoid spaces or special chars in attribute names.
ℹ️ Test in NiFi Expression Language Tester
Use the built-in EL tester (right-click property → Test) to validate expressions with sample attribute values before deploying flows.

Syntax Deep Dive

The Expression Language (EL) is triggered by ${...} delimiters. Every expression has a subject (an attribute name or literal) and optionally one or more function calls chained with :.

Key rules:

  • Attribute names with spaces must be quoted: ${'product name'}
  • String literals use single quotes: 'dairy'
  • Numbers are unquoted: 3.99, 100
  • Booleans: true, false (unquoted)
  • Nested expressions: inner ${...} inside function args
  • allAttributes / anyAttribute: match multiple attrs by prefix
// Escaping attribute names with special chars ${'shelf.aisle.number'} ✓ ${shelf.aisle.number} ✓ (dots OK) ${'product name with spaces'} ✓ (needs quotes) // Using allAttributes for multi-SKU checks ${allAttributes('sku.')}:contains('organic') // True if ANY attribute prefixed 'sku.' contains 'organic'

String Functions — Full Reference

String functions operate on the string value of the subject attribute. Chaining is left-to-right: each result becomes the input to the next function.

// Full supplier name pipeline // Input: " FRESH FARMS LLC " → "fresh-farms-llc" ${supplier.name} :trim() :toLower() :replaceAll('[^a-z0-9 ]','') :trim() :replaceAll(' +','-') // Count how many dashes in a product code ${product.code}:replaceAll('[^-]',''):length() // Build S3 key path for inventory file inventory/ ${store.region}:toLower() / ${product.category}:toLower() / ${now()}:format('yyyy/MM/dd') /stock_update.json

Math Functions — Deep Dive

Math functions work on numeric string attributes. NiFi EL will attempt to coerce a string to a number. Use toNumber() explicitly when type safety matters.

// Gross margin calculation // margin = (selling_price - cost) / selling_price * 100 ${selling.price} :minus(${cost.price}) :divide(${selling.price}) :multiply(100) :format('%.1f') // Safe reorder calculation with min enforcement ${max.stock}:minus(${current.stock}) :max(0) // Never negative :min(${max.reorder}) // Cap at supplier max

Date & Time Functions

Date handling in NiFi EL converts between strings, Date objects, and epoch milliseconds. The workflow is: parse → manipulate → format.

// Common date format patterns (Java SimpleDateFormat) 'yyyy-MM-dd' ISO date: 2025-08-14 'MM/dd/yyyy' US format: 08/14/2025 'yyyyMMdd' Compact: 20250814 'yyyy-MM-dd HH:mm:ss' ISO datetime 'EEE, dd MMM yyyy' RFC-like: Wed, 14 Aug 2025 // Check if delivery was today ${delivery.date}:toDate('yyyy-MM-dd') :format('yyyyMMdd') :equals(${now():format('yyyyMMdd')}) // Age of stock in days ${now():toNumber()} :minus(${received.date}:toDate('yyyy-MM-dd'):toNumber()) :divide(86400000) :toWholeNumber()

Conditionals In Depth

ifElse is the primary conditional in NiFi EL. Both branches are evaluated (no short-circuit), so avoid side effects.

⚠️ Both branches of ifElse are always evaluated — if you reference a missing attribute in either branch, it will produce an error even when that branch is not "taken".
// ANTI-PATTERN: risky — null-unsafe branch evaluated regardless ${flag:equals('Y'):ifElse(${complex.attr:divide(${other})}, '0')} // SAFE: guard with isNull/isEmpty before using in ifElse branches ${flag}:equals('Y'):and(${other}:gt(0)) :ifElse( ${complex.attr}:divide(${other}):format('%.2f'), '0.00' )

Function Chaining Patterns

Chaining transforms the output of one function into the input of the next. The type of the result determines what functions can follow.

A chain that produces a Boolean can then call :ifElse(), :and(), or :or(). A chain producing a Number can call math functions. A chain producing a String can call string functions.
// Type-aware chaining example ${barcode} → String :trim() → String :length() → Number :equals(13) → Boolean :ifElse( → String (result) 'VALID_EAN13', 'BAD_BARCODE' ) // Build dynamic SQL WHERE clause fragment "category = '" ${product.category}:trim():toLower():replaceAll("'",'') "' AND qty > 0"

🔀 Routing Patterns — Extended

RouteOnAttribute evaluates each route expression independently. The FlowFile is sent to the first matching route (or ALL matching routes with "Route to All"). Unmatched FlowFiles go to unmatched.

💡 Order routes from most-specific to least-specific. Always include a catch-all route using ${true} to handle unexpected cases gracefully.
// catch_all route — always true, last priority ${true} // Multi-store routing by region prefix ${store.id}:startsWith('EAST-') // Route oversized files to S3 direct upload ${fileSize}:ge(10485760) // ≥ 10MB // Route JSON vs CSV inventory files ${filename}:endsWith('.json')

Complete Best Practices Guide

The 7 Golden Rules of NiFi EL:

  1. Null-safe everything — use isNull() or isEmpty() before any function that requires a value
  2. Guard all divisions — check divisor gt(0) before any divide()
  3. Normalize before comparing — always trim():toLower() before equals()
  4. Limit nesting depth — max 3 levels of ifElse; deeper logic belongs in a script
  5. Break into steps — use UpdateAttribute to create named intermediate attributes
  6. Format numeric output — always format('%.2f') on prices before downstream use
  7. Test with the EL Tester — right-click any property → Test Expression
