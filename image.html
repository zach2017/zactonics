What are we building?

Imagine you run a busy kitchen. Every cook needs the same clean, fully-stocked workstation before service begins. In AWS, that pre-stocked workstation is an AMI — a saved snapshot of a server with the operating system and software already installed. Rather than have every team set up Java and Python by hand (slow, and easy to get wrong), we bake one trusted "golden" image and share it.

This guide uses EC2 Image Builder, the AWS service that bakes machine images on a repeatable, automated pipeline. Here is the whole story, start to finish:

  1. A message on an SNS topic says: "a new base image is needed — go build."
  2. A small Lambda function catches that message and starts the pipeline.
  3. Image Builder pulls the latest Amazon Linux 2023 AMI as its starting point.
  4. It installs Java 21 (Amazon Corretto) and Python 3 on top.
  5. It runs a test phase that proves both actually work — if not, the build fails and nothing ships.
  6. On success, Image Builder publishes to a second SNS topic, which fans out to an SQS queue.
  7. Downstream app teams read that queue and update their applications to the new image.

Before you start

  1. An AWS account with permissions for EC2 Image Builder, IAM, SNS, SQS, and Lambda.
  2. The AWS CLI v2 installed and configured with aws configure.
  3. A chosen region — this guide uses us-east-1. Replace ACCOUNT_ID with your 12-digit account number throughout.
  4. Basic comfort copying commands into a terminal. No prior Image Builder experience required.
Tip Every CLI block below has a Copy button. The matching Terraform version near the end builds the same thing in one file, if you prefer Infrastructure as Code.

Step-by-step guide

1

Create the IAM role Image Builder uses

The temporary server that Image Builder spins up to bake your image needs a "badge" — an IAM role — to act on your behalf. AWS publishes two managed permission policies that cover everything needed, so we attach those rather than writing our own. We then wrap the role in an instance profile, which is how a role gets handed to an EC2 instance.

bash · aws cli
# 1. Create the role and allow EC2 to assume it
aws iam create-role --role-name ImageBuilderRole \
  --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"ec2.amazonaws.com"},"Action":"sts:AssumeRole"}]}'

# 2. Attach the two AWS-managed policies
aws iam attach-role-policy --role-name ImageBuilderRole \
  --policy-arn arn:aws:iam::aws:policy/EC2InstanceProfileForImageBuilder
aws iam attach-role-policy --role-name ImageBuilderRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore

# 3. Wrap it in an instance profile and attach the role
aws iam create-instance-profile --instance-profile-name ImageBuilderProfile
aws iam add-role-to-instance-profile --instance-profile-name ImageBuilderProfile \
  --role-name ImageBuilderRole
2

Create the SNS topics and SQS queue

We need two SNS topics: one that triggers a build, and one that announces a finished image. We also create an SQS queue and subscribe it to the "ready" topic, so downstream teams receive every announcement reliably even if they're momentarily offline.

bash · aws cli
# Topic that kicks off a build
aws sns create-topic --name new-base-image-trigger
# Topic that announces a finished image
aws sns create-topic --name image-ready-topic
# Durable queue for downstream app teams
aws sqs create-queue --queue-name image-update-queue

# Subscribe the queue to the "ready" topic (fan-out)
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:image-ready-topic \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:ACCOUNT_ID:image-update-queue
Don't skip: the queue access policy An SNS topic can only deliver into an SQS queue if the queue grants it permission. Attach a queue policy allowing the sns.amazonaws.com service the sqs:SendMessage action, scoped to your topic's ARN — otherwise messages silently vanish. The Terraform version below wires this for you automatically.
3

Write the BUILD component (install Java 21 + Python)

A component is a YAML recipe of steps Image Builder runs while baking. Components have phases: build runs during image creation, validate just after, and test against the finished image. Save this as build-component.yaml.

yaml · build-component.yaml
name: InstallJavaPython
description: Install Java 21 (Corretto) and Python 3
schemaVersion: 1.0
phases:
  - name: build
    steps:
      - name: InstallJava21
        action: ExecuteBash
        inputs:
          commands:
            - sudo dnf install -y java-21-amazon-corretto-devel
      - name: InstallPython
        action: ExecuteBash
        inputs:
          commands:
            - sudo dnf install -y python3 python3-pip
            - python3 -m pip install --upgrade pip
bash · register the component
aws imagebuilder create-component \
  --name InstallJavaPython --semantic-version 1.0.0 \
  --platform Linux --data file://build-component.yaml
4

Write the TEST component (prove it works)

Tests run in the test phase against the freshly built image. The trick: if any command exits with a non-zero code, the phase fails, the whole pipeline stops, and no broken image is ever published. Save as test-component.yaml.

yaml · test-component.yaml
name: VerifyJavaPython
description: Confirm Java 21 and Python are present and runnable
schemaVersion: 1.0
phases:
  - name: test
    steps:
      - name: CheckJava
        action: ExecuteBash
        inputs:
          commands:
            - 'java -version 2>&1 | grep "21" || (echo "Java 21 missing" && exit 1)'
      - name: CheckPython
        action: ExecuteBash
        inputs:
          commands:
            - 'python3 --version || (echo "Python missing" && exit 1)'
bash · register the component
aws imagebuilder create-component \
  --name VerifyJavaPython --semantic-version 1.0.0 \
  --platform Linux --data file://test-component.yaml
Why exit 1 matters In Linux, exit code 0 means success and anything else means failure. Forcing exit 1 when Java or Python is missing makes the test phase fail loudly, so Image Builder refuses to publish a faulty image. This is your quality gate.
5

Create the image recipe

A recipe combines a base ("parent") image with your components. Pointing the parent at the AWS-managed Amazon Linux 2023 image ARN with a wildcard version means every build automatically starts from the newest, fully-patched release — no manual updating.

bash · aws cli
aws imagebuilder create-image-recipe \
  --name base-java-python-recipe --semantic-version 1.0.0 \
  --parent-image "arn:aws:imagebuilder:us-east-1:aws:image/amazon-linux-2023-x86/x.x.x" \
  --components \
    'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/installjavapython/1.0.0' \
    'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/verifyjavapython/1.0.0'
6

Create infrastructure config + the pipeline

The infrastructure configuration says which IAM profile and instance type to build on — and, crucially, which SNS topic to notify when a build finishes. That single field is what produces the "image-ready" announcement for free, no extra code needed. The pipeline then binds the recipe and the infra config together.

bash · aws cli
aws imagebuilder create-infrastructure-configuration \
  --name base-infra \
  --instance-profile-name ImageBuilderProfile \
  --instance-types t3.medium \
  --sns-topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:image-ready-topic

aws imagebuilder create-image-pipeline \
  --name base-java-python-pipeline \
  --image-recipe-arn arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:image-recipe/base-java-python-recipe/1.0.0 \
  --infrastructure-configuration-arn arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:infrastructure-configuration/base-infra
7

Wire the trigger topic to a Lambda

When the new-base-image-trigger topic receives a message, a tiny Lambda starts the pipeline. Separating "when to build" (the trigger) from "how to build" (the pipeline) keeps the system flexible — you can later swap in an EventBridge schedule without touching the pipeline.

python · lambda handler (index.py)
import boto3

def handler(event, context):
    ib = boto3.client('imagebuilder')
    ib.start_image_pipeline_execution(
        imagePipelineArn='arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:image-pipeline/base-java-python-pipeline'
    )
    return {"status": "build started"}
bash · subscribe Lambda to the trigger topic
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:new-base-image-trigger \
  --protocol lambda \
  --notification-endpoint arn:aws:lambda:us-east-1:ACCOUNT_ID:function:StartImageBuild
8

Downstream teams consume the queue

When a build succeeds, Image Builder posts a JSON message (including the new AMI ID and status) to image-ready-topic, which fans out to the SQS queue. App teams poll the queue, read the AMI ID, and roll it into their launch templates or Auto Scaling groups.

bash · poll the queue
aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/ACCOUNT_ID/image-update-queue \
  --max-number-of-messages 1 \
  --wait-time-seconds 10
9

Test the whole pipeline end-to-end

Publish one message to the trigger topic and watch the chain react.

bash · fire the trigger
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:new-base-image-trigger \
  --message "New Amazon Linux available, please build"

In the EC2 Image Builder console, watch the pipeline move through Build → Test → Distribute. If the test phase fails, no image is created and no "ready" message is sent — exactly the safety behaviour you want.

Bonus: build it all with Terraform

The CLI commands are great for learning, but typing them one by one invites mistakes. Terraform lets you describe the entire setup — role, topics, queue, components, recipe, pipeline — in one file, then create or destroy it with a single command. It also wires the ARNs together automatically, so there is no ACCOUNT_ID to paste anywhere.

Three commands to run it terraform init downloads the AWS plugin · terraform plan previews changes · terraform apply builds it. Run terraform destroy to remove everything.
hcl · main.tf
provider "aws" { region = "us-east-1" }

# 1. Always start from the latest Amazon Linux 2023 AMI
data "aws_ami" "al2023" {
  most_recent = true
  owners      = ["amazon"]
  filter { name = "name"  values = ["al2023-ami-2023.*-x86_64"] }
}

# 2. IAM role + instance profile
resource "aws_iam_role" "ib" {
  name = "ImageBuilderRole"
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{ Effect = "Allow", Action = "sts:AssumeRole",
      Principal = { Service = "ec2.amazonaws.com" } }]
  })
}
resource "aws_iam_role_policy_attachment" "core" {
  role = aws_iam_role.ib.name
  policy_arn = "arn:aws:iam::aws:policy/EC2InstanceProfileForImageBuilder"
}
resource "aws_iam_role_policy_attachment" "ssm" {
  role = aws_iam_role.ib.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore"
}
resource "aws_iam_instance_profile" "ib" {
  name = "ImageBuilderProfile"
  role = aws_iam_role.ib.name
}

# 3. SNS topics + SQS queue (with policy so SNS can deliver)
resource "aws_sns_topic" "trigger" { name = "new-base-image-trigger" }
resource "aws_sns_topic" "ready"   { name = "image-ready-topic" }
resource "aws_sqs_queue"  "updates" { name = "image-update-queue" }

resource "aws_sqs_queue_policy" "allow_sns" {
  queue_url = aws_sqs_queue.updates.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{ Effect = "Allow", Principal = { Service = "sns.amazonaws.com" },
      Action = "sqs:SendMessage", Resource = aws_sqs_queue.updates.arn,
      Condition = { ArnEquals = { "aws:SourceArn" = aws_sns_topic.ready.arn } } }]
  })
}
resource "aws_sns_topic_subscription" "queue_sub" {
  topic_arn = aws_sns_topic.ready.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.updates.arn
}

# 4. Build + 5. Test components (inline YAML via heredoc)
resource "aws_imagebuilder_component" "install" {
  name = "InstallJavaPython"  platform = "Linux"  version = "1.0.0"
  data = <<-EOT
    name: InstallJavaPython
    schemaVersion: 1.0
    phases:
      - name: build
        steps:
          - name: Java21
            action: ExecuteBash
            inputs: { commands: ["sudo dnf install -y java-21-amazon-corretto-devel"] }
          - name: Python
            action: ExecuteBash
            inputs: { commands: ["sudo dnf install -y python3 python3-pip"] }
  EOT
}
resource "aws_imagebuilder_component" "verify" {
  name = "VerifyJavaPython"  platform = "Linux"  version = "1.0.0"
  data = <<-EOT
    name: VerifyJavaPython
    schemaVersion: 1.0
    phases:
      - name: test
        steps:
          - name: CheckJava
            action: ExecuteBash
            inputs: { commands: ['java -version 2>&1 | grep "21" || (echo missing && exit 1)'] }
          - name: CheckPython
            action: ExecuteBash
            inputs: { commands: ['python3 --version || (echo missing && exit 1)'] }
  EOT
}

# 6. Recipe → 7. Infra config (with SNS) → 8. Pipeline
resource "aws_imagebuilder_image_recipe" "recipe" {
  name = "base-java-python-recipe"  version = "1.0.0"
  parent_image = data.aws_ami.al2023.id
  component { component_arn = aws_imagebuilder_component.install.arn }
  component { component_arn = aws_imagebuilder_component.verify.arn }
}
resource "aws_imagebuilder_infrastructure_configuration" "infra" {
  name = "base-infra"
  instance_profile_name = aws_iam_instance_profile.ib.name
  instance_types        = ["t3.medium"]
  sns_topic_arn         = aws_sns_topic.ready.arn
}
resource "aws_imagebuilder_image_pipeline" "pipeline" {
  name = "base-java-python-pipeline"
  image_recipe_arn                 = aws_imagebuilder_image_recipe.recipe.arn
  infrastructure_configuration_arn = aws_imagebuilder_infrastructure_configuration.infra.arn
}

output "pipeline_arn" { value = aws_imagebuilder_image_pipeline.pipeline.arn }
output "queue_url"    { value = aws_sqs_queue.updates.id }

The Lambda trigger from Step 7 fits the same way — add a aws_lambda_function, an aws_sns_topic_subscription with protocol = "lambda", and an aws_lambda_permission so SNS may invoke it.

You've built an image factory.

A single trigger message now bakes a fresh Amazon Linux image with Java 21 and Python, tests it, and announces it to every downstream team — repeatably, and with a quality gate that blocks broken builds. Re-run anytime by publishing to the trigger topic, or attach an EventBridge schedule to run it weekly.
