AWS Image Builder Pipeline

Build a base Amazon Linux AMI with Java 21 + Python, test it, and notify other teams.

Font size 16px

What are we building? (Read this first)

Imagine you run a kitchen. Every cook needs the same clean, pre-stocked workstation before they start cooking. In AWS, that "pre-stocked workstation" is an AMI (Amazon Machine Image) — a saved snapshot of a server with software already installed. Instead of every team setting up Java and Python by hand, we build one good base image and share it.

This tutorial uses EC2 Image Builder, an AWS service that automatically bakes images for you. Here is the full story of what happens:

  1. 1. A message (SNS trigger) says: "A new Amazon Linux image is available — go build!"
  2. 2. Image Builder takes the latest Amazon Linux image as a starting point.
  3. 3. It installs Java 21 and Python on top.
  4. 4. It runs tests to confirm both actually work.
  5. 5. When tests pass, it sends an SNS + SQS message so other teams know a fresh image is ready for their apps.

The flow at a glance

SNS "new base image" topic ──▶ Lambda ──▶ EC2 Image Builder Pipeline │ (latest Amazon Linux AMI) ──────┤ 1. Build → install Java 21 + Python │ 2. Test → verify versions ▼ New hardened AMI │ ▼ SNS topic "image-ready" ──▶ SQS queue │ ▼ App teams poll queue & update their stacks

Before you start (Prerequisites)

  • An AWS account with permissions for EC2 Image Builder, IAM, SNS, SQS, and Lambda.
  • The AWS CLI installed and configured (aws configure).
  • A region picked (this guide uses us-east-1).
  • Basic comfort copying commands into a terminal.

Step-by-step guide

1

Create the IAM role Image Builder will use

Image Builder needs a "badge" (IAM role) to act on your behalf. AWS gives ready-made permission policies, so we just attach them.

aws iam create-role --role-name ImageBuilderRole \ --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"ec2.amazonaws.com"},"Action":"sts:AssumeRole"}]}' aws iam attach-role-policy --role-name ImageBuilderRole \ --policy-arn arn:aws:iam::aws:policy/EC2InstanceProfileForImageBuilder aws iam attach-role-policy --role-name ImageBuilderRole \ --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore aws iam create-instance-profile --instance-profile-name ImageBuilderProfile aws iam add-role-to-instance-profile --instance-profile-name ImageBuilderProfile \ --role-name ImageBuilderRole
2

Create the notification topics (SNS) and queue (SQS)

We make two SNS topics: one that triggers the build, and one that announces the finished image. We also create an SQS queue subscribed to the "ready" topic so app teams can pull messages reliably.

# Topic that kicks off a build aws sns create-topic --name new-base-image-trigger # Topic that announces a finished image aws sns create-topic --name image-ready-topic # A queue for downstream app teams aws sqs create-queue --queue-name image-update-queue # Subscribe the queue to the "ready" topic aws sns subscribe \ --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:image-ready-topic \ --protocol sqs \ --notification-endpoint arn:aws:sqs:us-east-1:ACCOUNT_ID:image-update-queue

Replace ACCOUNT_ID with your 12-digit AWS account number.

3

Write the BUILD component (install Java 21 + Python)

A component is a recipe written in YAML telling Image Builder what to install. Save this as build-component.yaml.

name: InstallJavaPython description: Install Java 21 and Python on Amazon Linux schemaVersion: 1.0 phases: - name: build steps: - name: InstallJava21 action: ExecuteBash inputs: commands: - sudo dnf install -y java-21-amazon-corretto-devel - name: InstallPython action: ExecuteBash inputs: commands: - sudo dnf install -y python3 python3-pip - python3 -m pip install --upgrade pip
aws imagebuilder create-component \ --name InstallJavaPython --semantic-version 1.0.0 \ --platform Linux --data file://build-component.yaml
4

Write the TEST component (verify it works)

Tests run in the special validate/test phase. If any command exits with an error, the whole build fails and no broken image is published. Save as test-component.yaml.

name: VerifyJavaPython description: Confirm Java 21 and Python are installed schemaVersion: 1.0 phases: - name: test steps: - name: CheckJava action: ExecuteBash inputs: commands: - 'java -version 2>&1 | grep "21" || (echo "Java 21 missing" && exit 1)' - name: CheckPython action: ExecuteBash inputs: commands: - 'python3 --version || (echo "Python missing" && exit 1)'
aws imagebuilder create-component \ --name VerifyJavaPython --semantic-version 1.0.0 \ --platform Linux --data file://test-component.yaml
5

Create the recipe (latest Amazon Linux + your components)

A recipe = base image + components. Using the SSM-managed "latest" parameter means each build automatically starts from the newest Amazon Linux, no manual updates needed.

aws imagebuilder create-image-recipe \ --name base-java-python-recipe --semantic-version 1.0.0 \ --parent-image "arn:aws:imagebuilder:us-east-1:aws:image/amazon-linux-2023-x86/x.x.x" \ --components \ 'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/installjavapython/1.0.0' \ 'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/verifyjavapython/1.0.0'
6

Create infrastructure config + the pipeline (with SNS notification)

The infrastructure config tells Image Builder which IAM role and instance type to use — and which SNS topic to ping when an image finishes. This is how the "image-ready" announcement happens automatically.

aws imagebuilder create-infrastructure-configuration \ --name base-infra \ --instance-profile-name ImageBuilderProfile \ --instance-types t3.medium \ --sns-topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:image-ready-topic aws imagebuilder create-image-pipeline \ --name base-java-python-pipeline \ --image-recipe-arn arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:image-recipe/base-java-python-recipe/1.0.0 \ --infrastructure-configuration-arn arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:infrastructure-configuration/base-infra
7

Connect the TRIGGER topic to a Lambda that starts the build

When the "new-base-image-trigger" topic gets a message, a tiny Lambda function starts the pipeline. Here is the Python handler:

import boto3 def handler(event, context): ib = boto3.client('imagebuilder') ib.start_image_pipeline_execution( imagePipelineArn='arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:image-pipeline/base-java-python-pipeline' ) return {"status": "build started"}

Then subscribe the Lambda to the trigger topic:

aws sns subscribe \ --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:new-base-image-trigger \ --protocol lambda \ --notification-endpoint arn:aws:lambda:us-east-1:ACCOUNT_ID:function:StartImageBuild
8

Downstream teams read the SQS queue

When a build succeeds, Image Builder posts to "image-ready-topic" → which fans out to your SQS queue. App teams just poll the queue and grab the new AMI ID to update their launch templates.

aws sqs receive-message \ --queue-url https://sqs.us-east-1.amazonaws.com/ACCOUNT_ID/image-update-queue \ --max-number-of-messages 1
9

Test the whole pipeline end-to-end

Publish a message to the trigger topic and watch the chain react.

aws sns publish \ --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:new-base-image-trigger \ --message "New Amazon Linux available, please build"

In the EC2 Image Builder console, watch the pipeline move through Build → Test → Distribute. If the test phase fails, no image is created and no "ready" message is sent — exactly what you want.

Bonus: build it all with Terraform

The CLI commands above are great for learning, but typing them one by one is easy to get wrong. Terraform is an "Infrastructure as Code" tool: you describe everything you want in one text file, and Terraform creates (or deletes) it all for you with a single command. Think of it as a recipe card for your entire AWS setup — not just the image, but the topics, queue, role, and pipeline too.

Three commands to run it:

terraform init  →  terraform plan  →  terraform apply

init downloads the AWS plugin, plan shows what will be created, and apply actually builds it. Run terraform destroy to remove everything.

Save this as main.tf:

provider "aws" { region = "us-east-1" } # --- 1. Latest Amazon Linux 2023 as the base image --- data "aws_ami" "al2023" { most_recent = true owners = ["amazon"] filter { name = "name" values = ["al2023-ami-2023.*-x86_64"] } } # --- 2. IAM role + instance profile for Image Builder --- resource "aws_iam_role" "ib" { name = "ImageBuilderRole" assume_role_policy = jsonencode({ Version = "2012-10-17" Statement = [{ Effect = "Allow" Principal = { Service = "ec2.amazonaws.com" } Action = "sts:AssumeRole" }] }) } resource "aws_iam_role_policy_attachment" "ib_core" { role = aws_iam_role.ib.name policy_arn = "arn:aws:iam::aws:policy/EC2InstanceProfileForImageBuilder" } resource "aws_iam_role_policy_attachment" "ssm" { role = aws_iam_role.ib.name policy_arn = "arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore" } resource "aws_iam_instance_profile" "ib" { name = "ImageBuilderProfile" role = aws_iam_role.ib.name } # --- 3. SNS topics + SQS queue --- resource "aws_sns_topic" "trigger" { name = "new-base-image-trigger" } resource "aws_sns_topic" "ready" { name = "image-ready-topic" } resource "aws_sqs_queue" "updates" { name = "image-update-queue" } # Let the "ready" topic deliver into the queue resource "aws_sqs_queue_policy" "allow_sns" { queue_url = aws_sqs_queue.updates.id policy = jsonencode({ Version = "2012-10-17" Statement = [{ Effect = "Allow" Principal = { Service = "sns.amazonaws.com" } Action = "sqs:SendMessage" Resource = aws_sqs_queue.updates.arn Condition = { ArnEquals = { "aws:SourceArn" = aws_sns_topic.ready.arn } } }] }) } resource "aws_sns_topic_subscription" "queue_sub" { topic_arn = aws_sns_topic.ready.arn protocol = "sqs" endpoint = aws_sqs_queue.updates.arn } # --- 4. Build component: install Java 21 + Python --- resource "aws_imagebuilder_component" "install" { name = "InstallJavaPython" platform = "Linux" version = "1.0.0" data = <<-EOT name: InstallJavaPython schemaVersion: 1.0 phases: - name: build steps: - name: Java21 action: ExecuteBash inputs: commands: - sudo dnf install -y java-21-amazon-corretto-devel - name: Python action: ExecuteBash inputs: commands: - sudo dnf install -y python3 python3-pip EOT } # --- 5. Test component: verify versions --- resource "aws_imagebuilder_component" "verify" { name = "VerifyJavaPython" platform = "Linux" version = "1.0.0" data = <<-EOT name: VerifyJavaPython schemaVersion: 1.0 phases: - name: test steps: - name: CheckJava action: ExecuteBash inputs: commands: - 'java -version 2>&1 | grep "21" || (echo missing && exit 1)' - name: CheckPython action: ExecuteBash inputs: commands: - 'python3 --version || (echo missing && exit 1)' EOT } # --- 6. Recipe = base image + components --- resource "aws_imagebuilder_image_recipe" "recipe" { name = "base-java-python-recipe" parent_image = data.aws_ami.al2023.id version = "1.0.0" component { component_arn = aws_imagebuilder_component.install.arn } component { component_arn = aws_imagebuilder_component.verify.arn } } # --- 7. Infra config: role + the SNS topic to notify when done --- resource "aws_imagebuilder_infrastructure_configuration" "infra" { name = "base-infra" instance_profile_name = aws_iam_instance_profile.ib.name instance_types = ["t3.medium"] sns_topic_arn = aws_sns_topic.ready.arn } # --- 8. The pipeline itself --- resource "aws_imagebuilder_image_pipeline" "pipeline" { name = "base-java-python-pipeline" image_recipe_arn = aws_imagebuilder_image_recipe.recipe.arn infrastructure_configuration_arn = aws_imagebuilder_infrastructure_configuration.infra.arn } output "pipeline_arn" { value = aws_imagebuilder_image_pipeline.pipeline.arn } output "queue_url" { value = aws_sqs_queue.updates.id }

Notice what Terraform did for you:

No ACCOUNT_ID to copy-paste anywhere — Terraform wires the ARNs together automatically (e.g. aws_sns_topic.ready.arn). It also figured out the order: the role before the profile, the topics before the subscription. Change one line, re-run apply, and only what changed gets updated.

The Lambda trigger from Step 7 still fits the same way — add this to the file to create the function and subscribe it to the trigger topic:

resource "aws_lambda_function" "starter" { function_name = "StartImageBuild" runtime = "python3.12" handler = "index.handler" role = aws_iam_role.lambda.arn filename = "lambda.zip" # zip containing index.py from Step 7 } resource "aws_sns_topic_subscription" "lambda_sub" { topic_arn = aws_sns_topic.trigger.arn protocol = "lambda" endpoint = aws_lambda_function.starter.arn } resource "aws_lambda_permission" "allow_sns" { statement_id = "AllowSNS" action = "lambda:InvokeFunction" function_name = aws_lambda_function.starter.function_name principal = "sns.amazonaws.com" source_arn = aws_sns_topic.trigger.arn }

You're done! 🎉

You now have a fully automated factory: a trigger message builds a fresh Amazon Linux image with Java 21 and Python, tests it, and tells everyone else when it's ready. Re-run anytime by publishing to the trigger topic.
