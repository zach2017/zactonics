What are we building?

Imagine you run a busy kitchen. Every cook needs the same clean, fully-stocked workstation before service begins. In AWS, that pre-stocked workstation is an AMI — a saved snapshot of a server with the operating system and software already installed. Rather than have every team set up Java and Python by hand (slow, and easy to get wrong), we bake one trusted "golden" image and share it. We'll also make this image Kafka-ready — the right system libraries so any future application can talk to a Kafka cluster (Amazon MSK or self-hosted) without extra setup.

This guide uses EC2 Image Builder, the AWS service that bakes machine images on a repeatable, automated pipeline. Here is the whole story, start to finish:

  1. A message on an SNS topic says: "a new base image is needed — go build."
  2. A small Lambda function catches that message and starts the pipeline.
  3. Image Builder pulls the latest Amazon Linux 2023 AMI as its starting point.
  4. It installs Java 21 (Amazon Corretto), Python 3, and the system libraries needed for Kafka clients (librdkafka, openssl, cyrus-sasl, zlib).
  5. It runs a test phase that proves Java, Python, and the Kafka libraries actually work — if not, the build fails and nothing ships.
  6. On success, Image Builder publishes to a second SNS topic, which fans out to an SQS queue.
  7. Downstream app teams read that queue and update their applications to the new image.

Setting up your AWS user (least-privilege)

Before clicking anything, give yourself an IAM user with only the permissions this tutorial actually needs — no more. "Least privilege" is the AWS security principle that an identity should hold the smallest set of permissions that lets it do its job. If credentials ever leak, the blast radius stays small.

There are three different identities at play here, and it's worth knowing which is which before you start clicking:

  1. You, the human builder — the IAM user that runs the AWS CLI or clicks through the Console to create the pipeline. Needs setup permissions.
  2. The Image Builder build instance — the temporary EC2 server that bakes the image. Needs runtime permissions via an instance profile (the role from Step 1).
  3. The Lambda trigger — the function that starts the pipeline. Needs permission to call imagebuilder:StartImagePipelineExecution and write logs.

Step A · Create the human builder user (AWS Console)

The recommended modern path is IAM Identity Center (formerly AWS SSO), but a plain IAM user works fine for a single-account learning setup. Both routes appear below.

  1. Sign in to the AWS Console as a root or admin user, then go to IAM → Users → Create user.
  2. Name the user (e.g. image-pipeline-builder) and tick Provide user access to the AWS Management Console if you want browser access. Pick I want to create an IAM user.
  3. On the permissions page, choose Attach policies directly, then click Create policy (we'll attach our own minimal one — not AdministratorAccess).
  4. Paste the JSON policy below, name it ImageBuilderTutorialMinimal, save, and attach it to the user.
  5. Open the user and create an Access key for Command Line Interface (CLI). Copy the key and secret somewhere safe — they appear once.
  6. On your machine, run aws configure and paste the key, secret, default region (us-east-1), and output format (json).

Nothing here grants account-wide admin.

Step B · The Image Builder service role (runtime)

This is the role the build EC2 instance assumes while baking — the one created in Step 1 of the main guide. The two AWS-managed policies attached to it are the official minimum:

  1. EC2InstanceProfileForImageBuilder — permission for the build instance to talk to the Image Builder service, write artifacts to its working S3 buckets, and publish logs.
  2. AmazonSSMManagedInstanceCore — permission for AWS Systems Manager to send build commands to the instance and stream the output back. This is how Image Builder controls the box without SSH.

You don't add anything extra for installing Java/Python/Kafka libraries — those installs only need internet access from the build subnet to pull packages from dnf repositories.

Step C · The Lambda trigger role

The function from Step 7 needs its own tiny role. Create it with this trust policy and inline permissions:

Where things live in the AWS Console

Prefer clicking to typing? Every CLI step in this guide has an equivalent path in the AWS Management Console. Here's the menu map. The Console URL for each service is also listed so you can bookmark them.

EC2 Image Builder

From the main Console, the search bar is the fastest way in. Type Image Builder and select EC2 Image Builder. The left-hand navigation pane contains every resource type you'll create:

  1. Image pipelines — list view and "Create image pipeline" wizard. This is the end-to-end flow: it lets you create components, recipe, infrastructure config, and the pipeline in one guided wizard, if you prefer the all-in-one approach over building each piece separately.
  2. Images — every AMI Image Builder has ever produced for your account, with build status and logs.
  3. Components — "Create component" lets you paste in the YAML from Steps 3 and 4. Phase = build for the install component, test for the verify component.
  4. Image recipes — "Create image recipe" lets you pick the parent image (search "Amazon Linux 2023") and add your components in order.
  5. Infrastructure configurations — pick the IAM role, instance type, and the SNS topic for notifications.
  6. Distribution settings — optional; only needed when you copy AMIs to other regions or share with other accounts.

Direct URL: https://console.aws.amazon.com/imagebuilder/home

IAM (roles, users, policies)

  1. IAM → Users — create the human builder user.
  2. IAM → Roles — create ImageBuilderRole (trusted entity = EC2) and LambdaImageBuilderTrigger (trusted entity = Lambda).
  3. IAM → Policies — paste the least-privilege JSON above and save as a customer-managed policy.
  4. IAM → Roles → [your role] → Add permissions → Attach policies — attach the two managed policies (EC2InstanceProfileForImageBuilder, AmazonSSMManagedInstanceCore).

Direct URL: https://console.aws.amazon.com/iam/home

SNS & SQS

  1. Amazon SNS → Topics → Create topic — type Standard. Create two: new-base-image-trigger and image-ready-topic.
  2. Amazon SQS → Queues → Create queue — type Standard. Name it image-update-queue.
  3. On the queue's detail page, click SNS subscriptions → Subscribe to Amazon SNS topic and pick image-ready-topic. SQS will automatically offer to write the queue access policy for you — accept it.

Direct URLs: SNS Console · SQS Console

Lambda

  1. Lambda → Functions → Create function. Choose Author from scratch, runtime Python 3.12, and use the LambdaImageBuilderTrigger role.
  2. Paste the handler code from Step 7 into the inline editor and click Deploy.
  3. Click Add trigger on the function overview, choose SNS, and pick new-base-image-trigger.

Direct URL: https://console.aws.amazon.com/lambda/home

Watching a build run

After firing the trigger (Step 9), open EC2 Image Builder → Images. Click the in-progress image to see its phases tick through. For deeper logs:

  1. CloudWatch → Log groups → /aws/imagebuilder/<your-image-name> shows every command the build instance ran.
  2. EC2 → Instances shows the temporary build instance (it terminates automatically once the build completes).
  3. SQS → image-update-queue → Send and receive messages → Poll for messages is the easiest way to see the "image-ready" announcement land.

Before you start

  1. An AWS account with permissions for EC2 Image Builder, IAM, SNS, SQS, and Lambda.
  2. The AWS CLI v2 installed and configured with aws configure.
  3. A chosen region — this guide uses us-east-1. Replace ACCOUNT_ID with your 12-digit account number throughout.
  4. Basic comfort copying commands into a terminal. No prior Image Builder experience required.

Step-by-step guide

1

Create the IAM role Image Builder uses

The temporary server that Image Builder spins up to bake your image needs a "badge" — an IAM role — to act on your behalf. AWS publishes two managed permission policies that cover everything needed, so we attach those rather than writing our own. We then wrap the role in an instance profile, which is how a role gets handed to an EC2 instance.

bash · aws cli
# 1. Create the role and allow EC2 to assume it
aws iam create-role --role-name ImageBuilderRole \
  --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"ec2.amazonaws.com"},"Action":"sts:AssumeRole"}]}'

# 2. Attach the two AWS-managed policies
aws iam attach-role-policy --role-name ImageBuilderRole \
  --policy-arn arn:aws:iam::aws:policy/EC2InstanceProfileForImageBuilder
aws iam attach-role-policy --role-name ImageBuilderRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore

# 3. Wrap it in an instance profile and attach the role
aws iam create-instance-profile --instance-profile-name ImageBuilderProfile
aws iam add-role-to-instance-profile --instance-profile-name ImageBuilderProfile \
  --role-name ImageBuilderRole
2

Create the SNS topics and SQS queue

We need two SNS topics: one that triggers a build, and one that announces a finished image. We also create an SQS queue and subscribe it to the "ready" topic, so downstream teams receive every announcement reliably even if they're momentarily offline.

bash · aws cli
# Topic that kicks off a build
aws sns create-topic --name new-base-image-trigger
# Topic that announces a finished image
aws sns create-topic --name image-ready-topic
# Durable queue for downstream app teams
aws sqs create-queue --queue-name image-update-queue

# Subscribe the queue to the "ready" topic (fan-out)
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:image-ready-topic \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:ACCOUNT_ID:image-update-queue
Don't skip: the queue access policy An SNS topic can only deliver into an SQS queue if the queue grants it permission. Attach a queue policy allowing the sns.amazonaws.com service the sqs:SendMessage action, scoped to your topic's ARN — otherwise messages silently vanish. The Terraform version below wires this for you automatically.
3

Write the BUILD component (install Java 21 + Python)

A component is a YAML recipe of steps Image Builder runs while baking. Components have phases: build runs during image creation, validate just after, and test against the finished image. Save this as build-component.yaml.

Why these extra packages for Kafka readiness Future applications on this AMI will need to connect to Kafka (Amazon MSK or self-managed). Java Kafka clients use SSL/SASL, which needs the JDK's truststore (already in Corretto). Python Kafka clients like confluent-kafka link against librdkafka, which in turn needs openssl, cyrus-sasl, zlib, and a compiler. Installing these system libraries now means each app team can pip install confluent-kafka later without any compile errors.
yaml · build-component.yaml
name: InstallJavaPythonKafka
description: Install Java 21 (Corretto), Python 3, and Kafka client system libs
schemaVersion: 1.0
phases:
  - name: build
    steps:
      - name: SystemUpdate
        action: ExecuteBash
        inputs:
          commands:
            - sudo dnf update -y
      - name: InstallJava21
        action: ExecuteBash
        inputs:
          commands:
            - sudo dnf install -y java-21-amazon-corretto-devel
      - name: InstallPython
        action: ExecuteBash
        inputs:
          commands:
            - sudo dnf install -y python3 python3-pip python3-devel
            - python3 -m pip install --upgrade pip
      - name: KafkaClientLibs
        action: ExecuteBash
        inputs:
          commands:
            # System libs Kafka clients (Java & Python) need
            - sudo dnf install -y gcc openssl openssl-devel cyrus-sasl cyrus-sasl-devel zlib zlib-devel
            - sudo dnf install -y librdkafka librdkafka-devel
            # Pre-install Python Kafka clients so apps don't compile at runtime
            - sudo python3 -m pip install confluent-kafka kafka-python
bash · register the component
aws imagebuilder create-component \
  --name InstallJavaPythonKafka --semantic-version 1.0.0 \
  --platform Linux --data file://build-component.yaml
4

Write the TEST component (prove it works)

Tests run in the test phase against the freshly built image. The trick: if any command exits with a non-zero code, the phase fails, the whole pipeline stops, and no broken image is ever published. Save as test-component.yaml.

yaml · test-component.yaml
name: VerifyJavaPythonKafka
description: Confirm Java 21, Python, and Kafka client libs are present
schemaVersion: 1.0
phases:
  - name: test
    steps:
      - name: CheckJava
        action: ExecuteBash
        inputs:
          commands:
            - 'java -version 2>&1 | grep "21" || (echo "Java 21 missing" && exit 1)'
      - name: CheckPython
        action: ExecuteBash
        inputs:
          commands:
            - 'python3 --version || (echo "Python missing" && exit 1)'
      - name: CheckKafkaLibs
        action: ExecuteBash
        inputs:
          commands:
            - 'rpm -q librdkafka || (echo "librdkafka missing" && exit 1)'
            - 'python3 -c "import confluent_kafka; print(confluent_kafka.version())" || (echo "confluent-kafka import failed" && exit 1)'
            - 'python3 -c "import kafka; print(kafka.__version__)" || (echo "kafka-python import failed" && exit 1)'
bash · register the component
aws imagebuilder create-component \
  --name VerifyJavaPythonKafka --semantic-version 1.0.0 \
  --platform Linux --data file://test-component.yaml
Why exit 1 matters In Linux, exit code 0 means success and anything else means failure. Forcing exit 1 when Java or Python is missing makes the test phase fail loudly, so Image Builder refuses to publish a faulty image. This is your quality gate.
5

Create the image recipe

A recipe combines a base ("parent") image with your components. Pointing the parent at the AWS-managed Amazon Linux 2023 image ARN with a wildcard version means every build automatically starts from the newest, fully-patched release — no manual updating.

bash · aws cli
aws imagebuilder create-image-recipe \
  --name base-java-python-recipe --semantic-version 1.0.0 \
  --parent-image "arn:aws:imagebuilder:us-east-1:aws:image/amazon-linux-2023-x86/x.x.x" \
  --components \
    'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/installjavapythonkafka/1.0.0' \
    'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/verifyjavapythonkafka/1.0.0'
6

Create infrastructure config + the pipeline

The infrastructure configuration says which IAM profile and instance type to build on — and, crucially, which SNS topic to notify when a build finishes. That single field is what produces the "image-ready" announcement for free, no extra code needed. The pipeline then binds the recipe and the infra config together.

bash · aws cli
aws imagebuilder create-infrastructure-configuration \
  --name base-infra \
  --instance-profile-name ImageBuilderProfile \
  --instance-types t3.medium \
  --sns-topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:image-ready-topic

aws imagebuilder create-image-pipeline \
  --name base-java-python-pipeline \
  --image-recipe-arn arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:image-recipe/base-java-python-recipe/1.0.0 \
  --infrastructure-configuration-arn arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:infrastructure-configuration/base-infra
7

Wire the trigger topic to a Lambda

When the new-base-image-trigger topic receives a message, a tiny Lambda starts the pipeline. Separating "when to build" (the trigger) from "how to build" (the pipeline) keeps the system flexible — you can later swap in an EventBridge schedule without touching the pipeline.

python · lambda handler (index.py)
import boto3

def handler(event, context):
    ib = boto3.client('imagebuilder')
    ib.start_image_pipeline_execution(
        imagePipelineArn='arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:image-pipeline/base-java-python-pipeline'
    )
    return {"status": "build started"}
bash · subscribe Lambda to the trigger topic
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:new-base-image-trigger \
  --protocol lambda \
  --notification-endpoint arn:aws:lambda:us-east-1:ACCOUNT_ID:function:StartImageBuild
8

Downstream teams consume the queue

When a build succeeds, Image Builder posts a JSON message (including the new AMI ID and status) to image-ready-topic, which fans out to the SQS queue. App teams poll the queue, read the AMI ID, and roll it into their launch templates or Auto Scaling groups.

bash · poll the queue
aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/ACCOUNT_ID/image-update-queue \
  --max-number-of-messages 1 \
  --wait-time-seconds 10
9

Test the whole pipeline end-to-end

Publish one message to the trigger topic and watch the chain react.

bash · fire the trigger
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:ACCOUNT_ID:new-base-image-trigger \
  --message "New Amazon Linux available, please build"

In the EC2 Image Builder console, watch the pipeline move through Build → Test → Distribute. If the test phase fails, no image is created and no "ready" message is sent — exactly the safety behaviour you want.

Bonus: build it all with Terraform

The CLI commands are great for learning, but typing them one by one invites mistakes. Terraform lets you describe the entire setup — role, topics, queue, components, recipe, pipeline — in one file, then create or destroy it with a single command. It also wires the ARNs together automatically, so there is no ACCOUNT_ID to paste anywhere.

Three commands to run it terraform init downloads the AWS plugin · terraform plan previews changes · terraform apply builds it. Run terraform destroy to remove everything.
hcl · main.tf

  owners      = ["amazon"]
  filter { name = "name"  values = ["al2023-ami-2023.*-x86_64"] }
}

resource "aws_sns_topic" "trigger" { name = "new-base-image-trigger" }
resource "aws_sns_topic" "ready"   { name = "image-ready-topic" }
resource "aws_sqs_queue"  "updates" { name = "image-update-queue" }

resource "aws_sqs_queue_policy" "allow_sns" {
  queue_url = aws_sqs_queue.updates.id

    Version = "2012-10-17"

resource "aws_sns_topic_subscription" "queue_sub" {

resource "aws_imagebuilder_infrastructure_configuration" "infra" {
  name = "base-infra"
  instance_profile_name = aws_iam_instance_profile.ib.name
  instance_types        = ["t3.medium"]
  sns_topic_arn         = aws_sns_topic.ready.arn
}
resource "aws_imagebuilder_image_pipeline" "pipeline" {
  name = "base-java-python-pipeline"
  image_recipe_arn                 = aws_imagebuilder_image_recipe.recipe.arn
  infrastructure_configuration_arn = aws_imagebuilder_infrastructure_configuration.infra.arn
}

output "pipeline_arn" { value = aws_imagebuilder_image_pipeline.pipeline.arn }
output "queue_url"    { value = aws_sqs_queue.updates.id }

The Lambda trigger from Step 7 fits the same way — add a aws_lambda_function, an aws_sns_topic_subscription with protocol = "lambda", and an aws_lambda_permission so SNS may invoke it.

Starting from your own custom AMI

Up to now, the recipe's parent has been the AWS-published Amazon Linux 2023 image. But maybe your company already has a "company-approved" AMI — perhaps one your security team has already hardened with a CIS benchmark, an antivirus agent, or company SSL certs. Image Builder is happy to build on top of that instead. You have three choices for what to point the recipe at:

Option A · An ordinary EC2 AMI you own

If your security team gave you an AMI ID like ami-0abcd1234efgh5678, you can pass that ID directly. Image Builder will boot a temporary instance from your AMI, run your components, and snapshot the result.

bash · use a custom AMI as parent
aws imagebuilder create-image-recipe \
  --name kafka-on-company-base --semantic-version 1.0.0 \
  --parent-image "ami-0abcd1234efgh5678" \
  --components 'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/installjavapythonkafka/1.0.0'
The AMI must have the SSM Agent installed and running Image Builder controls the build instance through AWS Systems Manager. If your custom AMI doesn't have the SSM Agent, or the agent can't reach the SSM endpoints from your subnet, the build will hang and eventually time out. Amazon Linux 2023, Ubuntu 22.04+ from the Marketplace, and the official Windows Server AMIs all include it.

Option B · An AMI from SSM Parameter Store

If you want "always the newest" pointing at your own custom AMI, store its ID in SSM Parameter Store and reference the parameter. When your security team publishes a new hardened AMI, they update the parameter value — and your next Image Builder run picks it up automatically.

bash · resolve latest from SSM
# Security team publishes new AMI and updates the parameter
aws ssm put-parameter --name /company/golden/al2023 \
  --type String --value ami-0abcd1234efgh5678 --overwrite

# You read the current value when creating the recipe
PARENT_AMI=$(aws ssm get-parameter --name /company/golden/al2023 \
  --query Parameter.Value --output text)

aws imagebuilder create-image-recipe \
  --name base-on-company-golden --semantic-version 1.0.0 \
  --parent-image "$PARENT_AMI" \
  --components 'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/installjavapythonkafka/1.0.0'

Option C · Stack on top of a previous Image Builder output

You can chain pipelines. Pipeline 1 produces a "base-with-Java-Python" image; Pipeline 2 takes that AMI as its parent and adds Kafka tools; Pipeline 3 takes Pipeline 2's output and adds your app code. Each layer is small, reusable, and quick to rebuild. The parent ARN looks like the AWS-managed one but with your AMI ID:

bash · chain pipelines via the prior AMI
aws imagebuilder create-image-recipe \
  --name kafka-layer --semantic-version 1.0.0 \
  --parent-image "ami-PRIOR_PIPELINE_OUTPUT" \
  --components 'componentArn=arn:aws:imagebuilder:us-east-1:ACCOUNT_ID:component/installkafkacli/1.0.0'

Variant images: Kafka, Ansible, ZooKeeper, NiFi

Once you have a good Java-and-Python base image, making specialized images for other tools becomes easy. Each tool just needs its own component (a YAML recipe of install steps) added to a new image recipe. Below are starter components for four common ones. All assume the base image already has Java 21 and Python — so you just add the tool itself on top.

Kafka client / broker tools

If apps will only connect to Amazon MSK or another Kafka cluster, the libraries already in the base image are enough. If you want command-line tools (kafka-topics.sh, kafka-console-producer.sh) or a local broker for testing, install Apache Kafka itself:

yaml · kafka-tools.yaml
name: InstallKafkaTools
schemaVersion: 1.0
phases:
  - name: build
    steps:
      - name: DownloadKafka
        action: ExecuteBash
        inputs:
          commands:
            - KAFKA_VERSION=3.7.0
            - SCALA_VERSION=2.13
            - curl -fLO https://downloads.apache.org/kafka/${KAFKA_VERSION}/kafka_${SCALA_VERSION}-${KAFKA_VERSION}.tgz
            - sudo tar -xzf kafka_${SCALA_VERSION}-${KAFKA_VERSION}.tgz -C /opt
            - sudo ln -s /opt/kafka_${SCALA_VERSION}-${KAFKA_VERSION} /opt/kafka
            - echo 'export PATH=$PATH:/opt/kafka/bin' | sudo tee /etc/profile.d/kafka.sh

Ansible (config management on this AMI)

Ansible is a tool for automating server configuration. Putting it on the base image lets you run playbooks against the instance, or use the instance as an Ansible control node that manages others.

yaml · ansible.yaml
name: InstallAnsible
schemaVersion: 1.0
phases:
  - name: build
    steps:
      - name: AnsibleAndDeps
        action: ExecuteBash
        inputs:
          commands:
            # Ansible runs over SSH; sshpass helps with password-auth hosts (optional)
            - sudo dnf install -y openssh-clients sshpass git
            - sudo python3 -m pip install ansible ansible-lint boto3 botocore
            - ansible --version

Apache ZooKeeper

ZooKeeper is a coordination service used by older Kafka clusters and many other distributed systems. It needs Java (already there) plus its own user, data directory, and a config file.

yaml · zookeeper.yaml
name: InstallZooKeeper
schemaVersion: 1.0
phases:
  - name: build
    steps:
      - name: DownloadZK
        action: ExecuteBash
        inputs:
          commands:
            - ZK_VERSION=3.9.2
            - curl -fLO https://downloads.apache.org/zookeeper/zookeeper-${ZK_VERSION}/apache-zookeeper-${ZK_VERSION}-bin.tar.gz
            - sudo tar -xzf apache-zookeeper-${ZK_VERSION}-bin.tar.gz -C /opt
            - sudo ln -s /opt/apache-zookeeper-${ZK_VERSION}-bin /opt/zookeeper
      - name: ZKUserAndDirs
        action: ExecuteBash
        inputs:
          commands:
            - sudo useradd -r -s /sbin/nologin zookeeper
            - sudo mkdir -p /var/lib/zookeeper /var/log/zookeeper
            - sudo chown -R zookeeper:zookeeper /opt/zookeeper /var/lib/zookeeper /var/log/zookeeper
            - sudo cp /opt/zookeeper/conf/zoo_sample.cfg /opt/zookeeper/conf/zoo.cfg

Apache NiFi (data flow)

NiFi is a visual data-flow tool — useful for moving data between systems with little or no code. It is a Java app, so installation means downloading the tarball, creating a service user, and dropping a systemd unit so it starts on boot.

yaml · nifi.yaml
name: InstallNiFi
schemaVersion: 1.0
phases:
  - name: build
    steps:
      - name: DownloadNiFi
        action: ExecuteBash
        inputs:
          commands:
            - NIFI_VERSION=1.27.0
            - curl -fLO https://downloads.apache.org/nifi/${NIFI_VERSION}/nifi-${NIFI_VERSION}-bin.zip
            - sudo dnf install -y unzip
            - sudo unzip -q nifi-${NIFI_VERSION}-bin.zip -d /opt
            - sudo ln -s /opt/nifi-${NIFI_VERSION} /opt/nifi
      - name: NiFiServiceUser
        action: ExecuteBash
        inputs:
          commands:
            - sudo useradd -r -s /sbin/nologin nifi
            - sudo chown -R nifi:nifi /opt/nifi-${NIFI_VERSION}
      - name: SystemdUnit
        action: ExecuteBash
        inputs:
          commands:
            - |
              sudo tee /etc/systemd/system/nifi.service >/dev/null <<'EOF'
              [Unit]
              Description=Apache NiFi
              After=network.target
              [Service]
              Type=forking
              User=nifi
              ExecStart=/opt/nifi/bin/nifi.sh start
              ExecStop=/opt/nifi/bin/nifi.sh stop
              [Install]
              WantedBy=multi-user.target
              EOF
            - sudo systemctl daemon-reload
            - sudo systemctl enable nifi
How to combine these Register each component once with aws imagebuilder create-component. Then create a separate recipe per image type — a "Kafka edition" recipe lists the base + Kafka tools components; a "NiFi edition" recipe lists the base + NiFi component. One pipeline per recipe means each variant rebuilds independently when its source tool releases a new version.

Gotchas & fixes

Things that bite people the first time they build an image. Each one tells you the symptom, the cause in plain language, and how to fix it.

!

"Image not found" or "parent image does not exist"

Symptom: Creating the recipe (or starting a build) fails with InvalidParameterValueException: The parent image does not exist.

Likely causes:

  1. Wrong region in the ARN. Image Builder's managed AMIs are per-region. An ARN with us-east-1 won't resolve if your CLI is configured for eu-west-2.
  2. Wrong AMI architecture. The ARN ends in amazon-linux-2023-x86 for Intel/AMD, or amazon-linux-2023-arm64 for Graviton. Pick the one matching your instance type.
  3. Custom AMI not shared with your account. If you're pointing at someone else's AMI ID, they must explicitly share it with your account ID (EC2 → AMIs → Permissions).
  4. SSM parameter is empty or has the wrong value. Resolve it first with aws ssm get-parameter and confirm it returns a valid ami-... ID.
  5. Account doesn't have permission to use the image. Even AWS-managed images can be region-restricted; switch region in your CLI or in the ARN.
!

Build hangs forever in "BUILDING" state

Symptom: The pipeline run sits at "BUILDING" for 30+ minutes with no logs.

Cause: The build instance can't reach AWS Systems Manager. Image Builder controls the box through SSM, so without that channel it sits there silently.

Fixes (try in order):

  1. Confirm AmazonSSMManagedInstanceCore is attached to the role you gave the infra config.
  2. If the build subnet is private with no NAT gateway, add VPC endpoints for ssm, ssmmessages, ec2messages, and s3.
  3. If using a custom parent AMI, confirm it has the SSM Agent installed and configured to start on boot.
  4. Check the security group on the build instance permits outbound HTTPS (port 443).
!

"dnf install" fails with "Cannot find a valid baseurl for repo"

Symptom: A build step that calls sudo dnf install -y … fails. The CloudWatch log shows the package manager can't fetch its repository metadata.

Cause: The build instance has no internet access. Amazon Linux package mirrors live on the public internet (or AWS S3 mirrors).

Fixes: place the build in a subnet with a NAT gateway, or use a subnet inside a VPC that has an S3 gateway endpoint plus the AL2023 internal repos. If your security policy forbids internet from build subnets, mirror your packages to a private repo and override /etc/dnf/dnf.conf in your component.

!

"Component already exists" when re-running

Symptom: Re-running create-component with the same --semantic-version errors out.

Cause: Components are immutable. Once InstallJavaPythonKafka/1.0.0 exists, you can't overwrite it.

Fix: bump the version (1.0.1) and update any recipe that referenced it. This is a feature, not a bug — old AMIs always know exactly which component versions baked them.

!

"PassRole" denied when creating the infra config

Symptom: User: arn:aws:iam::…:user/X is not authorized to perform: iam:PassRole.

Cause: Your human user can create roles but isn't allowed to hand the Image Builder role to the service.

Fix: add iam:PassRole to your user policy scoped to the exact role ARN (arn:aws:iam::*:role/ImageBuilderRole). The least-privilege policy earlier in this guide already includes this.

!

Test phase fails with "command not found" — even though the build phase installed it

Symptom: java: command not found in the test phase, although the build phase ran dnf install successfully.

Cause: The test phase runs on a fresh instance launched from the finished AMI, with a fresh shell session. If your install put a binary somewhere not on the default PATH (e.g. /opt/kafka/bin), the test shell won't see it.

Fix: in the build phase, write a file in /etc/profile.d/ that exports the new PATH for all future logins. The Kafka component above already does this.

!

SNS subscription confirmed but no messages arrive

Symptom: Builds complete, but your SQS queue never receives the "image-ready" message.

Causes:

  1. The queue's access policy doesn't allow sns.amazonaws.com to sqs:SendMessage. Without it SNS silently drops messages.
  2. The subscription is in a different region from the topic.
  3. If using server-side encryption (SQS KMS keys), the KMS key policy must also allow SNS.

Fix: in the SQS Console, open the queue → Access policy → Edit, and paste the policy from Step 2's warning box (or use Terraform which writes it correctly).

!

"Apache download failed" or "404 Not Found" mid-build

Symptom: A NiFi or ZooKeeper component fails because the download URL returns 404.

Cause: Apache rotates its mirrors and removes old versions from the primary downloads server. The variant components above pin a version — that exact version may have aged out.

Fixes:

  1. Pull from the Apache archive instead: https://archive.apache.org/dist/<project>/... — older releases stay there permanently.
  2. Better: copy the tarball to your own S3 bucket once and have the component download from there. Survives the open-source mirror going offline.
!

Pipeline says SUCCESS but AMI doesn't appear in EC2 → AMIs

Cause: By default the AMI appears in the region the pipeline ran in. If your team looks in a different region, they won't see it.

Fix: add a distribution configuration to the pipeline listing every region the AMI should be copied into, and which accounts may launch it.

!

Quick checklist before pressing "Run"

  1. Replace every ACCOUNT_ID placeholder with your real 12-digit account number.
  2. Confirm your CLI region matches the region in every ARN.
  3. Confirm the parent image ARN/ID exists in that region.
  4. Confirm the SQS queue access policy allows the SNS topic.
  5. Bump component --semantic-version if you've edited any YAML.

You've built an image factory.

A single trigger message now bakes a fresh Amazon Linux image with Java 21 and Python, tests it, and announces it to every downstream team — repeatably, and with a quality gate that blocks broken builds. Re-run anytime by publishing to the trigger topic, or attach an EventBridge schedule to run it weekly.
