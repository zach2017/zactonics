Build a Real Cloud Server with Terraform & AWS

This guide takes you from a fresh Windows computer all the way to running your own Amazon Linux server in the cloud — and then making it do advanced things. Every step is explained in plain language. No experience needed.

Start Here: What Is This and Why Should I Care?

›

Imagine you want a computer that lives on the internet — a "server." You could rent one from Amazon (called AWS, short for Amazon Web Services). Normally you'd click around dozens of web pages to set it up. That's slow and easy to mess up.

Terraform is a tool that lets you write down what you want in a simple text file, then build it all automatically. Think of it like a recipe card: you write the recipe once, and Terraform "cooks" the exact same server every time. This idea is called Infrastructure as Code — your cloud setup is just code you can save, share, and reuse.

What you'll have by the end

  • Terraform installed and working on your Windows PC.
  • An AWS account connected safely to your computer.
  • A live Amazon Linux server you built with a single command.
  • That server set up to run a real web service, with the right network "doors" (ports) open.
Money warning: AWS can charge you for servers. We use the free tier where possible, but always do the Clean Up step when you're done so you don't get a surprise bill.

Install Terraform on Windows

›

There are two easy ways. Pick Method A if you want the simplest path. Use Method B if you like doing things the manual way and understanding each piece.

Method A — The easy way (using a "package manager")

A package manager is an app that installs other apps for you with one command. On Windows the popular one is called Chocolatey. First, open PowerShell as Administrator (click Start, type "PowerShell", right-click it, choose "Run as administrator").

Install Chocolatey by pasting this and pressing Enter:

Set-ExecutionPolicy Bypass -Scope Process -Force; `
[System.Net.ServicePointManager]::SecurityProtocol = 3072; `
iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))

Now install Terraform with one line:

choco install terraform -y

Method B — The manual way (download the file yourself)

  1. Go to the official Terraform download page (link below) and download the Windows AMD64 ZIP file.
  2. The ZIP contains one file: terraform.exe. Unzip it.
  3. Make a folder like C:\terraform and move terraform.exe into it.
  4. Add that folder to your PATH so Windows can find it from anywhere (see the pop-up).

Check that it worked

Close and reopen PowerShell, then type:

terraform -version

You should see something like Terraform v1.9.x. If you do — congratulations, Terraform is installed! If you see "not recognized," your PATH isn't set right (revisit the pop-up above).

Tip: You can use Windows PowerShell, Command Prompt, or Windows Terminal — they all work. We'll use PowerShell in this guide.

Set Up an AWS Account

›

To rent servers from Amazon, you need an AWS account. It's free to create — you only pay for what you use, and many things are free for the first year.

  1. Go to aws.amazon.com/free and click "Create a Free Account."
  2. Enter your email and pick an account name.
  3. You'll need a credit/debit card. AWS uses it to verify you and only charges if you go over free limits.
  4. Choose the Basic (Free) support plan.

Create a safe access key (very important)

Terraform needs permission to act on your behalf. Instead of using your main login, we create a special user with its own "key" — like giving a house-sitter one specific key instead of your master key.

  1. In the AWS console, search for IAM (Identity and Access Management).
  2. Create a new user, e.g. terraform-user.
  3. Attach a permission policy. For learning, AdministratorAccess is easiest; for real work, use narrower permissions.
  4. Create an Access Key for that user. You'll get an Access Key ID and a Secret Access Key.
Guard your secret key! Treat the Secret Access Key like a password. Never put it on the internet, in screenshots, or in code you share. If it leaks, delete it immediately in IAM.

Install & Configure the AWS CLI

›

The AWS CLI (Command Line Interface) is a tool that lets your computer talk to AWS. Terraform can use the login info that the CLI stores, so this is the cleanest way to connect them.

Install it

Easy way with Chocolatey:

choco install awscli -y

Or download the official Windows installer (.msi) from the link at the bottom and double-click it.

Check it works (you may need to reopen PowerShell):

aws --version

Connect it to your account

Now tell the CLI about the access key you made in Step 2. Run:

aws configure

It will ask four questions. Paste your keys and choose a region:

AWS Access Key ID [None]: AKIA................
AWS Secret Access Key [None]: wJalrXUtnFEMI................
Default region name [None]: us-east-1
Default output format [None]: json

Test the connection

This command asks AWS "who am I?" If it answers with your account number, everything is wired up:

aws sts get-caller-identity
Why this matters: Terraform automatically looks for these saved credentials. So once the CLI works, Terraform "just works" too — you won't paste keys into your Terraform files (which is the safe way to do it).

Create Your First Terraform Project

›

A Terraform "project" is just a folder with one or more text files ending in .tf. Let's make one.

mkdir C:\my-first-terraform
cd C:\my-first-terraform

Now create a file named main.tf. You can use Notepad, VS Code, or any editor. Paste in this starter content:

# main.tf — tells Terraform we want to use AWS

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# The "provider" block configures AWS and the region
provider "aws" {
  region = "us-east-1"
}

What does this mean?

  • Lines starting with # are comments — notes for humans that Terraform ignores.
  • The terraform block lists the "plugins" (called providers) we need. Here, the AWS provider.
  • The provider "aws" block says which region to build in.

The Core Terraform Commands

›

Terraform has a small set of commands you'll use over and over. Think of them as the four magic words. Run them from inside your project folder.

Build a Basic EC2 Server (Amazon Linux)

›

Now the fun part — let's create a real server. We'll add to main.tf. Instead of hard-coding which Linux image (AMI) to use, we'll ask AWS for the latest Amazon Linux 2023 automatically. That way it's always up to date.

Step 6a — Find the latest Amazon Linux image

Add this data block. A "data" block looks up information from AWS instead of creating something.

# Ask AWS for the newest Amazon Linux 2023 image
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }
}

Step 6b — Create the EC2 instance

Now add a resource block. A "resource" is a thing Terraform actually creates. Here it's an EC2 server.

# Create the EC2 virtual server
resource "aws_instance" "my_server" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t2.micro"   # free-tier eligible size

  tags = {
    Name = "MyFirstTerraformServer"
  }
}

What do these lines mean?

  • ami — which operating system image to use. We point it at the data block we just made, so it's always the newest Amazon Linux.
  • instance_type — the "size" of the computer. t2.micro is small and free-tier eligible.
  • tags — friendly labels. The "Name" tag is what you'll see in the AWS console.

Step 6c — Build it!

Run the workflow:

terraform plan
terraform apply

Type yes when asked. In under a minute, you'll have a live server! Open the AWS console → EC2 → Instances to see it running.

You just did it. You described a server in text and Terraform built it in the cloud. That's Infrastructure as Code in action.

Test & Connect to Your Server

›

A server is only useful if you can reach it. To log in over SSH (a secure remote connection), you need a key pair — like a matching lock and key.

Step 7a — Make a key pair

Generate a key on your PC with this command (it creates two files):

ssh-keygen -t rsa -b 4096 -f my-key

This makes my-key (private — keep secret!) and my-key.pub (public — safe to upload).

Step 7b — Tell Terraform about the key

Add a key-pair resource and attach it to your instance:

resource "aws_key_pair" "my_key" {
  key_name   = "my-key"
  public_key = file("my-key.pub")   # reads your public key file
}

Then add one line inside your aws_instance "my_server" block:

  key_name = aws_key_pair.my_key.key_name

Step 7c — See the server's address (Outputs)

Add an output so Terraform prints the public address after building:

output "public_ip" {
  value = aws_instance.my_server.public_ip
}

Run terraform apply again. When it finishes you'll see the IP. Connect:

ssh -i my-key ec2-user@PASTE_THE_IP_HERE
Can't connect yet? That's expected! By default AWS blocks all incoming traffic. You must open port 22 (SSH) with a security group — that's exactly the next step.

Advanced: Security Groups & Opening Ports

›

A security group is like a bouncer for your server. It decides which network "doors" (ports) are open and who's allowed through. By default everything is closed.

Common ports you'll open

PortUsed for
22SSH — remote login to the server
80HTTP — normal websites
443HTTPS — secure websites

Add a security group

resource "aws_security_group" "web_sg" {
  name        = "web-sg"
  description = "Allow SSH and web traffic"

  # Inbound: SSH (port 22)
  ingress {
    description = "SSH"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]   # any IP (see warning)
  }

  # Inbound: HTTP (port 80)
  ingress {
    description = "HTTP web traffic"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # Outbound: allow the server to reach the internet
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"            # -1 means "all"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

Now attach it to your instance — add this line inside the aws_instance block:

  vpc_security_group_ids = [aws_security_group.web_sg.id]

Key words explained

  • ingress = incoming traffic (people connecting TO your server).
  • egress = outgoing traffic (your server reaching OUT, e.g. to download updates).
  • cidr_blocks = which IP addresses are allowed. 0.0.0.0/0 means "the whole internet."
  • protocol = usually tcp for these ports.
Security tip: 0.0.0.0/0 on port 22 lets anyone in the world try to log in. For real use, replace it with just your own IP, like ["203.0.113.25/32"].

Run terraform apply again. Now your SSH connection from Step 7 should work!

Advanced: Install Services Automatically (User Data)

›

What if you want your server to set itself up — install a web server, start it, and show a page — the moment it boots? That's what user data does. It's a startup script that runs once, automatically, when the server first turns on.

Add a startup script to your instance

Put this inside your aws_instance "my_server" block. It installs the Apache web server and creates a simple home page:

  user_data = <<-EOF
    #!/bin/bash
    # This script runs once when the server boots
    dnf update -y
    dnf install -y httpd
    systemctl start httpd
    systemctl enable httpd
    echo "<h1>Hello from Terraform on Amazon Linux!</h1>" > /var/www/html/index.html
  EOF

Line-by-line

  • #!/bin/bash — tells Linux this is a bash script.
  • dnf install -y httpd — installs Apache (the httpd web server). dnf is Amazon Linux's app installer.
  • systemctl start httpd — turns the web server on now.
  • systemctl enable httpd — makes it auto-start after future reboots.
  • The echo ... > index.html line writes the web page you'll see.
Make sure port 80 is open! The web server uses HTTP on port 80, which you opened in Step 8. After applying, paste your server's public IP into a browser (http://YOUR_IP) and you'll see your page.

Apply your changes. Because user data only runs on first boot, you may need to recreate the instance to see it take effect (Terraform will tell you when a change forces a replacement):

terraform apply

Advanced: Variables, Outputs & State (Make It Reusable)

›

Right now values like the region are typed directly in your code. Variables let you pull those out so your project is flexible and reusable — like fill-in-the-blank slots.

Define variables (variables.tf)

variable "region" {
  description = "AWS region to build in"
  type        = string
  default     = "us-east-1"
}

variable "instance_type" {
  description = "Size of the EC2 server"
  type        = string
  default     = "t2.micro"
}

Now use them in your code with var.:

provider "aws" {
  region = var.region
}

resource "aws_instance" "my_server" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type
  # ...rest stays the same
}

Outputs (outputs.tf)

Outputs print useful info after a build. Group them in their own file:

output "public_ip" {
  description = "The server's public IP address"
  value       = aws_instance.my_server.public_ip
}

output "website_url" {
  value = "http://${aws_instance.my_server.public_ip}"
}

Understanding the State file

After you apply, Terraform creates a file called terraform.tfstate. This is Terraform's "memory" — it records what it built so it knows what to change or delete next time.

Never delete or hand-edit terraform.tfstate, and never commit it to public GitHub — it can contain secrets and is what Terraform relies on to manage your servers.

A clean project layout

FileHolds
main.tfProvider + resources (the server, security group)
variables.tfYour fill-in-the-blank inputs
outputs.tfInfo to print after building
terraform.tfstateAuto-made memory (don't touch)

Clean Up — Avoid Surprise Charges

›

When you're done experimenting, tear everything down so you stop using resources. One command removes everything this project created:

terraform destroy

Terraform shows what it will delete and asks you to type yes. After it finishes, check the AWS console (EC2 → Instances) to confirm your server is gone.

Habit to build: every time you finish a learning session, run terraform destroy. A stopped/terminated server can't cost you money.

Full Command Reference — Every Command & Option Explained

›

This is the complete list of Terraform commands, exactly as you'd see them when you type terraform -help. For each one you'll get a plain-language explanation, its most useful options (called flags or arguments), a real example, and what you should expect to see happen.

How to read a command: the shape is always terraform [global options] <command> [flags]. A flag starts with a dash, like -auto-approve. Flags that take a value look like -var="region=us-east-1".

Main Commands

terraform initmain

Prepares your working directory. It downloads the providers (like the AWS plugin) and sets up the "backend" where your state is stored. Always the first command you run in a new project, or after you add a new provider/module.

Common options
FlagWhat it does
-upgradeUpgrade providers/modules to the newest allowed versions.
-reconfigureReconfigure the backend, ignoring any saved settings.
-migrate-stateMove existing state to a new backend (e.g. local → S3).
-backend=falseSkip backend setup; only install plugins.
-get=falseSkip downloading modules.
-input=falseDon't ask interactive questions (good for automation).
Example
terraform init -upgrade
Expected result
Initializing the backend... Initializing provider plugins... - Finding hashicorp/aws versions matching "~> 5.0"... - Installing hashicorp/aws v5.xx.0... Terraform has been successfully initialized!

A hidden folder .terraform/ and a lock file .terraform.lock.hcl appear in your project.

terraform planmain

Shows a preview of what Terraform would do — what it will add, change, or destroy — but changes nothing. It contacts AWS to compare your code against reality. Reading this carefully is the single most important safety habit.

Common options
FlagWhat it does
-out=FILESave the plan to a file so apply runs exactly it.
-destroyShow a plan for destroying everything.
-var="key=value"Set an input variable. Repeatable.
-var-file=FILELoad variables from a file.
-target=ADDRLimit the plan to one resource (use sparingly).
-refresh=falseSkip checking real state first (faster, less accurate).
-detailed-exitcodeExit 0=no changes, 1=error, 2=changes (great for CI).
Example

Title

Content

"Infrastructure" means your servers, networks, and other cloud pieces. Normally you build them by clicking around a website. Infrastructure as Code means you write down what you want in a text file instead.

Benefits: you can save it, share it, undo it, and rebuild the exact same setup anytime. If your server breaks, you just run the code again and get an identical one.

EC2 stands for "Elastic Compute Cloud." An instance is one virtual computer you rent from Amazon. "Virtual" means it's a slice of a big real machine in an Amazon data center.

You choose its size, its operating system, and how long it runs. You pay only while it's on (and some sizes are free for the first year).

AMI = "Amazon Machine Image." It's a snapshot of an operating system that your new server starts from — like choosing which version of Windows or Linux to install, except it's pre-packaged.

We use Amazon Linux 2023, Amazon's own free Linux made to work smoothly on EC2. Asking for the "latest" one means you always start fresh and secure.

Chocolatey is a "package manager" for Windows — an app store you control from the command line. When you type choco install terraform, it downloads the right Terraform file, puts it in the correct folder, and adds it to your PATH automatically.

This saves you from doing those steps by hand (which is Method B).

Installing software changes protected parts of Windows. Running PowerShell "as Administrator" gives it permission to make those changes. Without it, the install may fail with "access denied."

The PATH is a list of folders Windows searches when you type a command. If terraform.exe lives in a folder that's on the PATH, you can type terraform from anywhere.

To edit it: press Start → type "environment variables" → "Edit the system environment variables" → "Environment Variables…" → under System variables select Path → Edit → New → add C:\terraform → OK. Reopen PowerShell afterward.

The root user is the very first login for your AWS account — it can do absolutely anything, including closing the account. Because it's so powerful, you should rarely use it.

For everyday work (and for Terraform) you make a limited IAM user instead. Turn on multi-factor authentication (MFA) on the root user for safety.

AWS has data centers all over the world, grouped into Regions like us-east-1 (Virginia) or eu-west-1 (Ireland). Your servers physically live in the region you pick.

Choose one close to you (or your users) for faster speeds. us-east-1 is the biggest and a common default.

IAM = "Identity and Access Management." It's where you create users and decide what each one is allowed to do. Think of it as the security desk that hands out ID badges with different access levels.

We make a dedicated user for Terraform so its key can be revoked anytime without touching your main login.

aws configure saves your keys to a hidden file at C:\Users\YourName\.aws\credentials and your region to .aws\config.

Terraform automatically reads these files, which is why you never paste keys into your .tf files — keeping secrets out of your code.

Pick the region nearest you for speed, or stick with us-east-1 for the widest service availability and the most tutorials/examples online. Just be consistent — resources in one region can't see resources in another by default.

A provider is a plugin that teaches Terraform how to talk to a specific service. The hashicorp/aws provider knows how to create AWS things (servers, networks, etc.).

There are providers for Google Cloud, Azure, GitHub, and hundreds more. terraform init downloads the ones your project lists.

Terraform files are written in HCL — "HashiCorp Configuration Language." It's designed to be easy for humans to read and write. You mostly write blocks like resource "type" "name" { ... } that describe what you want.

terraform plan is a safe preview. It compares what you wrote against what currently exists and lists what it would add (+), change (~), or destroy (-) — without doing anything.

Always read it. It's how you catch mistakes (like accidentally deleting something) before they happen.

Terraform records what it built in a file called terraform.tfstate. This is its "memory." Next time you run a command, it compares your code to this file to figure out what changed.

Don't edit or delete it by hand, and keep it private — it can hold sensitive values.

A data block reads existing information from AWS instead of creating anything. We use one to ask "what's the newest Amazon Linux image ID?" so we never have to hard-code an ID that goes stale.

Compare: a resource block creates; a data block looks up.

An instance type sets how much CPU and memory your server has. t2.micro and t3.micro are tiny and free-tier eligible — perfect for learning.

Bigger types like m5.large have more power but cost more. You can change the type anytime by editing one line and re-applying.

SSH (Secure Shell) is a way to safely log into a remote Linux machine from your terminal. It uses a key pair: a private key that stays secret on your PC and a public key that goes on the server.

Only someone holding the matching private key can log in — much safer than a password.

A security group is a virtual firewall wrapped around your server. It has rules saying which ports are open and which IP addresses are allowed in (ingress) or out (egress).

By default, nothing is allowed in — you must explicitly open what you need.

A single server can run many services at once — a website, an email server, etc. Ports are numbered doors that keep them separate. Web traffic uses port 80, secure web uses 443, and SSH login uses 22.

Opening a port means "allow traffic to knock on this specific door."

This is CIDR notation for describing groups of IP addresses. 0.0.0.0/0 means "every possible address" (the whole internet). 1.2.3.4/32 means "exactly this one address."

Smaller numbers after the slash = more addresses. For SSH, narrowing to your own /32 is much safer.

User data is a script you hand to the instance. AWS runs it automatically once, the first time the server boots. It's how you make a server set itself up — installing software, starting services, writing files — with no manual login needed.

systemctl is the Linux command that controls background services. systemctl start httpd turns the web server on now; systemctl enable httpd makes it start automatically every time the server reboots.

A variable's default is used if you don't override it. To override, you can:

1) Pass it on the command line: terraform apply -var="instance_type=t3.small"
2) Make a file terraform.tfvars with lines like instance_type = "t3.small" — Terraform reads it automatically.

The state file maps your code to the real resources. If you delete it, Terraform "forgets" what it made and may try to create duplicates. If you edit it wrong, you can corrupt your tracking.

Keep it safe, keep it private (it may hold secrets), and never commit it to a public repository.

When several people share a project, keeping the state file on one person's laptop is risky. Remote state stores it in a shared, locked location — commonly an AWS S3 bucket with DynamoDB locking — so everyone uses the same up-to-date memory and no two people apply at once.
