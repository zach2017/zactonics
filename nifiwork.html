NiFi Flow Deployment Architecture

🗂️ What is NiFi Registry?

NiFi Registry is a standalone service that provides flow version control. It stores snapshots of Process Group flows, tied to a Git-backed or local storage provider.

Every save creates an immutable snapshot. Flows can be imported into any NiFi instance by referencing the Registry bucket + flow ID + version number.

ℹ️ Registry is separate from NiFi itself — it runs on its own port (default 18080) and can serve multiple NiFi clusters.
🔁 Promotion Model

Flows are version-pinned. Each environment imports a specific version from a Registry bucket. Promotion means pointing the next environment at a validated version.

EnvironmentBucketPolicy
DEVdev-flowsAny version, frequent saves
STAGINGstaging-flowsPromoted + reviewed only
PRODprod-flowsChange-board approved

Registry Setup & NiFi Connection

1. nifi-registry.properties — Core Registry Config

properties 
# /opt/nifi-registry/conf/nifi-registry.properties
nifi.registry.web.http.port=18080
nifi.registry.web.https.port=18443

# Flow persistence — where snapshot JSON files live on disk
nifi.registry.db.directory=./database
nifi.registry.flow.provider.id=git-flow-provider

# Security
nifi.registry.security.keystore=./conf/keystore.jks
nifi.registry.security.keystorePasswd=changeit
nifi.registry.security.truststore=./conf/truststore.jks

# LDAP / identity provider
nifi.registry.security.identity.provider=ldap-identity-provider
nifi.registry.security.authorization.manager=managed-authorizer

2. providers.xml — Git-backed Flow Storage

xml 
<!-- /opt/nifi-registry/conf/providers.xml -->
<providers>

  <!-- Git provider: flows stored as JSON in a Git repo -->
  <flowPersistenceProvider>
    <class>org.apache.nifi.registry.provider.flow.git.GitFlowPersistenceProvider</class>
    <property name="Flow Storage Directory">./flow-storage</property>
    <property name="Remote To Push">origin</property>
    <property name="Remote Access User">git-nifi-svc</property>
    <property name="Remote Access Password">${GIT_TOKEN}</property>
  </flowPersistenceProvider>

  <!-- Event hook: notify Slack on new version committed -->
  <eventHookProvider>
    <class>org.apache.nifi.registry.provider.hook.ScriptEventHookProvider</class>
    <property name="Script Path">./conf/hooks/notify-slack.sh</property>
  </eventHookProvider>

</providers>

3. Connect NiFi Instance to Registry (nifi.properties)

properties 
# /opt/nifi/conf/nifi.properties  (repeat on each cluster: dev, staging, prod)

# DEV cluster points to registry
nifi.registry.client.url=https://nifi-registry.internal:18443
nifi.registry.client.id=dev-nifi-client
nifi.registry.client.name=Dev NiFi Client
✅ After restart, go to NiFi UI → Controller Settings → Registry Clients → Add, then verify the connection shows Connected status with the bucket list populated.

4. Create Buckets via Registry REST API

bash 
# Create the three environment buckets
for ENV in dev staging prod; do
  curl -s -X POST "https://nifi-registry.internal:18443/nifi-registry-api/buckets" \
    -H "Content-Type: application/json" \
    -H "Authorization: Bearer ${REGISTRY_TOKEN}" \
    -d "{\"name\": \"${ENV}-flows\", \"description\": \"Flows for ${ENV} environment\"}" \
    | jq '.identifier'
done

# Output (save these bucket IDs!):
# dev    → "a1b2c3d4-0001-..."
# staging→ "a1b2c3d4-0002-..."
# prod   → "a1b2c3d4-0003-..."

curl -X POST "https://nifi-dev:8443/nifi-api/versions/process-groups/{pgId}" \

  -H "Content-Type: application/json" \

What the Registry Stores (Flow Snapshot JSON)

The Registry persists a .snapshot.json (or git-committed JSON) per version:

json — flow snapshot excerpt 
// flow-storage/dev-flows/IngestCustomerOrders/5.snapshot.json
{
  "flowContents": {
    "identifier": "flow-uuid-xxxx",
    "name": "IngestCustomerOrders",
    "processors": [
      {
        "identifier": "proc-001",
        "name": "ConsumeKafka_2_6",
        "type": "org.apache.nifi.processors.kafka.pubsub.ConsumeKafka_2_6",
        "properties": {
          "bootstrap.servers": "#{KAFKA_BROKERS}",    // ← parameter ref
          "topic": "#{KAFKA_TOPIC}",
          "group.id": "nifi-ingest-group"
        }
      },
      {
        "name": "PutS3Object",
        "properties": {
          "Bucket": "#{S3_BUCKET}",          // ← parameter ref
          "Region": "us-east-1"
        }
      }
    ],
    "parameterContextName": "#{ENV_PARAM_CONTEXT}"
  },
  "timestamp": 1716300000000,
  "author": "j.smith@company.com",
  "comments": "Add error handling + retry loop on S3 put"
}
⚠️ Sensitive values (passwords, keys) must use Parameter Contexts with #{VAR} references — never hardcoded in the snapshot JSON. Each environment provides its own Parameter Context.

Promoting Flows: DEV → STAGING → PROD

ℹ️ Promotion is not a copy of the NiFi canvas — it's importing a specific version number from Registry into the next environment's NiFi instance, then updating the Parameter Context to that environment's values.

Step-by-step: DEV → STAGING

Export the versioned flow from DEV Registry
bash 
# Export version 5 of the flow to a JSON file
curl -s \
  "https://nifi-registry.internal:18443/nifi-registry-api/buckets/BUCKET_ID/flows/FLOW_ID/versions/5" \
  -H "Authorization: Bearer ${REGISTRY_TOKEN}" \
  -o IngestCustomerOrders_v5.json

ls -lh IngestCustomerOrders_v5.json
# -rw-r--r-- 1 nifi nifi 47K May 21 09:12 IngestCustomerOrders_v5.json
Import flow version into STAGING Registry bucket
bash 
# Import the same snapshot into the staging bucket
curl -s -X POST \
  "https://nifi-registry.internal:18443/nifi-registry-api/buckets/STAGING_BUCKET_ID/flows/STAGING_FLOW_ID/versions" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${REGISTRY_TOKEN}" \
  -d @IngestCustomerOrders_v5.json

# Returns version "1" in staging bucket (staging versions are independent)
Update Staging NiFi — change flow to new version
bash 
# Get current PG info
PG_ID="staging-pg-uuid"
PG_VERSION=$(curl -s "https://nifi-staging:8443/nifi-api/process-groups/${PG_ID}" \
  -H "Authorization: Bearer ${NIFI_TOKEN}" | jq '.revision.version')

# Update to new registry version
curl -X PUT \
  "https://nifi-staging:8443/nifi-api/versions/process-groups/${PG_ID}" \
  -H "Authorization: Bearer ${NIFI_TOKEN}" \
  -H "Content-Type: application/json" \
  -d "{
    \"processGroupRevision\": {\"version\": ${PG_VERSION}},
    \"versionedFlow\": {
      \"registryId\": \"staging-registry-client\",
      \"bucketId\":   \"STAGING_BUCKET_ID\",
      \"flowId\":     \"STAGING_FLOW_ID\",
      \"version\":    1
    }
  }"

Promote STAGING → PRODUCTION (change-board gated)

bash — promote-to-prod.sh 
#!/bin/bash
# promote-to-prod.sh — run after CAB approval
# Usage: ./promote-to-prod.sh IngestCustomerOrders 5

FLOW_NAME="$1"
SRC_VERSION="$2"
PROD_BUCKET="a1b2c3d4-0003-..."
PROD_NIFI="https://nifi-prod:8443"

set -euo pipefail

echo "[+] Promoting ${FLOW_NAME} v${SRC_VERSION} → PROD"

# 1. Export from staging registry
curl -s -o /tmp/${FLOW_NAME}_v${SRC_VERSION}.json \
  "https://nifi-registry.internal:18443/nifi-registry-api/buckets/${STAGING_BUCKET}/flows/${FLOW_ID}/versions/${SRC_VERSION}" \
  -H "Authorization: Bearer ${REGISTRY_TOKEN}"

# 2. Import to prod registry bucket
PROD_VERSION=$(curl -s -X POST \
  "https://nifi-registry.internal:18443/nifi-registry-api/buckets/${PROD_BUCKET}/flows/${PROD_FLOW_ID}/versions" \
  -H "Content-Type: application/json" -H "Authorization: Bearer ${REGISTRY_TOKEN}" \
  -d @/tmp/${FLOW_NAME}_v${SRC_VERSION}.json | jq '.versionedFlowSnapshotMetadata.version')

echo "[+] Imported as PROD v${PROD_VERSION}"

# 3. Update prod NiFi process group
./update-pg-version.sh "${PROD_NIFI}" "${PROD_PG_ID}" "${PROD_VERSION}"

# 4. Apply prod parameter context
./apply-params.sh "${PROD_NIFI}" "prod-params.json"

echo "✓ Production deployment complete: ${FLOW_NAME} v${PROD_VERSION}"

CI/CD Pipeline & Rollback

GitHub Actions — nifi-deploy.yml

yaml 
# .github/workflows/nifi-deploy.yml
name: NiFi Flow Deploy

on:
  workflow_dispatch:
    inputs:
      flow_name:    {description: "Flow name",    required: true}
      flow_version: {description: "Version to deploy", required: true}
      target_env:   {description: "staging or prod", required: true}

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.target_env }}   # requires env approval in prod
    steps:
      - uses: actions/checkout@v4

      - name: Promote flow
        env:
          REGISTRY_TOKEN: ${{ secrets.NIFI_REGISTRY_TOKEN }}
          NIFI_TOKEN:     ${{ secrets.NIFI_API_TOKEN }}
        run: |
          ./scripts/promote-flow.sh \
            "${{ inputs.flow_name }}" \
            "${{ inputs.flow_version }}" \
            "${{ inputs.target_env }}"

      - name: Run smoke tests
        run: ./scripts/smoke-test.sh "${{ inputs.target_env }}"

      - name: Notify Slack on failure
        if: failure()
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {"text": "❌ NiFi deploy FAILED: ${{inputs.flow_name}} v${{inputs.flow_version}} → ${{inputs.target_env}}"}

Rollback Script

bash — rollback-flow.sh 
#!/bin/bash
# rollback-flow.sh — revert prod to a known-good version
# Usage: ./rollback-flow.sh IngestCustomerOrders 4

FLOW_NAME="$1"
TARGET_VERSION="$2"
NIFI_PROD="https://nifi-prod:8443"

echo "⚠️  Rolling back ${FLOW_NAME} to v${TARGET_VERSION} in PRODUCTION"

# Stop the process group first (prevent data loss)
curl -s -X PUT "${NIFI_PROD}/nifi-api/flow/process-groups/${PROD_PG_ID}" \
  -H "Authorization: Bearer ${NIFI_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"id":"'${PROD_PG_ID}'","state":"STOPPED"}'

sleep 5   # allow in-flight FlowFiles to drain

# Revert to target version via versions API
PG_REV=$(curl -s "${NIFI_PROD}/nifi-api/process-groups/${PROD_PG_ID}" \
  -H "Authorization: Bearer ${NIFI_TOKEN}" | jq '.revision.version')

curl -X PUT "${NIFI_PROD}/nifi-api/versions/process-groups/${PROD_PG_ID}" \
  -H "Authorization: Bearer ${NIFI_TOKEN}" \
  -H "Content-Type: application/json" \
  -d "{
    \"processGroupRevision\": {\"version\": ${PG_REV}},
    \"versionedFlow\": {
      \"registryId\": \"prod-registry-client\",
      \"bucketId\":   \"${PROD_BUCKET_ID}\",
      \"flowId\":     \"${PROD_FLOW_ID}\",
      \"version\":    ${TARGET_VERSION}
    }
  }"

# Restart
curl -s -X PUT "${NIFI_PROD}/nifi-api/flow/process-groups/${PROD_PG_ID}" \
  -H "Authorization: Bearer ${NIFI_TOKEN}" \
  -d '{"id":"'${PROD_PG_ID}'","state":"RUNNING"}'

echo "✓ Rollback complete. Running v${TARGET_VERSION}"
⚠️ Always stop the Process Group before rolling back. FlowFiles in the queue stay intact — the processor configuration changes atomically, but the queue contents from the broken version may need manual review in the Data Provenance / Queue viewer.
