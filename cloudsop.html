Building & Running a Cloud Data Platform on AWS with Terraform

A complete, plain-language guide and Standard Operating Procedure (SOP) for the Cloud team. It explains what we run, why we run it, and the exact steps to build, change, and look after it safely.

Audience: Cloud / Platform Engineers, Data Engineers, On-call · Reading level: beginner-friendly · Last reviewed: 2026

00

Start Here

Imagine our company has a lot of data — customer records, website clicks, sensor readings, log files. A data application unit is the team and the software that collect, store, search, and make sense of all that data. The Cloud team builds and maintains the "ground" that all of this software stands on: the servers, the networks, the databases, the security, and the alarms that tell us when something breaks.

Instead of clicking buttons by hand in the AWS website (which is slow and easy to get wrong), we describe our whole setup in text files using a tool called Terraform. Those text files are like a recipe. Anyone on the team can read the recipe, suggest changes, and re-cook the exact same meal every time. This idea is called Infrastructure as Code.

The big idea in one sentence

We write down our cloud setup as code, review it like any other code, and let automation build it — so the same system can be created again and again, exactly the same, with no guesswork.

Why this matters (the problems it solves)

PROBLEM 01

"It worked on my machine"

Code makes every environment — dev, test, production — match. No more mystery differences.

PROBLEM 02

Slow, risky changes

A reviewed code change is safer than someone clicking in a console at midnight.

PROBLEM 03

"How is this built?"

The code is the documentation. New teammates read it to understand everything.

01

Key Concepts, Explained Simply

Before the steps, here are the building-block words. Tap any i through the guide for a deeper pop-up.

Cloud / AWS
Renting computers and services over the internet from Amazon instead of buying our own machines. We only pay for what we use.
Terraform
The tool that reads our recipe files and tells AWS to create, change, or delete things to match.
Module
A reusable, pre-written recipe for one thing (like a database). We use trusted public modules so we don't write everything from scratch.
State file
Terraform's memory. It records what it has already built so it knows what to change next time. We keep it safe in the cloud, not on a laptop.
Environment
A separate copy of the system. dev is for trying things, staging is for final testing, prod is the real one customers use.
Plan & Apply
plan = a preview of what would change. apply = actually make the change. Always read the plan first.

What the Cloud Team Must Support

This is the full list of jobs the Cloud team is responsible for when running a data application unit. Think of it as our "menu of duties." Each card groups related tasks.

A · Provisioning & Environments

  • ▸ Build & tear down dev / staging / prod that match
  • ▸ Networking: VPC, subnets, security groups, private links

  • ▸ Secrets & encryption keys (rotation included)

B · Data Services Lifecycle

  • ▸ Streaming (Kafka), Search (OpenSearch)
  • ▸ Databases (Postgres), Compute (Spark), Flow (NiFi)
  • ▸ Storage: EFS, EBS, S3 data lake
  • ▸ Manage topics, indexes, schemas, partitions

C · Reliability & Operations

  • ▸ Backups, snapshots, point-in-time recovery
  • ▸ Disaster recovery across zones / regions
  • ▸ Patching & version upgrades
  • ▸ Capacity planning & autoscaling

D · Observability

  • ▸ Metrics (Prometheus) & dashboards (Grafana)
  • ▸ Logs & traces collection
  • ▸ Alerts, SLOs, on-call routing
  • ▸ Runbooks for common problems

E · Security & Compliance

  • ▸ Encryption in transit & at rest, everywhere
  • ▸ Least-privilege access & audit logs
  • ▸ Data classification, retention, lineage
  • ▸ Regular access reviews

F · Enablement & AI Platform

  • ▸ Reusable module registry & "golden paths"
  • ▸ CI/CD for infra, drift detection, policy-as-code
  • ▸ GPU compute, model hosting, vector search
  • ▸ Embedding pipelines & agent runtimes

Rule of thumb: if a task is done more than once, or must be the same across environments, it belongs in Terraform — not in a person's memory or a console click.

How the Pieces Fit Together

We build in layers, like a cake. Each layer sits on the one below it. We keep each layer's Terraform memory (state) separate, so a mistake in one layer can't accidentally break another. This is called limiting the "blast radius."

Build bottom-up · Destroy top-down

Recommended folder layout

Keep reusable recipes in modules/ and per-environment settings in envs/. Each environment just fills in the blanks with its own values file.

# Our infrastructure repository
infra/
├── modules/            # reusable wrappers (we write once)
│   ├── network/       # VPC, subnets, security groups
│   ├── eks/           # Kubernetes cluster
│   ├── msk/           # Kafka
│   ├── opensearch/    # search + vector index
│   ├── aurora/        # Postgres database
│   ├── storage/       # EFS, EBS defaults, S3 lake
│   └── observability/ # Prometheus + Grafana
├── envs/
│   ├── dev/      # { main.tf, dev.tfvars, backend.tf }
│   ├── staging/
│   └── prod/
└── policies/          # automated safety rules (OPA / Sentinel)
04

Service Catalog & Terraform Examples

For each data service we list: what it does (in plain words), the trusted module to use, a starter example, and safety notes. Always pin a version (the version = line) so an upgrade never surprises us. Tap i for deeper notes and official docs.

Kafka — Amazon MSK

Streaming

Kafka is like a super-fast, reliable conveyor belt for data. Apps drop messages on one end ("producers") and other apps pick them up on the other end ("consumers") in real time. MSK is Amazon's managed version, so we don't babysit the servers. 

module "msk" {
  source  = "terraform-aws-modules/msk-kafka-cluster/aws"
  version = "~> 2.0"              # pin the version

  name                   = "data-platform-${var.env}"
  kafka_version          = "3.6.0"
  number_of_broker_nodes = 6     # must be a multiple of how many AZs you use
  enhanced_monitoring    = "PER_TOPIC_PER_PARTITION"

  broker_node_client_subnets = module.network.private_subnets
  broker_node_storage_info = {
    ebs_storage_info = { volume_size = 500 }
  }
}

Safety notes: use 3 copies of each message (replication factor 3) and require 2 copies confirmed before success, prefer IAM login over passwords, and send logs to CloudWatch + S3. A cluster takes 15–30 minutes to build, so set long timeouts.

Search & Indexing — OpenSearch

Search · Logs · Vectors

OpenSearch is a giant, smart index — like the index at the back of a book, but for millions of documents. It powers fast search, log analysis, and vector search for AI (finding things by meaning, not just exact words). One cluster can do double duty: app search and our logs. 

module "opensearch" {
  source  = "terraform-aws-modules/opensearch/aws"
  version = "~> 1.0"

  cluster_config = {
    instance_count           = 3
    instance_type            = "r6g.large.search"
    dedicated_master_enabled = true          # stable "managers"
    zone_awareness_enabled   = true          # spread across AZs
    zone_awareness_config    = { availability_zone_count = 3 }
  }
  auto_tune_options = { desired_state = "ENABLED" }
}

One-time setup: create the service-linked role first with aws iam create-service-linked-role --aws-service-name es.amazonaws.com. For company login (Okta/Azure AD), use the idealo SAML module.

Postgres — RDS / Aurora

Relational Database

A relational database stores neat tables of data (rows and columns) — like a giant, reliable spreadsheet that many apps share at once. Aurora is Amazon's faster, self-healing version of Postgres. Serverless v2 can even shrink to almost nothing when idle to save money. 

module "aurora_postgresql" {
  source  = "terraform-aws-modules/rds-aurora/aws"
  version = "~> 10.0"

  name           = "data-${var.env}"
  engine         = "aurora-postgresql"
  engine_version = "16"
  instance_class = "db.serverless"            # Serverless v2
  instances      = { writer = {}, reader = {} } # 1 writes, 1 reads

  vpc_id            = module.network.vpc_id
  storage_encrypted = true                    # always on
}

Safety notes: turn on encryption and automated backups, export Postgres logs to CloudWatch, and for disaster recovery use the AWS global-database module to copy to a second region.

Aurora module ↗ Cloud Posse ↗ Aurora Docs ↗

Spark & NiFi — Data on EKS

Compute · Data Flow

Spark is a powerful engine for crunching huge amounts of data quickly by splitting the work across many machines. NiFi is a drag-and-drop tool for moving and transforming data between systems. Both run nicely on EKS (Amazon's managed Kubernetes). AWS's "Data on EKS" project gives ready-made Terraform blueprints. 

# Foundation: the Kubernetes cluster everything runs on
module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 20.0"

  cluster_name    = "data-${var.env}"
  cluster_version = "1.30"
  # Then add Spark Operator / NiFi via Data-on-EKS blueprints
}

Choosing Spark: EMR Serverless = zero cluster management; Spark Operator on EKS = Kubernetes-native alongside other jobs. The NiFi blueprint wires up Prometheus + Grafana automatically.

Storage — EFS, EBS & S3 Data Lake

Storage

Three kinds of storage, three jobs: EBS is a hard drive attached to one machine. EFS is a shared folder many machines can use at once. S3 is a near-infinite bucket for files — perfect for a data lake (raw → cleaned → ready-to-use zones). 

module "efs" {
  source  = "terraform-aws-modules/efs/aws"
  version = "~> 1.0"
  name            = "shared-${var.env}"
  encrypted       = true
  attach_policy   = true
}
# EBS: usually handled by the service that needs it (RDS, EKS CSI).
# Prefer gp3 volumes, always encrypted.
# S3 lake: one bucket per zone (raw / curated / consumption) + Glue Catalog.

Safety notes: encrypt every volume and bucket, block all public access on S3, and use Lake Formation for table- and column-level permissions on the lake.

EFS module ↗ S3 module ↗ Lake Formation ↗

Monitoring — Prometheus & Grafana

Observability

Prometheus collects numbers about how healthy our systems are (CPU, memory, request counts). Grafana turns those numbers into clear charts and dashboards. The AWS Observability Accelerator sets up managed versions of both, plus the collector, in one module. 

module "observability" {
  source     = "aws-observability/terraform-aws-observability-accelerator/aws"
  aws_region     = var.region
  eks_cluster_id = module.eks.cluster_name
  # Creates: Managed Prometheus + Managed Grafana + OpenTelemetry collector
}

Tip: managed (AMP + AMG) means less upkeep; self-hosted (Prometheus/Grafana in EKS via Helm) gives more control. Connect Grafana data sources to CloudWatch, Prometheus, and X-Ray; log in with single sign-on.

Accelerator ↗ Self-hosted ↗ Managed Grafana ↗
05

AI, Analytics & Search Infrastructure

Terraform builds the platform; the AI tools run on top of it. The trick is to reuse what we already have. For example, the same OpenSearch cluster that stores logs can also be our vector store for AI search. One EKS cluster can host Spark, NiFi, model servers, and agents together — one set of rules to manage instead of five.

fastText

A lightweight tool that turns words into numbers ("embeddings") and sorts text into categories — fast, on a normal CPU, no expensive GPU needed. Great for tagging huge piles of documents and detecting languages before heavier models run.

Ollama

Runs open-weight AI language models on our own servers. Because the data never leaves our network, it's a strong fit when privacy rules matter. We run it on a GPU node group inside EKS and keep it private to the VPC.

Vector search

Finds things by meaning, not exact words. We store embeddings in OpenSearch's k-NN engine, so "car" can match "automobile." Embedding pipelines run as Spark or Ray jobs that write vectors into the index.

Agents (Python / PyTorch / TensorFlow)

Small programs that take actions for us. We package them as containers on EKS, give each a tightly-scoped IAM role (read this, write that — nothing more), and use Karpenter to add GPUs only when needed so idle cost stays near zero.

⚠ Guardrail for AI agents in operations

Agents may read freely (logs, metrics, cost reports) and draft suggestions, but they must never run terraform apply or change production on their own. A human reviews and approves every change. Give agents read-only roles by default.

Standard Operating Procedure — Step by Step

This is the official routine for making any infrastructure change. Follow it every time, even for small changes. It keeps production safe and changes reviewable.

Part A — One-time setup (per engineer)

1

Install the tools

Install Terraform, the AWS CLI, and git. Then check they work.

terraform -version
aws --version
git --version
2

Log in to AWS securely

Use single sign-on. Never paste long-lived secret keys into files.

aws sso login --profile data-platform
3

Get the code

Copy the infrastructure repository to your machine.

git clone https://git.company.com/cloud/infra.git
cd infra

Part B — Making a change (every time)

1

Start a fresh branch

Never edit the main branch directly. Give your branch a clear name.

git checkout -b add-opensearch-vectors
2

Move into the right environment & edit

Always start with dev. Edit the .tf files or the values file.

cd envs/dev
# edit main.tf or dev.tfvars in your editor
3

Initialize & format

init downloads the modules. fmt tidies spacing. validate checks for typos.

terraform init
terraform fmt
terraform validate
4

Preview with plan — and read it carefully

This shows what will be added (+), changed (~), or destroyed (−). If you see unexpected "destroy" on production data, STOP.

terraform plan -out=tfplan
5

Open a pull request for review

Push your branch and ask a teammate to review. Automated checks (security scan, policy rules) run here.

git add . && git commit -m "Add OpenSearch vector index"
git push origin add-opensearch-vectors
6

Apply to dev after approval

Once approved, apply the exact plan you previewed. Then test that it works.

terraform apply tfplan
7

Promote: staging → production

Repeat steps 2–6 in staging, then prod. Production applies should require a second approver and run from CI, not a laptop.

cd ../staging && terraform plan -out=tfplan && terraform apply tfplan
# then envs/prod with extra approval

The golden rules

  • ① Always plan before apply, and read every line.
  • ② dev → staging → prod, never skip straight to prod.
  • ③ Every change goes through a reviewed pull request.
  • ④ Never store secrets in code; use Secrets Manager.
  • ⑤ If a plan wants to destroy data, stop and ask.
07

Operational Runbooks

A runbook is a short, agreed list of steps for a common situation. Keep them calm and simple — they're read during stressful moments.

▸ "Terraform state is locked"

Someone else is running Terraform, or a run crashed. Wait first. If you're sure no one is running it, confirm the lock owner and unlock carefully with terraform force-unlock <LOCK_ID>. Never force-unlock during an active apply.

▸ "The plan wants to destroy a database"

STOP. Do not apply. This usually means a name or setting changed in a way Terraform reads as "replace." Check recent edits, confirm with a teammate, and use lifecycle { prevent_destroy = true } on critical resources to block accidents.

▸ "Real setup doesn't match the code (drift)"

Someone changed something by hand in the console. Run terraform plan to see the difference, then decide: re-apply the code to fix it, or update the code to keep the manual change. Then ask everyone to stop manual edits.

▸ "An upgrade broke something"

Because modules are version-pinned, revert the version = line to the previous value, run plan/apply, and confirm recovery. Investigate the new version in dev before trying again.

▸ "Costs spiked unexpectedly"

Check AWS Cost Explorer and tags to find the source. Common causes: oversized instances, idle GPUs, or data transfer. Right-size in dev first, then promote the change.

Best Practices Checklist

Code & structure

  • ✓ Prefer trusted registry modules over hand-written resources
  • ✓ Pin every module & provider version
  • ✓ Separate state per layer (network / platform / apps)
  • ✓ One module per service, environments via .tfvars
  • ✓ Keep state remote (S3 + DynamoDB lock), never on laptops

Safety & review

  • ✓ Plan before apply, always read the output
  • ✓ Pull-request review for every change
  • ✓ Policy-as-code (OPA / Sentinel) + scans (checkov, tflint)
  • ✓ prevent_destroy on critical data stores
  • ✓ Production applies run from CI with a second approver

Security

  • ✓ Encryption in transit & at rest, on by default
  • ✓ Least-privilege IAM; use IRSA on EKS
  • ✓ Secrets in Secrets Manager / SSM, never in code
  • ✓ Private networking + VPC endpoints
  • ✓ Audit logging on (CloudTrail) & regular access reviews

Reliability & cost

  • ✓ Backups, snapshots & tested disaster recovery
  • ✓ Spread across Availability Zones
  • ✓ Autoscaling; Karpenter for just-in-time GPUs
  • ✓ Consistent tagging for cost & ownership
  • ✓ Budgets & anomaly alerts

Glossary

VPC — your private, fenced-off network inside AWS.

Subnet — a smaller section of that network; public or private.

IAM — the system that decides who can do what.

IRSA — gives each Kubernetes app its own scoped AWS permissions.

EKS — Amazon's managed Kubernetes for running containers.

Availability Zone (AZ) — a separate data-center building; use several for safety.

Embedding — text turned into a list of numbers capturing meaning.

k-NN — "nearest neighbor" search; finds the most similar items.

CI/CD — automation that tests and ships changes.

Drift — when real infrastructure no longer matches the code.

Blast radius — how much can break if one thing goes wrong.

Policy-as-code — safety rules written as code that block bad changes.
