Infrastructure Guide

Running Apache Kafka on a Kubernetes Cluster in AWS

A complete, hands-on tutorial for standing up a Kafka cluster on Kubernetes (self-managed on EC2), with three parallel infrastructure paths — manual, CloudFormation, and Terraform — plus broker, producer, and consumer workloads running on basic EC2 instances.

1. Overview ›

Apache Kafka is a distributed event-streaming platform. Running it on Kubernetes gives you declarative scaling, self-healing pods, and rolling upgrades — but Kafka is stateful, so persistent storage, stable network identity (StatefulSets), and careful broker configuration matter.

In this guide the Kubernetes cluster runs on self-managed EC2 instances (not EKS-managed nodes), using kubeadm. This keeps the example portable across the three IaC options and uses only "basic" EC2 instances.

What you'll build

  • A VPC with public/private subnets across two Availability Zones.
  • 3 EC2 instances forming a kubeadm Kubernetes cluster (1 control-plane, 2 workers).
  • A 3-broker Kafka StatefulSet (KRaft mode — no ZooKeeper) with EBS-backed persistent volumes.
  • Producer and consumer worker Deployments that stream messages through the brokers.
Cost note: Self-managed EC2 + EBS incurs hourly charges. Tear everything down when finished (each option below includes a teardown step).

4. Three Ways to Provision the Infrastructure ›

All three options build the same target: a VPC, a security group, and 3 EC2 instances ready for kubeadm. Pick the tab that fits your workflow.

Option A — Manual setup with the AWS CLI

Best for learning exactly what each resource is. You run discrete commands and wire IDs together yourself.

1. Create the VPC and subnet

VPC_ID=$(aws ec2 create-vpc --cidr-block 10.0.0.0/16 \
  --query 'Vpc.VpcId' --output text)
aws ec2 modify-vpc-attribute --vpc-id $VPC_ID --enable-dns-hostnames

SUBNET_ID=$(aws ec2 create-subnet --vpc-id $VPC_ID \
  --cidr-block 10.0.1.0/24 --availability-zone us-east-1a \
  --query 'Subnet.SubnetId' --output text)

2. Internet gateway + route

IGW_ID=$(aws ec2 create-internet-gateway \
  --query 'InternetGateway.InternetGatewayId' --output text)
aws ec2 attach-internet-gateway --vpc-id $VPC_ID --internet-gateway-id $IGW_ID

RT_ID=$(aws ec2 create-route-table --vpc-id $VPC_ID \
  --query 'RouteTable.RouteTableId' --output text)
aws ec2 create-route --route-table-id $RT_ID \
  --destination-cidr-block 0.0.0.0/0 --gateway-id $IGW_ID
aws ec2 associate-route-table --subnet-id $SUBNET_ID --route-table-id $RT_ID

3. Security group (Kafka + k8s ports)

SG_ID=$(aws ec2 create-security-group --group-name kafka-k8s-sg \
  --description "Kafka k8s" --vpc-id $VPC_ID --query 'GroupId' --output text)
# SSH, k8s API, kubelet, Kafka, NodePorts
for P in 22 6443 10250 2379-2380 9092-9094 30000-32767; do
  aws ec2 authorize-security-group-ingress --group-id $SG_ID \
    --protocol tcp --port $P --cidr 10.0.0.0/16; done
aws ec2 authorize-security-group-ingress --group-id $SG_ID \
  --protocol tcp --port 22 --cidr 0.0.0.0/0

4. Launch 3 EC2 instances

# Amazon Linux 2023 AMI, t3.medium (basic, 2 vCPU / 4 GiB)
AMI=$(aws ec2 describe-images --owners amazon \
  --filters "Name=name,Values=al2023-ami-*-x86_64" \
  --query 'sort_by(Images,&CreationDate)[-1].ImageId' --output text)

aws ec2 run-instances --image-id $AMI --count 3 \
  --instance-type t3.medium --key-name kafka-k8s \
  --security-group-ids $SG_ID --subnet-id $SUBNET_ID \
  --associate-public-ip-address \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Project,Value=kafka-k8s}]'

Teardown

aws ec2 describe-instances --filters Name=tag:Project,Values=kafka-k8s \
  --query 'Reservations[].Instances[].InstanceId' --output text \
  | xargs aws ec2 terminate-instances --instance-ids
# then delete SG, subnet, IGW, VPC in reverse order

Option B — AWS CloudFormation

Declarative AWS-native IaC. One template, one create-stack call, full rollback on failure.

1. Save the template as kafka-k8s.yaml

AWSTemplateFormatVersion: '2010-09-09'
Description: Kafka K8s cluster - 3 EC2 nodes
Parameters:
  KeyName: { Type: AWS::EC2::KeyPair::KeyName }
  InstanceType: { Type: String, Default: t3.medium }
  LatestAmi:
    Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
    Default: /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64
Resources:
  Vpc:
    Type: AWS::EC2::VPC
    Properties: { CidrBlock: 10.0.0.0/16, EnableDnsHostnames: true }
  Subnet:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref Vpc
      CidrBlock: 10.0.1.0/24
      MapPublicIpOnLaunch: true
  Igw: { Type: AWS::EC2::InternetGateway }
  Attach:
    Type: AWS::EC2::VPCGatewayAttachment
    Properties: { VpcId: !Ref Vpc, InternetGatewayId: !Ref Igw }
  Rt:
    Type: AWS::EC2::RouteTable
    Properties: { VpcId: !Ref Vpc }
  Route:
    Type: AWS::EC2::Route
    DependsOn: Attach
    Properties:
      RouteTableId: !Ref Rt
      DestinationCidrBlock: 0.0.0.0/0
      GatewayId: !Ref Igw
  RtAssoc:
    Type: AWS::EC2::SubnetRouteTableAssociation
    Properties: { SubnetId: !Ref Subnet, RouteTableId: !Ref Rt }
  Sg:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: kafka-k8s
      VpcId: !Ref Vpc
      SecurityGroupIngress:
        - { IpProtocol: tcp, FromPort: 22,   ToPort: 22,   CidrIp: 0.0.0.0/0 }
        - { IpProtocol: tcp, FromPort: 6443, ToPort: 6443, CidrIp: 10.0.0.0/16 }
        - { IpProtocol: tcp, FromPort: 9092, ToPort: 9094, CidrIp: 10.0.0.0/16 }
        - { IpProtocol: tcp, FromPort: 30000,ToPort: 32767,CidrIp: 10.0.0.0/16 }
  ControlPlane:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !Ref LatestAmi
      InstanceType: !Ref InstanceType
      KeyName: !Ref KeyName
      SubnetId: !Ref Subnet
      SecurityGroupIds: [ !Ref Sg ]
      Tags: [ { Key: Role, Value: control-plane } ]
  Worker1:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !Ref LatestAmi
      InstanceType: !Ref InstanceType
      KeyName: !Ref KeyName
      SubnetId: !Ref Subnet
      SecurityGroupIds: [ !Ref Sg ]
      Tags: [ { Key: Role, Value: worker } ]
  Worker2:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !Ref LatestAmi
      InstanceType: !Ref InstanceType
      KeyName: !Ref KeyName
      SubnetId: !Ref Subnet
      SecurityGroupIds: [ !Ref Sg ]
      Tags: [ { Key: Role, Value: worker } ]
Outputs:
  ControlPlaneIp:
    Value: !GetAtt ControlPlane.PublicIp

2. Deploy the stack

aws cloudformation create-stack \
  --stack-name kafka-k8s \
  --template-body file://kafka-k8s.yaml \
  --parameters ParameterKey=KeyName,ParameterValue=kafka-k8s \
  --capabilities CAPABILITY_NAMED_IAM

aws cloudformation wait stack-create-complete --stack-name kafka-k8s
aws cloudformation describe-stacks --stack-name kafka-k8s \
  --query 'Stacks[0].Outputs'

Teardown

aws cloudformation delete-stack --stack-name kafka-k8s
aws cloudformation wait stack-delete-complete --stack-name kafka-k8s

Option C — Terraform

Cloud-agnostic IaC with a rich provider ecosystem and a plan/apply workflow. Save as main.tf.

terraform {
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }
}
provider "aws" { region = "us-east-1" }

variable "key_name"      { default = "kafka-k8s" }
variable "instance_type" { default = "t3.medium" }

data "aws_ssm_parameter" "ami" {
  name = "/aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64"
}

resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  tags = { Name = "kafka-k8s" }
}

resource "aws_subnet" "main" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  map_public_ip_on_launch = true
}

resource "aws_internet_gateway" "igw" { vpc_id = aws_vpc.main.id }

resource "aws_route_table" "rt" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }
}
resource "aws_route_table_association" "a" {
  subnet_id      = aws_subnet.main.id
  route_table_id = aws_route_table.rt.id
}

resource "aws_security_group" "sg" {
  name   = "kafka-k8s-sg"
  vpc_id = aws_vpc.main.id
  ingress { from_port = 22   to_port = 22   protocol = "tcp" cidr_blocks = ["0.0.0.0/0"] }
  ingress { from_port = 6443 to_port = 6443 protocol = "tcp" cidr_blocks = ["10.0.0.0/16"] }
  ingress { from_port = 9092 to_port = 9094 protocol = "tcp" cidr_blocks = ["10.0.0.0/16"] }
  ingress { from_port = 30000 to_port = 32767 protocol = "tcp" cidr_blocks = ["10.0.0.0/16"] }
  egress  { from_port = 0 to_port = 0 protocol = "-1" cidr_blocks = ["0.0.0.0/0"] }
}

locals { roles = { cp = "control-plane", w1 = "worker", w2 = "worker" } }

resource "aws_instance" "node" {
  for_each               = local.roles
  ami                    = data.aws_ssm_parameter.ami.value
  instance_type          = var.instance_type
  key_name               = var.key_name
  subnet_id              = aws_subnet.main.id
  vpc_security_group_ids = [aws_security_group.sg.id]
  tags = { Name = "kafka-${each.key}", Role = each.value }
}

output "control_plane_ip" {
  value = aws_instance.node["cp"].public_ip
}

Run it

terraform init
terraform plan
terraform apply -auto-approve
terraform output control_plane_ip

Teardown

terraform destroy -auto-approve
5. Bootstrap Kubernetes & Deploy Kafka ›

5.1 Initialize the cluster with kubeadm

SSH into the control-plane EC2 instance, install a container runtime + kubeadm, then run the init. Repeat the install steps on each worker and join.

# on every node
sudo dnf install -y containerd kubelet kubeadm kubectl
sudo systemctl enable --now containerd kubelet
sudo swapoff -a

# control plane only
sudo kubeadm init --pod-network-cidr=192.168.0.0/16
mkdir -p $HOME/.kube && sudo cp /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.27.0/manifests/calico.yaml

# workers: run the kubeadm join command printed by init
sudo kubeadm join <CP_IP>:6443 --token ... --discovery-token-ca-cert-hash ...

5.2 Storage class for EBS

cat <<EOF | kubectl apply -f -
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata: { name: ebs-gp3 }
provisioner: ebs.csi.aws.com
parameters: { type: gp3 }
volumeBindingMode: WaitForFirstConsumer
EOF
Install the AWS EBS CSI driver and attach an IAM policy to the node role so pods can dynamically provision EBS volumes.

5.3 Kafka StatefulSet (KRaft, 3 brokers)

apiVersion: v1
kind: Service
metadata: { name: kafka-svc }
spec:
  clusterIP: None        # headless: stable DNS per broker
  selector: { app: kafka }
  ports: [ { port: 9092, name: client } ]
---
apiVersion: apps/v1
kind: StatefulSet
metadata: { name: kafka }
spec:
  serviceName: kafka-svc
  replicas: 3
  selector: { matchLabels: { app: kafka } }
  template:
    metadata: { labels: { app: kafka } }
    spec:
      containers:
      - name: kafka
        image: apache/kafka:3.7.0
        ports: [ { containerPort: 9092 } ]
        env:
        - name: KAFKA_NODE_ID
          valueFrom: { fieldRef: { fieldPath: metadata.name } }
        - name: KAFKA_PROCESS_ROLES
          value: "broker,controller"
        - name: KAFKA_LISTENERS
          value: "PLAINTEXT://:9092,CONTROLLER://:9093"
        volumeMounts:
        - { name: data, mountPath: /var/lib/kafka/data }
  volumeClaimTemplates:
  - metadata: { name: data }
    spec:
      accessModes: [ "ReadWriteOnce" ]
      storageClassName: ebs-gp3
      resources: { requests: { storage: 10Gi } }
kubectl apply -f kafka-statefulset.yaml
kubectl get pods -l app=kafka -w   # kafka-0, kafka-1, kafka-2 → Running
6. Brokers, Producer & Consumer Workers ›

The three pods above are the brokers. Now add a producer Deployment that publishes messages and a consumer Deployment that reads them — both scheduled onto the basic EC2 worker nodes.

6.1 Create a topic

kubectl exec -it kafka-0 -- /opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server kafka-svc:9092 \
  --create --topic events --partitions 3 --replication-factor 3

6.2 Producer Deployment

apiVersion: apps/v1
kind: Deployment
metadata: { name: producer }
spec:
  replicas: 2
  selector: { matchLabels: { app: producer } }
  template:
    metadata: { labels: { app: producer } }
    spec:
      containers:
      - name: producer
        image: apache/kafka:3.7.0
        command: ["/bin/sh","-c"]
        args:
        - while true; do echo "msg-$(date +%s)" | \
            /opt/kafka/bin/kafka-console-producer.sh \
            --bootstrap-server kafka-svc:9092 --topic events;
            sleep 2; done

6.3 Consumer Deployment

apiVersion: apps/v1
kind: Deployment
metadata: { name: consumer }
spec:
  replicas: 2
  selector: { matchLabels: { app: consumer } }
  template:
    metadata: { labels: { app: consumer } }
    spec:
      containers:
      - name: consumer
        image: apache/kafka:3.7.0
        command: ["/bin/sh","-c"]
        args:
        - /opt/kafka/bin/kafka-console-consumer.sh \
            --bootstrap-server kafka-svc:9092 \
            --topic events --group worker-group
kubectl apply -f producer.yaml -f consumer.yaml
kubectl get pods -o wide   # confirm spread across EC2 worker nodes
7. Verify & Test the Pipeline ›

Watch consumed messages

kubectl logs -f deploy/consumer
# expect: msg-1717000000, msg-1717000002, ...

Check broker & topic health

kubectl exec -it kafka-0 -- /opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server kafka-svc:9092 --describe --topic events
# verify Leader, Replicas, Isr columns are populated for all 3 partitions
Resilience test: delete kafka-1 with kubectl delete pod kafka-1; the StatefulSet recreates it, reattaches its EBS volume, and the consumer keeps receiving messages.
