2026-ready recommendation

Build reusable AWS data infrastructure that data users can safely use.

This page recommends a support model for Linux AMIs, custom golden images, service-specific AMIs, Ansible, GitLab CI/CD, AWS SSM, Terraform, CloudFormation, clustering, load management, failover, fault tolerance, and manual backup operations for Kafka, ZooKeeper, NiFi, PostgreSQL, OpenSearch, and ETL worker servers.

The goal is to provide approved infrastructure that data users can request and use without needing deep AWS administrator knowledge.

Data users should get

  • ETL worker servers
  • NiFi development or production nodes
  • Kafka topics and broker access
  • PostgreSQL schemas or databases
  • OpenSearch indexes and dashboards
  • S3 landing, raw, processed, and archive buckets
  • Backup restore support

Platform team should provide

  • Approved AMIs
  • Terraform modules
  • CloudFormation templates
  • Ansible playbooks
  • GitLab CI/CD templates
  • AWS Systems Manager runbooks
  • Clear backup and restore procedures

Recommended flow: user request → approval → GitLab pipeline → Terraform or CloudFormation → EC2 from approved AMI → Ansible/SSM configuration → monitored data service.

AMI creation and lifecycle support must be a core capability. Do not treat AMIs as a one-time server copy. Treat them as versioned platform products.

AMI build process

  1. Select latest approved Amazon Linux 2023 AMI.
  2. Build with EC2 Image Builder or Packer.
  3. Run Ansible roles for baseline and service setup.
  4. Patch OS and packages.
  5. Install SSM Agent, CloudWatch Agent, security tools, and backup helpers.
  6. Run validation tests.
  7. Create and tag AMI.
  8. Scan image.
  9. Promote only after approval.
  10. Store latest approved AMI ID in SSM Parameter Store.

Use Ansible for both image baking and post-launch configuration. Keep secrets and environment-specific values outside the AMI.

Data infrastructure should be designed for node failure, Availability Zone failure, software failure, bad deployments, storage limits, and network interruptions. The support team should define RTO, how quickly the service must recover, and RPO, how much data loss is acceptable.

High Availability

Run across at least two or three Availability Zones, avoid single EC2 nodes for critical services, and use health checks.

Fault Tolerance

Design so one node, disk, broker, processor, or AZ can fail without taking down the platform.

Failover

Use automated failover when available, and documented manual failover when full automation is too risky.

Recommended AWS Cluster Pattern

VPC
  Public subnets:
    - Load balancers only, if public access is required

  Private app/data subnets across 3 AZs:
    - Kafka brokers or MSK brokers
    - NiFi cluster nodes
    - OpenSearch nodes or managed OpenSearch
    - ETL worker Auto Scaling group
    - RDS / Aurora Multi-AZ database

  Shared services:
    - Route 53 private DNS
    - AWS Systems Manager
    - CloudWatch logs and metrics
    - AWS Backup
    - Secrets Manager / Parameter Store
    - S3 backup and archive buckets

Service-by-Service Fault Tolerance Guidance

Service Cluster Recommendation Failover / Tolerance Notes Team Warning
Kafka Minimum 3 brokers across 3 AZs for production. Use replication factor 3 for important topics. Partition leadership moves if a broker fails. Monitor under-replicated partitions and consumer lag. Do not run critical Kafka on one EC2 instance. Bad partition design can overload one broker.
ZooKeeper Use odd-number quorum, usually 3 or 5 nodes. Cluster remains healthy if quorum survives. Three nodes can lose one node; five nodes can lose two. Avoid 2-node ZooKeeper. Split-brain and quorum loss are common design risks.
NiFi Use 3+ NiFi nodes behind an internal load balancer. Use NiFi Registry for versioned flows. Cluster Coordinator and Primary Node roles can move. Repositories need careful disk planning. Content, FlowFile, and Provenance repositories can fill disks quickly and stop flows.
PostgreSQL Prefer RDS Multi-AZ or Aurora PostgreSQL for production. Managed failover reduces operational burden. Use read replicas for read scaling, not as the only backup. Self-managed PostgreSQL failover is complex and needs tested runbooks.
OpenSearch Prefer managed OpenSearch with dedicated masters and data nodes across AZs. Use replicas, shard allocation awareness, snapshots, and health checks. Too many shards or undersized JVM memory causes instability.
ETL Workers Use Auto Scaling groups, ECS, EKS, AWS Batch, Glue, or EMR depending on workload. Workers should be replaceable. Store state in S3, databases, queues, or managed services. Do not keep important state only on worker local disks.

Load management means keeping data services stable when usage grows, jobs spike, or one component slows down. The team should monitor throughput, latency, queue depth, disk usage, CPU, memory, network, and error rates.

Horizontal scaling

Add nodes, brokers, workers, or replicas. Best for Kafka, NiFi clusters, ETL workers, OpenSearch data nodes, ECS/EKS jobs, and stateless services.

Vertical scaling

Use larger instances, more memory, faster disks, or better network. Useful for PostgreSQL, OpenSearch memory pressure, and heavy NiFi repositories.

Load Balancer Options

Option Best For Pros Cons
Application Load Balancer NiFi UI/API, internal web apps, REST services. Layer 7 routing, TLS termination, path routing, health checks. Not ideal for all raw TCP workloads.

Simple Team Rule

Use managed services when the business needs reliability faster than deep customization. Use EC2 with custom AMIs when the team needs full control, special software, custom agents, unusual networking, or deep file-system access. Use containers when workloads are portable, job-based, or need fast scaling.

Use GitLab protected branches, merge approvals, IaC scanning, Ansible linting, and OIDC federation to AWS. Avoid long-lived AWS access keys in GitLab variables.

AWS Systems Manager should be the main operations control plane for EC2-based data infrastructure.

  • Session Manager: shell access without public SSH.
  • Run Command: execute approved scripts on nodes.
  • Automation: guided AMI creation, backups, restores, and patching.
  • Patch Manager: patch baseline and maintenance windows.
  • Parameter Store: AMI IDs, non-secret config, environment values.
  • Inventory: package and system visibility.
  • State Manager: desired-state enforcement for agents and configs.

Data User Platform Mindset

The infrastructure should feel like a product. Data users should not need to know every EC2, IAM, VPC, or backup detail. The support team should expose safe choices, approved templates, documented runbooks, and clear service-level expectations.

AMI Lifecycle Details

  • Build monthly and immediately after critical security fixes.
  • Keep the last three approved versions for rollback.
  • Deprecate old AMIs after migration.
  • Never bake static passwords, private keys, API tokens, or environment secrets into AMIs.
  • Tag AMIs with owner, service, patch level, build date, Git commit, Ansible role, and approval state.
  • Store latest approved AMI IDs in SSM Parameter Store.

GitLab to AWS Security

Use OIDC federation so GitLab jobs can assume AWS IAM roles with temporary credentials. This is safer than storing long-lived AWS access keys in GitLab variables. Restrict trust policies by project, branch, environment, and protected refs.

Manual Backup Process

  1. Confirm service owner and change window.
  2. Check disk space, service health, and current alerts.
  3. Run SSM Automation or Run Command backup script.
  4. Create EBS snapshot or application-level export.
  5. Tag backup with service, owner, date, environment, and retention.
  6. Verify backup artifact exists.
  7. Document backup ID, location, and restore command.
  8. Schedule restore test if backup protects production data.

HA, Failover, and Fault Tolerance Details

High availability means the platform stays usable when a normal failure happens. Fault tolerance means the system can absorb failures without data loss or major outage. Failover is the process of moving traffic or leadership from a failed component to a healthy one.

  • Use at least two Availability Zones for non-critical systems and three Availability Zones for important production clusters.
  • For Kafka, design topics with replication factor, partition count, and minimum in-sync replicas that match the business criticality.
  • For ZooKeeper, use odd-number quorums. Never use a two-node quorum for important systems.
  • For NiFi, plan repository disks carefully. Repository disk exhaustion is a common production outage cause.
  • For PostgreSQL, prefer RDS Multi-AZ or Aurora because database failover is difficult to self-manage safely.
  • For OpenSearch, keep shard count reasonable and test snapshot restores.
  • For every service, document who can fail over, when they can fail over, and how to roll back.

Load Management Details

Load problems usually show up as slow pipelines, growing queues, disk pressure, high memory, or users reporting that data is late. The team should set thresholds before production use.

  1. Measure baseline load: normal records per second, average job duration, average query latency, and normal disk growth.
  2. Set alarms: warn before saturation, not after outage.
  3. Control back pressure: NiFi queues, Kafka retention, consumer lag, and worker concurrency should be intentionally configured.
  4. Separate workloads: avoid mixing heavy batch jobs, user queries, and ingestion spikes on the same under-sized nodes.
  5. Test failure: stop a broker, stop a NiFi node, restore a database snapshot, and verify the team knows what happens.

How to Choose the Right Platform Option

Use these questions during architecture reviews:

  • Does AWS offer a managed service that meets the need?
  • Does the team truly need root access, custom plugins, or special OS configuration?
  • What is the recovery time objective and recovery point objective?
  • Can the workload be rebuilt from code, AMI, Ansible, and data backup?
  • Can a data user safely request this through a controlled process?
  • Is the cost of self-managing lower after including patching, support, on-call, backups, and outages?

The safest default is managed services for critical databases, streams, and search platforms; custom AMIs for controlled EC2 patterns; and containers or serverless options for elastic ETL jobs.
