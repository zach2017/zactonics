2026-ready recommendation

Build running AWS data infrastructure that data users can safely use.

This page recommends a support model for secure Linux base AMIs, running EC2 servers, Ansible-based service installation, GitLab CI/CD, AWS SSM, Terraform, CloudFormation, clustering, load management, failover, fault tolerance, and manual backup operations for Kafka, ZooKeeper, NiFi, PostgreSQL, OpenSearch, and ETL worker servers. In this model, the service pipeline uses the secure base AMI and installs/configures Kafka and other services with Ansible at deployment time instead of requiring a separate service AMI for every product.

The goal is to provide approved infrastructure that data users can request and use without needing deep AWS administrator knowledge.

Data users should get

  • ETL worker servers
  • NiFi development or production nodes
  • Kafka topics and broker access
  • PostgreSQL schemas or databases
  • OpenSearch indexes and dashboards
  • S3 landing, raw, processed, and archive buckets
  • Backup restore support

Platform team should provide

  • Approved AMIs
  • Terraform modules
  • CloudFormation templates
  • Ansible playbooks
  • GitLab CI/CD templates
  • AWS Systems Manager runbooks
  • Clear backup and restore procedures

Recommended flow: user request → approval → GitLab pipeline → Terraform or CloudFormation → EC2 servers from approved AMIs → Ansible/SSM runtime configuration → running Kafka/NiFi/OpenSearch/PostgreSQL/ETL services → health checks → monitored data service ready for users.

AMI creation and lifecycle support must be a core capability. Do not treat AMIs as a one-time server copy. Treat them as versioned platform products that are used to launch running, supported EC2 servers and clusters.

Base AMI

Amazon Linux 2023, patched, minimal baseline.

Golden AMI

Security, SSM, CloudWatch, standard users, hardening.

Service AMI

Kafka, ZooKeeper, NiFi, OpenSearch, PostgreSQL tools.

Custom AMI

Project-specific drivers, ETL tools, Java/Python packages.

AMI build process

  1. Select latest approved Amazon Linux 2023 AMI.
  2. Build with EC2 Image Builder or Packer.
  3. Run Ansible roles for baseline and service setup.
  4. Patch OS and packages.
  5. Install SSM Agent, CloudWatch Agent, security tools, and backup helpers.
  6. Run validation tests.
  7. Create and tag AMI.
  8. Scan image.
  9. Promote only after approval.
  10. Store latest approved AMI ID in SSM Parameter Store.

Separate the platform into two major lifecycle tracks. The first track builds the base and golden AMIs. The second track deploys the final service infrastructure, such as Kafka, ZooKeeper, NiFi, PostgreSQL tools, OpenSearch, or ETL workers, by launching EC2 from the secure base AMI and using Ansible to install and configure services at deployment time. This separation keeps security-controlled operating system images stable while allowing service teams to move faster.

Repo 1: Base Image Factory

Purpose: Build Amazon Linux 2023 base AMIs and company golden AMIs.

Owned by: Cloud platform, security, or infrastructure engineering team.

Output: Approved AMI IDs published to SSM Parameter Store.

Repo 2: Service Infrastructure

Purpose: Use the approved secure base AMI to launch servers, then install and configure services with Ansible.

Owned by: Data platform, ETL, or application infrastructure team.

Output: Running Kafka, NiFi, OpenSearch, PostgreSQL tooling, or ETL worker services with health checks and user-ready endpoints.

Why Two Repositories?

  • Security separation: base operating system hardening should be controlled by cloud/security teams.
  • Change control: base AMIs change slower than service infrastructure.
  • Cleaner ownership: infrastructure teams own OS baseline; data teams own service patterns.
  • Safer promotion: dev can test a new golden AMI before production service clusters use it.
  • Better rollback: service teams can roll back to a prior approved AMI parameter.

Where Each Tool Fits

Tool Base Image Repo Service Infrastructure Repo Best Use
EC2 Image Builder Primary AWS-native AMI builder. Optional for service-specific AMI variants. Repeatable AMI pipelines with AWS integration.
Packer Alternative AMI builder. Useful for teams that prefer portable image templates. Cross-cloud or developer-friendly image build process.
Ansible Install baseline packages, hardening, agents. Configure runtime service settings after EC2 launch. Reusable configuration roles across bake and runtime.
Shell Scripts Small bootstrap, validation, package checks. Health checks, simple service tests, emergency scripts. Simple commands where Ansible would be too heavy.
Terraform Provision Image Builder pipelines and SSM parameters. Primary tool for VPC, EC2, ASG, RDS, load balancers, DNS, backups. Reusable modules and strong plan/apply workflow.
CloudFormation Alternative for Image Builder and IAM setup. Alternative for AWS-native stacks and StackSets. AWS-native deployments where Terraform is not preferred.
AWS SSM Store AMI IDs, run image validation, patch support. Runtime config, Run Command, Session Manager, Automation. Operations control plane without public SSH.

Ansible vs Shell vs User Data vs SSM

Option Use When Pros Cons
Ansible Configuration has many steps, templates, handlers, users, packages, and idempotent changes. Readable, reusable, idempotent, strong role structure. Requires Ansible skills and inventory design.
Shell Scripts Small, simple tasks such as health checks, smoke tests, and emergency operations. Simple, fast, easy to run from SSM. Can become messy; weaker idempotency and error handling.
EC2 User Data Small first-boot bootstrap, metadata registration, or calling SSM/Ansible. Runs automatically at launch. Harder to rerun and debug; should not become a large installer.
SSM Run Command Post-launch commands, health checks, patching, backups, restore tasks. Auditable, no SSH needed, works across fleets. Requires SSM Agent, IAM role, and well-controlled documents.
Cloud-Init Linux bootstrapping, package install, file creation, service startup. Native Linux boot mechanism. Can be difficult for large configuration logic.

This recommended option uses a secure approved base AMI for all EC2 servers. The service pipeline launches servers from that base AMI, then uses Ansible to install and configure Kafka, ZooKeeper, NiFi, OpenSearch, PostgreSQL tools, and ETL worker packages. This avoids creating a separate service AMI for every service version while still keeping the operating system secure and standardized.

Base Image Pipeline

Builds and publishes one secure Amazon Linux 2023 base/golden AMI.

Does not install: Kafka, NiFi, OpenSearch, or service-specific runtimes unless they are approved common dependencies.

Service Pipeline

Uses the approved base AMI, launches servers, runs Ansible, starts services, checks health, and publishes endpoints.

Installs: Kafka, ZooKeeper, NiFi, OpenSearch, PostgreSQL tools, ETL libraries, and runtime configuration.

End-to-End Service Pipeline Flow

1. GitLab pipeline starts from merge request, tag, schedule, or manual approval
2. Pipeline authenticates to AWS using OIDC and assumes an approved IAM role
3. Terraform or CloudFormation reads secure base AMI ID from SSM Parameter Store
4. Pipeline creates EC2 instances, Auto Scaling groups, EBS volumes, security groups, IAM roles, and DNS
5. Pipeline waits for EC2 instances to become reachable through AWS Systems Manager
6. Dynamic inventory discovers new EC2 instances by tags
7. Ansible installs Kafka, ZooKeeper, NiFi, OpenSearch, PostgreSQL tools, or ETL packages
8. Ansible writes runtime config from SSM Parameter Store and Secrets Manager
9. Ansible enables and starts systemd services
10. Pipeline runs service health checks
11. Pipeline performs smoke tests such as Kafka produce/consume or NiFi API checks
12. Pipeline registers load balancer targets and Route 53 DNS records
13. Pipeline confirms CloudWatch metrics, logs, alarms, and backup coverage
14. Pipeline publishes endpoint and user handoff details

Terraform Reads the Secure Base AMI

variable "base_ami_parameter" {
  type    = string
  default = "/data-platform/amis/golden/latest"
}

data "aws_ssm_parameter" "base_ami" {
  name = var.base_ami_parameter
}

resource "aws_instance" "kafka_broker" {
  count         = 3
  ami           = data.aws_ssm_parameter.base_ami.value
  instance_type = "m6i.large"
  subnet_id     = element(var.private_subnet_ids, count.index)

  iam_instance_profile = aws_iam_instance_profile.data_service_profile.name

  tags = {
    Name        = "kafka-${var.environment}-${count.index + 1}"
    Service     = "kafka"
    Environment = var.environment
    ManagedBy   = "terraform"
    ConfigureBy = "ansible"
  }
}

Pros and Cons of Base AMI + Ansible Install

Pros Cons Best Fit
Fewer AMIs to manage and patch. Service install takes longer during deployment. Teams that want one secure base image and flexible service versions.
Service version changes do not require rebuilding the base AMI. Ansible roles must be reliable, tested, and idempotent. Kafka/NiFi/OpenSearch versions that change independently.
Clear separation between OS security and service runtime. More work happens at deployment time, so health checks are critical. Organizations with strong CI/CD and operations discipline.
Works well with Terraform, CloudFormation, SSM, and dynamic inventory. Bad network access to package repositories can break deployments. Private subnets with controlled artifact repositories or S3 package mirrors.

The AMI strategy must produce more than image files. The platform should launch running EC2 servers and clusters that are ready for data users. A data user should not receive a blank Linux server. They should receive a working service endpoint, access instructions, monitoring, backup coverage, and support ownership.

Server

Kafka Cluster Bootstrapping Example

A Kafka service AMI should contain Java, Kafka binaries, users, systemd service files, storage layout, logging config, and helper scripts. Runtime configuration should set broker ID, advertised listeners, cluster ID, security settings, certificates, topic defaults, and monitoring. 

Example production Kafka launch target:

- 3 brokers across 3 Availability Zones
- 1 broker per private subnet/AZ
- Dedicated EBS or instance-store layout based on performance needs
- Replication factor 3 for critical topics
- Minimum in-sync replicas set to protect writes
- CloudWatch metrics and logs enabled
- Alarms for disk usage, consumer lag, offline partitions, and under-replicated partitions
- Bootstrap DNS name published to Route 53
- Data users receive approved topic request process and connection examples

Data infrastructure should be designed for node failure, Availability Zone failure, software failure, bad deployments, storage limits, and network interruptions. The support team should define RTO, how quickly the service must recover, and RPO, how much data loss is acceptable.

High Availability

Run across at least two or three Availability Zones, avoid single EC2 nodes for critical services, and use health checks.

Fault Tolerance

Design so one node, disk, broker, processor, or AZ can fail without taking down the platform.

Failover

Use automated failover when available, and documented manual failover when full automation is too risky.

Service-by-Service Fault Tolerance Guidance

Service Cluster Recommendation Failover / Tolerance Notes Team Warning
Kafka Minimum 3 brokers across 3 AZs for production. Use replication factor 3 for important topics. Partition leadership moves if a broker fails. Monitor under-replicated partitions and consumer lag. Do not run critical Kafka on one EC2 instance. Bad partition design can overload one broker.
ZooKeeper Use odd-number quorum, usually 3 or 5 nodes. Cluster remains healthy if quorum survives. Three nodes can lose one node; five nodes can lose two. Avoid 2-node ZooKeeper. Split-brain and quorum loss are common design risks.
NiFi Use 3+ NiFi nodes behind an internal load balancer. Use NiFi Registry for versioned flows. Cluster Coordinator and Primary Node roles can move. Repositories need careful disk planning. Content, FlowFile, and Provenance repositories can fill disks quickly and stop flows.
PostgreSQL Prefer RDS Multi-AZ or Aurora PostgreSQL for production. Managed failover reduces operational burden. Use read replicas for read scaling, not as the only backup. Self-managed PostgreSQL failover is complex and needs tested runbooks.
OpenSearch Prefer managed OpenSearch with dedicated masters and data nodes across AZs. Use replicas, shard allocation awareness, snapshots, and health checks. Too many shards or undersized JVM memory causes instability.
ETL Workers Use Auto Scaling groups, ECS, EKS, AWS Batch, Glue, or EMR depending on workload. Workers should be replaceable. Store state in S3, databases, queues, or managed services. Do not keep important state only on worker local disks.

Load management means keeping data services stable when usage grows, jobs spike, or one component slows down. The team should monitor throughput, latency, queue depth, disk usage, CPU, memory, network, and error rates.

Horizontal scaling

Add nodes, brokers, workers, or replicas. Best for Kafka, NiFi clusters, ETL workers, OpenSearch data nodes, ECS/EKS jobs, and stateless services.

Vertical scaling

Use larger instances, more memory, faster disks, or better network. Useful for PostgreSQL, OpenSearch memory pressure, and heavy NiFi repositories.

Load Balancer Options

Option Best For Pros Cons
Application Load Balancer NiFi UI/API, internal web apps, REST services. Layer 7 routing, TLS termination, path routing, health checks. Not ideal for all raw TCP workloads.
Network Load Balancer High-throughput TCP services and stable IP needs. Fast, scalable, static IP support, TCP/TLS/UDP. Less application-aware than ALB.
Route 53 DNS Private service discovery and failover records. Simple, supports health checks and weighted routing. DNS caching can delay failover.
Service Mesh EKS/ECS microservices with advanced traffic control. mTLS, retries, traffic splitting, observability. Adds operational complexity.

Simple Team Rule

Use managed services when the business needs reliability faster than deep customization. Use EC2 with custom AMIs when the team needs full control, special software, custom agents, unusual networking, or deep file-system access. Use containers when workloads are portable, job-based, or need fast scaling.

Use GitLab protected branches, merge approvals, IaC scanning, Ansible linting, and OIDC federation to AWS. Avoid long-lived AWS access keys in GitLab variables. Two GitLab pipelines should exist: one for the secure base/golden AMI lifecycle and one for final service deployment. The service pipeline should use the secure base AMI, run Ansible to install Kafka and other services, then test health and verify operation.

AWS Systems Manager should be the main operations control plane for EC2-based data infrastructure.

  • Session Manager: shell access without public SSH.
  • Run Command: execute approved scripts on nodes.
  • Automation: guided AMI creation, backups, restores, and patching.
  • Patch Manager: patch baseline and maintenance windows.
  • Parameter Store: AMI IDs, non-secret config, environment values.
  • Inventory: package and system visibility.
  • State Manager: desired-state enforcement for agents and configs.

Use Terraform modules or CloudFormation stacks to consume approved AMIs from SSM Parameter Store. Do not hardcode AMI IDs in production templates.

Terraform example

data "aws_ssm_parameter" "etl_worker_ami" {
  name = "/data-platform/amis/etl-worker/latest"
}

resource "aws_instance" "etl_worker" {
  ami           = data.aws_ssm_parameter.etl_worker_ami.value
  instance_type = "t3.large"
}

Data User Platform Mindset

The infrastructure should feel like a product. Data users should not need to know every EC2, IAM, VPC, or backup detail. The support team should expose safe choices, approved templates, documented runbooks, and clear service-level expectations.

AMI Lifecycle Details

  • Build monthly and immediately after critical security fixes.
  • Keep the last three approved versions for rollback.
  • Deprecate old AMIs after migration.
  • Never bake static passwords, private keys, API tokens, or environment secrets into AMIs.
  • Tag AMIs with owner, service, patch level, build date, Git commit, Ansible role, and approval state.
  • Store latest approved AMI IDs in SSM Parameter Store.

GitLab to AWS Security

Use OIDC federation so GitLab jobs can assume AWS IAM roles with temporary credentials. This is safer than storing long-lived AWS access keys in GitLab variables. Restrict trust policies by project, branch, environment, and protected refs.

Manual Backup Process

  1. Confirm service owner and change window.
  2. Check disk space, service health, and current alerts.
  3. Run SSM Automation or Run Command backup script.
  4. Create EBS snapshot or application-level export.
  5. Tag backup with service, owner, date, environment, and retention.
  6. Verify backup artifact exists.
  7. Document backup ID, location, and restore command.
  8. Schedule restore test if backup protects production data.

HA, Failover, and Fault Tolerance Details

High availability means the platform stays usable when a normal failure happens. Fault tolerance means the system can absorb failures without data loss or major outage. Failover is the process of moving traffic or leadership from a failed component to a healthy one.

  • Use at least two Availability Zones for non-critical systems and three Availability Zones for important production clusters.
  • For Kafka, design topics with replication factor, partition count, and minimum in-sync replicas that match the business criticality.
  • For ZooKeeper, use odd-number quorums. Never use a two-node quorum for important systems.
  • For NiFi, plan repository disks carefully. Repository disk exhaustion is a common production outage cause.
  • For PostgreSQL, prefer RDS Multi-AZ or Aurora because database failover is difficult to self-manage safely.
  • For OpenSearch, keep shard count reasonable and test snapshot restores.
  • For every service, document who can fail over, when they can fail over, and how to roll back.

Load Management Details

Load problems usually show up as slow pipelines, growing queues, disk pressure, high memory, or users reporting that data is late. The team should set thresholds before production use.

  1. Measure baseline load: normal records per second, average job duration, average query latency, and normal disk growth.
  2. Set alarms: warn before saturation, not after outage.
  3. Control back pressure: NiFi queues, Kafka retention, consumer lag, and worker concurrency should be intentionally configured.
  4. Separate workloads: avoid mixing heavy batch jobs, user queries, and ingestion spikes on the same under-sized nodes.
  5. Test failure: stop a broker, stop a NiFi node, restore a database snapshot, and verify the team knows what happens.

How to Choose the Right Platform Option

Use these questions during architecture reviews:

  • Does AWS offer a managed service that meets the need?
  • Does the team truly need root access, custom plugins, or special OS configuration?
  • What is the recovery time objective and recovery point objective?
  • Can the workload be rebuilt from code, AMI, Ansible, and data backup?
  • Can a data user safely request this through a controlled process?
  • Is the cost of self-managing lower after including patching, support, on-call, backups, and outages?

The safest default is managed services for critical databases, streams, and search platforms; custom AMIs for controlled EC2 patterns; and containers or serverless options for elastic ETL jobs.

Ready-to-Use Server Details

A good data platform does not stop after AMI creation. The team must prove that the AMI can launch into a working server or cluster. Each service should have a launch module, health test, monitoring rule, backup policy, and user handoff document.

  • Kafka: brokers should form a cluster, advertise the correct private DNS names, pass produce/consume tests, and expose metrics.
  • NiFi: nodes should join the cluster, repositories should mount correctly, and a test flow should run.
  • OpenSearch: nodes should join the cluster, shard allocation should work, and snapshot repository should be configured.
  • PostgreSQL: RDS/Aurora should be preferred; if EC2-based, backup and restore must be tested before users depend on it.
  • ETL workers: users should be able to log in through SSM and run approved scripts against S3, Kafka, PostgreSQL, or OpenSearch.

The handoff to data users should include endpoints, access method, sample commands, naming rules, support contact, backup policy, and expected service limits.
