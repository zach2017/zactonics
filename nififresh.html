1 · Why use Apache NiFi at all?

Picture a real grocery store. Every minute, things are happening:

  • A cashier scans a box of cereal at the register.
  • A truck arrives at the loading dock with 240 cases of bananas.
  • A shopper fills out a form saying the milk smelled bad.
  • A manager moves the chips from Aisle 5 to Aisle 12.

Each one of those is a small piece of information — we call them events. FreshMart needs all of them to land in the right places — the sales database, the warehouse system, an alert system, an email inbox. Doing this by hand would be impossible. So we use a tool that moves data automatically.

That tool is Apache NiFi. Think of NiFi like a giant kitchen with conveyor belts running between stations. You drop ingredients (data) onto a belt, and tiny robot chefs (called processors) chop, mix, season, and plate them as they roll by. At the end, the finished dish lands in the right place. NiFi is the kitchen. You design it by dragging pictures around on a screen — no programming needed.

In one sentence

NiFi is a visual tool for building data pipelines — programs that take data from one place, change it, and send it somewhere else — all by drag-and-drop instead of writing code.

What NiFi features does FreshMart use?

Drag-and-drop processors

NiFi has 400+ built-in "Lego blocks" called processors. We pick the ones we need (like ConsumeKafka, PutDatabaseRecord, PublishSNS) and connect them with arrows.

NiFi ships with hundreds of built-in processors. You can also write your own in Java or Python.

Names follow a pattern: Get* and Listen* bring data in. Put* and Publish* send data out. Route* chooses which path to take. Transform* changes the data.

', [{label:'NiFi processor list (official)', url:'https://nifi.apache.org/components/'}, {label:'NiFi User Guide', url:'https://nifi.apache.org/docs/nifi-docs/html/user-guide.html'}])">More info

FlowFile model

Every piece of data inside NiFi is wrapped in a FlowFile — a small package with the data itself plus tags about it (size, source, time received). More info

Back-pressure

If the database gets slow, NiFi automatically pauses the upstream processors so data doesn't pile up and crash anything. Like a traffic light at a busy intersection. More info

Data provenance

NiFi remembers every single thing that ever happened to every piece of data — like a security camera for your pipeline. You can replay any event, see what changed it, when, and why.

Later you can search those notes: \"Show me every event for store FM-042 from 2pm-3pm yesterday\", and NiFi will replay each step, including what the data looked like before and after.

This is huge for auditing, debugging, and compliance — especially in regulated industries like banking or healthcare.

', [{label:'NiFi User Guide · Provenance', url:'https://nifi.apache.org/docs/nifi-docs/html/user-guide.html#data_provenance'}])">More info

Expression Language

A mini scripting language inside NiFi that lets you write things like ${filename:substringBefore('_')} right inside a processor's config — no Java required. More info

Record-oriented processing

Instead of handling one row at a time, NiFi can grab a batch of a thousand records and process them together — 100× faster than one-by-one. QueryRecord, SplitRecord, PutDatabaseRecord) work with batches. They read 1000 records, do something, and write them in one operation. Much faster.

To use them, you configure a Reader (how to parse incoming data) and a Writer (how to format outgoing data) as Controller Services that are shared across processors.

', [{label:'RecordPath Guide', url:'https://nifi.apache.org/docs/nifi-docs/html/record-path-guide.html'}])">More info

Why not just write a Python script?

You could, for small jobs. But a real grocery chain has dozens of pipelines running at once, each needing to be monitored, paused, replayed, and updated without taking the whole system down. NiFi gives you a visual control panel for all of that, plus security, encryption, audit logs, and back-pressure — without writing a line of Java or Python for the most common cases. When you do need custom logic, you can drop in a small ExecuteScript processor.

2 · Core concepts you need first

Before we look at FreshMart, here are five ideas you'll see over and over.

1. Pipeline
A pipeline is a chain of steps that data flows through. Like an assembly line in a car factory: a chassis goes in one end, gets bolted together, painted, inspected, and rolls out as a finished car. In FreshMart, a sale at the register flows through a pipeline that validates it, stores it, alerts other systems, and updates the inventory count. More info
2. Message / Topic
A message is a small package of data (usually JSON) sent over a network. A topic is like a labeled mailbox where messages land. FreshMart has a topic called pos-sales — every sale event is dropped into that mailbox, and anyone who cares about sales reads from there. Kafka is the post office.
  • pos-sales
  • vendor-deliveries
  • customer-feedback
  • reorder-alerts
  • planogram-updates
  • product-recalls
  • hr-notifications
  • dead-letter-queue (for errors)

    • Topics are split into partitions for parallel processing. A message goes to a specific partition based on its key (we use store_id), so all messages for the same store stay in order.

    ', [{label:'Kafka Introduction', url:'https://kafka.apache.org/intro'}, {label:'Kafka quickstart', url:'https://kafka.apache.org/quickstart'}])">More info
    3. Producer / Consumer
    A producer creates messages and sends them to a topic. A consumer reads from a topic and does something with each message. The dashboard is a producer (it sends new sale events). NiFi is a consumer (it reads them and stores them in the database).
    4. Database (Postgres)
    A database is software that stores data in tables, like a giant spreadsheet that many programs can read and write at the same time without colliding. FreshMart uses PostgreSQL, a free, very reliable, very popular database.

    FreshMart uses Postgres for the data warehouse. The schema has 11 tables: sales, inventory, employees, stores, etc.

    Why Postgres and not MySQL or MongoDB? Postgres handles structured data (rows and columns) extremely well, supports the kind of joins and aggregations a grocery chain needs, and has features like JSONB columns for semi-structured data (like our EVENT_LOG audit table).

    ', [{label:'PostgreSQL official site', url:'https://www.postgresql.org/about/'}, {label:'PostgreSQL tutorial', url:'https://www.postgresql.org/docs/current/tutorial.html'}])">More info
    5. Container / Docker
    A container is a lightweight package containing one program plus everything it needs to run (its libraries, settings, files). You can run a container on any computer without installing all those things yourself. Docker is the tool that builds and runs containers. FreshMart runs each piece (NiFi, Kafka, the dashboard) in its own container, so they can all talk to each other but not get tangled up. More info

    confluentinc/cp-kafka:7.7.1  ·  The message bus
    localhost:9092 (host) · kafka:29092 (containers)
    ↓

    Kafka is a distributed log system. You write messages to topics; consumers read them in order. Each topic is split into partitions for parallelism. Messages have a key — all messages with the same key go to the same partition, so order is preserved within a key.

    FreshMart runs Kafka in KRaft mode. The old way required a separate service called ZooKeeper. KRaft (Kafka Raft) replaces it with the broker managing itself. Less complexity, fewer containers.

    Two listeners trick: Kafka has two ports inside the network. One (kafka:29092) is for other containers. The other (localhost:9092) is for programs on your host machine. This is essential because the address a container sees is different from what your laptop sees.

    5 · The six workflows in detail

    FreshMart has six pipelines, one per business scenario. Click each workflow to expand it — you'll see the trigger, the processors used, and what data ends up where.

    1
    POS Sales: Register → Database → Kafka
    Role: Cashier · Trigger: every barcode scan

    When Alice the cashier scans a box of cereal, a JSON message gets sent from the register to NiFi. This is the "hello world" of FreshMart — the most common event by far (thousands per minute at peak).

    What goes in
    {
  "store_id":   "FM-042",
  "sku":        "BAN-CAVENDISH-1LB",
  "qty":        3,
  "price":      0.99,
  "cashier_id": "EMP-1147",
  "timestamp":  "2024-11-01T14:23:00Z"
}
    What comes out
    • 1 row in DW_SALES_FACT
    • 1 decrement on INVENTORY
    • 1 Kafka msg on pos-sales
    • 1 audit row in EVENT_LOG
    Processors used (in order)
    1. 1.
      ListenHTTP — opens port 9090, accepts JSON. docs
    2. 2.
      ValidateRecord — uses Avro schema to check fields. Bad ones go to a "failure" queue. docs
    3. 3.
      JoltTransformJSON — renames fields (price → unit_price). docs
    4. 4.
      UpdateAttribute — adds context like environment=prod.
    5. 5.
      MergeRecord — batches 100 records together for one DB insert (much faster than 100 individual inserts).
    6. 6a.
      PutDatabaseRecord → Postgres. docs
    7. 6b.
      PublishKafkaRecord → Kafka topic pos-sales. docs
    2
    Inventory Tracking & Auto-Reorder
    Role: Store Manager · Trigger: timer, every 5 min

    Every 5 minutes, NiFi wakes up, looks at the inventory table, and asks: "are we running low on anything?" If yes, it automatically places a reorder with the vendor.

    In the demo, click Run Inventory Sweep on the Manager tab to fire it manually. Items below their threshold go red in the table; new rows appear in REORDER_LOG; messages land in Kafka topic reorder-alerts and SQS queue freshmart-reorder-alerts.fifo.

    Processors used
    1. 1.
      QueryDatabaseTableRecord — polls INVENTORY for new/changed rows.
    2. 2.
      RouteOnAttribute — checks ${qty:lt(${reorder_threshold})}.
    3. 3.
      InvokeHTTP → calls the vendor API to place the order.
    4. 4.
      PutSQS → AWS SQS FIFO queue (ordered alerts). docs
    5. 5.
      PutDatabaseRecord — UPSERT into REORDER_LOG.
    3
    Vendor CSV Ingestion
    Role: Vendor · Trigger: file drop in vendor-data/

    DOLE uploads a CSV called DOLE_20241101_delivery.csv via SFTP. NiFi sees the file, parses each row, looks up the vendor's product code in PRODUCT_CATALOG to find FreshMart's SKU, then writes to VENDOR_DELIVERIES and Kafka.

    The lookup is key. DOLE calls the product DOLE-BAN-001; FreshMart calls it BAN-CAVENDISH-1LB. The catalog table maps between them so we don't have to make the vendor learn our SKUs.

    Processors used
    1. 1.
      GetFile — polls /sftp/vendor/incoming/*.csv. docs
    2. 2.
      UpdateAttribute — extract vendor_id with EL: ${filename:substringBefore('_')}
    3. 3.
      SplitRecord — one CSV → many FlowFiles, one per row.
    4. 4.
      LookupRecord — uses DatabaseRecordLookup to map vendor codes → FreshMart SKUs.
    5. 5a.
      PublishKafkaRecord → topic vendor-deliveries
    6. 5b.
      PutS3Object → archive to S3 bucket freshmart-data-lake
    4
    Planogram Sync (SNS Fan-out)
    Role: Operations / HQ · Trigger: manual push or scheduled

    When HQ changes the store layout (planogram = where each product sits on the shelf), they need to tell everyone: the POS register, the mobile employee app, the electronic shelf labels (ESL), and the warehouse system. One publish → four delivery channels. This is SNS fan-out.

    In the demo, picking store_id = ALL on the Ops tab fans out to every store in the database.

    Processors used
    1. 1.
      InvokeHTTP → pulls latest planogram from HQ's API.
    2. 2.
      SplitJson → one FlowFile per SKU update.
    3. 3.
      UpdateAttribute — extracts aisle, section, shelf level into attributes.
    4. 4.
      PutDatabaseRecord — UPSERT into PRODUCT_LOCATIONS.
    5. 5.
      PublishSNS — fires once; SNS replicates to 4 subscribed SQS queues. docs
    5
    Staff Scheduling & HR Sync
    Role: HR · Trigger: cron, Monday 6 AM

    Every Monday at 6 AM, NiFi pulls next week's shift schedule from the HR system, sends each employee an email with their hours, and updates the payroll attendance table.

    How does Monday 6 AM work? NiFi processors have a property called "Run Schedule" with a CRON expression: 0 0 6 ? * MON *.

  • 0 0 6 ? * MON * — Monday at 6:00:00 AM
  • 0 */15 * * * ? — Every 15 minutes
  • 0 0 0 1 * ? — First of every month, midnight

    • NiFi uses Quartz CRON syntax (with seconds), slightly different from Linux cron.

    1. 1.
      InvokeHTTP → GET schedule from HR API.
    2. 2.
      SplitRecord → one FlowFile per shift.
    3. 3.
      RouteOnAttribute → split by department.
    4. 4.
      ReplaceText → fill in an email template.
    5. 5.
      PutEmail → SMTP. docs

    6
    Customer Feedback & Recall Detection
    Role: Customer · Trigger: feedback form submission

    A customer types "this milk was spoiled" into the feedback form. NiFi sees the word "spoiled" and classifies the feedback as SAFETY. If 5 or more SAFETY complaints come in on the same product within an hour, NiFi automatically triggers a product recall — broadcasts an urgent alert to every store via SNS.

    In the demo, click Fire 5 SAFETY → Trigger Recall on the Customer tab to see it happen.

    Processors used
    1. 1.
      ConsumeKafka → reads from topic customer-feedback.
    2. 2.
      RouteOnContent — regex match on safety keywords. docs
    3. 3.
      ExecuteGroovyScript — counts SAFETY complaints per SKU in NiFi's cluster state; sets recall_triggered=true if count ≥ 5.
    4. 4.
      RouteOnAttribute → if recall, go to PublishSNS; else just log.
    5. 5a.
      PublishSNS → topic freshmart-product-recalls, severity CRITICAL.
    6. 5b.
      PutDatabaseRecord → INSERT into CUSTOMER_FEEDBACK with recall flag.

    8 · How security works

    The dashboard isn't open to anyone. There's a login screen, every action is tied to a username, and the server checks — on every single request — whether your role is allowed to do that thing.

    8.1 · The login screen

    When you open the dashboard, the first thing you see is a login form with seven demo accounts. Pick one (or type a username and password), and the dashboard asks the server: "Is this person real?" The server checks the password against a bcrypt hash and, if it matches, hands back a eyJhbGciOiJIUzI1Ni...

    Inside (decoded) it's just JSON: {\"sub\": \"alice\", \"role\": \"CASHIER\", \"exp\": 1700000000}. The whole thing is signed with a secret key — if anyone changes the JSON, the signature breaks.

    So the server doesn't need to remember who's logged in. It just checks the signature on every request. Stateless, fast, easy to scale.

    FreshMart tokens last 8 hours by default. After that you log in again.

    ', [{label:'JWT.io introduction', url:'https://jwt.io/introduction'}, {label:'OWASP JWT cheat sheet', url:'https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html'}])">JWT token. The dashboard stores that token and sends it with every request from then on.

    Try it yourself
    curl -X POST http://localhost:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"cashier123"}'

# → { "ok": true,
#     "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
#     "role": "CASHIER", ... }

    8.2 · The seven roles

    Every user has exactly one role, and each role can do certain things and not others. This is called RBAC.

    CASHIER · alice

    Single POS sales. Read sales + inventory.

    MANAGER · bob

    Bulk sales, inventory sweeps, planograms, all reads.

    VENDOR · dole

    Submit deliveries. Read own deliveries.

    CUSTOMER · shopper

    Submit feedback. Read feedback.

    HR · frank

    HR notifications. Read employees.

    OPS · ops

    Super-user. Can do everything.

    AUDITOR · audit

    Read-only across every table. Cannot change anything, but can investigate everything. Great for compliance officers and incident response.

    8.3 · How permission checks actually happen

    Every protected endpoint in the Python code has a one-line decorator: 

    1@require_perm("pos.sale")
2def pos_sale():
3    ...

    The decorator reads the Authorization: Bearer eyJhbGc...

    The word "Bearer" just means "whoever has this token is the user" — like a movie ticket. Don't let it leak!

    The server reads the header, validates the JWT signature, extracts the role from the token, and decides whether to allow the request.

    ', [{label:'RFC 6750: Bearer tokens', url:'https://datatracker.ietf.org/doc/html/rfc6750'}, {label:'MDN Authorization header', url:'https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Authorization'}])">Authorization header, validates the token, looks up the user's role, and checks a table of which roles get which permissions. If the role doesn't have it, the server returns HTTP 403 immediately and the actual endpoint code never runs.

    See for yourself

    Try logging in as alice (a cashier) and then call an endpoint she's not allowed to:

    TOKEN=$(curl -sf -X POST http://localhost:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"cashier123"}' | jq -r .token)

curl -i -X POST http://localhost:8000/api/inventory/check \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" -d '{}'

# HTTP/1.1 403 Forbidden
# {"errors":["role CASHIER lacks permission inventory.check"],"ok":false}

    8.4 · The dashboard hides what you can't do

    When you log in, the dashboard asks the server "what permissions do I have?" and uses the answer to hide tabs you can't use. So Alice the cashier never even sees the Operations tab.

    UI hiding is not security!

    Hiding tabs is just for user experience. A determined attacker could edit the JavaScript in the browser and show the hidden tabs anyway. That's why the real permission check happens on the server, on every request. The UI hiding is a nicety; the 403 is the actual security.

    8.5 · Other protections layered in

    Rate limiting

    If anyone tries to brute-force the login or flood the API, Flask-Limiter blocks them after 10 attempts per minute (login) or 60 per minute (most endpoints). more info

    CORS lockdown

    The API only accepts requests from http://localhost:8000 — the dashboard origin. A malicious site can't trick your browser into firing requests on your behalf. more info

    Loopback-only ports

    By default, Kafka, Postgres, and LocalStack bind to 127.0.0.1 — only your laptop can reach them. Nobody else on your Wi-Fi can poke at the database. more info

    Audit trail

    Every action is logged to EVENT_LOG with the actor's username. Six months later you can answer "who triggered that recall on May 21st at 2:47 PM?" with a single SQL query. more info

    Secrets in .env

    No passwords or signing keys are hardcoded. Everything lives in .env, which is gitignored. Docker Compose refuses to start the stack if any required secret is missing. more info

    Input validation

    Before any data reaches Kafka or the database, it's validated: required fields present, numbers positive, timestamps in ISO format. Bad input gets HTTP 400 with a clear error message; no garbage ever lands in storage.

    8.6 · What's not included (yet)

    This is a demo. Real production would add:

    • TLS / HTTPS on the dashboard and API (we only have it on NiFi, with a self-signed cert).
    • Users in a real database with proper signup, password reset, account lockout after 5 bad attempts, and optional 2FA / MFA.
    • A secrets manager like Vault instead of .env.
    • NiFi multi-user mode with LDAP or OIDC login and per-process-group permissions.
    • Kafka SASL_SSL with per-client ACLs (right now Kafka has no auth on its broker port).
    • Refresh tokens so users aren't forced to re-login every 8 hours.
    • A WAF (Web Application Firewall) at the edge to block obvious attacks.
    • Audit log shipping to a SIEM like Splunk, Datadog, or OpenSearch for real-time alerting.
    Verifying it works

    The test suite has 39 dedicated security assertions: anonymous requests get 401, wrong-role requests get 403, audit logs include the actor, and each table check enforces its own permission. Run python3 tests/test_pos_simulator.py and you'll see them all pass.

    9 · Glossary

    Every technical word from this page, in alphabetical order, plain English.

    API (Application Programming Interface)
    A set of rules letting one program ask another to do something. Like a menu at a restaurant — you don't go in the kitchen, you order through the menu.
    Avro
    A way to write down what a piece of data looks like (its schema) so different programs can agree on the shape.
    Audit log
    A tamper-resistant record of who did what and when. FreshMart's EVENT_LOG table is the audit log.
    bcrypt
    A password hashing algorithm designed to be deliberately slow, making brute-force guessing impractical.
    Bearer token
    A token where "whoever has it is the user." Sent in the Authorization: Bearer ... HTTP header.
    Broker
    A Kafka server. It receives messages from producers and gives them to consumers. FreshMart has one broker.
    CRON
    A way to write a schedule, like "every Monday at 6 AM", as a short text string: 0 0 6 ? * MON *.
    Container
    A bundled-up program that runs the same way on any computer. Like a sealed lunch box.
    CORS (Cross-Origin Resource Sharing)
    A browser rule that blocks websites on one domain from calling APIs on another, unless the API explicitly allows it.
    DLQ (Dead Letter Queue)
    A separate place where messages that failed processing go, so they can be investigated later without blocking the main pipeline.
    FIFO
    "First In, First Out". The order you put things in is the order you get them back. Like a normal grocery line.
    FlowFile
    NiFi's name for one piece of data flowing through. Has Content (the data) and Attributes (tags).
    JSON (JavaScript Object Notation)
    A simple text format for data. Looks like {"name": "Alice", "age": 12}. Easy for both humans and computers to read.
    JWT (JSON Web Token)
    A small signed string that proves who you are. Lets the server check identity without storing sessions.
    Kafka topic
    A named mailbox in Kafka. Producers put messages in; consumers take them out (without removing them — they stay for replaying).
    KPI (Key Performance Indicator)
    A number that tells you how something is going. "Sales today" is a KPI.
    KRaft
    The new way Kafka manages itself, without needing a separate ZooKeeper service.
    Loopback (127.0.0.1)
    The address a computer uses to talk to itself. Services bound here are not reachable from other machines.
    Partition
    A slice of a Kafka topic. A topic with 3 partitions can be read by 3 consumers at once, in parallel.
    Pipeline
    An automated chain of steps that moves data from start to finish.
    Planogram
    A map of the store: which products go on which aisle, section, and shelf.
    Process Group
    A folder in NiFi that groups related processors. Each FreshMart workflow is one process group.
    Processor
    A single building block in NiFi. Does one job: read a file, send to Kafka, validate JSON, etc.
    Producer / Consumer
    A producer writes messages to Kafka. A consumer reads them.
    Rate limit
    A cap on how often one user or IP can hit an endpoint. Stops brute-force and runaway loops.
    RBAC (Role-Based Access Control)
    A system where users are grouped into roles (CASHIER, MANAGER, etc.) and permissions are attached to roles, not individuals.
    Recall
    An urgent notice that a product is unsafe and should be removed from shelves.
    S3
    Amazon's file storage service. Stores big files cheaply.
    Schema
    A description of what data should look like: which fields, which types.
    SKU (Stock Keeping Unit)
    A unique code for one product. BAN-CAVENDISH-1LB is one.
    SMTP
    The protocol for sending emails between mail servers.
    SNS (Simple Notification Service)
    AWS service for pub/sub messaging. Publish once → many subscribers each get a copy.
    SQS (Simple Queue Service)
    AWS service for message queues. Like Kafka but simpler — one message, one receiver.
    UPSERT
    Insert if new, update if already exists. Saves an extra round-trip to check first.
    UTC (Coordinated Universal Time)
    The world's "master clock" time. Always store timestamps in UTC; convert to local time only for display.
    YAML
    A human-readable text format used for config files (like docker-compose.yml). Cousin of JSON.
