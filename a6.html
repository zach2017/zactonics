Prerequisites — Get Your Windows Computer Ready

Before we write any code, we need to install four free tools on your Windows computer. Think of them as the basic kitchen tools you need before cooking a recipe.

1. Git for Windows

This lets you talk to GitLab (like saving your school project to Google Drive but for code).

Download Git for Windows
Install tip: During installation, choose "Git from the command line and also from 3rd-party software" and accept all default options.

2. Terraform (by HashiCorp)

The magic tool that reads your .tf files and builds things in AWS automatically.

Download Terraform
Windows install: Download the 64-bit zip. Extract to C:\terraform. Then add that folder to your PATH (search "Environment Variables" in Windows Start menu → Edit the system environment variables → Environment Variables → Path → New → type C:\terraform).

3. AWS Command Line Interface (CLI) v2

Helps you talk to AWS from your computer (we mostly use it once to set up security).

Download AWS CLI v2 for Windows
Just run the installer. After installing, open Command Prompt or Git Bash and type aws --version to check it worked.

4. Visual Studio Code (Recommended)

A free code editor that makes writing Terraform and YAML files easy with colors and helpful tips.

Download VS Code
Install these extensions: HashiCorp Terraform GitLab Workflow
Pro tip for Windows users: After installing everything, open Git Bash (installed with Git) and type these commands one by one to check everything works:
git --version
terraform --version
aws --version

Understanding the Tools — Simple Explanations

THE RECIPE WRITER

Terraform

Terraform is a tool that lets you describe what your cloud should look like using simple code files (ending in .tf). Instead of clicking around the AWS website, you write "I want an S3 bucket called my-app-bucket with encryption turned on" and Terraform builds it for you — every single time, exactly the same way.

THE CLOUD KITCHEN

Amazon Web Services (AWS)

AWS is the biggest "cloud" company in the world. Instead of buying and maintaining your own computers (servers), you rent them from Amazon by the hour or minute. They have hundreds of services. In this guide we will mainly use S3 (simple storage) — think of it like a super-secure, super-reliable Google Drive that you control completely with code.

THE ROBOT FACTORY

GitLab CI/CD

GitLab is a website where you store your code (like GitHub). The "CI/CD" part means Continuous Integration / Continuous Deployment. It has built-in robots (called runners) that watch your code. When you save new code, the robots automatically run tests and deploy your changes to AWS. No more "it works on my computer" problems.

AWS Security Foundation — IAM User + Terraform State Bucket

Important security note: We will create a special "robot user" in AWS just for GitLab. This user will only have permission to create and manage our demo S3 bucket and the state files. Never use your main AWS account root login for automation.

4.1 Create a Dedicated IAM User for GitLab

  1. 1. Log into AWS Console → IAM
  2. 2. Click Users in left menu → Create user
  3. 3. User name: gitlab-terraform-ci
  4. 4. Select Access key - Programmatic access only
  5. 5. Permissions: Click Attach policies directly. For learning, attach AmazonS3FullAccess and AmazonDynamoDBFullAccess. (Later we will make a stricter custom policy)
  6. 6. Click through to create the user. Download the CSV with Access key ID and Secret access key. Keep this file safe and never share it.

4.2 Create Your Terraform State S3 Bucket + DynamoDB Lock Table (One Time)

Why do we need this? Terraform needs to remember what it has already created in AWS. It stores this information in a file called terraform.tfstate. If two people (or two pipelines) try to change things at the same time, they can break everything. So we store the state file safely in S3 with locking in DynamoDB.

Create the State S3 Bucket (via Console)
  1. Go to S3 in AWS Console
  2. Click Create bucket
  3. Bucket name: yourname-terraform-state-2026 (must be globally unique!)
  4. Region: Choose us-east-1 (or your preferred region)
  5. Block Public Access: Leave all checked (default)
  6. Enable Bucket Versioning
  7. Enable Default encryption (AES-256)
  8. Create bucket
Create the DynamoDB Lock Table
  1. Go to DynamoDB → Create table
  2. Table name: terraform-state-locks
  3. Partition key: LockID (String)
  4. Capacity mode: On-demand (cheap for learning)
  5. Leave everything else default
  6. Create table

This table prevents two pipelines from deploying at the exact same moment.

All Terraform Files — Complete Code + Explanations

Below are every single file you need. Copy them exactly. We explain what each line does in simple language.

versions.tf Tells Terraform which tools and versions to use 
terraform {
  required_version = ">= 1.8.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.55"
    }
  }
}
This file makes sure everyone (you and the GitLab robot) uses the same version of Terraform and the AWS plugin so nothing breaks unexpectedly.
backend.tf Where to store the state file safely 
terraform {
  backend "s3" {
    bucket         = "yourname-terraform-state-2026"   # ← CHANGE THIS to your bucket name
    key            = "demo/aws-gitlab-terraform/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-state-locks"
    encrypt        = true
  }
}
Important: Replace yourname-terraform-state-2026 with the exact name you created in Step 4.2. This tells Terraform: "Store my memory file in this S3 bucket and use DynamoDB to stop fights."
variables.tf Settings you can change without editing main code 
variable "aws_region" {
  description = "Which AWS region to create resources in"
  type        = string
  default     = "us-east-1"
}

variable "bucket_name" {
  description = "Name for the S3 bucket we will create (must be globally unique!)"
  type        = string
  # No default — you MUST provide this value
}

variable "environment" {
  description = "dev, staging, or production"
  type        = string
  default     = "dev"
}
main.tf The actual instructions for what to build in AWS 
provider "aws" {
  region = var.aws_region
}

# Create the S3 bucket with best-practice security
resource "aws_s3_bucket" "demo_bucket" {
  bucket = var.bucket_name

  tags = {
    Name        = "GitLab Terraform Demo Bucket"
    Environment = var.environment
    ManagedBy   = "Terraform"
    CreatedVia  = "GitLab-CI"
  }
}

# Turn on versioning (keeps old versions of files)
resource "aws_s3_bucket_versioning" "demo_bucket_versioning" {
  bucket = aws_s3_bucket.demo_bucket.id
  versioning_configuration {
    status = "Enabled"
  }
}

# Turn on encryption so files are scrambled and safe
resource "aws_s3_bucket_server_side_encryption_configuration" "demo_bucket_encryption" {
  bucket = aws_s3_bucket.demo_bucket.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

# Block all public access (very important for security!)
resource "aws_s3_bucket_public_access_block" "demo_bucket_pab" {
  bucket = aws_s3_bucket.demo_bucket.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Extra security: Force all traffic to use HTTPS (no unencrypted access)
resource "aws_s3_bucket_policy" "demo_bucket_policy" {
  bucket = aws_s3_bucket.demo_bucket.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid       = "EnforceHTTPSOnly"
        Effect    = "Deny"
        Principal = "*"
        Action    = "s3:*"
        Resource = [
          aws_s3_bucket.demo_bucket.arn,
          "${aws_s3_bucket.demo_bucket.arn}/*"
        ]
        Condition = {
          Bool = {
            "aws:SecureTransport" = "false"
          }
        }
      }
    ]
  })
}
Security features we enabled: Versioning • AES-256 encryption • Complete public access block • HTTPS-only policy. These are the exact settings professional teams use.
outputs.tf Shows you useful information after deployment 
output "bucket_name" {
  description = "The name of the S3 bucket we created"

output "bucket_arn" {

The GitLab Pipeline — .gitlab-ci.yml Explained

This is the most important file. It tells GitLab's robots exactly what to do when you push code.

.gitlab-ci.yml 
stages:
  - validate
  - plan
  - apply

variables:
  TF_VERSION: "1.9.5"
  AWS_DEFAULT_REGION: "us-east-1"

# Use the official Terraform Docker image
image: 
  name: hashicorp/terraform:1.9.5
  entrypoint: [""]

cache:
  paths:
    - .terraform/

before_script:
  - terraform --version
  - echo "AWS Access Key ID length: ${#AWS_ACCESS_KEY_ID}"   # Just to confirm variable exists (never print secret!)

validate:
  stage: validate
  script:
    - terraform fmt -check -recursive
    - terraform init -backend=false
    - terraform validate
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH

plan:
  stage: plan
  script:
    - terraform init
    - terraform plan -out=tfplan
  artifacts:
    paths:
      - tfplan
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH

apply:
  stage: apply
  needs: ["plan"]
  when: manual          # ← Very important! You must click to approve
  environment:
    name: production
    url: https://console.aws.amazon.com/s3/buckets/$TF_VAR_bucket_name
  script:
    - terraform init
    - terraform apply tfplan
  rules:
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH

Security Highlights in This Pipeline

  • • when: manual — Apply never happens automatically. You must click "Play" in GitLab.
  • • Variables are masked in GitLab settings (never shown in logs).
  • • We use terraform plan first so you can review exactly what will change.
  • • Only runs on protected branches (main) or merge requests.

How to Add the AWS Keys to GitLab (Safely)

  1. Go to your GitLab project → Settings → CI/CD
  2. Expand Variables
  3. Add these three (all masked + protected):
  4. AWS_ACCESS_KEY_ID
  5. AWS_SECRET_ACCESS_KEY
  6. TF_VAR_bucket_name
  7. TF_VAR_aws_region

Never put keys in the .gitlab-ci.yml file itself!

Run Everything — From Your Laptop to AWS

Option A: Test Locally on Windows First (Recommended)

  1. 1. Copy terraform.tfvars.example to terraform.tfvars and edit the bucket_name to something unique.
  2. 2. Open Git Bash in the project folder.
  3. 3. Run: aws configure (paste the keys from the CSV you downloaded earlier).
  4. 4. Run: terraform init
  5. 5. Run: terraform plan
  6. 6. Run: terraform apply and type yes when asked.
  7. 7. Go to AWS Console → S3 and see your new bucket!

Option B: Push to GitLab and Let the Pipeline Deploy

  1. 1. Make sure you added the CI/CD variables in GitLab (Step 6).
  2. 2. In Git Bash: git add .
  3. 3. git commit -m "Add complete Terraform + GitLab pipeline"
  4. 4. git push origin main
  5. 5. Go to your GitLab project → CI/CD → Pipelines
  6. 6. Watch the pipeline run (validate → plan). It will pause at "apply".
  7. 7. Click the play button on the apply job to deploy to AWS.
  8. 8. Check AWS Console to see your new secure S3 bucket.
You did it!

You now have a professional-grade deployment pipeline. Every time you change the Terraform code and push, GitLab will show you exactly what will happen before it touches AWS.

Security Best Practices Checklist

✅ Do These Things

  • • Always use least privilege IAM policies
  • • Mask and protect all CI/CD variables in GitLab
  • • Require manual approval for production deploys
  • • Use remote state with encryption + locking
  • • Block public access on every S3 bucket
  • • Enable S3 versioning and encryption
  • • Use terraform plan before every apply
  • • Protect your main branch in GitLab
  • • Rotate AWS keys regularly
  • • Add .tfvars to .gitignore

❌ Never Do These Things

  • • Never commit AWS keys or secrets to Git
  • • Never give IAM users more permissions than needed
  • • Never run terraform apply -auto-approve in CI without review
  • • Never leave S3 buckets publicly readable
  • • Never skip terraform plan
  • • Never use root AWS account for automation
  • • Never store state files locally in shared projects
  • • Never ignore pipeline failures

Cleanup, Troubleshooting & Next Steps

How to Destroy Everything (Stop Paying)

When you're done learning:

terraform destroy

Or run the destroy job in GitLab pipeline (add a destroy stage the same way as apply).

Also delete the state S3 bucket and DynamoDB table manually if you want everything gone.

Common Problems & Fixes

"S3 bucket name already exists"
→ Change bucket_name in terraform.tfvars or CI variable to something more unique.
"Access Denied"
→ Check that your IAM user has the right policies and that keys are correctly added as masked variables.
" state lock"
→ Someone else is deploying. Wait or force-unlock with terraform force-unlock LOCK-ID.
