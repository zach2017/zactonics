Deploying a Kafka Cluster on AWS with Terraform & GitOps

An end-to-end, pipeline-driven tutorial: bake an immutable AMI with Packer + Ansible, provision the cluster with Terraform, manage configuration through SSM, and ship everything through GitHub Actions and GitLab CI/CD across promoted environments.

AWS EC2 / VPC Terraform Packer + Ansible SSM Parameter Store GitHub Actions GitLab CI/CD KRaft (no ZooKeeper)
0

Overview & what you'll build

▶

You'll provision a 3-broker Apache Kafka cluster (KRaft mode, no ZooKeeper) running on EC2 inside a private VPC. The whole lifecycle is GitOps-driven: every change is a Git commit that flows through a pipeline with promotion gates.

Design principles

  • Immutable infrastructure — brokers boot from a pre-baked AMI; no in-place mutation.
  • Declarative everything — Terraform is the single source of truth for infra.
  • Promotion through stages — dev → staging → prod, same code, different tfvars/state.
  • No secrets in Git — SSM Parameter Store + OIDC short-lived credentials.
Note  This guide assumes Kafka 3.7+ which supports production KRaft mode. ZooKeeper is deprecated and removed in Kafka 4.0.

Prerequisites

▶
  • AWS account with an IAM role for OIDC federation (no long-lived keys in CI).
  • Tooling: terraform >= 1.7, packer >= 1.10, ansible >= 2.16, aws-cli v2.
  • An S3 bucket + DynamoDB table for Terraform remote state & locking.
  • A registered domain/Route53 zone (optional, for internal DNS of brokers).

Bake the AMI with Packer + Ansible

▶

An immutable AMI means brokers boot ready-to-run. Packer launches a temporary EC2 instance, Ansible installs and hardens Kafka, then Packer snapshots it to an AMI.

AWS AMI — Packer template

packer/kafka.pkr.hcl 
packer {
  required_plugins {
    amazon = { version = "~> 1.3", source = "github.com/hashicorp/amazon" }
    ansible = { version = "~> 1.1", source = "github.com/hashicorp/ansible" }
  }
}

source "amazon-ebs" "kafka" {
  region        = "us-east-1"
  instance_type = "t3.medium"
  ssh_username  = "ubuntu"
  ami_name      = "kafka-kraft-{{timestamp}}"
  source_ami_filter {
    filters = { name = "ubuntu/images/*ubuntu-jammy-22.04-amd64-server-*" }
    owners      = ["099720109477"]   # Canonical
    most_recent = true
  }
  tags = { Name = "kafka-kraft", BakedBy = "packer" }
}

build {
  sources = ["source.amazon-ebs.kafka"]
  provisioner "ansible" {
    playbook_file = "./ansible/playbook.yml"
  }
}

Ansible playbook

packer/ansible/playbook.yml 
- hosts: all
  become: true
  vars:
    kafka_version: "3.7.1"
    scala_version: "2.13"
  tasks:
    - name: Install Java 17
      apt: { name: openjdk-17-jre-headless, update_cache: yes }

    - name: Download Kafka
      get_url:
        url: "https://downloads.apache.org/kafka/{{ kafka_version }}/kafka_{{ scala_version }}-{{ kafka_version }}.tgz"
        dest: /tmp/kafka.tgz

    - name: Unpack to /opt/kafka
      unarchive:
        src: /tmp/kafka.tgz
        dest: /opt/
        remote_src: yes

    - name: Symlink /opt/kafka
      file:
        src: "/opt/kafka_{{ scala_version }}-{{ kafka_version }}"
        dest: /opt/kafka
        state: link

    - name: Install systemd unit (config injected at boot via SSM)
      copy:
        dest: /etc/systemd/system/kafka.service
        content: |
          [Unit]
          Description=Apache Kafka (KRaft)
          After=network.target
          [Service]
          ExecStartPre=/opt/kafka/bin/render-config.sh
          ExecStart=/opt/kafka/bin/kafka-server-start.sh /opt/kafka/config/server.properties
          Restart=on-failure
          [Install]
          WantedBy=multi-user.target
Key idea  The AMI is config-agnostic. Broker ID, advertised listeners, and the KRaft quorum are rendered at boot from SSM/instance metadata, so one AMI serves all brokers and all environments.

Terraform infrastructure module

▶

The module looks up the latest AMI by tag, builds the network, and launches brokers across AZs.

terraform/modules/kafka-cluster/main.tf 
data "aws_ami" "kafka" {
  most_recent = true
  owners      = ["self"]
  filter { name = "name"; values = ["kafka-kraft-*"] }
}

resource "aws_instance" "broker" {
  count                  = var.broker_count
  ami                    = data.aws_ami.kafka.id
  instance_type          = var.instance_type
  subnet_id              = element(var.private_subnet_ids, count.index)
  vpc_security_group_ids = [aws_security_group.kafka.id]
  iam_instance_profile   = aws_iam_instance_profile.kafka.name

  user_data = templatefile("${path.module}/user_data.tftpl", {
    node_id      = count.index + 1
    cluster_name = var.cluster_name
    env          = var.environment
  })

  root_block_device { volume_size = var.disk_gb; volume_type = "gp3" }
  tags = { Name = "kafka-${var.environment}-${count.index + 1}" }
}
terraform/envs/dev/dev.tfvars 
environment   = "dev"
broker_count  = 3

disk_gb       = 50

SSM Parameter Store & secrets

▶

Brokers read configuration at boot from SSM. Pipelines write non-secret config via Terraform; sensitive values are SecureString parameters seeded out-of-band.

terraform — config params 
resource "aws_ssm_parameter" "cluster_id" {
  name  = "/kafka/${var.environment}/cluster-id"
  type  = "String"
  value = var.cluster_id
}

resource "aws_ssm_parameter" "controller_quorum" {
  name  = "/kafka/${var.environment}/controller-quorum-voters"
  type  = "String"
  value = join(",", [for i, ip in aws_instance.broker[*].private_ip :
            "${i + 1}@${ip}:9093"])
}
read at boot (on instance) 
aws ssm get-parameter --name "/kafka/${ENV}/controller-quorum-voters" \
  --query 'Parameter.Value' --output text
Never  commit SASL passwords, TLS keys, or cluster IDs to Git. Use SecureString + KMS and grant the broker instance role ssm:GetParameter scoped to its env path.

GitHub Actions pipeline

▶

OIDC-authenticated, with plan on PRs and gated apply on merge to main via environments.

.github/workflows/kafka.yml 
name: kafka-platform
on:
  pull_request: { paths: ['terraform/**','packer/**'] }
  push: { branches: [main] }

permissions:
  id-token: write   # OIDC
  contents: read

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - run: terraform -chdir=terraform/envs/dev fmt -check
      - run: terraform -chdir=terraform/envs/dev init
      - run: terraform -chdir=terraform/envs/dev validate

  plan:
    needs: validate
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::ACCT:role/gh-kafka-deploy
          aws-region: us-east-1
      - uses: hashicorp/setup-terraform@v3
      - run: terraform -chdir=terraform/envs/dev init
      - run: terraform -chdir=terraform/envs/dev plan -var-file=dev.tfvars

  apply-dev:
    needs: validate
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: dev          # required reviewers / protection rules
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::ACCT:role/gh-kafka-deploy
          aws-region: us-east-1
      - uses: hashicorp/setup-terraform@v3
      - run: terraform -chdir=terraform/envs/dev init
      - run: terraform -chdir=terraform/envs/dev apply -auto-approve -var-file=dev.tfvars

GitLab CI/CD pipeline

▶

The equivalent in GitLab using staged jobs, OIDC (id_tokens), and when: manual gates for promotion.

.gitlab-ci.yml 
stages: [validate, plan, deploy-dev, deploy-staging, deploy-prod]

variables:
  AWS_REGION: us-east-1
  TF_ROOT: terraform/envs

image: hashicorp/terraform:1.7

.aws_oidc: &aws_oidc
  id_tokens:
    AWS_TOKEN:
      aud: https://gitlab.com
  before_script:
    - export $(aws-oidc-assume --role arn:aws:iam::ACCT:role/gl-kafka-deploy)

validate:
  stage: validate
  script:
    - cd $TF_ROOT/dev
    - terraform fmt -check && terraform init -backend=false && terraform validate

plan:
  stage: plan
  <<: *aws_oidc
  script:
    - cd $TF_ROOT/dev
    - terraform init && terraform plan -var-file=dev.tfvars -out=plan.cache
  artifacts: { paths: [terraform/envs/dev/plan.cache] }
  rules: [{ if: '$CI_PIPELINE_SOURCE == "merge_request_event"' }]

deploy-dev:
  stage: deploy-dev
  <<: *aws_oidc
  environment: { name: dev }
  script:
    - cd $TF_ROOT/dev
    - terraform init && terraform apply -auto-approve -var-file=dev.tfvars
  rules: [{ if: '$CI_COMMIT_BRANCH == "main"' }]

deploy-staging:
  stage: deploy-staging
  <<: *aws_oidc
  environment: { name: staging }
  when: manual          # promotion gate
  script:
    - cd $TF_ROOT/staging
    - terraform init && terraform apply -auto-approve -var-file=staging.tfvars
  rules: [{ if: '$CI_COMMIT_BRANCH == "main"' }]

deploy-prod:
  stage: deploy-prod
  <<: *aws_oidc
  environment: { name: prod }
  when: manual          # protected, manual approval
  script:
    - cd $TF_ROOT/prod
    - terraform init && terraform apply -auto-approve -var-file=prod.tfvars
  rules: [{ if: '$CI_COMMIT_BRANCH == "main"' }]

GitOps flow & pipeline stages

▶

Every environment change is a Git operation. The pipeline enforces the same path on both platforms:

01
Validate
fmt, init, validate, lint
02
Plan
on PR/MR, posted for review
03
Deploy dev
auto on merge to main
04
Deploy staging
manual gate
05
Deploy prod
manual + protected approval

The GitOps loop

  1. Developer opens a PR/MR changing tfvars, module, or Ansible role.
  2. Pipeline runs validate + plan; the plan is the reviewable diff.
  3. Reviewer approves & merges → main becomes desired state.
  4. Merge triggers auto-apply to dev; staging/prod require manual promotion.
  5. State in S3 reflects reality; drift is caught by scheduled plan runs.
AMI promotion  A new AMI is just a new input. Bump the AMI filter/version in a tfvars commit; the same staged pipeline rolls it through environments with a controlled instance refresh.

Security hardening

▶

A production Kafka cluster needs encryption in transit, authentication, and authorization. The plaintext listener in earlier examples is for bootstrapping only.

Layered controls

  • TLS everywhere — enable SSL listeners for client and inter-broker traffic; terminate at the broker, not the NLB, so the chain stays encrypted end-to-end.
  • SASL authentication — SASL_SSL with SCRAM-SHA-512 or mTLS client certs; credentials live in SSM SecureString.
  • ACL authorization — enable the standard authorizer so each principal only touches its own topics.
  • Network isolation — brokers in private subnets, no public IPs, access via NLB + VPC peering/PrivateLink only.
  • Encryption at rest — gp3 volumes encrypted with a customer-managed KMS key.
server.properties (hardened) 
listeners=SASL_SSL://:9092,CONTROLLER://:9093
listener.security.protocol.map=CONTROLLER:SSL,SASL_SSL:SASL_SSL
sasl.enabled.mechanisms=SCRAM-SHA-512
ssl.keystore.location=/etc/kafka/ssl/broker.keystore.jks
ssl.truststore.location=/etc/kafka/ssl/broker.truststore.jks
ssl.client.auth=required
authorizer.class.name=org.apache.kafka.metadata.authorizer.StandardAuthorizer
super.users=User:admin
Encrypted EBS  Set encrypted = true and a kms_key_id on the root_block_device (and any data volumes) in Terraform — it cannot be toggled on an existing volume without recreating it.

Troubleshooting & common pitfalls

▶
SymptomLikely cause & fix
Clients connect to bootstrap but then time outadvertised.listeners points to an unreachable address. Clients use the advertised host after the initial handshake — it must be routable from the client.
Broker won't start: No readable meta.propertiesStorage not formatted. The boot script must run kafka-storage.sh format with the shared cluster ID before first start.
InconsistentClusterIdExceptionBrokers formatted with different cluster IDs. All nodes must share one ID from /kafka/<env>/cluster-id.
Controller quorum never elects a leadercontroller.quorum.voters mismatch — IDs/IPs differ across nodes, or :9093 blocked by the security group.
Terraform: Error acquiring the state lockA prior run crashed. Verify no live run, then terraform force-unlock <LOCK_ID>.
CI fails assuming the AWS roleOIDC trust policy sub claim doesn't match the repo/branch/environment. Check the condition in the IAM role trust relationship.
AMI data source returns nothingPacker build hasn't run yet or the name filter doesn't match. Confirm with aws ec2 describe-images --owners self.

Line-by-line explainers — read every file like a story

▶

This part explains the GitLab and Terraform files one line at a time, in plain language — imagine a friend reading over your shoulder. We also explain how GitLab runners work, because that's the piece most people find confusing. The files below are the same ones in the EC2 starter project (a single web server on AWS).

First, the big picture: how the pieces talk to each other

Think of it like ordering food. You write down what you want (the code). GitLab is the restaurant that takes your order. The runner is the cook who actually makes it. Terraform is the recipe-follower, and AWS is the kitchen where the real food (servers) gets made.

Key idea  You never log into a server to build things by hand. You write files, push them, and robots (the runner + Terraform) do the work the exact same way every time.

How GitLab runners work (the "cook" explained)

A runner is just a computer (or a small program on a computer) whose only job is to run the steps in your pipeline. When you push code, GitLab makes a list of jobs to do. A runner raises its hand, says "I'll take that job," does it in a fresh clean workspace, and reports back pass or fail.

Runner
The machine/program that actually runs your jobs. You can use GitLab's shared runners (free, already set up) or host your own.
Executor
The way a runner runs a job. The most common is "Docker," which means each job runs inside a fresh, clean container.
Container
A throwaway mini-computer that starts from a clean image every time. Nothing from the last job leaks in.
Tags
Labels you can put on jobs and runners so the right runner picks up the right job (e.g. only a runner tagged aws).
You git push your code. GitLab notices the new commit.
GitLab reads .gitlab-ci.yml and builds a to-do list of jobs (validate, plan, apply…).
A free runner volunteers for the next job that's ready.
The runner starts a fresh container from the image you picked (here, the Terraform image).
It downloads your code, then runs the job's commands line by line.
It reports pass or fail back to GitLab, saves any artifacts, and deletes the container.

File 1: .gitlab-ci.yml — the recipe card for the pipeline

This file lives at the top of your project. GitLab reads it to learn what jobs to run and in what order. YAML is just a way of writing lists and settings using spaces for nesting. Here it is, line by line:

.gitlab-ci.yml
Picking the computer image every job starts from
image: name: hashicorp/terraform:1.9 entrypoint: [""]
"Start each job inside this pre-made computer." The hashicorp/terraform:1.9 image already has Terraform installed, so we don't have to install it ourselves. entrypoint: [""] turns off the image's auto-start command so GitLab can run our own commands instead.
Variables = sticky notes the whole pipeline can read
variables: TF_ROOT: ${CI_PROJECT_DIR}/terraform TF_STATE_NAME: ec2-demo AWS_DEFAULT_REGION: us-east-1
Reusable values. TF_ROOT is the folder holding our .tf files. CI_PROJECT_DIR is a built-in GitLab variable meaning "where the code was downloaded." TF_STATE_NAME names our saved state. AWS_DEFAULT_REGION tells AWS which part of the world to build in.
TF_ADDRESS: ${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/terraform/state/${TF_STATE_NAME}
The web address where GitLab keeps our Terraform "memory." Terraform needs to remember what it already built; GitLab stores that for us at this URL. CI_API_V4_URL and CI_PROJECT_ID are filled in automatically.
Cache = keep some files between jobs to go faster
cache: key: "${TF_STATE_NAME}" paths: - ${TF_ROOT}/.terraform
"Save this folder so the next job doesn't re-download everything." The .terraform folder holds downloaded plugins. Re-using it makes later jobs start quicker.
A reusable block of setup steps (an "anchor")
.terraform_init: &terraform_init before_script: - cd "${TF_ROOT}" - terraform --version
A shortcut we can paste into many jobs. The dot in .terraform_init means "this is a template, don't run it on its own." &terraform_init gives it a nickname. before_script = steps that run before the main job. First we move into the Terraform folder and print the version.
- export TF_HTTP_ADDRESS="${TF_ADDRESS}" - export TF_HTTP_USERNAME="gitlab-ci-token" - export TF_HTTP_PASSWORD="${CI_JOB_TOKEN}" - export TF_HTTP_LOCK_METHOD="POST" - export TF_HTTP_UNLOCK_METHOD="DELETE"
Tell Terraform how to log into GitLab's memory storage. CI_JOB_TOKEN is a temporary password GitLab makes for each job. We use environment variables (not command flags) on purpose — see the warning below. The lock settings stop two pipelines from editing at the same time.
- terraform init -reconfigure
"Get ready." init downloads plugins and connects to the saved state. -reconfigure means "use the settings I just gave you, don't trust old ones."
The order the jobs happen in
stages: - validate - plan - apply - destroy
The four steps, top to bottom. All jobs in validate finish before plan starts, and so on. Think of them as levels in a game you clear in order.
Job 1 — check the code is written correctly
validate: stage: validate <<: *terraform_init script: - terraform fmt -check -recursive - terraform validate
"Is the code tidy and valid?" <<: *terraform_init pastes in our setup shortcut. fmt -check makes sure the formatting is neat. validate checks for spelling/grammar mistakes in the code. This runs on every push.
Job 2 — make a preview of the changes
plan: stage: plan <<: *terraform_init script: - terraform plan -out=plan.cache - terraform show -no-color plan.cache > plan.txt artifacts: paths: - ${TF_ROOT}/plan.cache - ${TF_ROOT}/plan.txt expire_in: 1 week
"Show me what WOULD change, but don't change anything yet." plan writes the preview to plan.cache. We also save a readable copy as plan.txt. artifacts = files saved after the job so the next job (and you) can download them. They auto-delete after a week.
Job 3 — actually build it (only when YOU say go)
apply: stage: apply <<: *terraform_init script: - terraform apply -auto-approve plan.cache - terraform output dependencies: - plan when: manual rules: - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
"Build exactly what the preview showed." apply creates the server. dependencies: plan grabs the saved preview. when: manual means it waits for you to click the ▶ button — nothing builds by surprise. rules only allows this on the main branch, so test branches can't deploy.
Job 4 — tear it down so you stop paying
destroy: stage: destroy <<: *terraform_init script: - terraform destroy -auto-approve when: manual rules: - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
"Delete everything we made." Also a manual ▶ button so it never deletes by accident. Great for demos so a server isn't left running and charging money.
Why environment variables and not -backend-config flags?  If you pass the job token as a command flag, Terraform secretly saves it inside plan.cache. That token dies when the job ends, so the later apply job would fail with a "state lock" error. Using TF_HTTP_* variables avoids this because each job reads a fresh token.

File 2: versions.tf — which tools and where to save memory

Terraform files end in .tf. Splitting them into several files is just for tidiness — Terraform reads them all together. This one sets up the rules of the game.

terraform/versions.tf
terraform { required_version = ">= 1.5"
"You need at least Terraform version 1.5 to run this." Stops someone using a too-old version that might behave differently.
required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" } }
"To talk to AWS, download the official AWS plugin." A provider is a plugin that lets Terraform control a specific service. ~> 5.0 means "any 5.x version" — safe small updates, no surprise big jumps.
backend "http" {} }
"Save my memory using an internet address." Empty braces because the CI pipeline fills in the real address at run time (that's the TF_HTTP_* stuff from File 1). This is how GitLab stores state for us.
provider "aws" { region = var.aws_region
Set up the AWS connection. region = which AWS data-center region. var.aws_region pulls the value from our variables file, so we don't hard-code it.
default_tags { tags = { Project = "gitlab-ec2-demo" ManagedBy = "terraform" } } }
Sticky labels on everything we create. Tags are name-tags on AWS resources. Putting ManagedBy = terraform on everything makes it easy to find later and know "a human shouldn't edit this by hand."
Where are the AWS passwords?  Notice there's no access key in this file. The AWS plugin automatically reads AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY from the environment — which you set as masked variables in GitLab. Secrets never live in the code.

File 3: variables.tf — the knobs you can turn

Variables are like settings on a game before you start — change them without touching the main code.

terraform/variables.tf
variable "aws_region" { description = "AWS region to deploy into." type = string default = "us-east-1" }
A knob named aws_region. description explains it to humans. type = string means it's text. default is the value used if you don't pick one — so this is optional.
variable "instance_type" { type = string default = "t3.micro" }
How big the server is. t3.micro is a tiny, cheap size — perfect for a demo. Change it to t3.small for more power.
variable "ingress_cidr" { type = string default = "0.0.0.0/0" }
Who is allowed to reach the server. 0.0.0.0/0 means "anyone on the internet" — fine for a quick test, but you should narrow it to your own IP for anything real.
Changing a knob from GitLab  Add a CI/CD variable named TF_VAR_instance_type with value t3.small, and Terraform automatically uses it. The TF_VAR_ prefix is the magic word.

File 4: main.tf — the actual server, step by step

This is the heart of it. It does three things: find a network to use, make a firewall, and launch the server.

terraform/main.tf
Part A — borrow the network that already exists
data "aws_vpc" "default" { default = true }
"Find the default network AWS already gave me." A data block looks up something that exists (it doesn't create anything). A VPC is your private network in AWS. Every account gets a default one.
data "aws_subnets" "default" { filter { name = "vpc-id" values = [data.aws_vpc.default.id] } }
"List the smaller sections inside that network." A subnet is a slice of the VPC. We grab them so we can place the server in one.
data "aws_ami" "al2023" { most_recent = true owners = ["amazon"] filter { name = "name" values = ["al2023-ami-*-x86_64"] } }
"Find the newest Amazon Linux 2023 disk image." An AMI is like a saved snapshot of an operating system. most_recent + the name filter means we always get the latest one without hard-coding an ID that goes stale.
Part B — build a firewall (security group)
resource "aws_security_group" "demo" { name = "${var.instance_name}-sg" vpc_id = data.aws_vpc.default.id
A resource block CREATES something real. A security group is a firewall — a list of who's allowed in and out. We give it a name and attach it to our network.
ingress { from_port = 22 to_port = 22 protocol = "tcp" cidr_blocks = [var.ingress_cidr] }
"Allow people to connect on port 22." Ingress = traffic coming IN. Port 22 is SSH (the way you remotely log into a Linux server).
ingress { from_port = 80 to_port = 80 protocol = "tcp" cidr_blocks = [var.ingress_cidr] }
"Allow web traffic on port 80." Port 80 is normal websites (HTTP). This lets a browser load the demo page.
egress { from_port = 0 to_port = 0 protocol = "-1" cidr_blocks = ["0.0.0.0/0"] } }
"Let the server reach out anywhere." Egress = traffic going OUT. protocol = "-1" and port 0 mean "all of it" — needed so the server can download updates.
Part C — launch the actual server
resource "aws_instance" "demo" { ami = data.aws_ami.al2023.id instance_type = var.instance_type subnet_id = data.aws_subnets.default.ids[0] vpc_security_group_ids = [aws_security_group.demo.id]
Create the EC2 server. It uses the AMI we found, the size from our variable, the first subnet from the list ([0] = first item), and the firewall we just built. Notice how blocks reference each other — Terraform figures out the right build order automatically.
user_data = <<-EOF #!/bin/bash dnf install -y httpd systemctl enable --now httpd echo "<h1>Hello from GitLab CI</h1>" > /var/www/html/index.html EOF
A startup script the server runs the first time it boots. user_data = "do this on first start." Here it installs a web server (httpd), turns it on, and writes a hello page. The <<-EOF ... EOF is just a way to write several lines of text.
tags = { Name = var.instance_name } }
Give the server a friendly name tag so you can spot it in the AWS console.

File 5: outputs.tf — the answers Terraform hands back

After building, Terraform can print useful facts — like the server's address — so you don't have to go hunting in the AWS console.

terraform/outputs.tf
output "instance_id" { value = aws_instance.demo.id }
"Print the server's ID number." output = a value shown after apply finishes. aws_instance.demo.id reads the ID from the server we built.
output "public_ip" { value = aws_instance.demo.public_ip }
"Print the public IP address." This is the internet address of your server.
output "web_url" { value = "http://${aws_instance.demo.public_ip}" }
"Print a ready-to-click link." It glues http:// in front of the IP so you can paste it into a browser and see your "Hello" page.
Putting it together  Push code → runner starts → validate checks it → plan previews it → you click ▶ on apply → AWS builds the server → outputs show you the web_url → click ▶ on destroy when you're done. Same steps, every time, no surprises.
