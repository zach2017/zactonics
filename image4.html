Infrastructure Support Team Handbook

Building a Reusable AWS Data Infrastructure Platform

This guide teaches an infrastructure support team how to build and operate a self-service AWS data platform that data users, data engineers, ETL developers, analysts, and operations teams can safely request and consume. It is written to be clear for newer cloud engineers while still useful as a day-to-day operational reference.

Reading tip: Sections are collapsible. Click any More Details button to open a focused modal. Use the font controls and dark mode toggle at the top to suit your environment.

1. The Goal: Safe, Self-Service Infrastructure

The platform's purpose is to let data users request standardized infrastructure (a Kafka cluster, a NiFi node, an ETL worker fleet) and use it safely, without each team reinventing security, patching, and networking. The support team owns the "paved road": approved images, repeatable pipelines, and guardrails.

  • Self-service: users pick from a catalog of approved building blocks.
  • Safe by default: hardening, logging, patching, and backups are built in, not optional.
  • Repeatable: everything is code (AMIs, Ansible, Terraform/CloudFormation) and version-controlled.
  • Auditable: approvals, scans, and inventory give a clear trail of what is running and why.

2–6. Base, Golden & Service AMIs

▶ Amazon Linux 2023 base & company golden AMIs

Base AMI: Start from the current Amazon Linux 2023 (AL2023) AMI. It is the supported, long-term successor to Amazon Linux 2, with a deterministic update model and modern kernel/toolchain.

Golden AMI: A "company-approved golden AMI" is AL2023 plus your mandatory baseline — security hardening (CIS-aligned), CloudWatch Agent, SSM Agent, logging/audit config, corporate CA certs, and tagging. Every service image is built on top of the golden AMI, never directly on the raw base.

Layering model: AL2023 base → Golden AMI (baseline) → Service AMI (Kafka, NiFi, …). Each layer is its own pipeline so a baseline fix re-flows into every service image.
▶ Custom service-specific AMIs

Build one purpose-built AMI per service role, each from the golden AMI:

Kafka broker — JVM, Kafka binaries, tuned page-cache & ulimits
ZooKeeper — JVM, ZK binaries, dedicated data/log disks
Apache NiFi — JVM, NiFi, content/flowfile/provenance repos
PostgreSQL tools / self-managed PG — client tools or full engine
OpenSearch — JVM, OpenSearch, vm.max_map_count tuning
ETL worker — Python/Spark/clients, job runner, agents

Support both AMIs built from AWS Linux base images and approved internal AMIs as parents — the pipeline takes the parent image ID as a parameter.

▶ EC2 Image Builder vs. Packer

EC2 Image Builder — AWS-native, managed pipelines, built-in test & distribution, integrates with SSM and AMI lifecycle. Best when you want minimal infra to maintain and tight AWS integration.

HashiCorp Packer — cloud-agnostic, huge plugin/provisioner ecosystem, easy to run Ansible inline. Best when you want portability or already standardize on HashiCorp tooling.

Either tool produces repeatable AMI pipelines; both can invoke Ansible to do the actual configuration.

From AMI to a Running, Ready-to-Use Service

An AMI by itself is just a template — it does nothing until it is used to launch a running EC2 server (instance). The whole point of the approved images is this: when Terraform or CloudFormation launches EC2 instances from a service AMI, those servers must boot up and bootstrap themselves into a working data service — a live Kafka cluster, a NiFi cluster, an OpenSearch cluster, and so on — that data users can immediately request and use.

The core promise: users get up-and-running services built from the safest, company-approved images. They never hand-build a server or harden an OS — they request a service and receive a healthy, secured, monitored cluster.

What "ready to use after bootstrapping" means

Because the heavy, identical work is already baked into the AMI (hardened OS, JVM, Kafka/ZooKeeper/NiFi/OpenSearch binaries, CloudWatch & SSM agents), the running instance only needs a short bootstrap step at first boot to turn from a generic image into a specific, clustered node. That bootstrap is where per-instance and per-environment details are applied.

Already in the AMI (no boot delay)

  • Hardened Amazon Linux 2023 + security baseline
  • Service software (Kafka, ZooKeeper, NiFi, OpenSearch, PostgreSQL tools, ETL runtime)
  • CloudWatch Agent, SSM Agent, backup agent
  • Service users, directories, tuned kernel/JVM settings

Applied at bootstrap (first boot)

  • Cluster identity: broker/node IDs, ZooKeeper quorum members, seed/peer lists
  • Environment endpoints & secrets pulled from Parameter Store / Secrets Manager
  • Attach & mount the correct data EBS volumes
  • Join the cluster, start the service, register with the load balancer / target group
  • Report healthy so traffic is allowed in

How the bootstrap runs

The first-boot configuration is driven by EC2 user-data and/or SSM State Manager / Run Command, which invoke the same Ansible runtime roles described later. A typical Kafka node's journey from image to working cluster member:

  1. Terraform/CloudFormation launches EC2 instances from /platform/ami/kafka/latest across Multi-AZ subnets.
  2. Each instance boots the pre-baked, hardened image — no OS build needed.
  3. User-data / SSM triggers the Ansible runtime role.
  4. The role fetches this node's broker ID, ZooKeeper connect string, and listeners from Parameter Store.
  5. Data EBS volumes are attached and mounted; Kafka config is templated in.
  6. The broker service starts and joins the existing brokers to form (or extend) the cluster.
  7. Health checks pass → the node registers with its NLB target group → producers/consumers can use it.

This applies to every service: ZooKeeper nodes form a quorum, NiFi nodes form a coordinated cluster, OpenSearch nodes elect masters and balance shards, and ETL workers join their Auto Scaling group — all from the same approved-image-plus-bootstrap pattern, so what users receive is always a working service built on the safest image.

Reference Architecture

The pipeline flows left to right: source config → image build → image registry → infrastructure deployment across environments.

7–8. Configuration with Ansible (Bake vs. Runtime)

Ansible is the single configuration engine, used both inside the AMI pipeline and at runtime via SSM. Roles cover: base Linux setup, security hardening, CloudWatch Agent, SSM Agent, disk mounts, Kafka, ZooKeeper, NiFi, PostgreSQL tools, OpenSearch, ETL workers, and backup agents.

Bake into the AMI

  • OS packages, JVM, service binaries
  • Security hardening & CIS baseline
  • Agents: CloudWatch, SSM, backup
  • Static directory layout & users
  • Anything slow or identical across all instances

Configure at runtime

  • Cluster membership / broker & node IDs
  • Environment-specific endpoints & secrets
  • Data disk mounts tied to attached EBS
  • Anything that differs per instance or per env
  • Fast-changing config pulled from Parameter Store

Rule of thumb: bake what is stable and identical; inject what is unique or secret. This keeps images cache-friendly and avoids rebuilding for every config change.

AMI ID handoff: when a pipeline publishes an approved image, it writes the new ID to a well-known Parameter Store path, e.g. /platform/ami/kafka/latest. Terraform and CloudFormation read that parameter instead of hard-coding IDs, so promotions are a parameter update, not a code change.

11–12. IaC & GitLab CI/CD

Use Terraform and/or CloudFormation to deploy infrastructure from approved AMIs. GitLab CI/CD orchestrates the whole flow with security built in.

▶ Secure GitLab pipeline design

AWS OIDC auth: GitLab assumes an IAM role via OpenID Connect — no long-lived AWS keys stored in CI variables.

Guardrails: protected branches, merge approvals, and environment-scoped permissions.

Quality & security stages: SAST/secret/dependency scans, Ansible lint, terraform validate + plan review.

Image stages: AMI build → automated test → publish → write AMI ID to Parameter Store.

Promotion: dev → stage → production, each gated by approvals and successful tests.

Two Pipelines, Two Repositories

Separate the base/golden AMI build from the service AMI build and deployment. These change at different speeds, have different owners, and need different approvals — so they live in two GitLab repositories, each with its own pipeline. The base pipeline produces a hardened, approved parent image; the service pipeline consumes it, layers the service on top, and deploys infrastructure with Terraform or CloudFormation.

The handoff: the base pipeline publishes its approved AMI ID to SSM Parameter Store (e.g. /platform/ami/golden/latest). The service pipeline reads that parameter as its parent image. This decouples the two — a base re-bake never forces a service code change, and a service build always starts from the current approved base.

Repo 1 — Base / Golden AMI

Owner: platform/security team · Changes: slow (patches, baseline updates)

Builds AL2023 → hardened golden AMI. Output is an approved parent image, nothing service-specific.

platform-base-ami/
├─ .gitlab-ci.yml
├─ packer/ (or imagebuilder/)
│  └─ golden.pkr.hcl
├─ ansible/
│  ├─ baseline.yml
│  └─ roles/ (hardening, agents, cloudwatch, ssm)
├─ tests/ (smoke + compliance)
└─ README.md

Repo 2 — Service AMI + Deploy

Owner: data platform / service team · Changes: faster (service config, infra)

Reads the golden AMI from Parameter Store, layers the service (Kafka, etc.), then deploys with Terraform/CloudFormation.

├─ .gitlab-ci.yml
├─ packer/ (parent = golden AMI param)
├─ ansible/roles/ (kafka, zookeeper, nifi, opensearch, etl)
├─ terraform/ (or cloudformation/)
│  ├─ modules/
│  └─ envs/ dev · stage · prod
├─ scripts/ (bootstrap user-data)
└─ tests/

Pipeline 1 — Base / Golden AMI (detailed stages)

Triggered on merge to main in platform-base-ami, or on a schedule (e.g. monthly patch cycle) so the golden image stays current. Authenticates to AWS via OIDC; no static keys.

▶ Stage walkthrough
1. validate — ansible-lint, yamllint, packer validate/packer fmt -check. Fails fast on style/syntax before any cloud cost is incurred.
2. security-scan — secret scan (e.g. gitleaks), dependency/SAST scan, and a static policy check of the Packer/Ansible config. Blocks merge on high-severity findings.
3. build — Packer (or EC2 Image Builder) launches a temporary EC2 from the current AL2023 base, then runs the baseline Ansible roles: OS update, CIS-style hardening, CloudWatch Agent, SSM Agent, audit/logging, corporate CA certs, tagging. Produces an unvalidated AMI.
4. test — boots an instance from the new AMI and runs automated checks: agents report in, hardening controls present (e.g. via a compliance tool / InSpec-style tests), no unexpected open ports, expected packages and versions installed.
5. scan-image — vulnerability scan of the built AMI (e.g. Amazon Inspector). Fails the pipeline if critical CVEs are present.
6. publish — tags the AMI as approved (commit SHA, build date, pipeline ID), optionally copies it cross-region, and writes the new AMI ID to SSM Parameter Store at /platform/ami/golden/latest (plus a versioned path for rollback).
7. notify / trigger — optionally calls the Pipeline 2 trigger API so dependent service images rebuild on the new base.
Approvals & guardrails: protected main, mandatory MR review by the platform/security team, environment-scoped OIDC role limited to image-build actions, and manual approval before the publish stage promotes an image to "approved".
▶ Inputs, outputs & rollback

Inputs: AL2023 source AMI (looked up by SSM public parameter), baseline Ansible roles, hardening policy, allowed region list.

Outputs: one approved golden AMI per region; its ID in Parameter Store; build metadata and scan reports as pipeline artifacts.

Rollback: because each build writes a versioned parameter (e.g. /platform/ami/golden/2026-06-01) alongside /latest, you can repoint /latest to a previous known-good AMI without rebuilding.

Config tool: Ansible does the work; a tiny shell/cloud-init step only bootstraps Ansible inside the builder instance.

Pipeline 2 — Service AMI + Deploy (detailed stages)

Lives in data-service-platform. Runs per service (Kafka, ZooKeeper, NiFi, OpenSearch, ETL) — typically one pipeline configuration parameterized by service, or a child pipeline per service. Starts from the approved golden AMI it reads from Parameter Store.

▶ Stage walkthrough
1. validate — ansible-lint, terraform fmt -check + terraform validate (or cfn-lint), and a service config schema check.
2. security-scan — secret scan, SAST, IaC security scan (e.g. tfsec/checkov) on the Terraform/CloudFormation, plus dependency scanning of service config.
3. resolve-parent — reads /platform/ami/golden/latest from Parameter Store to pin the parent image; the build fails if no approved golden AMI exists.
4. build service AMI — Packer launches from the golden AMI and runs the service Ansible role (e.g. kafka): installs JVM + Kafka binaries, lays down config templates, sets tuned kernel/JVM/ulimit values, places the bootstrap script. Only the service layer is added — the hardened base is inherited.
5. test service AMI — boots an instance, confirms the service binary is present and starts, ports listen, and a single-node smoke test passes (e.g. create a topic and produce/consume for Kafka).
6. scan + publish — vulnerability scan, tag as approved, then write the service AMI ID to its own path, e.g. /platform/ami/kafka/latest.
7. deploy-dev — Terraform/CloudFormation reads the service AMI ID and launches a Multi-AZ cluster in dev: an Auto Scaling group / launch template per node role, EBS data volumes, security groups, load balancer/target groups, and SSM associations that run the runtime bootstrap so nodes join the cluster.
8. integration-test (dev) — validates the live cluster: brokers register, quorum forms, health checks green, end-to-end produce/consume or query succeeds.
9. deploy-stage — same Terraform with stage variables; manual approval gate; prod-like validation.
10. deploy-prod — manual approval + change record; deploys the HA Multi-AZ cluster. Rollout is controlled (rolling/canary) so the cluster stays available.
Approvals & guardrails: protected branches, MR review by the service team, per-environment OIDC roles (dev role can't touch prod), terraform plan posted to the MR for review, and manual approvals before stage and prod.
▶ Inputs, outputs & rollback

Inputs: golden AMI ID (from Parameter Store), service Ansible role, Terraform/CFN modules, per-environment variable sets, bootstrap user-data script.

Outputs: an approved service AMI and its Parameter Store ID; a running, health-checked Multi-AZ cluster per environment; Terraform state and plan artifacts.

Rollback: repoint the service AMI parameter to the prior version and re-run deploy, or use Terraform to revert to the last applied state; Auto Scaling replaces nodes with the rolled-back image automatically.

Config tool: Ansible builds the AMI and runs at first boot (same roles, tag-gated --tags build vs --tags runtime); a short shell/cloud-init bootstrap in user-data fetches and invokes it.

Configuration layer: Ansible vs. shell vs. other

Both pipelines need a tool to actually configure the image and the running instance. Ansible is the recommended default for anything non-trivial; shell scripts are fine for tiny, fast bootstraps. Use them together where it makes sense (a thin shell wrapper that invokes Ansible).

13–14. Manual & Emergency Operations

Automation is the default, but the team must be able to act manually under pressure. Support all three paths for both AMI creation and backup/restore:

AWS Console
Fastest for one-off, human-driven emergency actions.
AWS CLI
Scriptable, repeatable, good for documented runbooks.
SSM Automation
Pre-approved runbooks with logging & least privilege.

Discipline: any emergency/manual AMI or restore should be tagged, logged, and back-filled into code afterward so the pipeline remains the source of truth.

16–18. HA, Clustering & Multi-AZ

Production systems must avoid single-node designs. A single node is a single point of failure: one AZ outage, one bad patch, or one disk loss takes the whole service down. Multi-AZ spreads nodes across at least two (ideally three) Availability Zones so the service survives the loss of an entire AZ.

ServiceCluster recommendation
Kafka brokers≥3 brokers across AZs; replication factor 3, min ISR 2.
ZooKeeperOdd quorum (3 or 5) across AZs for majority voting.
NiFiMulti-node cluster (≥3) coordinated by ZooKeeper.
PostgreSQLRDS Multi-AZ or Aurora with reader replicas; auto-failover.
OpenSearch3 dedicated masters + data nodes across AZs; shard replicas.
ETL workersAuto Scaling group across AZs; scale on queue depth.

Add failover (health-checked replacement), load management (balancers + autoscaling), and fault tolerance (replication, retries, idempotent processing) at every layer.

19. Load Balancing & Traffic Routing

Application Load Balancer (ALB)
Layer 7, HTTP/HTTPS, path/host routing — good for NiFi UIs, REST/dashboards.
Network Load Balancer (NLB)
Layer 4, ultra-low latency, static IPs — good for Kafka/TCP workloads.
Route 53 failover
DNS health checks route away from unhealthy endpoints/regions.
Service mesh
App Mesh / Istio for mTLS, retries, and traffic shaping in ECS/EKS.

20. Scaling Options

  • Horizontal scaling — add more nodes/instances; preferred for stateless and shardable workloads.
  • Vertical scaling — bigger instance types; simplest but bounded and disruptive.
  • Auto Scaling groups — automatic capacity for ETL workers and stateless tiers.
  • Managed services — offload scaling to AWS (MSK, RDS/Aurora, OpenSearch Service).
  • Containers — package workloads for portability and density.
  • AWS Batch / Glue / EMR / ECS / EKS — purpose-built compute for batch, serverless ETL, big-data, and orchestrated containers.
