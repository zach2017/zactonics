Ansible, step by step

A practical walkthrough using your Docker Compose lab — one controller, two managed targets, one bridge network. From docker compose up to roles and vault.

01What you already have

Your Docker Compose file describes three containers on a single bridge network:

ContainerRoleOSNotable ports
ansible-controllerRuns AnsibleUbuntu 24.04None exposed
target1Managed nodeUbuntu 24.048081 → 80, sshd 22
target2Managed nodeUbuntu 24.048082 → 80, sshd 22

A few things baked in by the Dockerfiles that matter for the tutorial:

  • The controller has Ansible installed via pipx, plus openssh-client, sshpass, ping, and editors. The host directory ./ansible is bind-mounted to /ansible inside the controller.
  • Each target runs sshd on port 22 (only reachable from lab-net, not from your host), has python3 and python3-apt installed, has nginx pre-installed, and accepts SSH login as root with the password root.
  • Because Compose puts all three containers on lab-net, the controller can ssh root@target1 directly. No IP juggling needed.

02Starting the lab

From the directory containing your docker-compose.yml, on your host:

docker compose up -d --build

--build ensures images are built the first time. -d runs in the background. Verify everything is healthy:

docker compose ps

All three containers should be running, with target1 and target2 showing (healthy). The healthcheck probes TCP port 22 inside each target, so "healthy" means sshd is actually accepting connections — which is why ansible-controller is configured to wait via depends_on: condition: service_healthy.

03Getting a shell on the controller

All Ansible commands run from the controller. Open a shell:

docker exec -it ansible-controller bash

You'll land in /ansible. This directory is bind-mounted from ./ansible on your host, so files persist across container restarts.

ansible --version

You should see ansible [core 2.x.x] plus the Python interpreter path.

04Bootstrapping SSH access

Ansible talks to managed nodes over SSH. The controller must trust each target's host key, and we need working credentials.

4a · Accept host keys

mkdir -p ~/.ssh && chmod 700 ~/.ssh
ssh-keyscan -H target1 target2 >> ~/.ssh/known_hosts

4b · Set up SSH keys (recommended)

ssh-keygen -t ed25519 -N "" -f ~/.ssh/id_ed25519

sshpass -p root ssh-copy-id -o StrictHostKeyChecking=no root@target1
sshpass -p root ssh-copy-id -o StrictHostKeyChecking=no root@target2

ssh root@target1 hostname
ssh root@target2 hostname

The last two commands should return the hostname without asking for a password.

05Your first inventory

The inventory tells Ansible what hosts exist and how to reach them. Create /ansible/inventory.ini:

[web]
target1
target2

[web:vars]
ansible_user=root
ansible_ssh_private_key_file=~/.ssh/id_ed25519
ansible_python_interpreter=/usr/bin/python3

If you skipped key setup, swap the key line for ansible_password=root. Verify:

ansible-inventory -i inventory.ini --list
Tip. Create /ansible/ansible.cfg with inventory = ./inventory.ini, host_key_checking = False, and stdout_callback = yaml to avoid typing -i inventory.ini every time and get readable output.

06Your first ad-hoc commands

Ad-hoc commands run a single module against your inventory — no playbook required.

ansible -i inventory.ini web -m ping
ansible -i inventory.ini web -m command -a "uptime"
ansible -i inventory.ini target1 -m setup
ansible -i inventory.ini web -m apt -a "name=htop state=present update_cache=yes" --become

The pattern after ansible is a host pattern. You can combine groups, exclude hosts, use wildcards — web:!target2 means "web group except target2".

07Your first playbook

A playbook is YAML describing tasks to run against hosts. Create /ansible/01-webserver.yml:

---
- name: Configure a basic nginx web server
  hosts: web
  become: true
  gather_facts: true

  tasks:
    - name: Ensure nginx is installed
      ansible.builtin.apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Ensure nginx is running
      ansible.builtin.service:
        name: nginx
        state: started

    - name: Drop a custom index page
      ansible.builtin.copy:
        dest: /var/www/html/index.html
        content: "<h1>Hello from {{ inventory_hostname }}</h1>\n"
        mode: "0644"
ansible-playbook -i inventory.ini 01-webserver.yml

From your host machine verify the result:

curl http://localhost:8081   # target1
curl http://localhost:8082   # target2

Ansible is idempotent: a second run reports ok instead of changed — state already matches what you declared.

Why no enabled: yes? These containers run sshd as PID 1, not systemd. There's no init system to enable against — state: started is enough for the lab.

08Variables, facts, and templates

Real playbooks separate data (variables) from logic (tasks). Create /ansible/group_vars/web.yml:

---
site_title: "Ansible Lab"
site_admin_email: "admin@example.test"
nginx_worker_processes: 2

And the template /ansible/templates/index.html.j2:

<!doctype html>
<html>
<head><title>{{ site_title }}</title></head>
<body>
  <h1>{{ site_title }} — {{ inventory_hostname }}</h1>
  <p>Running {{ ansible_distribution }} {{ ansible_distribution_version }}.</p>
  <p>IPv4: {{ ansible_default_ipv4.address }}</p>
</body>
</html>

Anything {{ ansible_* }} is a fact — discovered automatically when gather_facts: true. Replace the copy task with:

- name: Render the site index from a template
  ansible.builtin.template:
    src: templates/index.html.j2
    dest: /var/www/html/index.html
    mode: "0644"

09Handlers — only run when something changes

If you change nginx config, you want to reload nginx — but only if it actually changed. That's what handlers do.

---
- name: Configure nginx
  hosts: web
  become: true

  tasks:
    - name: Deploy nginx config
      ansible.builtin.template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        mode: "0644"
      notify: Reload nginx

    - name: Ensure nginx is running
      ansible.builtin.service:
        name: nginx
        state: started

  handlers:
    - name: Reload nginx
      ansible.builtin.service:
        name: nginx
        state: reloaded

First run: template task is changed → handler fires at end of play. Second run: nothing changed → handler does not fire. Restart-only-when-needed is built in.

10Loops and conditionals

- name: Install handy tools
  ansible.builtin.apt:
    name: "{{ item }}"
    state: present
  loop:
    - htop
    - tree
    - jq

- name: Create app users
  ansible.builtin.user:
    name: "{{ item.name }}"
    shell: "{{ item.shell }}"
  loop:
    - { name: alice, shell: /bin/bash }
    - { name: bob,   shell: /bin/sh }

- name: Place a flag file only on target1
  ansible.builtin.copy:
    dest: /etc/primary.flag
    content: "I am the primary\n"
  when: inventory_hostname == "target1"

11Roles — organizing real projects

A role bundles tasks, handlers, templates, and defaults into a reusable directory. Scaffold one:

mkdir -p roles
ansible-galaxy role init roles/webserver

This creates the standard layout:

roles/webserver/
├── defaults/main.yml      # default vars (lowest precedence)
├── files/                 # static files
├── handlers/main.yml      # handlers
├── meta/main.yml          # role metadata
├── tasks/main.yml         # the tasks themselves
├── templates/             # Jinja2 templates
└── vars/main.yml          # role vars (higher than defaults)

Move your templates into roles/webserver/templates/, then write a thin top-level /ansible/site.yml:

---
- name: Configure web servers
  hosts: web
  become: true
  roles:
    - webserver

From inside a role, the template module looks in the role's templates/ directory automatically — no path prefix needed in src.

12Ansible Vault — encrypting secrets

You don't want passwords sitting in plaintext YAML in git. Vault encrypts files with a passphrase.

ansible-vault create group_vars/web_secrets.yml
# (opens editor; save secrets, exit)

ansible-vault edit   group_vars/web_secrets.yml
ansible-vault view   group_vars/web_secrets.yml
ansible-vault rekey  group_vars/web_secrets.yml

ansible-playbook -i inventory.ini site.yml --ask-vault-pass

Variables in the encrypted file are referenced as normal — {{ db_password }} — and Ansible decrypts at runtime.

13Tags, check mode, and diff

Three quality-of-life features you'll use constantly.

# Run only tasks tagged 'config'
ansible-playbook -i inventory.ini site.yml --tags config

# Dry run — show what WOULD change, change nothing
ansible-playbook -i inventory.ini site.yml --check

# Dry run + line-by-line diff of file changes
ansible-playbook -i inventory.ini site.yml --check --diff

Combining --check --diff before a real run on production is the closest thing Ansible has to a "preview" — get in the habit.

14Tearing down

docker compose down                 # stop and remove containers
docker compose down --rmi all -v    # also remove images and volumes

Files under ./ansible/ on your host stay put — they aren't in the container's filesystem.

Common gotchas in this specific lab

  • Targets aren't reachable from your host on port 22 — only 8081/8082 are published. Do all Ansible work from inside the controller.
  • service: enabled: yes will fail on the targets because they run sshd as PID 1, not systemd. Use state: started only.
  • Host key checking — pre-scan keys, or set host_key_checking = False in ansible.cfg, or the first connection hangs on yes/no.
  • /ansible is bind-mounted from ./ansible/ on your host — files persist across docker compose down.
