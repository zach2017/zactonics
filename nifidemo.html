🏪 POS Scanner Producer

Emits one FlowFile per checkout transaction. Connects to the register TCP socket or Redis stream from the POS middleware. Uses GenerateFlowFile in dev, ListenTCP in production.

Volume: ~14 txns/min per register × 4 registers = ~56 FlowFiles/min at STORE-042.

Each FlowFile wraps a JSON array containing all line items from one transaction. A custom Java processor (POSScannerProcessor) stamps all required FlowFile attributes, then hands off to ValidateRecord which enforces the Avro schema before the FlowFile enters the main pipeline.

🔒 PII & Data Rules

cashier_id is an employee ID (PII). It is preserved in full for Inventory and Billing, but masked to *** on the analytics routing path by a ReplaceText processor placed before Kafka publish.

loyalty_card numbers are SHA-256 hashed at the producer before entering the pipeline. The raw card number never appears in any FlowFile attribute or content body.

ValidateRecord is placed immediately downstream. Malformed scans (wrong types, missing required fields, enum violations) are routed to a dead-letter queue and trigger an ops alert via PagerDuty.

POSScannerProcessor.java — Full Implementation PRODUCER 1 JAVA
@Tags({"grocery", "pos", "producer", "scanner"})
@CapabilityDescription("Reads POS scan events and emits FlowFiles with full attribute metadata")
@SupportsBatching
public class POSScannerProcessor extends AbstractProcessor {

    static final PropertyDescriptor STORE_ID = new PropertyDescriptor.Builder()
        .name("store-id").displayName("Store ID")
        .required(true).defaultValue("STORE-042")
        .addValidator(StandardValidators.NON_EMPTY_VALIDATOR).build();

    static final PropertyDescriptor SCHEMA_VER = new PropertyDescriptor.Builder()
        .name("schema-version").required(true)
        .defaultValue("pos-scan-event-v2")
        .addValidator(StandardValidators.NON_EMPTY_VALIDATOR).build();

    static final Relationship SUCCESS = new Relationship.Builder()
        .name("success").description("Scan event emitted OK").build();
    static final Relationship FAILURE = new Relationship.Builder()
        .name("failure").description("Malformed or unreadable scan").build();

    @Override
    public void onTrigger(ProcessContext ctx, ProcessSession session) {
        FlowFile ff = session.create();
        try {
            String storeId  = ctx.getProperty(STORE_ID).getValue();
            String schema   = ctx.getProperty(SCHEMA_VER).getValue();
            POSScanEvent evt = readNextScanEvent(); // TCP socket / Redis stream

            // Stamp all FlowFile attributes ───────────────────
            Map<String,String> attrs = new HashMap<>();
            attrs.put("source.type",     "pos_scanner");
            attrs.put("store.id",        storeId);
            attrs.put("item.category",   evt.getPrimaryCategory());
            attrs.put("event.id",        evt.getTransactionId().toString());
            attrs.put("event.time",      evt.getTimestamp().toString());
            attrs.put("mime.type",       "application/json");
            attrs.put("schema.name",     schema);
            attrs.put("pipeline.version","2.4.1");
            ff = session.putAllAttributes(ff, attrs);

            // Write JSON body ──────────────────────────────────
            String json = ObjectMapper.writeValueAsString(evt);
            ff = session.write(ff, out -> out.write(json.getBytes(StandardCharsets.UTF_8)));
            session.transfer(ff, SUCCESS);
            session.commitAsync();
            getLogger().info("POS event emitted txn={} store={} cat={}",
                evt.getTransactionId(), storeId, evt.getPrimaryCategory());
        } catch (Exception e) {
            getLogger().error("POS scan failed: {}", e.getMessage(), e);
            session.transfer(ff, FAILURE);
        }
    }
}

🚚 Supplier Feed Producer

Inbound HTTP listener using NiFi's built-in ListenHTTP on port 9999. External supplier systems (distributors, farms, wholesalers) POST JSON webhook payloads whenever stock events occur.

Three event types are supported: restock (new delivery), price_update (new cost/retail price), and recall (product safety recall — highest priority, bypasses batching).

Endpoint: https://nifi.internal:9999/supplier/ingest  |  Each supplier has a unique mTLS cert from the Grocery CA. Cert CN is validated against the X-Supplier-ID header to prevent spoofing.

⚠ Recall Fast-Path

Events where delivery_type=recall receive a priority=HIGH attribute via UpdateAttribute. A separate connection from the listener bypasses MergeContent entirely and routes directly to RouteOnAttribute, then to InventoryDB and a PagerDuty alerting processor.

Recall events must reach InventoryDB within 3 seconds of receipt. Back-pressure on the to_alert relationship queue triggers a P1 incident alert. Monitor this queue in the NiFi UI.

Recall events also trigger a downstream RecallAlertProcessor that calls the store PA system API to broadcast an in-store announcement and locks the affected SKU from further POS transactions.

🔃 RouteOnAttribute — Fan-out Routing Rules PROCESSOR
Routes each FlowFile to one or more consumer queues. A single FlowFile can match multiple routes simultaneously — analytics receives everything; billing receives POS sales only; recalls also trigger the alert path.
// NiFi Expression Language routing predicates:

"to_inventory"  : "${literal(true)}"
// ALL events update stock — sales decrement, restocks increment

"to_billing"    : "${source.type:equals('pos_scanner')}"
// Only POS sales — supplier cost data never enters billing path

"to_analytics"  : "${literal(true)}"
// ALL events, PII scrubbed before Kafka publish

"to_alert"      : "${delivery.type:equals('recall')}"
// Recall events → PagerDuty alert processor (zero batching delay)

Routing Strategy: Route to Property Name
Unmatched:        dead-letter-queue connection → ops alert

📦 InventoryDB Consumer

Receives ALL FlowFiles from both producers via the to_inventory route. Upserts rows into inventory.stock_ledger in PostgreSQL. Sale events decrement stock quantities; restock events increment them; recall events set a recalled=true flag.

Idempotent — uses po.number / event.id as the primary key. Duplicate deliveries are silently skipped, logged, and transferred to the success relationship.

The processor uses a JsonTreeReader record reader with field translation (camelCase → snake_case) so JSON field names map directly to column names without explicit field mapping.

🔒 RBAC: inventory-consumer

This role has write access to the INVENTORY-DB-001 processor and its input connection only. It cannot read billing processor configs, analytics configs, or any other consumer group resources.

DB password stored as encrypted Parameter Context variable #{db.inventory.password} using AES-256-GCM. The value never appears in flow.xml in plain text. The JDBC driver credentials are injected only at runtime.

The nifi_writer PostgreSQL role used by this processor has INSERT and UPDATE privileges on inventory.stock_ledger only — no DELETE, no DDL, no access to other schemas.

💳 BillingSystem Consumer

Receives only POS sale FlowFiles (filtered by the to_billing route). POSTs the canonical event JSON to the billing settlement REST API over HTTPS with mutual TLS. Implements retry with exponential back-off (1s → 2s → 4s, max 3 attempts). Uses the event.id FlowFile attribute as the HTTP Idempotency-Key header to prevent double-billing on retry.

Supplier cost_price data is never routed to this consumer. The Jolt transform strips cost_price for the billing path. This is enforced architecturally, not just by configuration.

🔒 RBAC: billing-consumer (most restricted)

The most restricted role in the pipeline. Can only read from the billing input queue and write to BILLING-001. Has no access to other consumer configs, data provenance views, cost price fields, or policy management.

API bearer token stored as sensitive Parameter Context variable #{billing.api.token}, encrypted AES-256-GCM. Explicitly marked sensitive=true in processor config so it is redacted from NiFi UI and audit logs.
BillingHttpConsumer.java — Full Implementation CONSUMER 2 JAVA
@Tags({"billing", "http", "pos", "grocery"})
@CapabilityDescription("POSTs POS sale events to billing API with idempotency and retry")
public class BillingHttpConsumer extends AbstractProcessor {

    static final PropertyDescriptor BILLING_URL = new PropertyDescriptor.Builder()
        .name("billing-url").required(true).sensitive(false)
        .defaultValue("https://billing.internal/api/v2/transactions")
        .addValidator(StandardValidators.URL_VALIDATOR).build();

    static final PropertyDescriptor API_TOKEN = new PropertyDescriptor.Builder()
        .name("api-token").required(true).sensitive(true)  // encrypted in nifi.properties
        .addValidator(StandardValidators.NON_EMPTY_VALIDATOR).build();

    static final PropertyDescriptor MAX_RETRIES = new PropertyDescriptor.Builder()
        .name("max-retries").required(true).defaultValue("3")
        .addValidator(StandardValidators.POSITIVE_INTEGER_VALIDATOR).build();

    static final Relationship SUCCESS = new Relationship.Builder().name("success").build();
    static final Relationship RETRY   = new Relationship.Builder().name("retry").build();

    @Override
    public void onTrigger(ProcessContext ctx, ProcessSession session) {
        FlowFile ff = session.get(); if (ff == null) return;

        String txnId    = ff.getAttribute("event.id");
        String storeId  = ff.getAttribute("store.id");
        String token    = ctx.getProperty(API_TOKEN).getValue();
        String url      = ctx.getProperty(BILLING_URL).getValue();
        int    maxRetry = ctx.getProperty(MAX_RETRIES).asInteger();

        for (int attempt = 0; attempt < maxRetry; attempt++) {
            try {
                HttpRequest req = HttpRequest.newBuilder()
                    .uri(URI.create(url))
                    .header("Content-Type",   "application/json")
                    .header("Authorization",  "Bearer " + token)
                    .header("X-Store-ID",     storeId)
                    .header("Idempotency-Key", txnId)  // prevents double-billing
                    .POST(HttpRequest.BodyPublishers.ofInputStream(
                        () -> session.read(ff)))
                    .timeout(Duration.ofSeconds(10)).build();

                int status = httpClient.send(req,
                    HttpResponse.BodyHandlers.ofString()).statusCode();

                if (status == 200 || status == 201 || status == 409) {
                    // 409 = Conflict: already processed (idempotent OK)
                    ff = session.putAttribute(ff, "billing.status", "settled");
                    ff = session.putAttribute(ff, "billing.http.code", String.valueOf(status));
                    session.transfer(ff, SUCCESS); return;
                }
                // Exponential back-off: 1s, 2s, 4s
                Thread.sleep((long) Math.pow(2, attempt) * 1000L);
            } catch (Exception e) {
                getLogger().warn("Billing attempt {} failed txn={}: {}",
                    attempt + 1, txnId, e.getMessage());
            }
        }
        session.penalize(ff);
        session.transfer(ff, RETRY);
    }
}

📊 Analytics Sink Consumer

Receives all events from both producers after PII scrubbing (cashier_id replaced with *** by a ReplaceText processor). Publishes to Kafka topic grocery.events.all, partitioned by item.category.

Downstream consumers of this Kafka topic include: a Flink streaming job for real-time inventory dashboards, a Spark Streaming ML pipeline for demand forecasting, and a Python consumer for shrinkage / loss detection.

Partitioning by category means department-specific consumers (e.g. the dairy forecasting model) never have to filter through unrelated events — they just consume their partition.

🔒 RBAC: analytics-consumer

Can read all event types but only receives the PII-scrubbed version. Kafka client authenticates using SASL/SCRAM-SHA-512 with a dedicated nifi-analytics-producer service account.

This role cannot access the billing processor config, billing connection, or supplier cost_price fields — those are removed in the Jolt transform before the analytics routing path receives the FlowFile.

Kafka SASL password stored as encrypted Parameter Context var #{kafka.sasl.password}. The Kafka truststore at /opt/nifi/certs/kafka-truststore.jks contains only the Kafka cluster CA cert.

🔒 Security Architecture Overview

NiFi uses a file-based authorizer (upgradeable to LDAP/OIDC in production) defined in /conf/authorizations.xml and /conf/users.xml. Every processor, process group, connection, and controller service is an authorizable resource with discrete read, write, and delete policies.

The grocery pipeline enforces least-privilege: each consumer role can access only the connections and processors it needs. Producers cannot see consumer configs. The NiFi Admin role is restricted to break-glass scenarios with full audit logging.

All inter-node, client, and cluster communication is secured with mutual TLS (mTLS). The NiFi keystore and truststore are provisioned by the internal Grocery CA. Sensitive properties in flow.xml.gz are AES-256-GCM encrypted using nifi.sensitive.props.key — the key itself is injected from HashiCorp Vault at startup via bootstrap.conf.

🏪 POS Scanner — Producer 1

What it does

Emits one FlowFile per checkout transaction. Connects to register TCP socket or Redis stream from POS middleware. GenerateFlowFile in dev, ListenTCP in production.

Volume: ~56 FlowFiles/min across 4 registers at STORE-042.

Key FlowFile Attributes

source.typepos_scanner
store.idSTORE-042
item.categoryproduce | dairy | frozen | deli …
event.idtransaction_id — Billing idempotency key
schema.namepos-scan-event-v2
mime.typeapplication/json

Security

RBAC role pos-producer — write to POS-SCANNER-001 only. loyalty_card SHA-256 hashed at source before pipeline entry.

🚚 Supplier Feed — Producer 2

What it does

Inbound HTTP listener on port 9999. Suppliers POST JSON webhooks for restock, price_update, or recall events. Authenticated via mutual TLS client certificates — each supplier has a unique cert from the Grocery CA.

Recall events bypass MergeContent — zero-delay routing to InventoryDB + PagerDuty alert.

Key FlowFile Attributes

source.typesupplier_feed
supplier.id${http.headers.X-Supplier-ID}
delivery.typerestock | price_update | recall
priorityHIGH (recall) | NORMAL
http.client.dnTLS cert DN — anti-spoofing

Security

RBAC role supplier-producer — cannot see POS processor or any other group. Cert CN validated against X-Supplier-ID header.

🔀 MergeContent Processor

What it does

Batches FlowFiles from both producers grouped by source.type. Emits a JSON array when 100 items accumulate or 5 seconds elapse. Recall events (priority=HIGH) bypass this processor via a separate direct connection.

Key Settings

Max Entries100
Max Bin Age5 sec
Correlation Keysource.type
FormatJSON Array (newline delimiter)

🔧 JoltTransformJSON

What it does

Normalises both POS and supplier payloads to a unified canonical grocery-event schema using a Jolt shift spec. Both shapes map to the same output before routing so all downstream consumers deal with one schema.

Key Field Mappings

transaction_id→ event.id
po_number→ event.id (supplier path)
items[].qty→ event.items[].quantity
timestamp→ event.occurred_at

🔃 RouteOnAttribute

Fan-out routing rules

Each FlowFile can match multiple routes simultaneously. Analytics receives everything; Billing receives POS sales only; Recalls also trigger the alert path.

Routes

to_inventoryALL events (true)
to_billingsource.type = pos_scanner only
to_analyticsALL events (PII scrubbed)
to_alertdelivery.type = recall only

📦 InventoryDB — Consumer 1

What it does

Receives ALL events. Upserts into inventory.stock_ledger (PostgreSQL). Sales decrement stock; restocks increment; recalls set the recalled flag. Idempotent on event.id primary key.

Duplicate event_id keys are silently skipped and logged — no double-counting.

RBAC: inventory-consumer

/processors/INVENTORY-DB-001READ+WRITE
/processors/BILLING-001DENY
/policies/*DENY

💳 BillingSystem — Consumer 2

What it does

POS sales only. POSTs JSON to billing settlement API (HTTPS + mTLS). Exponential retry (1s, 2s, 4s). Idempotency-Key header prevents double-billing on retry.

cost_price from supplier events is never routed here — stripped by JoltTransform.

RBAC: billing-consumer (most restricted)

/processors/BILLING-001READ+WRITE
/provenance-data/BILLING-001DENY
/processors/KAFKA-ANALYTICS-001DENY

📊 Analytics Sink — Consumer 3

What it does

ALL events after PII scrubbing. Publishes to Kafka grocery.events.all, partitioned by item.category (7 partitions). SASL/SCRAM-SHA-512 auth. Feeds Flink dashboards, Spark demand forecasting, and shrinkage detection.

RBAC: analytics-consumer

/processors/KAFKA-ANALYTICS-001READ+WRITE
/data (PII-scrubbed only)READ
/processors/BILLING-001DENY
