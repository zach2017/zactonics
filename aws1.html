Deployment model & security baseline

What we deploy

Start from a plain Amazon Linux 2023 base image only — no Kafka baked in at the infrastructure layer. The IaC (Terraform / CloudFormation) provisions bare EC2 instances, EBS, the NLB and security groups. Everything Kafka-specific is then layered on by Ansible.

  • 1 IaC launches base AL2023 EC2 + EBS + NLB
  • 2 Ansible installs JDK 17, Kafka, systemd units
  • 3 Ansible templates server.properties from cluster.yaml
  • 4 Ansible enforces the security baseline, then starts brokers

Security baseline (base image)

Keep the base image minimal and hardened. Ansible is the single tool responsible for configuration, so the attack surface stays small and auditable.

  • Minimal AL2023 — no extra packages, latest patches via dnf update
  • SSH key-only, no passwords; SSH limited to the bastion CIDR
  • Dedicated non-login kafka service user; brokers never run as root
  • SELinux enforcing; host firewall + tight security groups
  • TLS + SASL/SCRAM for all Kafka traffic; secrets pulled from SSM/Secrets Manager at config time
  • CloudWatch agent + auditd for an immutable audit trail

Migration: ZooKeeper → KRaft

A no-downtime path. Brokers run in dual-write mode (metadata to both ZooKeeper and KRaft) so you can roll back at any point until the final step. Click a stage for detail and references.

    Track progress via JMX: kafka.controller → ZkMigrationState  (target: MIGRATION_COMPLETED)

    Optional: bake a Kafka AMI

    The model above installs Kafka at boot with Ansible. Alternatively you can pre-bake Kafka into a custom AMI for faster, more repeatable launches. Three options, each with trade-offs — click for steps.
