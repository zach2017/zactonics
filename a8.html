Building in the Cloud: A Step-by-Step Guide

Learn how to automatically build a computer in the cloud using AWS, GitLab, and Terraform—explained simply, straight from your Windows PC.

1. What Are We Building? (The Core Concepts)

Instead of clicking buttons on a website to create a computer, we are going to write a "recipe" for a computer. Then, we will give that recipe to a robot helper who will go build it for us. Click the links below to learn about the tools we will use:

2. Getting Ready on Windows

Before we write our recipe, we need some tools on our Windows computer to read it. Think of this as buying your cooking utensils.

  1. Download Git for Windows: This gives you a program called "Git Bash." It's a black window where you can type commands. It speaks the same language as our cloud robots.
  2. Download Terraform: Go to the official website, download the Windows version, and put the .exe file in a special folder on your C: drive so your computer can find it.
  3. Download the AWS CLI: This is a Windows installer that lets your computer talk to AWS securely.
  4. Create Accounts: You will need a free account on AWS and GitLab.

3. The Terraform Files (The Recipe)

Create a new folder on your computer. Inside, we are going to make three plain text files. These files tell AWS exactly what kind of computer to build.

File 1: main.tf (The Main Instructions)

This file says: "I want to use AWS, and I want a small, free computer (EC2 instance)."

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = var.aws_region
}

resource "aws_instance" "my_cool_server" {
  ami           = "ami-0c55b159cbfafe1f0" # A basic Linux starter pack
  instance_type = "t2.micro"              # A tiny, free-tier computer

  tags = {
    Name = "MyMiddleSchoolProject"
  }
}

File 2: variables.tf (The Changeable Parts)

This file holds things that might change, like which city (region) the computer should be built in.

variable "aws_region" {
  description = "The AWS location to build our computer"
  type        = string
  default     = "us-east-1" # This means 'Virginia, USA'
}

File 3: outputs.tf (The Receipt)

When the robot finishes building, we want it to hand us the public IP address so we can connect to our new computer.

output "server_address" {
  description = "The public IP of our new computer"
  value       = aws_instance.my_cool_server.public_ip
}

4. GitLab Setup & The Robot Helper (Pipeline)

Now, we need to put our files into GitLab (our cloud folder) and give instructions to the GitLab Robot (the CI/CD Pipeline) to read our Terraform recipe and actually build the server.

  1. Log into GitLab and click "New Project". Create a blank project.
  2. Upload your three .tf files to this project.
  3. Create one more file in GitLab called exactly: .gitlab-ci.yml. This file is the instruction manual for the GitLab robot.

The Pipeline File: .gitlab-ci.yml

This tells the robot to do three things:
1. Validate: Check our spelling.
2. Plan: Do a practice run and show us what it will build.
3. Apply: Actually build it on AWS.

stages:
  - validate
  - plan
  - apply

image: hashicorp/terraform:latest

before_script:
  - terraform init # Wakes up Terraform and gets it ready

validate_recipe:
  stage: validate
  script:
    - terraform validate

plan_changes:
  stage: plan
  script:
    - terraform plan -out=myplan
  artifacts:
    paths:
      - myplan # Saves the practice run so the apply stage can use it

build_it_for_real:
  stage: apply
  script:
    - terraform apply -auto-approve myplan
  rules:
    - if: $CI_COMMIT_BRANCH == "main" # Only build if we save to the main folder

5. Security Best Practices (Very Important!)

Never put your passwords or secret keys in your code files!

If you put your AWS keys inside main.tf, anyone on the internet could steal them and rent millions of dollars worth of computers on your credit card. Here is how to be safe:

  • Use GitLab Variables: Go to your GitLab project -> Settings -> CI/CD -> Variables. Add your AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY here. Check the "Mask variable" box. This hides your keys so the robot can use them, but nobody can read them.
  • Principle of Least Privilege: Don't give your robot the master key to your whole house. Create a specific AWS user that is only allowed to build EC2 computers, and nothing else.
  • Protect the State File: Terraform creates a secret diary called a "state file" (.tfstate). Never upload this manually! GitLab has a built-in feature called "Managed Terraform State" that locks this file away safely in a digital vault.

Amazon Web Services (AWS)

Imagine a massive, secure warehouse filled with thousands of super-fast computers. Instead of saving up to buy your own heavy computer to run a website, AWS lets you rent a tiny piece of one of their computers over the internet by the minute.

Think of it like renting a booth at a giant digital science fair.

Terraform

Terraform is like a LEGO instruction manual for computers. It is a tool called "Infrastructure as Code" (IaC).

Instead of clicking buttons in AWS for an hour to set up a server, you write a text file saying "I want 1 computer with 2GB of memory." Terraform reads your text file and magically builds it for you.

GitLab

GitLab is two things in one:

  • A Cloud Folder: A place to safely store your code (like Google Drive, but for programmers).
  • An Assembly Line Robot (CI/CD): It watches your folder. Whenever you save a new Terraform recipe, the GitLab robot automatically grabs it, checks it for errors, and hands it to AWS to be built.
