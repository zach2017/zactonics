Chapter 1 — What Are We Building?

Welcome to class. I'll be your teacher, and I'll explain everything the way I'd explain it to a smart middle-school student: simple words, lots of word-pictures (analogies), and small quizzes so the ideas stick.

Our big goal: build a cluster of Kafka servers on Amazon's cloud (AWS) that keeps working even if a server breaks. That "keeps working" idea has a fancy name: high availability (HA) and fault tolerance.

1.1 The Restaurant Analogy (read this first!)

Picture a busy restaurant. This one picture will carry you through the whole tutorial:

  • Waiters take orders and shout them toward the kitchen — these are producers (programs that send messages).
  • Cooks pick up orders and make food — these are consumers (programs that read messages).
  • The order rail (the spinning wheel holding order tickets in line) is Kafka itself.

Kafka is the order rail of the software world: it sits in the middle, holds messages safely and in order, and lets many waiters add tickets while many cooks take them — without collisions.

💡 Teacher's Tip

Whenever you feel lost later, return to the restaurant: producers = waiters, consumers = cooks, Kafka = the rail, brokers = extra copies of the rail so the kitchen never stops.

Chapter 2 — The Cloud & AWS, Explained Simply

2.1 What Is 'The Cloud'?

"The cloud" just means someone else's computers, in a big warehouse, that you rent over the internet. Instead of buying a server and paying its electric bill, you rent one by the hour and give it back when done.

📘 Why this matters

Renting means you can ask for 3 computers today, 30 next week, and 3 again after. You pay only for what you use — exactly what lets us spin up a fault-tolerant cluster in minutes.

2.2 The AWS Services We Use

ServiceWhat it gives you
EC2Virtual computers (servers) you rent.
VPCYour own private network — a fenced yard for servers.
SubnetA section of that yard, tied to one location.
Security GroupA firewall: rules for who may talk to your server.
Availability Zone (AZ)A separate physical data-center building.
RegionA geographic area containing several AZs.
EBSA virtual hard drive attached to an EC2 server.

2.3 Regions & Availability Zones — The Heart of Fault Tolerance

AWS divides the world into Regions (like 'US East'). Inside each Region are several Availability Zones — separate buildings, miles apart, with their own power and internet. If a storm knocks out one building, the others keep running.

The golden rule: never put all your servers in one building. One broker in AZ-a, one in AZ-b, one in AZ-c means losing a whole data center costs only one broker.

⚠️ Common Mistake

Beginners launch all 3 brokers in the same AZ because it's simpler. That's fake fault tolerance — one power outage takes down all 3 at once. Always spread across at least 3 AZs.

Chapter 4 — EC2 & Amazon Linux: Our Servers

4.1 What Is an EC2 Instance?

EC2 = Elastic Compute Cloud. An EC2 instance is one virtual computer — a slice of Amazon's real hardware that behaves like a computer you can log into. 'Elastic' means stretchy: ask for more or fewer anytime.

4.2 Instance Types (T-shirt sizes)

TypevCPUsMemoryGood for
t3.micro21 GBTiny tests (often free-tier)
t3.large28 GBSmall clusters, dev/test
m5.xlarge416 GBProduction brokers (balanced)
r5.2xlarge864 GBHigh-throughput, memory-hungry Kafka
💡 Teacher's Tip

Kafka likes memory and fast disks more than raw CPU. Production often uses m5/r5 with fast EBS. For learning, t3.large is plenty and cheap.

4.3 Amazon Linux & AMIs

Every computer needs an operating system. Amazon Linux 2023 is a free Linux tuned for EC2. When you launch an instance you pick an AMI (Amazon Machine Image) — a frozen snapshot of an installed OS that boots a clean server in about a minute.

4.4 Logging In with SSH

We connect using SSH (Secure Shell), an encrypted tunnel. Instead of a password we use a key pair: a secret private key you keep and a public key AWS stores on the server.

connect-with-ssh.sh
# Connect to your server over SSH (run on your own laptop)
ssh -i my-key.pem ec2-user@<the-server-public-ip>

#  ssh            -> makes the secure connection
#  -i my-key.pem  -> use this private key to prove who you are
#  ec2-user       -> default username on Amazon Linux
#  @<public-ip>   -> address of the server

Chapter 5 — Kafka Deep Dive: The Order Rail

5.1 What Problem Does Kafka Solve?

In a ride-share app, your phone sends location data that the map, billing, driver-matching, and fraud systems all need. Without Kafka the app would send the same data to four places and resend if any were down. Messy!

Kafka fixes this: the app sends data once to Kafka; the four systems read it whenever ready. The sender doesn't know who's listening. This separation is called decoupling.

5.2 Core Kafka Vocabulary

TermPlain meaningRestaurant analogy
Message / RecordOne piece of data (an event).One order ticket
ProducerProgram that sends messages in.Waiter clipping a ticket
ConsumerProgram that reads messages.Cook grabbing a ticket
TopicA named category of messages.A labeled section of the rail
PartitionA slice of a topic for parallelism.Parallel rails for one section
OffsetA message's position number in a partition.The ticket's spot in line
BrokerOne Kafka server.One rail station
ClusterA group of brokers together.All rail stations
Consumer GroupA team of consumers sharing the work.A team of cooks sharing tickets

5.3 Topics & Partitions — Why Slicing Matters

A topic is a category. A single rail can only move so fast, so Kafka splits each topic into partitions — parallel lanes. More partitions = more consumers working at once. Each partition keeps strict order, numbered by offset (0,1,2,3…).

📘 Key insight

Order is guaranteed only inside a single partition — not across the whole topic. To keep two messages in order, send them with the same key so Kafka puts them in the same partition.

5.4 Replication — The Magic Behind Fault Tolerance

Replication keeps multiple copies of every partition on different brokers. One copy is the leader (handles reads/writes); the others are followers (they copy the leader). If the leader's broker dies, a follower is instantly promoted — no data lost.

SettingMeaningHA recommendation
Replication FactorTotal copies of each partition.3 (one leader + two followers)
ISR (In-Sync Replicas)Copies fully caught up with the leader.Keep ≥ 2 in sync
min.insync.replicasFewest in-sync copies to accept a write.2
acks=allProducer waits until all in-sync copies have it.Use for important data
⚠️ The classic safe combo

Replication Factor = 3, min.insync.replicas = 2, acks=all. A message is only 'accepted' once at least two brokers have it. Lose any one broker and never lose data. Memorize this — it's everywhere in real Kafka work.

✓ Quick Quiz — Check Your Understanding

Q1. A Kafka 'topic' is most like…
Topics are named categories; messages live inside.
Q2. Message order is guaranteed…
Each partition keeps strict order; no global order across partitions.
Q3. Which copy handles reads and writes?
The leader serves traffic; followers copy it.
Q4. The classic safe combo is…
Three copies, two in sync, producer waits for all in-sync.

Chapter 6 — CloudFormation Template, Line by Line

Now we put it to work. Below is a real CloudFormation template that builds the network and three brokers across three AZs. We show it in four parts and explain every line.

6.1 Header & Parameters

kafka-cluster.yaml (1 of 4)
AWSTemplateFormatVersion: "2010-09-09"
Description: >
  Highly-available Kafka cluster - 3 broker EC2 instances across 3 AZs.

Parameters:
  KeyName:
    Type: AWS::EC2::KeyPair::KeyName        # Lets you pick an existing SSH key
    Description: SSH key pair for logging into the brokers.
  InstanceType:
    Type: String
    Default: t3.large                       # Used if you don't choose
    AllowedValues: [t3.large, m5.large, m5.xlarge]
    Description: EC2 size for each broker.
  LatestAmiId:
    Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
    Default: /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64
    Description: Auto-fetches newest Amazon Linux 2023 AMI.
  • AWSTemplateFormatVersion — the template language version; always this exact string.
  • Description: > — human description; the > lets text wrap onto indented lines.
  • KeyName — its special type shows a dropdown of your SSH keys so you can log in later.
  • InstanceType — a text choice with a Default and AllowedValues so no invalid size can be typed.
  • LatestAmiId — asks AWS for the newest Amazon Linux AMI automatically, so it never goes stale.

6.2 Network: VPC, Subnets, Internet Access

kafka-cluster.yaml (2 of 4)
Resources:
  KafkaVPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.0.0.0/16                # ~65,000 private addresses
      EnableDnsSupport: true
      EnableDnsHostnames: true
  InternetGateway:
    Type: AWS::EC2::InternetGateway         # Door to the public internet
  AttachGateway:
    Type: AWS::EC2::VPCGatewayAttachment
    Properties:
      VpcId: !Ref KafkaVPC
      InternetGatewayId: !Ref InternetGateway
  SubnetA:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref KafkaVPC
      CidrBlock: 10.0.1.0/24
      AvailabilityZone: !Select [0, !GetAZs ""]   # 1st AZ
      MapPublicIpOnLaunch: true
  SubnetB:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref KafkaVPC
      CidrBlock: 10.0.2.0/24
      AvailabilityZone: !Select [1, !GetAZs ""]   # 2nd AZ
      MapPublicIpOnLaunch: true
  SubnetC:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref KafkaVPC
      CidrBlock: 10.0.3.0/24
      AvailabilityZone: !Select [2, !GetAZs ""]   # 3rd AZ
      MapPublicIpOnLaunch: true
  • KafkaVPC — our private network; CidrBlock: 10.0.0.0/16 reserves ~65,000 private addresses.
  • InternetGateway + AttachGateway — give the network a door to the internet to download Kafka and SSH in.
  • !Ref KafkaVPC — intrinsic function meaning 'insert the ID of KafkaVPC'; this links resources.
  • SubnetA/B/C — three subnets, one per AZ — the fault-tolerance secret written as code.
  • !GetAZs "" returns the region's AZs; !Select [0,…] picks the first, [1,…] the second, etc.
  • MapPublicIpOnLaunch: true — gives a public IP (fine for learning; production keeps brokers private).
⚠️ Production note

Public IPs on brokers are convenient but risky. In production, place brokers in private subnets and reach them via a bastion host or VPN. We keep it simple to focus on Kafka.

6.3 The Firewall: Security Group

kafka-cluster.yaml (3 of 4)
  KafkaSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow SSH and Kafka traffic
      VpcId: !Ref KafkaVPC
      SecurityGroupIngress:
        - IpProtocol: tcp           # Rule 1: SSH login
          FromPort: 22
          ToPort: 22
          CidrIp: 0.0.0.0/0         # WARNING: open to the world
        - IpProtocol: tcp           # Rule 2: Kafka client port
          FromPort: 9092
          ToPort: 9092
          CidrIp: 10.0.0.0/16       # Only inside our VPC
        - IpProtocol: tcp           # Rule 3: broker + controller
          FromPort: 9093
          ToPort: 9094
          CidrIp: 10.0.0.0/16
        - IpProtocol: tcp           # Rule 4: ZooKeeper (ZK mode only)
          FromPort: 2181
          ToPort: 2181
          CidrIp: 10.0.0.0/16
  • SecurityGroupIngress — list of allowed incoming connections; anything not listed is blocked.
  • Rule 1 (22) — SSH admin login. 0.0.0.0/0 means 'from anywhere' — convenient but insecure.
  • Rule 2 (9092) — main Kafka port for producers/consumers, limited to inside the VPC.
  • Rule 3 (9093-9094) — broker-to-broker and, in KRaft, controller traffic.
  • Rule 4 (2181) — ZooKeeper's port; only needed in ZooKeeper mode.
⚠️ Security mistake to avoid

Opening SSH (22) to 0.0.0.0/0 lets anyone on the internet attempt login. Restrict it to your own IP, or use AWS Systems Manager Session Manager and drop port 22 entirely.

6.4 The Brokers & Outputs

kafka-cluster.yaml (4 of 4)
  Broker1:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !Ref InstanceType
      ImageId: !Ref LatestAmiId
      KeyName: !Ref KeyName
      SubnetId: !Ref SubnetA            # Broker 1 in AZ-a
      SecurityGroupIds: [!Ref KafkaSecurityGroup]
      BlockDeviceMappings:
        - DeviceName: /dev/xvdb
          Ebs: { VolumeSize: 100, VolumeType: gp3 }
      Tags:
        - { Key: Name, Value: kafka-broker-1 }
        - { Key: BrokerId, Value: "1" }
  Broker2:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !Ref InstanceType
      ImageId: !Ref LatestAmiId
      KeyName: !Ref KeyName
      SubnetId: !Ref SubnetB            # Broker 2 in AZ-b
      SecurityGroupIds: [!Ref KafkaSecurityGroup]
      BlockDeviceMappings:
        - DeviceName: /dev/xvdb
          Ebs: { VolumeSize: 100, VolumeType: gp3 }
      Tags:
        - { Key: Name, Value: kafka-broker-2 }
        - { Key: BrokerId, Value: "2" }
  Broker3:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !Ref InstanceType
      ImageId: !Ref LatestAmiId
      KeyName: !Ref KeyName
      SubnetId: !Ref SubnetC            # Broker 3 in AZ-c
      SecurityGroupIds: [!Ref KafkaSecurityGroup]
      BlockDeviceMappings:
        - DeviceName: /dev/xvdb
          Ebs: { VolumeSize: 100, VolumeType: gp3 }
      Tags:
        - { Key: Name, Value: kafka-broker-3 }
        - { Key: BrokerId, Value: "3" }

Outputs:
  Broker1PrivateIp: { Value: !GetAtt Broker1.PrivateIp }
  Broker2PrivateIp: { Value: !GetAtt Broker2.PrivateIp }
  Broker3PrivateIp: { Value: !GetAtt Broker3.PrivateIp }
  • Broker1/2/3 — three identical EC2 instances, each pointed at a different subnet (SubnetA/B/C), one per AZ.
  • BlockDeviceMappings — attaches a 100 GB gp3 EBS disk (/dev/xvdb) for Kafka data, off the boot disk.
  • Tags — labels; the BrokerId tag lets Ansible give each broker a unique ID later.
  • Outputs — prints each broker's private IP; !GetAtt Broker1.PrivateIp gets that attribute.
💡 !Ref vs !GetAtt

Use !Ref for a resource's main ID or a parameter's value. Use !GetAtt to pull a specific attribute like .PrivateIp. Rule of thumb: !Ref = the whole thing's ID; !GetAtt = one detail about it.

Chapter 7 — Ansible: The Robot That Installs Kafka

CloudFormation built three empty Linux computers — but no Kafka yet. Rather than SSH into each and type install commands three times, we use Ansible: it runs your setup steps on many servers at once, automatically.

Ansible is declarative (you describe the end state, e.g. 'Java should be installed') and idempotent (running it twice does no harm — already-done steps are skipped). You can safely re-run it anytime.

7.1 Ansible Vocabulary

TermPlain meaning

Managed nodeA server Ansible sets up (our brokers).
InventoryA list of servers with their addresses.
PlaybookA YAML file listing tasks in order.
TaskOne step (install a package, copy a file).
ModuleThe built-in tool a task uses (e.g., 'dnf').
RoleA reusable bundle of tasks, files, templates.
TemplateA config file with blanks Ansible fills per-server.

7.2 The Inventory File

inventory.ini
# The list of brokers Ansible will configure
[kafka_brokers]
broker1 ansible_host=10.0.1.10 broker_id=1
broker2 ansible_host=10.0.2.10 broker_id=2
broker3 ansible_host=10.0.3.10 broker_id=3

[kafka_brokers:vars]
ansible_user=ec2-user
ansible_ssh_private_key_file=~/.ssh/my-key.pem
  • [kafka_brokers] — a group name; run tasks on the whole group at once.
  • ansible_host — each server's IP address.
  • broker_id=1/2/3 — a custom variable giving each broker its unique Kafka ID.
  • [kafka_brokers:vars] — shared settings: the login user and SSH key.

7.3 A Playbook to Install Kafka

install-kafka.yml
# Run: ansible-playbook -i inventory.ini install-kafka.yml
- name: Install and prepare Kafka on all brokers
  hosts: kafka_brokers          # Apply to every server in that group
  become: true                  # Run as root (sudo)
  vars:
    kafka_version: "3.7.0"
    kafka_scala: "2.13"
    kafka_home: /opt/kafka
  tasks:
    - name: Install Java 17 (Kafka requires Java)
      ansible.builtin.dnf:
        name: java-17-amazon-corretto-headless
        state: present          # Make sure it's installed
    - name: Create a dedicated kafka user
      ansible.builtin.user:
        name: kafka
        system: true
    - name: Download Kafka release
      ansible.builtin.get_url:
        url: "https://downloads.apache.org/kafka/{{ kafka_version }}/kafka_{{ kafka_scala }}-{{ kafka_version }}.tgz"
        dest: /tmp/kafka.tgz
    - name: Unpack Kafka into /opt/kafka
      ansible.builtin.unarchive:
        src: /tmp/kafka.tgz
        dest: /opt/
        remote_src: true
        extra_opts: [--strip-components=1, --one-top-level={{ kafka_home }}]
    - name: Make a data folder on the extra disk
      ansible.builtin.file:
        path: /data/kafka
        state: directory
        owner: kafka
        group: kafka
  • hosts: kafka_brokers — apply to all three brokers at once.
  • become: true — use sudo, needed to install software.
  • vars: — values reused below via {{ ... }}, so changing the version is a one-line edit.
  • dnf module — installs Java; state: present makes it idempotent.
  • get_url / unarchive — download Kafka, then unpack it into /opt/kafka.
  • file module — creates the data directory on the EBS disk and gives the kafka user ownership.
💡 Teacher's Tip

Notice we never typed an IP inside the playbook. The playbook says WHAT to do; the inventory says WHERE. The same playbook works for 3 brokers or 300.

Chapter 8 — Setup Path A: Kafka with ZooKeeper

8.1 What Is ZooKeeper?

For most of Kafka's history it needed ZooKeeper — think of it as the kitchen's shift manager. It doesn't carry tickets; it keeps the list of which brokers exist, who leads each partition, and the cluster settings. It's the shared notebook of truth.

ZooKeeper runs as a small cluster (usually 3 nodes), so it too is fault tolerant. An odd number is required because it decides by majority vote (a quorum), and odd numbers avoid ties.

📘 Why an odd number?

With 3 nodes you need 2 to agree; you can lose 1 and still vote. With 4 you'd need 3 and still only tolerate 1 failure — wasting a server. Odd numbers give the best failure tolerance per dollar. The same math applies to KRaft controllers.

8.2 ZooKeeper Config, Line by Line

zookeeper.properties
tickTime=2000
dataDir=/data/zookeeper
clientPort=2181
initLimit=10
syncLimit=5
# The 3-node ensemble: server.ID=host:peerPort:electionPort
server.1=10.0.1.10:2888:3888
server.2=10.0.2.10:2888:3888
server.3=10.0.3.10:2888:3888
  • tickTime=2000 — ZooKeeper's heartbeat in milliseconds; its basic time unit.
  • dataDir — where it stores data, including the myid file holding this node's number.
  • clientPort=2181 — the port Kafka brokers use to reach ZooKeeper.
  • initLimit / syncLimit — how long followers may take to connect and stay in sync, in ticks.
  • server.1/2/3 — the three members. Port 2888 is follower-to-leader chatter; 3888 is elections.

8.3 Kafka Broker Config (ZooKeeper mode), Line by Line

server.properties (ZooKeeper mode)
broker.id=1
listeners=PLAINTEXT://0.0.0.0:9092
advertised.listeners=PLAINTEXT://10.0.1.10:9092
log.dirs=/data/kafka
num.partitions=6

# --- Connect to the ZooKeeper ensemble ---
zookeeper.connect=10.0.1.10:2181,10.0.2.10:2181,10.0.3.10:2181

# --- High-availability / durability ---
default.replication.factor=3
min.insync.replicas=2
offsets.topic.replication.factor=3
transaction.state.log.replication.factor=3
transaction.state.log.min.isr=2
unclean.leader.election.enable=false
  • broker.id=1 — unique per broker; Ansible plugs in the inventory's broker_id.
  • listeners — the address/port the broker listens on; 0.0.0.0 means all interfaces.
  • advertised.listeners — the address the broker tells clients to use; must be reachable, so the private IP.
  • log.dirs=/data/kafka — where messages are stored, on the EBS disk.
  • zookeeper.connect — all 3 ZooKeeper nodes, so the broker finds the manager even if one is down.
  • default.replication.factor=3 + min.insync.replicas=2 — the safe combo applied to new topics.
  • offsets.topic… / transaction… — replicate Kafka's internal bookkeeping topics 3× too.
  • unclean.leader.election.enable=false — refuses to promote an out-of-date replica; safety over uptime.
⚠️ unclean.leader.election

Setting this false is a deliberate trade: if no in-sync replica is available, that partition pauses rather than serving stale data. For most businesses a brief pause beats silently losing records. Leave it false unless you truly prefer availability over correctness.

✓ Quick Quiz — Check Your Understanding

Q1. ZooKeeper's job is to…
It's the metadata manager, not the message store.
Q2. Why an ODD number of ZooKeeper nodes?
Quorum needs a majority; odd avoids waste and ties.
Q3. advertised.listeners controls…
Clients use the advertised address; it must reach them.
Q4. unclean.leader.election.enable=false means…
Avoids data loss by not electing a behind replica.

Chapter 9 — Setup Path B: Kafka with KRaft (No ZooKeeper)

9.1 What Is KRaft?

Running two systems (Kafka AND ZooKeeper) is extra work. So Kafka built the manager into itself. This mode is KRaft ('Kafka Raft', after the Raft consensus algorithm). ZooKeeper is being retired; KRaft is the present and future.

In KRaft, some nodes act as controllers (the old ZooKeeper role). You can run dedicated controllers or let brokers be both ('combined mode'). Controllers form a quorum, so the odd-number rule applies (typically 3).

ZooKeeper modeKRaft mode
Systems to runTwo (Kafka + ZooKeeper)One (Kafka only)
Who stores metadataZooKeeperKafka controllers (a special topic)
Failover speedSlowerFaster
Max partitionsLower limitMuch higher (millions)
Future supportBeing phased outThe recommended path

9.2 Generating a Cluster ID

format-kraft-storage.sh
# Run ONCE on any broker, then reuse everywhere
KAFKA_CLUSTER_ID=$(/opt/kafka/bin/kafka-storage.sh random-uuid)
echo "$KAFKA_CLUSTER_ID"     # e.g. MkU3OEVBNTcwNTJENDM2Qk

# On EVERY node, format storage with that shared ID + config
/opt/kafka/bin/kafka-storage.sh format \
    -t "$KAFKA_CLUSTER_ID" \
    -c /opt/kafka/config/kraft/server.properties
  • random-uuid — makes a fresh cluster ID. Generate it once; do NOT make a new one per node.
  • kafka-storage.sh format — stamps that ID onto each node's data dir so they belong to one cluster.

9.3 KRaft Broker Config, Line by Line

server.properties (KRaft mode)
process.roles=broker,controller          # This node is BOTH
node.id=1
controller.quorum.voters=1@10.0.1.10:9093,2@10.0.2.10:9093,3@10.0.3.10:9093

listeners=PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093
advertised.listeners=PLAINTEXT://10.0.1.10:9092
controller.listener.names=CONTROLLER
inter.broker.listener.name=PLAINTEXT

log.dirs=/data/kafka
num.partitions=6

# --- Same safe durability combo as ZooKeeper mode ---
default.replication.factor=3
min.insync.replicas=2
offsets.topic.replication.factor=3
transaction.state.log.replication.factor=3
transaction.state.log.min.isr=2
unclean.leader.election.enable=false
  • process.roles=broker,controller — this node does both jobs (combined mode); big clusters split them.
  • node.id=1 — the unique ID (KRaft's name for ZooKeeper-mode's broker.id).
  • controller.quorum.voters — controller nodes as id@host:port; this replaces zookeeper.connect.
  • listeners — now TWO entries: PLAINTEXT (9092) for clients and CONTROLLER (9093) for the quorum.
  • controller.listener.names — tells Kafka which listener is the controller one.
  • inter.broker.listener.name — which listener brokers use to talk to each other.
  • The durability block is identical to ZooKeeper mode — replication works the same regardless of manager.
💡 The big difference

Compare the two configs: durability settings are the SAME. Only metadata management changed — zookeeper.connect became controller.quorum.voters, plus a controller listener. Understand one config and you nearly understand both.

Chapter 10 — Designing for High Availability

10.1 The Four Layers of Fault Tolerance

  1. Spread across AZs. One broker per AZ means a whole data-center outage costs only one broker.
  2. Replicate the data. Replication factor 3 keeps three copies of every partition.
  3. Require a quorum of copies. min.insync.replicas=2 and acks=all mean a write is accepted only once two brokers have it.
  4. Protect the metadata. Run 3 ZooKeeper nodes (or 3 KRaft controllers) so the manager survives a failure.

10.2 How a Failover Actually Happens

  1. Broker 2, leader for partition P, suddenly crashes (power loss in AZ-b).
  2. The controller (ZooKeeper or KRaft) notices Broker 2 stopped sending heartbeats.
  3. It looks at the in-sync replicas for P and promotes one — say the copy on Broker 1 — to leader.
  4. Producers and consumers are told the new leader's address and reconnect automatically.
  5. Because of acks=all and min.insync.replicas=2, every acknowledged message was already on Broker 1. Nothing is lost.
  6. When Broker 2 returns, it becomes a follower again and catches up.
📘 The key takeaway

Fault tolerance isn't one switch. It's spreading out (AZs) + copying (replication) + waiting for copies (acks + min.insync.replicas) + a resilient manager (quorum). Remove any one and the guarantee weakens.

10.3 Producer & Consumer Settings for HA

Setting (side)ValueWhy
acks (producer)allWait until all in-sync replicas store the message.
retries (producer)highAuto-resend transient errors instead of dropping.
enable.idempotence (producer)truePrevents duplicate messages during retries.
max.in.flight… (producer)≤ 5Keeps ordering safe when idempotence is on.
enable.auto.commit (consumer)falseCommit offsets only after truly processing.

Chapter 11 — Testing: Break It on Purpose

A fault-tolerant system you've never tested is just a hope. Engineers practice chaos — deliberately breaking things in a safe environment to prove the system heals.

11.1 Create a Test Topic

create-and-describe-topic.sh
# Create a topic with 6 partitions and 3 copies each
/opt/kafka/bin/kafka-topics.sh --create \
  --topic orders --partitions 6 --replication-factor 3 \
  --bootstrap-server 10.0.1.10:9092

# Describe it to SEE leaders and replicas
/opt/kafka/bin/kafka-topics.sh --describe \
  --topic orders --bootstrap-server 10.0.1.10:9092

The --describe output lists each partition's Leader, Replicas, and Isr (in-sync replicas). Note these — you'll watch them change when you kill a broker.

11.2 Experiment 1 — Kill a Broker

experiment-kill-broker.sh
# On broker 2, simulate a crash
sudo systemctl stop kafka

# Re-run --describe: partitions led by 2 now have a NEW leader,
# and broker 2 disappears from the Isr lists.

# Keep producing -- it should keep working!
echo "test-after-failure" | /opt/kafka/bin/kafka-console-producer.sh \
  --topic orders --bootstrap-server 10.0.1.10:9092

Expected result: producing and consuming continue with only a tiny hiccup as leadership moves. Run systemctl start kafka on broker 2 and watch it rejoin the Isr as it catches up.

11.3 Experiment 2 — Verify No Data Loss

  1. Before killing anything, produce 1000 numbered messages.
  2. Kill the leader broker mid-stream.
  3. After failover, consume from the beginning and count.
  4. With acks=all and min.insync.replicas=2, every acknowledged message should still be there.

11.4 Experiment 3 — Lose a Whole AZ

Stop the broker in one AZ AND its ZooKeeper/controller node together, simulating a full data-center outage. With 3 AZs and quorum settings, the remaining two keep the cluster alive — the ultimate proof your AZ spread works.

⚠️ Test safely

Run chaos experiments in a dedicated test cluster, never first on production. Start small (one broker), confirm recovery, then escalate (a whole AZ). Keep monitoring on so you can SEE it heal.

Chapter 12 — Monitoring: Knowing Before It Breaks

Monitoring is the smoke detector of your cluster — it warns you early, ideally before users notice. Kafka exposes its inner numbers (metrics) through Java's JMX, which monitoring tools collect.

12.1 The Most Important Metrics

MetricWhat it tells youWhy you care
UnderReplicatedPartitionsPartitions missing a copy.Should be 0; above 0 means a broker is behind/down.
OfflinePartitionsCountPartitions with no leader.Should be 0; above 0 means data is unavailable!
ActiveControllerCountHow many controllers are active.Should be exactly 1 across the cluster.
Consumer LagHow far behind consumers are.Growing lag = consumers can't keep up.
RequestHandlerAvgIdlePercentHow busy broker threads are.Near 0% = broker overloaded.
Disk usageFree space on log.dirs.Full disks crash brokers — alert early.

12.2 A Simple Monitoring Stack

  • JMX Exporter — a small agent on each broker that turns JMX metrics into a Prometheus format.
  • Prometheus — a server that scrapes those metrics every few seconds and stores them over time.
  • Grafana — dashboards that draw the metrics as graphs you understand at a glance.
  • Alertmanager — emails/Slacks you when a metric crosses a danger line.

On AWS you can also push metrics to Amazon CloudWatch and set alarms there, which pairs with EC2's own health checks.

12.3 A Sample Alert Rule

prometheus-alert.yml
groups:
  - name: kafka-critical
    rules:
      - alert: OfflinePartitions
        expr: kafka_controller_offlinepartitionscount > 0
        for: 1m                       # Must stay true for 1 minute
        labels: { severity: page }    # 'page' = wake someone up
        annotations:
          summary: "Kafka has offline partitions (data unavailable)!"
  • expr — the condition checked continuously; here, any offline (leaderless) partition.
  • for: 1m — avoids false alarms by requiring the problem to persist a minute.
  • severity: page — routes this to an urgent channel that wakes an on-call engineer.

✓ Quick Quiz — Check Your Understanding

Q1. Which metric should normally be 0?

Q2. ActiveControllerCount across the cluster should be…
Exactly one active controller; 0 or 2+ signals trouble.
Q3. Grafana's job is to…
Prometheus collects; Grafana visualizes.
Q4. Why use 'for: 1m' in an alert?
Requiring persistence filters out noise.

Chapter 13 — Best Practices Checklist

A consolidated checklist — your pre-flight list before calling a cluster 'production-ready'.

Infrastructure & CloudFormation

  • Spread brokers across at least 3 Availability Zones.
  • Keep templates in Git; review changes before deploying.
  • Use Parameters and Mappings instead of hard-coding AMI IDs.
  • Put brokers in private subnets; reach them via a bastion or Session Manager.
  • Use separate fast EBS (gp3) volumes for Kafka data, sized with room to grow.
  • Restrict Security Group rules to the narrowest source ranges.

Kafka Configuration

  • Replication factor 3, min.insync.replicas 2, producers using acks=all.
  • unclean.leader.election.enable=false to protect against data loss.
  • Replicate internal topics (offsets, transactions) 3 times too.
  • Prefer KRaft for new clusters; plan migration off ZooKeeper.
  • Enable producer idempotence; commit consumer offsets only after processing.
  • Use TLS encryption and SASL authentication in production (we skipped these for clarity).

Operations

  • Automate everything with CloudFormation + Ansible; avoid manual clicks.
  • Monitor under-replicated/offline partitions and consumer lag; alert on them.
  • Run chaos tests regularly in a non-production cluster.
  • Plan capacity: watch disk, CPU, and network headroom.
  • Document a runbook: what to do when a broker or AZ fails.
💡 Final Teacher's Tip

The theme across all of this: redundancy + automation + observation. Have spares, build them the same way every time, and always be watching. Master those three habits and you can run almost any reliable system.

Chapter 14 — Apache NiFi: Moving Data Without Code

So far our waiters (producers) and cooks (consumers) were programs we wrote. But often you just need to move data from one place to another — read from Kafka, drop it in a file, push it to a database — without writing and deploying a whole program. That’s what Apache NiFi is for.

Think of NiFi as a smart conveyor-belt builder for data. You drag boxes onto a screen, draw arrows between them, and data starts flowing. Each box is a processor that does one small job; the arrows are connections that carry the data along.

💡 Restaurant analogy, extended

If Kafka is the order rail, NiFi is the runner who carries tickets between stations and also makes a photocopy for the records drawer. NiFi doesn’t replace Kafka — it connects Kafka to everything else (files, databases, other systems) using clicks instead of code.

14.1 The NiFi Vocabulary

TermPlain meaningConveyor-belt analogy
FlowFileOne piece of data moving through NiFi (content + attributes).A box on the belt with a label
ProcessorA block that does one job (read, transform, write).A station along the belt
ConnectionA queue carrying FlowFiles between processors.The belt segment between stations
RelationshipA labeled exit from a processor (success, failure).Which chute a box drops into
Processor GroupA folder of processors you can manage together.A whole sub-assembly line
Controller ServiceShared settings many processors reuse (e.g. a reader).A shared toolbox
Back PressureNiFi pauses upstream when a queue fills up.Belt stops so boxes don’t pile off the end

14.2 Why NiFi Pairs So Well with Kafka

  • No code to deploy. Need Kafka messages saved to files for an auditor? Drag two processors and connect them — done in minutes.
  • Built-in Kafka processors. ConsumeKafka reads a topic; PublishKafka writes to one. They handle offsets and consumer groups for you.
  • Hundreds of other processors. Write files (PutFile), call APIs (InvokeHTTP), load databases (PutDatabaseRecord), log data (LogAttribute).
  • Visibility. You can watch boxes flow live, see queue sizes, and right-click any FlowFile to inspect its contents — great for learning and debugging.
  • Back pressure & retries are built in, so a slow database downstream won’t crash the pipeline.
📘 NiFi vs. Kafka Connect

Kafka Connect is another popular tool for the same job (moving data in/out of Kafka via pre-built connectors). NiFi is broader and visual; Connect is Kafka-focused and config-driven. For learning and mixed sources, NiFi’s drag-and-drop is friendlier. Both are valid — many shops use both.

14.3 Our First Flow: Sales → File + Log

Here’s the exact flow used in the companion demo. It consumes the grocery.sales topic and fans each message out to two places: the NiFi log (so you can watch it) and a file on disk (so it’s saved). One input, two outputs:

nifi-flow-overview
# A NiFi flow is just boxes and arrows:

  ConsumeKafka (topic: grocery.sales)
        |  success
        +-------------------> LogAttribute   # writes each message to the NiFi log
        |
        +-------------------> PutFile        # writes each message as a file in /data/output

Three processors, two connections — both fed by the success relationship of ConsumeKafka. That’s the whole pipeline.

14.4 Running NiFi in Docker

NiFi 2.x runs secured (HTTPS + a single login) by default. For a local demo you set a username and a password of at least 12 characters:

docker-compose.yml (nifi service)
  nifi:
    image: apache/nifi:latest        # always pull the current image
    ports: ["8443:8443"]            # UI at https://localhost:8443/nifi
    environment:
      SINGLE_USER_CREDENTIALS_USERNAME: admin
      SINGLE_USER_CREDENTIALS_PASSWORD: freshchain-nifi-admin  # 12+ chars!
    volumes:
      - nifi-output:/data/output     # PutFile writes here; shared with other services
⚠️ Two NiFi gotchas

First, the password must be 12+ characters or NiFi ignores it and generates a random one (check the logs with docker logs nifi | grep Generated). Second, NiFi is heavy and slow to start — give it a minute or two before the UI or its API responds, and budget ~2 GB of memory for it.

14.5 Two Ways to Build the Flow

You can build it by hand in the UI — drag a ConsumeKafka, a LogAttribute, and a PutFile onto the canvas, then draw the success arrows. Great for learning because you watch it happen.

Or you build it automatically over NiFi’s REST API, so it’s repeatable and version-controlled (the same “infrastructure as code” idea from Chapter 3, applied to dataflows). The demo logs in for a token, then creates the processors and connections:

provision_flow.py (excerpt)
# 1) get a token (NiFi 2.x is secured by default)
token = post("/nifi-api/access/token", user, password)

# 2) create the ConsumeKafka processor
consume = create_processor("ConsumeKafka", props={
    "bootstrap.servers": "kafka1:19092,kafka2:19092",
    "topic": "grocery.sales",
    "Group ID": "nifi-freshchain"})

# 3) create LogAttribute + PutFile, then connect success -> both
log  = create_processor("LogAttribute", props={"Log Payload": "true"})
put  = create_processor("PutFile",      props={"Directory": "/data/output"})
connect(consume, log, ["success"])
connect(consume, put, ["success"])

Re-running the script is safe — it checks whether the flow group already exists first (the same idempotent idea Ansible taught us in Chapter 7).

14.6 Seeing It Work

  • In NiFi: open https://localhost:8443/nifi, accept the self-signed certificate, and watch FlowFiles travel along the connections.
  • On disk: docker compose exec nifi ls /data/output shows one file per sales message.
  • In the log: docker compose logs nifi | grep LogAttribute prints each message NiFi handled.
💡 Try this

Send a few sales, then add a third branch: drag a PutDatabaseRecord processor and connect ConsumeKafka’s success to it as well. One Kafka topic now feeds a log, a file, AND a database — and you never wrote a consumer program. That’s the power of NiFi.

Chapter 15 — Integration Training: Connecting Everything

You now know the pieces: Kafka (the rail), producers and consumers (waiters and cooks), NiFi (the runner), and the monitoring stack (the smoke detectors). Integration is the skill of wiring these pieces together so data flows end-to-end through a real system. This chapter is the training that ties it all together.

15.1 The Big Picture: One Event, Many Destinations

The whole point of Kafka-based integration is that a producer sends an event once, and any number of systems react to it independently. Picture a grocery store the moment a sale happens:

end-to-end event flow
  Cash register (producer)
        |  publishes one "sale" event
        v
   +===========================  Kafka topic: grocery.sales  ===========================+
        |              |                |                  |                    |
        v              v                v                  v                    v
   Inventory      Profit/Owner     NiFi -> file        Monitoring          Other apps
   consumer       consumer         + NiFi log         (metrics)           (subscribe later)

Each consumer reads at its own pace and can’t slow the others down. Add a new system tomorrow? It just subscribes to the topic — nobody else changes. That decoupling (Chapter 5) is what makes integration scale.

15.2 The Common Integration Patterns

PatternWhat it meansTool in our stack

Kafka → SinkGet events OUT to a file, DB, or API.Consumer code, or NiFi ConsumeKafka + PutFile

15.3 Hands-On: Trace a Message All the Way Through

The best integration training is following one message from birth to every resting place. Do this with the companion demo running:

  1. Produce: click “Simulate a sale” in the dashboard (or run the producer). One event lands on grocery.sales.
  2. Confirm in Kafka: the API’s consumer reads it; revenue and inventory update on the dashboard.
  3. Confirm in NiFi: within seconds a new file appears in /data/output and a line appears in the NiFi log.
  4. Confirm in monitoring: the “messages produced/consumed” counters tick up in the Monitoring tab and Grafana.
  5. Prove decoupling: stop NiFi, send more sales, restart NiFi — it resumes from where it left off (its consumer group remembered the offset), and the other consumers never noticed.
📘 Why offsets make integration robust

Each consumer group tracks its own offset (its place in line, Chapter 5). That’s why a system can go down, come back, and pick up exactly where it stopped — without the producer resending. Offsets are the quiet hero of reliable integration.

15.4 Integration Rules of Thumb

  • Give every topic one clear meaning. grocery.sales carries sales — not sales-and-returns-and-restocks mixed together. Clear topics keep consumers simple.
  • Use a key when order matters. Same key → same partition → ordered (Chapter 5). For per-store ordering, key by store ID.
  • Agree on a message format. JSON is easy to start; a schema (e.g. Avro with a schema registry) prevents one team breaking another’s consumer.
  • Make consumers idempotent. A message may arrive twice after a retry; processing it twice should be harmless (Chapter 10’s idempotence idea).
  • Separate “move data” from “business logic.” Let NiFi/Connect handle plumbing; keep real decisions in your apps.
  • Watch consumer lag. Growing lag (Chapter 12) is the earliest sign an integration can’t keep up.

15.5 Common Integration Mistakes

⚠️ Avoid these

One giant topic for everything (consumers waste effort filtering). No schema agreement (a field rename silently breaks a downstream system). Auto-committing offsets before processing (a crash then loses messages — commit AFTER, Chapter 10). Tight coupling (one consumer calling another directly instead of via a topic) recreates the very mess Kafka removes.
