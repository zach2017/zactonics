Step-by-step guide to learn AWS deployments with GitLab CI/CD and Terraform

This guide teaches a new learner how cloud deployment works using simple language. You will create a GitLab repository, write Terraform files, connect to AWS, run a safe pipeline, and deploy a small EC2 web server from Windows.

Goal

Build a repeatable deployment, not a one-time click in the AWS console.

Audience

New AWS, GitLab, and Terraform users working from Windows.

Safety

Use least privilege, protected variables, state locking, scanning, and manual approval.

1. Big picture: what are we building?

Think of this like building with instructions. AWS is the place where the computer server lives. Terraform is the instruction sheet that says what to build. GitLab is the robot helper that reads the instructions and runs them every time you update the project.

Simple flow

Windows Laptop
  ↓ git push
GitLab Repository
  ↓ pipeline runs
Terraform fmt → validate → plan → manual apply
  ↓ secure AWS access
AWS creates VPC, Security Group, EC2, and web page

3. AWS setup before Terraform

Step 1: Protect the account

Enable MFA, avoid using the root account, and create roles or users with only the permissions needed.

Step 2: Create a budget

Set a small budget alert before deploying. EC2, NAT Gateway, EBS, and data transfer can cost money.

Step 3: Pick a region

Use one region for learning, such as us-east-1. Keep it consistent in Terraform and GitLab variables.

Step 4: Choose authentication

Best is short-lived access with OIDC role federation. For a beginner lab, protected GitLab variables can work, but rotate keys and use least privilege.

terraform/versions.tf

  required_version = ">= 1.8.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 6.0"
    }
  }

  # For teams, use a remote backend like S3 + DynamoDB locking.
  # backend "s3" {}
}

terraform/provider.tf

provider "aws" {
  region = var.aws_region

  default_tags {
    tags = local.common_tags
  }
}

terraform/variables.tf

variable "aws_region" {
  description = "AWS region to deploy into."
  type        = string

variable "project_name" {

  type        = string
  default     = "gitlab-tf-demo"

  description = "Environment name, such as dev, test, or prod."

terraform/locals.tf

locals {
  name_prefix = "${var.project_name}-${var.environment}"

  common_tags = {
    Project     = var.project_name
    Environment = var.environment
    ManagedBy   = "Terraform"
    Owner       = "GitLab-CICD"
  }
}

terraform/data.tf

Finds the latest Amazon Linux 2023 AMI automatically.

data "aws_ami" "amazon_linux_2023" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}

terraform/network.tf

resource "aws_vpc" "main" {
  cidr_block           = "10.10.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = { Name = "${local.name_prefix}-vpc" }
}

resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.10.1.0/24"
  map_public_ip_on_launch = true

  tags = { Name = "${local.name_prefix}-public-subnet" }
}

resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "${local.name_prefix}-igw" }
}

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }

  tags = { Name = "${local.name_prefix}-public-rt" }
}

resource "aws_route_table_association" "public" {
  subnet_id      = aws_subnet.public.id
  route_table_id = aws_route_table.public.id
}

terraform/security-group.tf

resource "aws_security_group" "web" {
  name        = "${local.name_prefix}-web-sg"
  description = "Allow HTTP and limited SSH"
  vpc_id      = aws_vpc.main.id

  ingress {
    description = "HTTP from internet for demo web page"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    description = "SSH only from your IP"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = [var.my_ip_cidr]
  }

  egress {
    description = "Allow outbound updates"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = { Name = "${local.name_prefix}-web-sg" }
}

terraform/ec2.tf

resource "aws_instance" "web" {
  ami                    = data.aws_ami.amazon_linux_2023.id
  instance_type          = "t3.micro"
  subnet_id              = aws_subnet.public.id
  vpc_security_group_ids = [aws_security_group.web.id]
  user_data              = file("${path.module}/../scripts/user_data.sh")

  metadata_options {
    http_tokens = "required"
  }

  root_block_device {
    encrypted = true
  }

  tags = { Name = "${local.name_prefix}-web" }
}

6. GitLab CI/CD pipeline setup

A pipeline is a list of jobs. Each job is like a checkpoint. We check formatting, validate Terraform, make a plan, and only apply after a manual button click.

Better professional option: configure GitLab OIDC to assume an AWS IAM role so you do not store long-term AWS keys in GitLab.

.gitlab-ci.yml

image:
  name: hashicorp/terraform:1.10
  entrypoint: [""]

stages:
  - format
  - validate
  - plan
  - apply
  - destroy

variables:
  TF_ROOT: "terraform"
  TF_IN_AUTOMATION: "true"

cache:
  key: "${CI_COMMIT_REF_SLUG}-terraform"
  paths:
    - ${TF_ROOT}/.terraform/

before_script:
  - cd ${TF_ROOT}
  - terraform --version
  - terraform init

terraform_fmt:
  stage: format
  script:
    - terraform fmt -check -recursive

terraform_validate:
  stage: validate
  script:
    - terraform validate

terraform_plan:
  stage: plan
  script:
    - terraform plan -out=tfplan
  artifacts:
    name: "terraform-plan-${CI_COMMIT_SHORT_SHA}"
    paths:
      - ${TF_ROOT}/tfplan
    expire_in: 1 hour

terraform_apply:
  stage: apply
  script:
    - terraform apply -auto-approve tfplan
  dependencies:
    - terraform_plan
  when: manual
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'

terraform_destroy:
  stage: destroy
  script:
    - terraform destroy -auto-approve
  when: manual
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
Important: The destroy job is manual so nobody deletes infrastructure by accident. In real production, destroy should be even more restricted or removed.

7. Runbook: how to deploy safely

  1. Create or update Terraform files in a feature branch.
  2. Run locally: terraform fmt, terraform validate, and terraform plan.
  3. Commit and push to GitLab.
  4. Open the GitLab pipeline and read the plan output.
  5. Confirm the plan only adds or changes what you expect.
  6. Merge to main after review.
  7. Click the manual apply job.
  8. Copy the Terraform output URL and test the web page.
  9. When finished with the lab, click the manual destroy job to avoid charges.

8. Security best practices

Use least privilege

Give the pipeline only the permissions it needs. Do not use administrator access for daily deployments.

Use temporary credentials

Prefer GitLab OIDC to AWS IAM role assumption over long-term access keys.

Protect pipeline variables

Mark secrets as masked and protected. Limit them to protected branches.

Scan infrastructure code

Add tools like Checkov, tfsec/Trivy, GitLab secret detection, and GitLab IaC scanning.

Lock Terraform state

Use remote state, encryption, and locking so two people do not change the same infrastructure at the same time.

Use manual approvals

Plan can run automatically. Apply to production should require review and approval.

Optional security scanning job

# Add a security stage if your project supports it.
stages:
  - format
  - validate
  - security
  - plan
  - apply

checkov_scan:
  stage: security
  image:
    name: bridgecrew/checkov:latest
    entrypoint: [""]
  script:
    - checkov -d terraform --quiet
  allow_failure: false

9. Professional deployment patterns to learn next

Dev/Test/Prod folders

Use separate folders or workspaces for each environment.

Modules

Put reusable Terraform code in modules like modules/vpc and modules/ec2.

Remote state

Use S3 for state and DynamoDB for locking in team projects.

OIDC role federation

Let GitLab request temporary AWS credentials instead of storing keys.

Blue/green deployment

Run old and new versions side by side, then switch traffic.

ECS or EKS

After EC2, learn container deployments using ECS or Kubernetes.

AWS in simple words

AWS is like renting computers, storage, databases, and networks from Amazon. You only create what you need, and you should delete lab resources when done.

Terraform in simple words

Terraform turns text files into real cloud resources. The command terraform plan previews changes. The command terraform apply makes the changes.

GitLab CI/CD in simple words

GitLab CI/CD is an automatic checklist. When you push code, GitLab runs jobs like format, validate, plan, and apply.

Terraform state

Terraform state is Terraform's memory. It records what resources were created. Protect it because it can contain sensitive information.

AWS security setup

  • Use MFA for accounts.
  • Do not use the root account for daily work.
  • Use IAM roles and temporary credentials when possible.
  • Start with a narrow policy and add permissions only when needed.
  • Store secrets in GitLab protected variables or use OIDC.

Troubleshooting tips

  • Access denied: AWS permissions are missing or credentials are wrong.
  • Invalid CIDR: use your public IP with /32.
  • Pipeline says no runner: enable a GitLab shared runner or install a project runner.
  • Terraform state locked: wait for the other job to finish or unlock only if you are sure no job is running.
  • Web page not loading: check security group port 80 and EC2 user data logs.
