AWS EKS · Apache Kafka · Strimzi

Run Apache Kafka on Amazon's Kubernetes, the easy way.

This guide walks you through setting up Kafka on AWS EKS using the Strimzi operator. We start from zero, build the worker computers (node groups), add load balancing, make it survive failures, let it grow, and finish with logging and pretty Grafana dashboards. Plain language. Real commands. One step at a time.

/ 00

First, what are all these words?

Before we type anything, let's learn the four big words in plain English. Think of building a giant restaurant kitchen.

📨 Apache Kafka

Kafka is like a super-fast post office for computer programs. One program drops off messages, and other programs pick them up — millions per second, in order, without losing them.

📦 Kubernetes

Kubernetes (people write k8s) is a manager that runs apps inside containers. It decides which computer runs what, restarts crashed apps, and keeps everything healthy.

☁️ AWS EKS

EKS = Elastic Kubernetes Service. It's Amazon running the hard, boring parts of Kubernetes for you, so you only worry about your apps and your worker computers.

🤖 Strimzi

Strimzi is a robot helper (an "operator") that knows how to install and babysit Kafka on Kubernetes. You tell it "I want a Kafka cluster," and it builds and repairs it for you.

The kitchen picture: EKS is the building. Node groups are teams of cooks (worker computers). Kubernetes is the head chef giving orders. Strimzi is the specialist who sets up the Kafka "order conveyor belt." We'll build them in that order.

Big change to know: Old Kafka needed a sidekick called ZooKeeper. Modern Kafka (4.x) dropped it and uses KRaft instead. / 01

Step 0 — Install your tools

You need four command-line tools on your laptop. Think of them as your toolbox before any building starts.

  • aws — talks to your Amazon account.
  • eksctl — builds and changes EKS clusters with one command.
  • kubectl — sends orders to Kubernetes ("koob-control").
  • helm — installs big apps (like Strimzi) from ready-made packages.

Install on macOS (with Homebrew):

bash 
brew install awscli eksctl kubectl helm

# Check they all work
aws --version
eksctl version      # must be 0.215.0 or newer
kubectl version --client
helm version

Then connect to your AWS account:

bash 
# It will ask for your Access Key, Secret Key, region, and output format
aws configure

# Make sure it recognizes you
aws sts get-caller-identity

On Windows use choco install or download the installers; on Linux use the official scripts. The links are in the Official docs section.

/ 02

Step 1 — Create the EKS control plane

Every cluster has a control plane (the brain) and nodes (the muscle). Amazon runs the brain for you. Here we create the brain only, with zero nodes, so we can add node groups carefully in the next step (that's what you asked for).

Save this as cluster.yaml:

cluster.yaml 
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: kafka-cluster
  region: us-east-1
  version: "1.33"          # use the newest EKS offers (1.30+)

# Use 3 Availability Zones so we can survive a zone outage later
availabilityZones: ["us-east-1a", "us-east-1b", "us-east-1c"]

# Start with NO worker nodes — we add node groups in Step 2
managedNodeGroups: []

# Lets pods get their own AWS permissions (needed later for the load balancer)
iam:
  withOIDC: true

Build it (takes ~15 minutes — grab a snack):

bash 
eksctl create cluster -f cluster.yaml

# When done, point kubectl at the new cluster
aws eks update-kubeconfig --name kafka-cluster --region us-east-1

# Prove it works (there are no worker nodes yet — that's expected)
kubectl get nodes
kubectl get svc

What just happened? Amazon created the brain, a private network (VPC), and all the security plumbing. You didn't pay for any worker computers yet because we asked for none.

/ 03

Step 2 — Create the node groups ⭐

A node group is a team of identical worker computers (EC2 machines) that join your cluster together. We'll use managed node groups, which means Amazon patches and replaces the machines for you.

We spread the team across 3 Availability Zones. An Availability Zone is a separate Amazon data center. If one zone has a power cut, the other two keep working — that's the foundation of failover.

Save this as nodegroup.yaml:

nodegroup.yaml 
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: kafka-cluster
  region: us-east-1

managedNodeGroups:
  - name: kafka-workers
    instanceType: m5.xlarge      # 4 CPU, 16 GB RAM — good starter size
    desiredCapacity: 3           # start with 3 machines (one per zone)
    minSize: 3                   # never go below 3
    maxSize: 6                   # allowed to grow up to 6
    volumeSize: 100              # 100 GB disk on each machine
    privateNetworking: true      # keep workers off the public internet
    availabilityZones: ["us-east-1a", "us-east-1b", "us-east-1c"]
    labels:
      role: kafka                # a name tag so Kafka pods land here

    # Roll out upgrades one machine at a time (no downtime)
    updateConfig:
      maxUnavailable: 1

    # Let AWS auto-replace a broken machine (node-level self-healing)
    nodeRepairConfig:
      enabled: true

    # Use SSM for safe shell access instead of open SSH ports
    ssh:
      enableSsm: true

Add the node group to your existing cluster:

bash 
# Create the node group from the file
eksctl create nodegroup -f nodegroup.yaml

# Watch the 3 machines show up and turn "Ready"
kubectl get nodes -o wide --watch

# See node groups Amazon knows about
eksctl get nodegroup --cluster kafka-cluster

Prefer one line instead of a file? This does the same thing:

bash 
eksctl create nodegroup \
  --cluster kafka-cluster \
  --region us-east-1 \
  --name kafka-workers \
  --node-type m5.xlarge \
  --nodes 3 --nodes-min 3 --nodes-max 6 \
  --node-volume-size 100 \
  --managed

Why "managed"? Self-managed nodes give you more knobs, but managed node groups handle patching, draining, and replacing machines automatically. For most Kafka setups, managed is the calm choice.

/ IaC

Terraform — build the nodes (and Kafka) as code

Prefer writing your setup once as text files instead of typing eksctl commands? Terraform lets you describe the cluster, the node groups, and even Kafka in files, then build it all with one command — and rebuild or delete it the same way.

This is an alternative path. It replaces Steps 1–5 (cluster + node groups + operator + Kafka) with Terraform. Pick one approach — eksctl or Terraform — not both. If you already built things with eksctl, you can skip this section and continue at Step 3.

Versions used here (current as of June 2026 — pin these in your code):

Tool Version
Terraform>= 1.5.7
AWS provider>= 6.42
terraform-aws-modules/eks~> 21.0
hashicorp/helm provider~> 3.0
kubectl provideralekc/kubectl >= 2.0

A Terraform project is just a folder of .tf files. Put these side by side, plus a manifests/ folder holding the three YAML files from Step 5 (controller-pool.yaml, broker-pool.yaml, kafka.yaml).

1. providers.tf — tools and login

Declares which providers to download and wires Helm and kubectl to the cluster Terraform is about to create.

providers.tf 
terraform {
  required_version = ">= 1.5.7"

  required_providers {
    aws     = { source = "hashicorp/aws", version = ">= 6.42" }
    helm    = { source = "hashicorp/helm", version = "~> 3.0" }
    kubectl = { source = "alekc/kubectl", version = ">= 2.0" }  # maintained fork
  }
}

provider "aws" {
  region = var.region
}

# Helm v3 uses nested objects (kubernetes = { ... }), not blocks
provider "helm" {
  kubernetes = {
    host                   = module.eks.cluster_endpoint
    cluster_ca_certificate = base64decode(module.eks.cluster_certificate_authority_data)
    exec = {
      api_version = "client.authentication.k8s.io/v1beta1"
      command     = "aws"
      args        = ["eks", "get-token", "--cluster-name", module.eks.cluster_name, "--region", var.region]
    }
  }
}

provider "kubectl" {
  host                   = module.eks.cluster_endpoint
  cluster_ca_certificate = base64decode(module.eks.cluster_certificate_authority_data)
  load_config_file       = false
  exec {
    api_version = "client.authentication.k8s.io/v1beta1"
    command     = "aws"
    args        = ["eks", "get-token", "--cluster-name", module.eks.cluster_name, "--region", var.region]
  }
}

variables.tf

3. main.tf — network, cluster & the node groups ⭐

This is the heart of your request: the EKS cluster plus the managed node group and its nodes, spread across 3 zones, with rolling upgrades and node self-healing.

main.tf 
data "aws_availability_zones" "available" {
  state = "available"
}

locals {
  azs = slice(data.aws_availability_zones.available.names, 0, 3)
}

# ---------- Network (VPC across 3 zones) ----------
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"

  name = "${var.cluster_name}-vpc"
  cidr = "10.0.0.0/16"

  azs             = local.azs
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]

  enable_nat_gateway   = true
  single_nat_gateway   = true
  enable_dns_hostnames = true

  # Tags so AWS load balancers know which subnets to use (Step 6)
  public_subnet_tags  = { "kubernetes.io/role/elb" = "1" }
  private_subnet_tags = { "kubernetes.io/role/internal-elb" = "1" }
}

# ---------- EKS cluster + managed node group(s) ----------
module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 21.0"

  name               = var.cluster_name
  kubernetes_version = var.kubernetes_version

  endpoint_public_access                   = true
  enable_cluster_creator_admin_permissions = true

  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_subnets

  # Core add-ons (incl. EBS CSI so Kafka can get disks)
  addons = {
    coredns                = {}
    kube-proxy             = {}
    vpc-cni                = { before_compute = true }
    eks-pod-identity-agent = { before_compute = true }
    aws-ebs-csi-driver     = {}
  }

  # ===== THE NODE GROUPS AND NODES =====
  eks_managed_node_groups = {
    kafka = {
      ami_type       = "AL2023_x86_64_STANDARD"   # default on 1.30+
      instance_types = ["m5.xlarge"]              # 4 vCPU, 16 GB

      min_size     = 3   # one per zone, never fewer
      desired_size = 3
      max_size     = 6   # allowed to grow

      subnet_ids = module.vpc.private_subnets     # spread across 3 zones
      labels     = { role = "kafka" }

      update_config      = { max_unavailable = 1 }  # roll upgrades 1 node at a time
      node_repair_config = { enabled = true }       # auto-replace broken machines
    }
  }

  tags = {
    Project   = "kafka-on-eks"
    ManagedBy = "terraform"
  }
}

4. storage.tf — let Kafka create disks

Kafka stores data on EBS volumes. The disk driver needs permission to make them. We grant it with an EKS Pod Identity link. Skip this and your Kafka pods get stuck waiting for storage.

storage.tf 
data "aws_iam_policy_document" "ebs_csi_trust" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRole", "sts:TagSession"]
    principals {
      type        = "Service"
      identifiers = ["pods.eks.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "ebs_csi" {
  name               = "${var.cluster_name}-ebs-csi"
  assume_role_policy = data.aws_iam_policy_document.ebs_csi_trust.json
}

resource "aws_iam_role_policy_attachment" "ebs_csi" {
  role       = aws_iam_role.ebs_csi.name
  policy_arn = "arn:aws:iam::aws:policy/service-role/AmazonEBSCSIDriverPolicy"
}

resource "aws_eks_pod_identity_association" "ebs_csi" {
  cluster_name    = module.eks.cluster_name
  namespace       = "kube-system"
  service_account = "ebs-csi-controller-sa"
  role_arn        = aws_iam_role.ebs_csi.arn
}

5. strimzi.tf — operator + Kafka

Installs the Strimzi operator with Helm, then applies your Kafka YAML files. We use the kubectl provider (not kubernetes_manifest) on purpose — see the warning below.

strimzi.tf 
# ---------- Strimzi operator (Helm) ----------
resource "helm_release" "strimzi" {
  name             = "strimzi-operator"
  repository       = "https://strimzi.io/charts/"
  chart            = "strimzi-kafka-operator"
  version          = "1.0.1"
  namespace        = "kafka"
  create_namespace = true
}

# ---------- Kafka node pools + cluster ----------
# kubectl applies these AFTER the operator installs the CRDs.
resource "kubectl_manifest" "controller_pool" {
  yaml_body  = file("${path.module}/manifests/controller-pool.yaml")
  depends_on = [helm_release.strimzi]
}

resource "kubectl_manifest" "broker_pool" {
  yaml_body  = file("${path.module}/manifests/broker-pool.yaml")
  depends_on = [helm_release.strimzi]
}

resource "kubectl_manifest" "kafka" {
  yaml_body  = file("${path.module}/manifests/kafka.yaml")
  depends_on = [kubectl_manifest.controller_pool, kubectl_manifest.broker_pool]
}

⚠ Common Terraform gotcha: don't apply the Kafka resources with the built-in kubernetes_manifest. It checks the CRDs at plan time, but the operator hasn't created them yet — so the plan fails before anything is built. The kubectl_manifest resource applies at apply time, after the CRDs exist, which is why we use it here.

6. Build it

bash 
terraform init        # download providers + modules
terraform plan        # preview what will be created
terraform apply       # type "yes" to build it (~15-20 min)

# Point kubectl at the new cluster, then verify
aws eks update-kubeconfig --name kafka-cluster --region us-east-1
kubectl get nodes
kubectl get kafka,kafkanodepool -n kafka

7. Add more nodes later

Just change the numbers (or add another entry to eks_managed_node_groups) and apply again. Terraform figures out the difference.

main.tf (edit the kafka node group) 
      min_size     = 3
      desired_size = 5   # was 3
      max_size     = 8   # was 6
bash 
terraform apply

8. Tear it all down

bash 
terraform destroy     # removes everything this project created

Two tips for real projects: store your state in a shared, locked backend (an S3 bucket) so a team can collaborate safely , and if a load balancer lingers during destroy, delete the Kafka resource first with kubectl delete kafka my-cluster -n kafka so the operator can clean up its NLBs.

/ 04

Step 3 — Add more nodes & auto-heal

Three ways to get more muscle: change the size by hand, add a brand-new node group, or let the cluster grow itself automatically.

A) Scale by hand (fastest)

bash 
# Grow from 3 to 5 machines, allow up to 8
eksctl scale nodegroup \
  --cluster kafka-cluster \
  --name kafka-workers \
  --nodes 5 --nodes-min 3 --nodes-max 8 \
  --wait

B) Add a second node group

Handy when you want different machines — for example bigger boxes, or cheaper Spot machines for non-critical work.

bash 
eksctl create nodegroup \
  --cluster kafka-cluster \
  --name kafka-workers-2 \
  --node-type m5.2xlarge \
  --nodes 3 --nodes-min 3 --nodes-max 6 \
  --managed

C) Let it grow automatically

When pods can't fit, an autoscaler adds machines; when machines sit empty, it removes them to save money. The modern tool is Karpenter; the classic one is Cluster Autoscaler.

bash 
# Quick start: turn on the classic Cluster Autoscaler for our node group.
# (eksctl already tagged the node group for auto-discovery.)
kubectl apply -f https://raw.githubusercontent.com/kubernetes/autoscaler/master/cluster-autoscaler/cloudprovider/aws/examples/cluster-autoscaler-autodiscover.yaml

# Edit the deployment and set your cluster name:
#   --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/kafka-cluster
kubectl -n kube-system edit deployment cluster-autoscaler

Two kinds of self-healing: Node Repair (which we turned on in Step 2) replaces a broken machine. Kubernetes itself replaces a broken pod. Together they keep Kafka standing without you waking up at 3 a.m.

Step 4 — Install the Strimzi operator

The operator is the robot helper. We install it once with Helm. After this, building Kafka is as simple as writing a short YAML file.

bash 
# Make a tidy room (namespace) just for Kafka
kubectl create namespace kafka

# Add the official Strimzi Helm chart and install the operator
helm repo add strimzi https://strimzi.io/charts/
helm repo update

helm install strimzi-operator strimzi/strimzi-kafka-operator \
  --namespace kafka \
  --version 1.0.1

# Wait for the operator pod to be Running
kubectl get pods -n kafka -w

The operator also installs Custom Resource Definitions — new "word types" like Kafka and KafkaNodePool that Kubernetes now understands.

/ 06

Step 5 — Deploy Kafka (KRaft + node pools)

Modern Strimzi builds Kafka from node pools. We make two pools:

  • Controllers — the bookkeepers that agree on who's in charge (KRaft). Usually 3.
  • Brokers — the workers that actually store and serve your messages. Usually 3 or more.

1. Controller pool — save as controller-pool.yaml:

controller-pool.yaml 
apiVersion: kafka.strimzi.io/v1
kind: KafkaNodePool
metadata:
  name: controller
  namespace: kafka
  labels:
    strimzi.io/cluster: my-cluster
spec:
  replicas: 3
  roles:
    - controller
  storage:
    type: jbod
    volumes:
      - id: 0
        type: persistent-claim
        size: 20Gi
        kraftMetadata: shared
        deleteClaim: false

2. Broker pool — save as broker-pool.yaml:

broker-pool.yaml 
apiVersion: kafka.strimzi.io/v1
kind: KafkaNodePool
metadata:
  name: broker
  namespace: kafka
  labels:
    strimzi.io/cluster: my-cluster
spec:
  replicas: 3
  roles:
    - broker
  storage:
    type: jbod
    volumes:
      - id: 0
        type: persistent-claim
        size: 100Gi
        deleteClaim: false

3. The Kafka cluster itself — save as kafka.yaml:

This one file also sets up failover (rack awareness + replication) and an external load balancer. We explain each part in Steps 6 and 7.

kafka.yaml 
apiVersion: kafka.strimzi.io/v1
kind: Kafka
metadata:
  name: my-cluster
  namespace: kafka
  annotations:
    strimzi.io/node-pools: enabled    # use node pools (default now)
    strimzi.io/kraft: enabled         # use KRaft, no ZooKeeper
spec:
  kafka:
    version: 4.1.0
    # metadataVersion is optional; the operator picks one that matches

    listeners:
      # Inside-the-cluster, plain text (for our test apps)
      - name: plain
        port: 9092
        type: internal
        tls: false
      # Inside-the-cluster, encrypted
      - name: tls
        port: 9093
        type: internal
        tls: true
      # Outside the cluster, via an AWS load balancer (see Step 6)
      - name: external
        port: 9094
        type: loadbalancer
        tls: true

    # --- Failover settings (see Step 7) ---
    config:
      default.replication.factor: 3
      min.insync.replicas: 2
      offsets.topic.replication.factor: 3
      transaction.state.log.replication.factor: 3
      transaction.state.log.min.isr: 2

    # Spread brokers across the 3 zones automatically
    rack:
      topologyKey: topology.kubernetes.io/zone

  # Lets us manage topics and users with YAML
  entityOperator:
    topicOperator: {}
    userOperator: {}

4. Apply all three and wait:

bash 
kubectl apply -f controller-pool.yaml
kubectl apply -f broker-pool.yaml
kubectl apply -f kafka.yaml

# Wait until Kafka reports "Ready" (a few minutes)
kubectl wait kafka/my-cluster --for=condition=Ready --timeout=600s -n kafka

# Look at your shiny new pods
kubectl get pods -n kafka

You now have Kafka running! The operator created controller pods, broker pods, persistent disks, internal addresses, and TLS certificates — all from those three small files.

/ 07

Step 6 — Load balancing & outside access

A load balancer is like a receptionist at the front door: visitors arrive at one address, and the receptionist sends each one to a free broker. For Kafka on AWS we use a Network Load Balancer (NLB), not the web-style ALB.

1. Install the AWS Load Balancer Controller

This is the piece that turns a Kubernetes "LoadBalancer" request into a real AWS NLB.

bash 
# 1) Make sure OIDC is on (we set withOIDC in Step 1)
eksctl utils associate-iam-oidc-provider --cluster kafka-cluster --approve

# 2) Download the permission policy AWS publishes
curl -o iam_policy.json https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/main/docs/install/iam_policy.json

aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file://iam_policy.json

# 3) Give the controller its own identity (replace ACCOUNT_ID)
eksctl create iamserviceaccount \
  --cluster kafka-cluster \
  --namespace kube-system \
  --name aws-load-balancer-controller \
  --attach-policy-arn arn:aws:iam::ACCOUNT_ID:policy/AWSLoadBalancerControllerIAMPolicy \
  --approve

# 4) Install the controller with Helm
helm repo add eks https://aws.github.io/eks-charts
helm repo update
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=kafka-cluster \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller

2. Tell Strimzi to ask for an NLB

Add annotations to the external listener in kafka.yaml, then re-apply. Annotations are little sticky notes that tell AWS what kind of load balancer you want.

kafka.yaml (listener section) 
      - name: external
        port: 9094
        type: loadbalancer
        tls: true
        configuration:
          bootstrap:
            annotations:
              service.beta.kubernetes.io/aws-load-balancer-type: "external"
              service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
              service.beta.kubernetes.io/aws-load-balancer-scheme: "internet-facing"
          brokers:
            - broker: 0
              annotations:
                service.beta.kubernetes.io/aws-load-balancer-type: "external"
                service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
            - broker: 1
              annotations:
                service.beta.kubernetes.io/aws-load-balancer-type: "external"
                service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
            - broker: 2
              annotations:
                service.beta.kubernetes.io/aws-load-balancer-type: "external"
                service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
bash 
kubectl apply -f kafka.yaml

# Find the public address clients will connect to (port 9094)
kubectl get service my-cluster-kafka-external-bootstrap -n kafka

Why one address per broker? Kafka clients must reach every broker directly, not just one. So Strimzi creates a bootstrap address plus one address per broker. The NLB shares the load and routes traffic to healthy brokers.

/ 08

Step 7 — Failover & high availability

"Failover" means: when something breaks, the system keeps working. We already added the magic words in kafka.yaml. Here's what each one does, in plain English.

default.replication.factor: 3

Every message is copied to 3 brokers. Lose one broker, your data is still safe on the other two. Like keeping three photocopies in three rooms.

min.insync.replicas: 2

A write only "counts" once 2 copies are saved. This stops you from losing data if a broker dies right after a write.

rack: topologyKey: topology.kubernetes.io/zone

Rack awareness. Strimzi puts the 3 copies in 3 different zones (data centers). One whole zone can go dark and you still have copies elsewhere. Don't put all your eggs in one basket.

Optional but smart: Cruise Control auto-balances your brokers.

Add this near the bottom of kafka.yaml (same level as entityOperator). It keeps data evenly spread, and re-spreads it after a broker is replaced.

kafka.yaml (add to spec) 
  cruiseControl: {}

Test failover safely — delete a broker pod and watch it heal:

bash 
# Kill one broker on purpose
kubectl delete pod my-cluster-broker-0 -n kafka

# Watch Kubernetes bring it right back, no data lost
kubectl get pods -n kafka -w

Step 8 — Logging

Logs are the diary each broker writes about what it's doing. They're how you find out why something went wrong.

Read logs right now

bash 
# Live logs from one broker (Ctrl+C to stop)
kubectl logs -f my-cluster-broker-0 -n kafka

# Logs from the operator (great for troubleshooting setup)
kubectl logs -f deployment/strimzi-cluster-operator -n kafka

Set the log level (less noise, or more detail)

Add a logging block under spec.kafka in kafka.yaml. Use INFO normally, DEBUG when hunting a bug, WARN to quiet things down.

kafka.yaml (under spec.kafka) 
    logging:
      type: inline
      loggers:
        rootLogger.level: INFO
        # Quiet a chatty area as an example:
        logger.kafkacontroller.name: org.apache.kafka.controller
        logger.kafkacontroller.level: WARN

Going to production? Send all these logs to one place (CloudWatch, Loki, or OpenSearch) with a log shipper like Fluent Bit, so you can search every broker at once instead of pod by pod.

/ FIX

Troubleshooting — find it, fix it, prove it

Something not working? Don't panic. Almost every problem is found with a handful of commands. This section shows you how to (1) see what's running, (2) find what's broken, (3) fix the common ones, and (4) prove it's truly healthy.

Golden rule: when in doubt, run kubectl describe on the broken thing and read the Events at the very bottom — and check status.conditions on the Kafka object. They almost always say the real reason in plain words.

1. Is everything running? (the 30-second health check)

Run these top to bottom. You want pods Running with full readiness (like 1/1), and your Kafka object showing Ready.

bash · overall health 
# Are the Kafka pods up? (controllers, brokers, operators)
kubectl get pods -n kafka

# Is the Kafka cluster itself Ready? Are the node pools healthy?
kubectl get kafka,kafkanodepool -n kafka

# Topics and users the operators manage
kubectl get kafkatopic,kafkauser -n kafka

# Are the worker machines Ready?
kubectl get nodes -o wide

# Did the disks (storage) get created and bound?
kubectl get pvc -n kafka

# Are the network addresses (including load balancers) up?
kubectl get svc -n kafka

Pod status words and what they mean:

You see It means
Running 1/1Healthy and ready. 🎉
PendingCan't be placed yet — usually no room on nodes, or a disk problem.
ContainerCreatingStarting up. Fine for a minute; stuck longer means a disk or image issue.
CrashLoopBackOffIt starts, crashes, retries. A real error — read the logs.
ImagePullBackOffCan't download the container image — wrong tag, or no internet/registry access.
Init:0/1Waiting on a setup step before the main app starts.
Terminating (stuck)Trying to shut down. If stuck, a volume or finalizer is holding it.

2. The 5 commands that find almost any problem

When a pod or resource looks wrong, these five tell you why. Replace POD with the real name from kubectl get pods (for example my-cluster-broker-0).

bash · the detective toolkit 
# 1) DESCRIBE — read the Events at the bottom. This is the #1 tool.
kubectl describe pod POD -n kafka

# 2) LOGS — what the app itself printed. Add --previous to see a crashed run.
kubectl logs POD -n kafka
kubectl logs POD -n kafka --previous
kubectl logs POD -n kafka -c kafka          # a specific container

# 3) EVENTS — recent cluster-wide messages, newest last
kubectl get events -n kafka --sort-by='.lastTimestamp'

# 4) STATUS — the Kafka object explains itself in status.conditions
kubectl get kafka my-cluster -n kafka -o yaml | grep -A 30 "status:"

# 5) EXEC — open a shell inside a broker to look around / run Kafka tools
kubectl exec -ti my-cluster-broker-0 -n kafka -- bash

The operator log is gold when setup itself fails (bad YAML, unsupported version, cert trouble):

bash · operator log 
# Follow the Strimzi operator. Search it for WARN / ERROR.
kubectl logs -f deployment/strimzi-cluster-operator -n kafka | grep -i -E "warn|error|exception"

3. Common problems & how to fix them

Each card is one real issue: what you see, why it happens, the command to confirm it, and the fix.

⚠ Pod stuck on Pending

Why: no node has enough CPU/RAM, or the disk can't be placed in the broker's zone.

confirm
kubectl describe pod POD -n kafka      # Events say "Insufficient cpu" or "didn't match..."
kubectl describe pvc -n kafka          # or a volume/zone message

Fix: add capacity, then re-check. (More nodes = more room.)

fix
eksctl scale nodegroup --cluster kafka-cluster --name kafka-workers \
  --nodes 5 --nodes-min 3 --nodes-max 8 --wait

⚠ CrashLoopBackOff on a broker or controller

Why: bad config value, out-of-memory (OOMKilled), or a wrong Kafka/metadata version.

confirm
kubectl logs POD -n kafka --previous     # the crash message itself
kubectl describe pod POD -n kafka | grep -i -E "reason|oomkilled|exit"

Fix: if OOM, give it more memory in kafka.yaml and re-apply; if config, fix the bad key the log named.

kafka.yaml (under spec.kafka)
    resources:
      requests:
        memory: 4Gi
        cpu: "1"
      limits:
        memory: 6Gi

⚠ ImagePullBackOff / ErrImagePull

Why: the image tag is wrong, or private-subnet nodes can't reach the registry (no NAT gateway / no ECR endpoints).

confirm
kubectl describe pod POD -n kafka | grep -A3 -i "failed to pull"

Fix: correct the image tag to one that exists (match your Kafka version, e.g. latest-kafka-4.1.0). If it's networking, give private subnets a NAT gateway or add ECR/S3 VPC endpoints.

⚠ Disk (PVC) stuck on Pending / unbound

Why: no default StorageClass, or the EBS CSI driver add-on isn't installed, or the volume's zone doesn't match the node.

confirm
kubectl get storageclass                 # is one marked (default)?
kubectl get pods -n kube-system | grep ebs-csi   # is the driver running?
kubectl describe pvc -n kafka            # read the Events line

Fix: install the EBS CSI driver add-on.

fix
eksctl create addon --name aws-ebs-csi-driver --cluster kafka-cluster --force

⚠ Kafka object never turns Ready

Why: the operator hit an error reconciling — invalid listener, bad config, or version mismatch.

confirm
# The status.conditions message usually names the exact problem
kubectl get kafka my-cluster -n kafka -o jsonpath='{.status.conditions[*].message}{"\n"}'

# And the operator log shows the full error / stack trace
kubectl logs deployment/strimzi-cluster-operator -n kafka --tail=100

Fix: correct what the message names in your YAML, then kubectl apply -f kafka.yaml again. (A classic one: "Unsupported Kafka.spec.kafka.version" — set a version your operator supports.)

⚠ External address stuck on <pending> (no load balancer)

Why: the AWS Load Balancer Controller isn't installed/healthy, its IAM is missing, or your subnets aren't tagged for load balancers.

confirm
kubectl get pods -n kube-system | grep aws-load-balancer
kubectl logs -n kube-system deployment/aws-load-balancer-controller --tail=80
kubectl describe svc my-cluster-kafka-external-bootstrap -n kafka   # Events show LB errors

Fix: (re)install the controller from Step 6, and tag your public subnets so AWS knows where to put internet-facing load balancers:

fix · subnet tag
aws ec2 create-tags --resources SUBNET_ID \
  --tags Key=kubernetes.io/role/elb,Value=1

⚠ Load balancer exists, but clients can't connect

Why: the NLB targets are unhealthy, a security group blocks the port, or the client doesn't trust the cluster's TLS certificate.

confirm
# Get the external bootstrap address and port (9094)
kubectl get svc my-cluster-kafka-external-bootstrap -n kafka

# Check the NLB target health in AWS (replace the ARN from the console)
aws elbv2 describe-target-health --target-group-arn TG_ARN

Fix: open the broker port to your client in the security group; for TLS, export and trust the cluster CA so your client accepts the certificate:

fix · grab the CA cert
kubectl get secret my-cluster-cluster-ca-cert -n kafka \
  -o jsonpath='{.data.ca\.crt}' | base64 -d > ca.crt

⚠ A KafkaTopic you applied never appears

Why: the Topic Operator is down, or the topic is missing the strimzi.io/cluster label, or it's in the wrong namespace.

confirm
kubectl get kafkatopic my-topic -n kafka -o yaml | grep -A5 "status:"
kubectl logs my-cluster-entity-operator-XXXX -c topic-operator -n kafka --tail=50

Fix: make sure the topic YAML has labels: strimzi.io/cluster: my-cluster and lives in the kafka namespace, then re-apply.

⚠ A worker node shows NotReady

Why: the machine ran out of disk, lost network, or the kubelet is unhealthy.

confirm
kubectl describe node NODE_NAME | grep -A10 "Conditions:"
aws eks describe-nodegroup --cluster-name kafka-cluster \
  --nodegroup-name kafka-workers --query 'nodegroup.health'

Fix: with Node Repair enabled (Step 2) AWS replaces it automatically. To force it now, drain and delete the node — the node group launches a fresh one:

fix
kubectl drain NODE_NAME --ignore-daemonsets --delete-emptydir-data
kubectl delete node NODE_NAME

4. Prove it's actually working

"No errors" isn't the same as "working." These commands give you real proof.

Cluster-level health:

bash · is it Ready? 
# Wait for / confirm Ready
kubectl wait kafka/my-cluster --for=condition=Ready --timeout=300s -n kafka

# One-line Ready check
kubectl get kafka my-cluster -n kafka \
  -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}{"\n"}'

# See which node and ZONE each broker landed in (proves rack spread)
kubectl get pods -n kafka -o wide

Kafka-level health (run inside a broker):

bash · Kafka's own tools 
# List all topics
kubectl exec -ti my-cluster-broker-0 -n kafka -- \
  bin/kafka-topics.sh --bootstrap-server localhost:9092 --list

# Describe a topic: check that Isr (in-sync replicas) equals Replicas
kubectl exec -ti my-cluster-broker-0 -n kafka -- \
  bin/kafka-topics.sh --bootstrap-server localhost:9092 --describe --topic my-topic

# THE BIG ONE: any under-replicated partitions? Empty output = perfect health
kubectl exec -ti my-cluster-broker-0 -n kafka -- \
  bin/kafka-topics.sh --bootstrap-server localhost:9092 --describe --under-replicated-partitions

# Check a consumer group's lag (LAG column should stay small)
kubectl exec -ti my-cluster-broker-0 -n kafka -- \
  bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 --describe --all-groups

Metrics & node usage:

bash · metrics + capacity 
# Confirm the metrics endpoint Prometheus scrapes is alive
kubectl exec -ti my-cluster-broker-0 -n kafka -- \
  curl -s localhost:9404/metrics | head

# How busy are nodes and pods? (needs metrics-server)
kubectl top nodes
kubectl top pods -n kafka

End-to-end smoke test (write one message, read it back):

bash · prove the round trip 
# Send one message, then read the last one back. Seeing it = full success.
echo "hello-$(date +%s)" | kubectl exec -i my-cluster-broker-0 -n kafka -- \
  bin/kafka-console-producer.sh --bootstrap-server localhost:9092 --topic my-topic

kubectl exec -ti my-cluster-broker-0 -n kafka -- \
  bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 \
  --topic my-topic --from-beginning --max-messages 1 --timeout-ms 10000

Still stuck? Copy the exact error from describe or the operator log into the Strimzi community (Slack/GitHub Discussions) linked in the docs below. A precise error message gets a fast answer; "it doesn't work" does not.

Clean up (so you stop paying)

EKS, EC2 machines, disks, and load balancers all cost money every hour. When you're done practising, delete everything in this order.

bash 
# 1) Remove Kafka (this also removes its load balancers)
kubectl delete kafka my-cluster -n kafka

# 2) Remove the operators and monitoring
helm uninstall strimzi-operator -n kafka
helm uninstall kube-prometheus-stack -n monitoring

# 3) Finally, delete the whole cluster, node groups and all
eksctl delete cluster --name kafka-cluster --region us-east-1

Double-check in the AWS console that no EC2 instances, EBS volumes, or load balancers are left running. A forgotten NLB or disk can quietly add to your bill.

Built as a learning guide. Versions current as of June 2026 — always confirm against the official docs above. Cloud resources cost money; remember to clean up.

Kafka stores messages in topics, and each topic is split into partitions so the work can spread across many brokers. Programs that send messages are producers; programs that read them are consumers.

The magic is that consumers can read at their own speed, replay old messages, and many consumers can read the same data. That's why Kafka is used for things like tracking clicks, moving payments, and feeding data pipelines.

A container is an app packed up with everything it needs to run. Kubernetes runs containers inside pods (the smallest unit), schedules them onto worker machines (nodes), and restarts them if they crash.

You describe the result you want in YAML ("I want 3 copies of this app"), and Kubernetes constantly works to make reality match your description. That idea is called desired state.

With EKS, Amazon runs and protects the Kubernetes control plane (the brain) across multiple data centers. You don't patch it or babysit it.

You still provide the worker nodes (the EC2 machines that run your pods) — which is exactly what node groups are for. You pay for the control plane per hour, plus whatever machines and storage you use.

Strimzi is an open-source project (part of the Cloud Native Computing Foundation). It installs an operator — a program that watches for special Kubernetes objects like Kafka and turns them into a real, running Kafka cluster.

It handles the hard parts: rolling upgrades with no downtime, TLS certificates, user and topic management, scaling, and recovery. You mostly just edit small YAML files.

Old Kafka needed a separate system called ZooKeeper to keep track of which broker was in charge and where data lived. It was one more thing to run and break.

KRaft (Kafka Raft) moves that bookkeeping inside Kafka itself, handled by special controller nodes. Kafka 4.x is KRaft-only, and Strimzi dropped ZooKeeper support in version 0.46. That's why this guide creates a controller pool and a broker pool.

Helm is the "app store" for Kubernetes. Instead of applying dozens of YAML files by hand, you install a chart — a packaged app with sensible defaults.

You can override settings with --set flags or a values file, and upgrade or uninstall the whole thing with one command.

A VPC (Virtual Private Cloud) is your own private network inside AWS. It's split into subnets, and those subnets live in different Availability Zones.

Private subnets have no direct door to the internet (safer for workers), while public subnets do (used by internet-facing load balancers). eksctl sets all of this up for you.

A managed node group is an Auto Scaling group of EC2 machines that AWS looks after for your cluster. AWS handles graceful draining when it patches or replaces a machine, so pods move safely.

It can span multiple Availability Zones for high availability, and there's no extra charge — you only pay for the EC2 machines, disks, and other resources you use.

Managed node groups: AWS patches, drains, and replaces machines; the Kubernetes version follows the control plane; simplest to run.

Self-managed node groups: you control the launch template, the AMI, and the upgrade process — more flexible, but more work. For Windows nodes or very custom setups, you may need self-managed. Most Kafka clusters are happy on managed.

Cluster Autoscaler grows and shrinks existing node groups when pods can't be placed. It's mature and simple, but tied to node group sizes.

Karpenter is newer and smarter: it launches right-sized machines on demand, picks cheaper instance types, and removes them quickly when idle. For bursty Kafka workloads, Karpenter often saves money — but it's a bit more to set up.

A Custom Resource Definition teaches Kubernetes a new kind of object. Strimzi adds objects like Kafka, KafkaNodePool, KafkaTopic, and KafkaUser.

After the CRDs are installed, you can kubectl apply a Kafka object just like a built-in one, and the operator does the rest. This guide uses the modern v1 API that Strimzi switched to in version 1.0.

An ALB (Application Load Balancer) only understands web traffic (HTTP/HTTPS). Kafka does not speak HTTP — it has its own binary protocol over plain TCP.

So Kafka needs a Network Load Balancer (NLB), which works at the TCP level (Layer 4). The annotations in this guide tell the AWS Load Balancer Controller to build an NLB and route by IP for best performance.

"ISR" means in-sync replicas — the copies that are fully caught up. min.insync.replicas: 2 means a write is only accepted once at least 2 copies have it.

With replication factor 3 and min ISR 2, you can lose 1 broker and still accept writes safely. If you set min ISR equal to the replication factor, losing even one broker stops writes — so 2-of-3 is the common, balanced choice.

Rack awareness tells Kafka which "rack" each broker is in. On AWS we map a rack to an Availability Zone using the label topology.kubernetes.io/zone.

Kafka then makes sure the copies of a partition land in different zones. If one zone fails, another zone still has a full copy, and Kafka automatically picks a new leader there. This is the heart of zone-level failover.

Reading logs pod by pod is fine for learning, but painful at scale. A log shipper like Fluent Bit runs on every node, collects all container logs, and sends them to one searchable place.

On AWS that's often CloudWatch Logs or OpenSearch; many teams use Loki, which plugs right into the same Grafana you set up in Step 9.

Prometheus regularly "scrapes" numbers from your apps (called metrics) and stores them over time. Examples for Kafka: messages per second, under-replicated partitions, JVM memory.

Grafana reads Prometheus and draws dashboards and alerts. The kube-prometheus-stack chart installs both, plus the operator that manages them, in one go.

The classic way to expose Kafka metrics is the JMX Prometheus Exporter (used in this guide) — well supported and reliable.

Newer Strimzi versions also offer the Strimzi Metrics Reporter, which exposes Prometheus metrics directly without JMX. It was announced in Strimzi 0.47 and aims to be lighter and faster. Either works; the JMX exporter is the safe default while the reporter matures.

"Pending" means Kubernetes accepted the pod but can't place it on any node yet. The two usual reasons:

1. No room. Every node is already full on CPU or memory. The fix is more nodes (scale the node group) or smaller resource requests.

2. Storage zone mismatch. A broker's disk (EBS volume) lives in one Availability Zone, and a pod can only run on a node in that same zone. If there's no node there, it waits. Running 3 nodes across 3 zones (as in this guide) avoids this.

Run kubectl describe pod POD -n kafka and read the Events — it will literally say "Insufficient cpu" or a volume/zone message.

The container starts, crashes, and Kubernetes keeps restarting it — waiting a little longer each time (the "back-off"). It's a real error, not a temporary blip.

The crash reason is in the logs of the previous run: kubectl logs POD -n kafka --previous. Common causes for Kafka: OOMKilled (ran out of memory — give it more), a bad config value (the log names it), or a version mismatch between the Kafka version and the KRaft metadata version.

Check kubectl describe pod POD -n kafka for Reason: OOMKilled and the last exit code to tell memory problems from config problems.

Both tools create the same AWS resources — the difference is the style.

eksctl is quick and EKS-focused: great for getting started, demos, and one-off clusters. You run commands and it does the work.

Terraform is "infrastructure as code." You describe the end result in files, and Terraform keeps a record (called state) of what it built. That makes it easy to review changes in a pull request, repeat the exact same setup across dev/staging/prod, and tear everything down cleanly. Most teams managing long-lived infrastructure prefer Terraform.

Terraform remembers what it created in a file called terraform.tfstate. Next time you run it, it compares your files to that record and only changes what's different.

By default the state sits on your laptop, which is risky for teams (two people could clash, or it could be lost). For real projects, store it in a shared backend — commonly an S3 bucket with locking — so everyone works from the same source of truth and can't overwrite each other.

Strimzi adds new object types (CRDs) like Kafka and KafkaNodePool. Terraform's built-in kubernetes_manifest checks that those types exist during the plan step — but on a brand-new cluster the operator hasn't created them yet. So the plan fails before anything is built. It's a classic chicken-and-egg problem.

The kubectl_manifest resource (from the kubectl provider) applies your YAML at apply time instead, after the operator's Helm release has installed the CRDs. Pairing it with depends_on makes the order reliable. That's why this guide uses it for the Kafka resources.

Pods sometimes need AWS permissions — for example, the EBS disk driver needs permission to create volumes for Kafka.

EKS Pod Identity is the modern, simple way to give a specific Kubernetes service account an AWS IAM role. You create the role, attach the policy it needs, and make an "association" linking the role to the service account. No long-lived keys, and it's cleaner than the older IRSA approach. The eks-pod-identity-agent add-on (installed in main.tf) makes it work.
