Lesson 01

What is a load balancer?

Start with a pizza shop. Imagine a pizza shop with only one cashier. On a quiet Tuesday, that one cashier is fine. But on Friday night, fifty hungry people show up at once. The line stretches out the door, people wait forever, and some give up and leave. The cashier is overloaded.

Now imagine the shop hires three cashiers and one friendly greeter at the door. The greeter looks at the three lines and sends each new customer to whichever cashier is least busy. Suddenly everyone gets served quickly, no single cashier melts down, and if one cashier goes on break, the greeter simply stops sending people their way.

On the internet, that greeter is called a load balancer. The cashiers are — computers that do the actual work, like showing a website or saving your game score. The customers are — little messages your phone or laptop sends, like “please show me this web page.”

50 customers→ greeter (load balancer)→ cashier 1cashier 2cashier 3

The three jobs of a load balancer

A load balancer does three important jobs, all day, every second:

  1. Spread the work (balancing). It shares incoming requests across many servers so no single server gets crushed. “Load” just means “amount of work,” and “balance” means “share it fairly.”
  2. Check who is healthy (health checks). Every few seconds it pokes each server and asks, “Are you okay?” If a server stops answering, the load balancer stops sending it customers. Users never even notice a server broke. This is called a .
  3. Be the one front door. Users only need to remember one address (one ). Behind that door, you can add servers, remove servers, or replace servers, and nobody outside has to change anything.
Why this matters

Big sites like streaming services or game servers stay up during huge traffic spikes because of load balancers. They make systems scalable (add more servers when busy) and highly available (survive when one server dies). Those two words come up in almost every cloud job interview.

A few words you will see everywhere

Before we touch AWS, let’s lock in the basic vocabulary. Tap any word for the full popup explanation:

  • A is just a computer whose job is to answer requests.
  • An is a computer’s street address on a network, like 10.0.1.25.
  • A is a numbered door on that computer. Websites usually use door 80 (HTTP) or door 443 (, the locked version).
  • is the internet’s phone book — it turns a name like myapp.example.com into an IP address.
  • Traffic is the flow of all those requests and responses, like cars on a road.

Lesson 02

The AWS load balancer family

AWS (Amazon Web Services) is a huge collection of rentable computer services. Its load balancer service is called Elastic Load Balancing (ELB) — “elastic” because it stretches automatically when traffic grows. ELB is not one product; it’s a small family. Choosing the right family member is your first real engineering decision.

First, a 60-second idea: network “layers”

Engineers describe networking as a stack of , like floors in a building. Two floors matter for us. Layer 4 is the “envelope” floor: it only sees addresses and ports — where a message goes, never what’s inside. Layer 7 is the “letter” floor: it opens the envelope and reads the actual web request — the URL path, the website name, the headers. A Layer 7 balancer can make smart decisions like “send /videos to the video servers.” A Layer 4 balancer can’t read any of that, but because it does less, it is blazingly fast.

TypeLayerBest atUse it for
7 — reads the request Smart routing by URL path, hostname, or headers; HTTPS; redirects Websites, APIs, microservices, the NiFi web screen
4 — sees only address + port Extreme speed, millions of connections, raw TCP/UDP, fixed IPs Databases, game servers, Kafka-style TCP traffic
Gateway Load Balancer (GWLB) 3 — whole network packets Slipping firewalls and security inspection boxes into the traffic path Security appliance fleets (a more advanced, niche tool)
Classic Load Balancer (CLB) 4 & 7 (old) Nothing new — it’s the retired grandparent Legacy systems only. Never pick it for new work.
The simple rule of thumb

Talking HTTP/HTTPS (web pages, APIs, dashboards)? Choose the ALB. Talking raw TCP/UDP at high speed (Kafka brokers, databases, custom protocols)? Choose the NLB. We will build one of each in this course.

Lesson 03

Building blocks: the neighborhood your load balancer lives in

A load balancer never floats alone in space. It lives inside a private network you build in AWS, surrounded by a few pieces with intimidating names. Here’s the friendly version of each. Think of AWS as a giant country, and you are designing one gated neighborhood inside it.

— the country’s cities
AWS has data centers grouped into regions like us-east-1 (Northern Virginia) or eu-west-1 (Ireland). You pick one region and build there.
— separate buildings in that city
Each region contains several AZs — physically separate data centers with their own power and internet. If lightning takes out one building, the others keep running. Best practice: always spread across at least two AZs. Load balancers actually require it.
— your gated neighborhood
Your own private slice of AWS networking. Nothing gets in or out unless you allow it. You give it a range of IP addresses using , like 10.0.0.0/16 (about 65,000 addresses to hand out).
— the streets
You divide the VPC into smaller streets. Public subnets have a road to the internet (through an ) — load balancers that face the world live here. Private subnets have no direct road in — your servers, Kafka brokers, and databases hide here, safe from random internet strangers.
— bouncers at each door
A tiny firewall attached to each resource. Its rules say things like “allow door 443 from anywhere” or, even better, “allow door 80 only from the load balancer’s security group.” Security groups are stateful: if a request is allowed in, the reply is automatically allowed back out.
— the receptionist’s instructions
A listener tells the load balancer which door to watch and what to do: “Listen on port 443 for HTTPS, and forward everything to the web servers group.”
— the team roster
A named list of servers (the “targets”) that share work, plus the rules for deciding who on the roster is fit to play today.
internet→ Internet Gateway→ ALB · public subnets→ target group → servers · private subnets
Watch out — the #1 beginner mistake

Putting servers in public subnets and opening their ports to the whole internet (0.0.0.0/0). Only the load balancer should be public. Servers belong in private subnets, accepting traffic only from the load balancer’s security group. We’ll write the code exactly that way.

Lesson 04

What is Terraform?

You could build everything from Lesson 03 by clicking around the AWS website. But imagine clicking 40 buttons perfectly, three times, for three environments… and then doing it again next month when something changes. Humans forget steps. Clicks leave no record. That’s why professionals use .

Think of a LEGO set. Terraform is like writing the LEGO instruction booklet for your cloud. You describe the finished build in text files — “one VPC, two subnets, one load balancer, two servers” — and Terraform reads the booklet and assembles the real thing in AWS for you. Same booklet, same build, every single time. The booklet lives in , so every change is reviewed and remembered.

Five words that unlock all of Terraform

  • — the language Terraform files are written in. Files end in .tf and read almost like English.
  • — the plugin that teaches Terraform how to talk to a specific cloud. We use the aws provider; there are providers for GitLab, Kafka, and hundreds more.
  • — one thing you want to exist: one VPC, one load balancer, one security group. Each gets a resource block.
  • — Terraform’s memory file (terraform.tfstate) recording what it has already built. So important it gets its own lesson (08).
  • — a reusable folder of Terraform code, like a saved LEGO sub-assembly you can snap into many builds.

The four commands you’ll type forever

terraform init→ terraform validate→ terraform plan→ terraform apply
  1. terraform init — “unbox the kit”: downloads the providers and connects to your state storage.
  2. terraform validate — “spell-check the booklet”: catches typos and broken references.
  3. terraform plan — “preview the build”: shows exactly what would be created (+), changed (~), or destroyed (-) — without touching anything. Always read the plan.
  4. terraform apply — “build it for real.” And terraform destroy tears it all down when you’re done practicing (important for your wallet!).
Best practice from day one

Run terraform fmt to auto-tidy your files, and never edit the cloud by hand once Terraform manages it — hand edits create “drift” (the cloud quietly differing from the booklet), and the next plan will fight you.

Lesson 05

Build a real load balancer, step by step

Time to build! We’ll create a complete mini-system: a VPC, two public subnets (in two AZs), two tiny web servers, and an Application Load Balancer that shares traffic between them. Each step adds one file or one block, with every line explained. Total cost if you destroy it within an hour: a few cents.

1

Install your tools

You need three things on your computer: the Terraform CLI, the AWS CLI, and an AWS account with access keys configured (run aws configure and paste your keys). Ask an adult/teammate with the AWS account to create an IAM user for you — never use the root account for daily work.

Check it worked: terraform -version and aws sts get-caller-identity should both print happily.

2

Make a project folder

Terraform reads every .tf file in a folder and merges them, so we split code into clearly named files just for human sanity:

alb-tutorial/ ├── versions.tf ← which Terraform + providers we need ├── variables.tf ← knobs we might want to change ├── network.tf ← VPC, subnets, internet gateway, routes ├── security.tf ← the bouncers (security groups) ├── servers.tf ← two tiny web servers ├── alb.tf ← the star: load balancer + target group + listener └── outputs.tf ← useful answers printed after apply
3

Pin your versions versions.tf

Pinning versions means everyone on the team (and the GitLab robot later) builds with the same tools — no “works on my machine” mysteries.

4

Add your knobs variables.tf

A variable is a labeled knob. Instead of burying "us-east-1" in ten places, you define it once and reuse it everywhere as var.aws_region.

5

Build the neighborhood network.tf

One VPC, two public subnets in different AZs (the ALB requires at least two), an internet gateway (the road to the outside world), and a route table that says “to reach the internet, take the gateway.”

Notice how blocks reference each other: aws_vpc.main.id means “the ID of the VPC named main, once it exists.” Terraform reads these references and figures out the correct build order all by itself.

6

Hire the bouncers security.tf

Two security groups. The ALB’s bouncer lets the whole internet knock on port 80. The servers’ bouncer is stricter: it admits port-80 traffic only if it comes from the ALB’s security group. That chaining is the textbook best practice.

7

Launch two tiny servers servers.tf

We start two small machines. A startup script (user_data) installs the nginx web server and writes a page saying which server you reached — so you can watch the balancing happen. count = 2 is Terraform’s copy machine: one block, two servers.

(In real production you’d put servers in private subnets behind a NAT gateway, and use an Auto Scaling Group instead of fixed instances. We keep this lab small and cheap on purpose.)

8

The star of the show alb.tf — the load balancer

Six lines of meaning: its name, whether it’s public, its type, which bouncer guards it, and which two streets (in two AZs) it stands on.

9

The team roster alb.tf — target group + members

The defines the team and the health-check rules: visit / every 30 seconds; two passes in a row = healthy, three fails = benched.

10

The receptionist’s instructions alb.tf — listener

Translation: “Stand at door 80. Whatever arrives, forward it to the web team.” ALB listeners can also hold smarter rules — “if the path starts with /api, send it to the API team” — which is exactly the Layer-7 magic from Lesson 02.

11

Print the address outputs.tf

12

Build it, test it, tear it down

Try the magic trick

Before destroying, stop one EC2 instance in the AWS console. Wait ~90 seconds (3 failed health checks), then refresh your browser. Every request now goes to the surviving server — no errors, no downtime. You just witnessed high availability.

Level up: real HTTPS

Production sites use HTTPS. The pattern: request a free certificate in AWS Certificate Manager (ACM), add a listener on port 443 with protocol = "HTTPS" and certificate_arn, then change the port-80 listener’s action to a redirect to 443. Users typing the insecure address get bounced to the locked one automatically.

Lesson 06

Data pipelines: Kafka, Zookeeper, NiFi & friends

Websites aren’t the only things behind load balancers. Companies also run data pipelines — systems that move and clean rivers of information: every click, every sensor reading, every purchase. Picture a water system: Kafka is the high-pressure pipes, NiFi is the control room where you connect pipes visually, Zookeeper is the old supervisor keeping everyone coordinated, and analytics tools are the labs testing the water at the end.

Apache Kafka — the conveyor belt for events

is a super-fast conveyor belt for messages, called events (“player123 scored,” “sensor 7 reads 21°C”). Producers drop events on the belt; consumers pick them up — even hours later, because Kafka keeps everything for a configurable time. The belt is split into named lanes called , each chopped into partitions so many readers can work in parallel. The machines running the belt are called .

On AWS, the best practice is usually not to babysit Kafka yourself but to use (Managed Streaming for Apache Kafka): AWS patches, monitors, and repairs the brokers for you.

Kafka / MSK best practices (memorize these)
  • 3 brokers across 3 AZs minimum, with replication factor 3 — one building can burn down and no data is lost.
  • Private subnets only. A message system should never have a public IP address.
  • Encrypt everywhere: TLS between clients and brokers, and between brokers (client_broker = "TLS").
  • Turn on monitoring and broker logs to CloudWatch — when a pipeline jams at 2 a.m., logs are your flashlight.
  • Lock access with security groups: only your app and NiFi security groups may reach ports 9092–9098.
Honest engineering note: Kafka and load balancers are frenemies

Kafka clients are clever: they ask any broker “who holds partition 3 of my topic?” and then connect directly to that exact broker. A normal “spread it anywhere” load balancer breaks that. So the patterns that actually work are: (a) use MSK’s built-in bootstrap brokers address — no load balancer needed inside your VPC; or (b) if outside clients must reach self-managed Kafka, put an internal in front with one listener + one target group per broker (e.g., port 9092 → broker 1, 9093 → broker 2…), matching each broker’s advertised listener. Knowing this nuance is what separates copy-pasters from engineers.

Zookeeper — the referee (now retiring)

is a small coordination service — a referee that distributed systems use to agree on things: who is the leader broker, what configuration is current, who is alive. Classic Kafka could not start without it.

Two best-practice facts for today: (1) modern Kafka replaced Zookeeper with a built-in mode called — new clusters (including new MSK clusters) should use KRaft, and with MSK, AWS manages the coordination layer for you either way; (2) Zookeeper still matters because other tools — notably a NiFi cluster — use it as their referee. If you must run it: always an odd number of nodes (3 or 5, so votes can’t tie), in private subnets, spread across AZs.

Apache NiFi — pipelines you can see

lets you build data pipelines by dragging boxes on a canvas: “pull files from this folder → convert to JSON → push into Kafka.” It’s the visual control room. A NiFi cluster is several NiFi machines acting as one (coordinated by — you guessed it — Zookeeper), and its web screen is exactly the kind of HTTP traffic an ALB was born for.

One special trick is required: sticky sessions. After you log in to NiFi on node 2, your next click must also go to node 2, or you’d be “logged out” constantly. Stickiness makes the ALB hand your browser a cookie and keep sending you to the same node.

NiFi best practices
  • Cluster nodes in private subnets; only the internal ALB (and your VPN/office network) may reach port 8443.
  • Make the ALB internal = true — a pipeline control room does not belong on the public internet.
  • Use an external Zookeeper ensemble (3 nodes) for production clusters, not NiFi’s embedded one.
  • Give NiFi fast disks — its “flowfile” and “content” repositories are disk-hungry.

ETL vs. ELT, and the analytics shelf

You’ll hear constantly. Both mean Extract (collect data), Transform (clean/reshape it), Load (store it for analysis) — the only difference is the order of the last two. ETL: clean first, then store. ELT (the modern favorite): dump the raw data into cheap storage first, transform later inside a powerful warehouse — like keeping all your raw photos and editing copies whenever you need them.

The AWS analytics shelf, in one breath: (AWS’s own streaming belt, simpler than Kafka), S3 as the data lake (the giant raw-files bucket), (serverless Spark jobs + a catalog that acts as the lake’s index), (ask SQL questions directly against files in S3, pay per query), (the heavy-duty SQL warehouse for dashboards), and EMR (rent-a-Spark-cluster for big custom jobs).

apps · sensors→ Kafka / Kinesis→ NiFi / Glue→ S3 data lake→ Athena · Redshift→ dashboards
Best practices for the whole pipeline (these apply to everything above)
  • Everything in Terraform modules — one kafka module, one nifi module — reused identically in dev, staging, and prod.
  • Least privilege everywhere: each service gets an IAM role allowing only what it truly needs; security groups reference other security groups, never 0.0.0.0/0 on data ports.
  • Encrypt at rest and in transit: KMS keys on S3/MSK/Redshift, TLS on every connection.
  • Tag every resource (Project, Environment, Owner) so bills and incidents have a name attached.
  • Alarms before users notice: CloudWatch alarms on broker disk, consumer lag, NiFi back-pressure, and ALB 5xx errors.
  • One state file per service — the heart of Lesson 08.

Lesson 07

GitLab pipelines: robots that ship your code

Running terraform apply from your laptop is fine for learning, but dangerous for a team: whose laptop? with whose keys? did anyone review it? The professional answer is — an assembly line of robots that checks, previews, and (only after human approval) applies every change. In GitLab, the assembly line is described in one file at the root of your repo: .gitlab-ci.yml. The robots that execute it are called .

The safe workflow, as a story

  1. You create a branch (a personal copy of the code) and change alb.tf.
  2. You open a Merge Request (MR) — “please review my change.”
  3. The pipeline robot automatically runs fmt, validate, and plan, and pastes the plan into the MR for everyone to read.
  4. A teammate reviews the plan (“creates 1 listener, destroys nothing — approved ✅”).
  5. You merge to main. The pipeline plans again, then waits at a manual apply button. A human clicks it. Terraform applies the exact saved plan file — no surprises between preview and reality.
validate→ plan→ 👀 human review→ apply (manual)

A clean starter pipeline

Three details make this professional: the plan is saved as an and the apply consumes that exact file; the apply is when: manual; and it can only run from main. Your AWS keys never live in the file — store them as masked, protected CI/CD variables in GitLab settings (or, best of all, use GitLab’s OIDC to assume an AWS role with no long-lived keys at all).

Scaling up: many services, many environments

Real repos hold several components (network, ALB, Kafka, NiFi) and several environments (dev → staging → prod). Two GitLab features keep that tidy: a hidden template job (the .tf-component below — names starting with a dot don’t run by themselves) and rules: changes, so each component’s jobs run only when its own folder changed:

GitLab best practices checklist
  • Protect the prod environment (Settings → CI/CD → Protected environments) so only senior approvers can press apply.
  • Promote in order: dev applies automatically-ish, staging after review, prod last — same code, increasing ceremony.
  • Add tflint and a security scanner like trivy or checkov to the validate stage.
  • Touched several components? GitLab simply runs each affected plan side by side — small blast radius per job.
  • GitLab can even store your Terraform state for you (next lesson covers when to choose that vs. S3).

Lesson 08

Terraform state: the save file (and how to split it)

Here’s the lesson that separates hobby projects from real teams. is like a video-game save file: a JSON document recording every resource Terraform built and its real-world ID. On every plan, Terraform compares three things — your code (what you want), the state (what it remembers), and AWS (what actually exists) — and computes the difference. Lose the save file and Terraform forgets it ever built anything: the next apply tries to build duplicates of your whole world.

Rule 1 — never keep state on a laptop (or in Git)

By default state lands next to your code as terraform.tfstate. Three problems: teammates can’t see it, it can contain secrets (so it must never be committed to Git), and two people applying at once will corrupt it. The fix is a remote — shared storage for the save file. On AWS the classic choice is an S3 bucket (versioned + encrypted) plus a DynamoDB table for : while one person applies, everyone else is politely told to wait, like a “recording in progress” light.

Two modern alternatives, so nothing surprises you

(a) Terraform 1.10+ can lock using S3 itself — add use_lockfile = true to the S3 backend and skip DynamoDB on new projects. (b) GitLab offers GitLab-managed Terraform state: state is stored, versioned, and locked inside your GitLab project via an HTTP backend — wonderfully convenient when GitLab is already your CI home. Both are legitimate; what is not legitimate is local state for a team.

Rule 2 — many small states, never one giant one

Beginners put the VPC, the ALB, Kafka, NiFi, and analytics into one state. Then every plan takes ages, every lock blocks the whole team, and a mistake while touching NiFi can propose destroying the network. The pro move is splitting by environment × component, each with its own backend key. The damage any one mistake can cause — the — shrinks to one small folder.

Component folderBackend key (its own save file)Changes how often?

Rule 3 — let components read each other’s outputs

Split states still need to cooperate: the Kafka component must know the VPC ID that the network component created. The clean bridge is the data source — read-only access to another component’s outputs (only outputs are exposed, which is exactly the point):

Adding a brand-new service later (say, Redshift)

  1. Create modules/redshift/ (reusable code) and envs/dev/redshift/ (this environment’s copy).
  2. Give it its own backend key: dev/redshift/terraform.tfstate. Existing states are untouched — zero risk to what already runs.
  3. Pull the VPC and subnets it needs via terraform_remote_state.
  4. Add plan:redshift / apply:redshift jobs with rules: changes (Lesson 07).
  5. MR → robot plan → review → merge → manual apply in dev → promote to staging → prod. Same dance, every time.
State surgery exists — handle with gloves

Renaming a resource in code makes Terraform think “destroy old, create new.” To rename without rebuilding, use a moved {} block (or terraform state mv). To adopt something built by hand, use an import {} block. Run these on a branch, read the plan twice, and remember the S3 versioning you enabled is your undo button.

Lesson 09

Project structure & the golden rules

Here is the whole course condensed into one repository layout. Modules are the LEGO sub-assemblies (written once); envs folders are thin wrappers that call those modules with each environment’s sizes and names. Dev gets one small Kafka broker; prod gets three big ones — same module, different inputs.

infrastructure/ ← one Git repo ├── .gitlab-ci.yml ← the robot assembly line (Lesson 07) ├── .gitignore ← MUST ignore: .terraform/, *.tfstate* ├── bootstrap/ ← state bucket + lock table (applied once) ├── modules/ ← reusable LEGO sub-assemblies │ ├── network/ ← VPC, subnets, routes │ ├── alb/ ← load balancer + listeners + TGs │ ├── kafka/ ← MSK cluster (Lesson 06) │ ├── nifi/ ← NiFi cluster + sticky internal ALB │ └── analytics/ ← Glue, Athena workgroup, etc. └── envs/ ├── dev/ │ ├── network/ ← own backend key + state │ ├── alb/ ← own backend key + state │ ├── kafka/ ← own backend key + state │ └── nifi/ ← own backend key + state ├── staging/ ← same shape as dev └── prod/ ← same shape, protected applies

And an environment wrapper is genuinely this small — that’s the payoff of modules:

A quick word on workspaces

Terraform also offers — multiple states under one folder, switched with terraform workspace select. They’re handy for short-lived clones (like a per-MR preview environment), but for dev/staging/prod most teams prefer the explicit folders above: you can see every difference in Git, and it’s much harder to apply prod while believing you’re in dev.

The golden rules — print this

  1. Everything is code. If it isn’t in Git, it doesn’t exist. No console clicking on managed resources.
  2. Remote state, locked, versioned, encrypted. One state per environment × component. Never commit state.
  3. Plan before apply, human before prod. Apply the saved plan artifact, behind a manual, protected button.
  4. ALB for HTTP, NLB for TCP, at least two AZs, health checks always, deletion protection in prod.
  5. Private by default. Only load balancers face the internet; security groups reference security groups.
  6. Modules + variables, versions pinned, fmt + validate + linters in CI.
  7. Encrypt and tag everything; alarms on the metrics that page you (5xx, unhealthy hosts, consumer lag, disk).
  8. Small blast radius. New service = new module + new folder + new state key — never a bigger old state.
