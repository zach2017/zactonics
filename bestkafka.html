01

Overview of Expected Tasks

The cloud team owns the full lifecycle of the messaging platform and its underlying compute. At a high level, the recurring responsibilities fall into eight domains. Each is expanded in the sections below.

DOMAIN A

Immutable Base Images

Build hardened, patched golden AMIs with Packer on a regular cadence; rotate for CVE remediation.

DOMAIN B

Configuration Management

Convert base AMIs to role-specific nodes using Ansible, version-controlled in GitLab with CI gates.

DOMAIN C

Kafka / Zookeeper Operations

Provision, tune, secure, upgrade, and reliably scale brokers and the metadata quorum.

DOMAIN D

Cluster Lifecycle & Scaling

Capacity planning, partition rebalancing, rolling upgrades, and disaster recovery drills.

DOMAIN E

EC2 → EKS Migration

Evaluate workloads for containerization; migrate where cost and operability improve.

DOMAIN F

Logging & Observability

Centralize logs and metrics, ship to Grafana/Loki/Prometheus, and build actionable dashboards.

DOMAIN G

User Support & Onboarding

Self-service topic provisioning, quotas, SLAs, runbooks, and a clear support model.

DOMAIN H

Performance Management

Continuous benchmarking, lag monitoring, SLO tracking, and proactive tuning.

Guiding principle: Favor immutable, declarative, and automated approaches over manual, mutable ones. Every change should be reproducible from version control, observable in dashboards, and reversible via a documented rollback.

02

Bare-Bone AMI Updates & Security with Packer

The foundation of secure infrastructure is a golden image pipeline. Rather than patching live servers, you bake a known-good, hardened image and replace instances. This is the immutable infrastructure pattern.

What & Why

Packer produces machine images from a single source template. On a regular cadence (e.g. monthly, or on critical CVE), you start from an upstream base (Amazon Linux 2023, Ubuntu LTS), apply OS patches, run a CIS hardening baseline, install a baseline agent set (CloudWatch/SSM/log shipper), and publish a versioned, tagged AMI. Downstream automation then references the latest approved AMI ID. 

// packer/kafka-base.pkr.hcl  (HCL2)
source "amazon-ebs" "base" {
  ami_name      = "platform-base-{{timestamp}}"
  instance_type = "t3.medium"
  region        = "us-east-1"
  source_ami_filter {
    filters = { name = "al2023-ami-*-x86_64" }
    owners  = ["amazon"]
    most_recent = true
  }
  ssh_username = "ec2-user"
  tags = {
    Baseline = "CIS-Level1"
    BuildBy  = "gitlab-ci"
    Patched  = "{{timestamp}}"
  }
}

build {
  sources = ["source.amazon-ebs.base"]
  provisioner "ansible" {           // hand off to Ansible for hardening
    playbook_file = "ansible/harden.yml"
  }
  provisioner "shell" {
    inline = ["sudo dnf upgrade -y --security"]
  }
}
03

Configuring New AMIs from Base — Ansible, GitLab & Shell

The base image is generic. To turn it into a Kafka broker, a Zookeeper node, or a Connect worker, you layer role-specific configuration. Ansible is the configuration engine; GitLab is the source of truth and CI runner; shell scripts handle small glue tasks.

Ansible

Declarative, agentless roles for users, JVM, Kafka config, sysctl tuning. Idempotent and reusable.

GitLab CI

Lints, tests with Molecule, builds AMIs, gates merges, and stores artifacts/inventory.

Shell Scripts

Thin glue only — bootstrap, health probes, one-off ops. Kept minimal and idempotent.

// .gitlab-ci.yml
stages: [lint, test, bake, deploy]

ansible-lint:
  stage: lint
  script: ["ansible-lint roles/"]

molecule-test:
  stage: test
  script: ["molecule test -s kafka-broker"]   // converge + idempotence

bake-ami:
  stage: bake
  rules: [{ if: '$CI_COMMIT_BRANCH == "main"' }]
  script:
    - packer init .
    - packer build -var "role=kafka-broker" kafka.pkr.hcl
  artifacts: { paths: ["manifest.json"] }      // emits AMI id
04

Kafka & Zookeeper Operations

Kafka is the durable, partitioned commit log at the center of the platform. Zookeeper historically stored cluster metadata, but modern Kafka (3.x+) moves to KRaft mode, removing the Zookeeper dependency. A key strategic decision is whether to keep Zookeeper, adopt KRaft, or move to managed Amazon MSK.

Concern Best Practice
ReplicationRF=3, min.insync.replicas=2, acks=all for durability.
Rack awarenessMap broker.rack to AZ so replicas span availability zones.
MetadataPrefer KRaft for new clusters; keep ZK quorum at 3 or 5 (odd) if retained.
Storagegp3 EBS with provisioned IOPS; separate volume for logs.dirs.
SecurityTLS in transit, SASL/SCRAM or mTLS auth, ACLs per topic.
UpgradesRolling, one broker at a time, watch under-replicated partitions = 0.
05

Building & Managing Clusters, Scaling

Cluster management covers provisioning new clusters, capacity planning, partition reassignment, broker add/remove, and disaster recovery. Treat the cluster topology as code (Terraform) and partition layout as a reviewed, automated operation rather than an ad-hoc CLI run.

Scaling out

Add brokers, then rebalance partitions with Cruise Control or kafka-reassign-partitions with a throttle to protect client throughput.

Capacity planning

Size on peak write throughput × RF × retention, plus 30–40% headroom. Watch disk, network, and partition count per broker.

Disaster recovery

MirrorMaker 2 or managed replication to a second region; document RPO/RTO and rehearse failover quarterly.

Infrastructure as Code

Terraform for VPC, ASGs/node groups, security groups; state in S3 with locking. No console drift.

06

Migration from EC2 to EKS (Where Cost-Effective)

Not everything belongs in Kubernetes. The team's job is to evaluate each workload and migrate only where EKS clearly improves cost, density, or operability. Stateless services (Connect workers, REST proxies, consumers) are strong candidates; stateful Kafka brokers themselves are a more nuanced decision.

Decision heuristic

  • ✓ Many small, stateless, bursty services with low utilization → EKS bin-packing saves money.
  • ✓ Team already runs other K8s workloads → consolidate operational surface.
  • ~ Stateful Kafka brokers → only with Strimzi operator, EBS-backed StatefulSets, mature K8s expertise.
  • ✗ A few large, steady, well-tuned EC2 instances → EKS overhead may cost more, not less.
07

Logging & Sending Logs to Grafana

Observability has three pillars: logs, metrics, and traces. The Grafana ecosystem unifies them: Loki for logs, Prometheus/Mimir for metrics, Tempo for traces, all visualized in Grafana with a single alerting layer. The collection agent (Alloy / Promtail / OTel Collector) runs on every node.

COLLECT
Grafana Alloy / OTel Collector / Promtail on each host or as a DaemonSet.
STORE
Loki (logs), Prometheus/Mimir (metrics), Tempo (traces).
VISUALIZE
Grafana dashboards per service, with templating and drill-down.
ALERT
Grafana Alerting / Alertmanager → PagerDuty / Slack, tied to SLOs.
# Kafka exposes JMX → scrape with the JMX Prometheus exporter
# key broker metrics to watch:

kafka_server_BrokerTopicMetrics_BytesInPerSec
kafka_consumergroup_lag                                 # consumer health
08

User Support & Performance Management

The platform exists to serve users — application teams producing and consuming from Kafka. A mature team offers self-service, clear SLAs, runbooks, and proactive performance management so that issues are caught before users feel them.

Self-service onboarding

Topic + ACL requests via GitOps merge request or a portal; enforce naming conventions, quotas, and RF policy automatically.

SLOs & error budgets

Define availability and latency SLOs; track consumer lag and produce/consume p99 against budget.

Runbooks & on-call

Documented playbooks for broker failure, disk-full, rebalancing storms; rotation with clear escalation.

Performance tuning

Right-size partitions, batch/linger settings, compression (lz4/zstd), and JVM/GC; benchmark regularly.
