0 The Big Picture

Imagine you want to build a LEGO castle, but instead of placing each brick by hand every time, you write down instructions. Then a robot reads your instructions and builds the exact same castle — perfectly — as many times as you want. That is what we're doing here, but the "castle" is a computer server on the internet.

Here are the three main characters in our story:

  • AWS (Amazon Web Services) — A giant company that rents out computers, storage, and databases over the internet. This is the "cloud." Instead of buying a physical server, you rent one by the hour.
  • Terraform — A free tool that lets you describe the servers you want in a text file. It then talks to AWS and builds them for you. This idea is called Infrastructure as Code (IaC).
  • GitLab — A website that stores your code and can automatically run Terraform for you whenever you save changes. The auto-running part is called CI/CD.
You write code → push to GitLab → GitLab runs Terraform → Terraform builds it on AWS

3 Set Up Your AWS Account

  1. 1. Create a free AWS account at aws.amazon.com/free. You'll need a credit card, but the free tier covers small practice projects.
  2. 2. Create an IAM user instead of using your main login. IAM is AWS's "who is allowed to do what" system. Never use the root account for daily work.
  3. 3. Make an Access Key for that user. You get two secret strings: an Access Key ID and a Secret Access Key. Treat the secret like a password — never put it in your code.
  4. 4. Tell the AWS CLI your keys by running the command below and pasting them in:
aws configure
# It will ask for:
#   AWS Access Key ID     -> paste yours
#   AWS Secret Access Key -> paste yours
#   Default region        -> e.g. us-east-1
#   Default output format -> just press Enter

⚠️ Safety rule: If a secret key ever ends up on the internet, delete it immediately in AWS and make a new one.

→ AWS docs on access keys

4 Write Your Terraform Files

Make a new folder, e.g. my-first-infra. Inside it, create these files. Terraform reads every file ending in .tf in the folder. We split them up just to stay organized — Terraform glues them together automatically.

providers.tf

Tells Terraform which cloud we're using and which region. Think of it as "connect to AWS, in Virginia."

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # use any 5.x version
    }
  }
}

provider "aws" {
  region = "us-east-1"
}
variables.tf

Variables are like blanks you fill in. This keeps values reusable instead of typed everywhere.

variable "bucket_name" {
  description = "A globally-unique name for our storage bucket"
  type        = string
  default     = "my-first-tf-bucket-change-me-123"
}
main.tf

The actual thing we build. We'll make an S3 bucket (a cloud folder for files) because it's simple, cheap, and safe to practice with. 

resource "aws_s3_bucket" "my_bucket" {
  bucket = var.bucket_name

  tags = {
    Name        = "MyFirstBucket"
    Environment = "Learning"
  }
}
outputs.tf

After building, print useful info back to you — here, the bucket's name.

output "bucket_id" {
  description = "The name of the bucket we created"
  value       = aws_s3_bucket.my_bucket.id
}

5 Run Terraform (The 4 Magic Commands)

Open your terminal inside the folder and run these in order. Always read what each command prints before continuing.

1. terraform init — Get ready

Downloads the AWS provider plugin. Run once per project (and again if you change versions).

terraform init

2. terraform fmt & validate — Tidy & check

Auto-formats your files and checks for spelling/syntax mistakes.

terraform fmt
terraform validate

3. terraform plan — Preview

Shows exactly what will be created (a +), changed (~), or destroyed (-). Nothing is built yet. This is your safety net — always read the plan.

terraform plan

4. terraform apply — Build it

Does the work. It shows the plan again and asks you to type yes to confirm.

terraform apply

🎉 Go check the AWS Console → S3 — your bucket is really there! → All Terraform CLI commands

6 Automate It with GitLab CI/CD

Right now you run commands by hand. Let's make GitLab do it automatically every time you push code. GitLab looks for one special file in your project: .gitlab-ci.yml.

Step A — Push your code to GitLab

git init
git add .
git commit -m "My first terraform project"
# Make an empty project on gitlab.com first, then:
git remote add origin https://gitlab.com/YOUR_USERNAME/my-first-infra.git
git push -u origin main

Step B — Store AWS keys safely in GitLab

Go to your project → Settings → CI/CD → Variables. Add two masked, protected variables. This keeps secrets out of your code.

  • AWS_ACCESS_KEY_ID
  • AWS_SECRET_ACCESS_KEY

Step C — Create .gitlab-ci.yml

This file defines the pipeline: two stages, plan then apply. The apply only runs when you click a button (manual), so nothing changes by surprise.

.gitlab-ci.yml 
# Use the official Terraform image so commands exist
image:
  name: hashicorp/terraform:1.9
  entrypoint: [""]

variables:
  AWS_DEFAULT_REGION: "us-east-1"

stages:
  - plan
  - apply

# Runs before every job: get Terraform ready
before_script:
  - terraform init

plan:
  stage: plan
  script:
    - terraform validate
    - terraform plan -out=tfplan
  artifacts:
    paths:
      - tfplan        # save the plan for the apply step

apply:
  stage: apply
  script:
    - terraform apply -input=false tfplan
  when: manual       # YOU click to approve
  only:
    - main           # only on the main branch

Push this file. In GitLab go to Build → Pipelines and watch plan run, then click ▶ on apply to deploy.

7 Remote State (Sharing the "Memory")

Remember state — Terraform's memory of what it built? By default it's a file on your laptop called terraform.tfstate. That's a problem when GitLab and teammates also run Terraform: everyone needs the same memory, or they'll clash.

The fix: store the state file in a shared place. GitLab has a built-in spot for this (no extra AWS setup needed). Add this to providers.tf:

providers.tf (add the backend block) 
terraform {
  backend "http" {}   # GitLab fills in the details

  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }
}

Then change terraform init in your pipeline to point at GitLab's state backend.

→ GitLab managed Terraform state docs

8 Best Practices

  • 🔒 Never commit secrets. Use GitLab CI/CD variables or AWS roles — not hard-coded keys. Add *.tfstate and .terraform/ to a .gitignore file.
  • 👀 Always review the plan before apply. Make apply a manual, approved step.
  • 🗂️ Use remote state (Section 7) so your team shares one source of truth, with locking to prevent two people applying at once.
  • 🏷️ Tag everything (Name, Environment, Owner) so you can find and bill resources later.
  • 📦 Use modules once things grow — reusable bundles of Terraform code.
  • 🌱 Separate environments (dev / staging / prod) so a mistake in testing never touches real users.
  • 🔁 Pin versions (like ~> 5.0) so upgrades don't break things unexpectedly.
  • 📖 Run fmt and validate in your pipeline to keep code clean and catch typos early.

→ HashiCorp's official style & best-practice guide

9 Clean Up (Avoid Surprise Bills)

When you're done practicing, delete everything Terraform made so AWS doesn't keep charging you. One command does it:

terraform destroy
# It lists what will be removed, then asks you to type: yes

✅ Because Terraform tracks state, destroy knows exactly what to remove — no leftover servers quietly costing money.

What is AWS?

AWS (Amazon Web Services) is the world's largest cloud provider. "The cloud" just means computers in someone else's data center that you rent over the internet. Instead of buying a $5,000 server, you can rent computing power for pennies per hour and turn it off when done.

AWS offers 200+ services — storage (S3), virtual servers (EC2), databases (RDS), and much more.

Read more on AWS →

What is Terraform?

Terraform, made by HashiCorp, lets you describe infrastructure in simple text files using a language called HCL. You declare what you want ("I want one storage bucket"), and Terraform figures out how to create it and keeps it that way.

Because it's just text files, you can version it, review it, and reuse it — exactly like normal code. This whole approach is "Infrastructure as Code."

Intro to Terraform →

What is GitLab & CI/CD?

GitLab is a website that stores your code (using Git) and adds automation on top. CI/CD stands for Continuous Integration / Continuous Delivery — fancy words for "automatically test and ship code whenever it changes."

You define jobs in a .gitlab-ci.yml file, and GitLab runs them for you on its own servers, called runners.

GitLab CI/CD intro →

Why not use the root account?

Your root (main) account can do anything, including closing the account or running up huge bills. If those keys leak, the damage is unlimited.

An IAM user can be given just the permissions it needs (least privilege). If it's compromised, you simply delete it. This is a core security habit.

IAM best practices →

What is an S3 bucket?

S3 (Simple Storage Service) is AWS's file storage. A "bucket" is like a top-level folder where you put files (called objects). Bucket names must be unique across all of AWS worldwide — that's why we add random numbers.

We use it for practice because it's cheap, fast to create, and easy to delete.

S3 docs →

Adding masked CI/CD variables

In GitLab: Settings → CI/CD → Variables → Add variable. For each key:

  • Set the Key (e.g. AWS_ACCESS_KEY_ID) and paste the Value.
  • Tick Mask variable so it's hidden in logs.
  • Tick Protect variable so only protected branches can use it.

Terraform's AWS provider automatically reads these standard variable names — no extra code needed.

Init command for GitLab state

Replace the simple terraform init in your pipeline's before_script with this. GitLab provides these variables automatically.

# In .gitlab-ci.yml before_script
- terraform init
    -backend-config="address=${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/terraform/state/default"
    -backend-config="lock_address=${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/terraform/state/default/lock"
    -backend-config="username=gitlab-ci-token"
    -backend-config="password=${CI_JOB_TOKEN}"
    -backend-config="lock_method=POST"
    -backend-config="unlock_method=DELETE"

What are modules?

A module is a reusable folder of Terraform files. Instead of copy-pasting the same 50 lines for every server, you write it once as a module and call it with different inputs.

It's like a recipe you can reuse with different ingredients. The public Terraform Registry has thousands of ready-made modules.
