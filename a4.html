A complete, beginner-friendly walkthrough written for people working on Windows 10 or Windows 11. Every file, every command, and every security setting is explained in plain English. Tap the info buttons for deeper background, and use the official documentation links at every step.

Part 00

00 Overview

Think of building cloud servers the old way like assembling furniture by hand every time — slow, easy to mess up, hard to repeat. The modern way is to write down the instructions in a file, hand them to a robot, and let the robot build it. The instructions are code. The robot is Terraform. The furniture warehouse is AWS. The shared workshop where teams keep their plans (and where the robot lives) is GitLab.

The three tools, in plain words

ToolWhat it doesMade by
AWSRents out computers, storage, and databases over the internet.Amazon
TerraformReads text files describing what you want and builds it on AWS.HashiCorp
GitLabStores your code and automatically runs Terraform when you change it.GitLab Inc.

The complete workflow

From keyboard to cloud

You edit Terraform files on your Windows PC → Git pushes them to GitLab → GitLab's pipeline runs terraform plan → you review and approve → GitLab runs terraform apply → AWS builds the resources.

What you'll have at the end

  • A working AWS account with a secure non-root user.
  • A GitLab repository with a clean folder structure.
  • Terraform code that creates real AWS resources (an S3 bucket and a small web server).
  • An automated CI/CD pipeline with separate dev and prod environments.
  • Remote state storage, secrets handled securely, and approval gates before anything ships.

We'll install everything using winget, the package manager built into modern Windows. Open PowerShell as Administrator: press Win, type "PowerShell," right-click, and choose Run as administrator.

Git is the version-control tool that talks to GitLab.

# Close PowerShell and reopen it, then verify:

PS> winget install --id Microsoft.VisualStudioCode -e

Inside VS Code, install these extensions for a much nicer experience:

  • HashiCorp Terraform — syntax highlighting and autocomplete.
  • GitLab Workflow — see pipelines and merge requests in the editor.

Part 03

03 AWS Account & IAM Setup

Step 1 — Create an AWS account

Sign up at aws.amazon.com/free. A credit card is required but most learning fits inside the free tier.

Step 2 — Enable MFA on the root account immediately

The root account is the "owner" of everything. Add multi-factor authentication (MFA) before doing anything else: sign in → top-right menu → Security Credentials → "Assign MFA device." Use an authenticator app like Microsoft Authenticator.

Critical

Never use the root account for daily work. Never create access keys for the root user. Lock it away after setup.

Step 3 — Create an IAM user for yourself

From the AWS console, search IAM → Users → Create user:

  1. Name it terraform-admin (you can rename later).
  2. Enable console access with a password.
  3. On the permissions screen choose Attach policies directly and pick AdministratorAccess for now. (We'll narrow this later — see Part 04.)
  4. Finish and enable MFA on this user too.

Step 4 — Create access keys for the CLI

From your new IAM user → Security credentials → Create access key → Command Line Interface (CLI). You'll see two strings:

  • Access Key ID — like a username.
  • Secret Access Key — like a password. Shown only once. Save it safely (a password manager is ideal).

Step 5 — Configure the AWS CLI on Windows Windows

PS> aws configure
AWS Access Key ID [None]:     <paste your key id>
AWS Secret Access Key [None]: <paste your secret>
Default region name [None]:   us-east-1
Default output format [None]: json

This saves credentials to C:\Users\YourName\.aws\credentials. Verify with:

PS> aws sts get-caller-identity

You should see your user ARN. → AWS CLI configuration docs

Heads-up

These keys live on your laptop. They are powerful. We will not put them in GitLab — we'll use a safer method (OIDC) in Part 10.

Part 04

04 AWS Security Foundations

Before pushing anything to the cloud, lock down the basics. These are the same rules professionals follow.

1. Least privilege

Give every user and service the smallest set of permissions it needs. Start with AdministratorAccess while learning, then trim it down. Later in this guide, our GitLab role will only get permission to manage S3 and EC2 — nothing else.

2. Set a billing alarm

In Billing → Budgets, create a $5 (or any number) monthly alert. You'll get an email if something runs up costs. → AWS Budgets docs

3. Turn on CloudTrail

CloudTrail records every action taken in your account — invaluable for spotting trouble. The first trail is free. → CloudTrail docs

4. Rotate or remove access keys often

Every 90 days, generate a new key, update aws configure, then delete the old one. Better still — graduate to OIDC (Part 10) so long-lived keys go away entirely.

5. Never store secrets in code

No passwords, no keys, no certificates in your Terraform files. Use AWS Secrets Manager or GitLab CI/CD variables — both covered later.

Here is the exact folder layout we're going to build. Each box below explains what it does and why.

File 2 — README.md

Documents the project for your future self and teammates.

README.md 
# AWS Terraform Demo

Infrastructure as Code for our learning project, deployed via GitLab CI/CD.

## Structure
- `terraform/`        — Terraform configuration files
- `environments/`     — per-environment variable values
- `.gitlab-ci.yml`    — automated pipeline (plan + apply)

File 3 — environment tfvars

Each environment gets its own values file so dev and prod can differ without code changes.

environments/dev/terraform.tfvars 
environment   = "dev"
instance_type = "t3.micro"
bucket_suffix = "dev-001"
environments/prod/terraform.tfvars 
environment   = "prod"
instance_type = "t3.small"
bucket_suffix = "prod-001"
Why separate folders?

Splitting environments physically (in folders) prevents the classic mistake of accidentally applying dev settings to production. The pipeline picks the right file based on the environment.

Part 07

07 Terraform Code, File by File

All four files live inside the terraform/ folder. Terraform reads every .tf file in the folder as one program — we split them only to stay organized.

File 1 — providers.tf

Declares which clouds we use and pins versions so future upgrades don't break us by surprise.

terraform/providers.tf 
terraform {
  required_version = ">= 1.9.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.60"   # any 5.60.x or newer in 5.x
    }
  }
}

provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      ManagedBy   = "Terraform"
      Environment = var.environment
      Project     = "aws-terraform-demo"
    }
  }
}

default_tags attaches the same labels to every resource — huge time-saver for billing reports and audits.

File 2 — variables.tf

Variables turn hard-coded values into inputs. Always give them a description and a type.

terraform/variables.tf 
variable "aws_region" {
  description = "AWS region to deploy into"
  type        = string
  default     = "us-east-1"
}

variable "environment" {
  description = "Environment name (dev or prod)"
  type        = string

  validation {
    condition     = contains(["dev", "prod"], var.environment)
    error_message = "environment must be dev or prod."
  }
}

variable "instance_type" {
  description = "EC2 instance size"
  type        = string
  default     = "t3.micro"
}

variable "bucket_suffix" {
  description = "Suffix to keep bucket name globally unique"
  type        = string
}

File 3 — main.tf

The actual resources. We'll create two: a private S3 bucket (cloud storage) and a small EC2 instance (a virtual server). Both are great practice resources.

terraform/main.tf 
# ---------------------------------------------------------------
# A unique S3 bucket for storing files
# ---------------------------------------------------------------
resource "aws_s3_bucket" "app_data" {
  bucket = "my-app-data-${var.environment}-${var.bucket_suffix}"
}

# Block ALL public access — security best practice
resource "aws_s3_bucket_public_access_block" "app_data" {
  bucket                  = aws_s3_bucket.app_data.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Enable versioning so deleted/overwritten files can be recovered
resource "aws_s3_bucket_versioning" "app_data" {
  bucket = aws_s3_bucket.app_data.id
  versioning_configuration {
    status = "Enabled"
  }
}

# Encrypt every object at rest with AWS-managed keys
resource "aws_s3_bucket_server_side_encryption_configuration" "app_data" {
  bucket = aws_s3_bucket.app_data.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

# ---------------------------------------------------------------
# A small EC2 server running Amazon Linux
# ---------------------------------------------------------------
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]
  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }
}

resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type

  tags = {
    Name = "web-${var.environment}"
  }
}

File 4 — outputs.tf

Outputs print useful values after a successful apply. Great for confirming what was built.

terraform/outputs.tf 
output "bucket_name" {
  description = "Name of the created S3 bucket"
  value       = aws_s3_bucket.app_data.id
}

output "instance_id" {
  description = "ID of the EC2 server"
  value       = aws_instance.web.id
}

output "instance_public_ip" {
  description = "Public IP of the EC2 server"
  value       = aws_instance.web.public_ip
}
Part 08

08 Run Terraform Locally First Windows

Always test from your Windows machine before automating. The four commands are init, fmt/validate, plan, and apply.

1. Initialize the project

PS> cd $env:USERPROFILE\Documents\aws-terraform-demo\terraform
PS> terraform init

Downloads the AWS provider into a hidden .terraform folder. Run again any time you change provider versions.

2. Format and validate

PS> terraform fmt -recursive
PS> terraform validate

fmt auto-styles your code; validate checks for syntax errors without contacting AWS.

3. Plan against the dev environment

PS> terraform plan -var-file="..\environments\dev\terraform.tfvars" -out=tfplan

Read the output carefully. Look for the summary line: Plan: 5 to add, 0 to change, 0 to destroy.

4. Apply

PS> terraform apply tfplan

This actually creates the resources. After a minute or two, Terraform prints the outputs (bucket name, EC2 ID, public IP). Open the AWS console and verify they exist.

Habit to build

Run plan every time, even when you "know" what's going to happen. Surprises in apply are how production outages start.

Part 09

09 Remote State with GitLab

Right now your terraform.tfstate lives only on your laptop. That's fine alone but breaks the moment GitLab (or a teammate) runs Terraform — everyone needs the same memory or you'll create duplicates and conflicts.

GitLab includes a free, secure place to store this file with built-in locking (so two jobs can't change it at once).

Update providers.tf

Add a backend "http" block. The actual URL is supplied by GitLab at run time.

terraform/providers.tf (add this block) 
terraform {
  required_version = ">= 1.9.0"

  backend "http" {}     # GitLab provides the address

  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.60" }
  }
}

GitLab will inject the correct backend configuration during pipeline init. We'll see the exact terraform init command in the pipeline in Part 10.

→ GitLab-managed Terraform state docs

Migration note

If you ran Terraform locally first (Part 08), running terraform init again with the new backend block will offer to copy your existing local state to the remote backend. Type yes when prompted.

Part 10

10 Build the GitLab CI/CD Pipeline

A pipeline is defined in a single file: .gitlab-ci.yml at the root of the repo. GitLab automatically picks it up.

Step 1 — Set up OIDC instead of long-lived keys (recommended)

OIDC lets GitLab prove its identity to AWS and receive temporary credentials — no AWS_SECRET_ACCESS_KEY stored anywhere. Setup is a one-time job in AWS:

  1. In AWS IAM → Identity providers → Add provider. Choose OpenID Connect.
  2. Provider URL: https://gitlab.com. Audience: https://gitlab.com.
  3. Create an IAM Role that trusts that provider, scoped to your project path.
  4. Attach a tightly-scoped policy (S3 + EC2 only — not Administrator).
  5. Copy the Role ARN into a GitLab CI/CD variable named AWS_ROLE_ARN.

→ GitLab OIDC with AWS — full step-by-step

If OIDC feels like too much right now

It's fine to start with stored access keys. In GitLab go to Settings → CI/CD → Variables and add AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY — both Masked and Protected. Plan to upgrade to OIDC once comfortable.

Step 2 — Create .gitlab-ci.yml

This is the heart of the automation. The pipeline has three stages: validate, plan, apply. The apply step is manual, so nothing changes without a human approving it.

.gitlab-ci.yml (at the repository root) 
image:
  name: hashicorp/terraform:1.9
  entrypoint: [""]

variables:
  TF_ROOT: "${CI_PROJECT_DIR}/terraform"
  TF_VAR_FILE: "${CI_PROJECT_DIR}/environments/${ENVIRONMENT}/terraform.tfvars"
  TF_STATE_NAME: "${ENVIRONMENT}"
  AWS_DEFAULT_REGION: "us-east-1"

stages:
  - validate
  - plan
  - apply

# Reusable block: log in to AWS via OIDC, then init Terraform
.terraform_setup: &terraform_setup
  - cd "$TF_ROOT"
  - export AWS_WEB_IDENTITY_TOKEN_FILE=$(mktemp)
  - echo "$CI_JOB_JWT_V2" > $AWS_WEB_IDENTITY_TOKEN_FILE
  - export AWS_ROLE_SESSION_NAME="gitlab-${CI_PIPELINE_ID}"
  - terraform init
      -backend-config="address=${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/terraform/state/${TF_STATE_NAME}"
      -backend-config="lock_address=${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/terraform/state/${TF_STATE_NAME}/lock"
      -backend-config="username=gitlab-ci-token"
      -backend-config="password=${CI_JOB_TOKEN}"
      -backend-config="lock_method=POST"
      -backend-config="unlock_method=DELETE"
      -backend-config="retry_wait_min=5"

# ----------- VALIDATE -----------
validate:
  stage: validate
  script:
    - cd "$TF_ROOT"
    - terraform fmt -check -recursive
    - terraform init -backend=false
    - terraform validate

# ----------- PLAN (dev) -----------
plan:dev:
  stage: plan
  variables:
    ENVIRONMENT: "dev"
  script:
    - *terraform_setup
    - terraform plan -var-file="$TF_VAR_FILE" -out=tfplan
  artifacts:
    paths: [terraform/tfplan]
    expire_in: 1 week

# ----------- APPLY (dev — manual) -----------
apply:dev:
  stage: apply
  variables:
    ENVIRONMENT: "dev"
  needs: [plan:dev]
  script:
    - *terraform_setup
    - terraform apply -input=false tfplan
  when: manual
  environment:
    name: dev
  only:
    - main

# ----------- PLAN (prod) -----------
plan:prod:
  stage: plan
  variables:
    ENVIRONMENT: "prod"
  script:
    - *terraform_setup
    - terraform plan -var-file="$TF_VAR_FILE" -out=tfplan
  artifacts:
    paths: [terraform/tfplan]
    expire_in: 1 week
  only:
    - main

# ----------- APPLY (prod — manual, protected) -----------
apply:prod:
  stage: apply
  variables:
    ENVIRONMENT: "prod"
  needs: [plan:prod]
  script:
    - *terraform_setup
    - terraform apply -input=false tfplan
  when: manual
  environment:
    name: prod
  only:
    - main

Part 11

11 Environments & Approval Gates

Production deserves more protection than dev. GitLab gives you several tools for this.

Protected environments

In GitLab: Settings → CI/CD → Protected environments. Add prod and restrict "Allowed to deploy" to specific people (Maintainers, a release manager, etc.). Now no one else can press the apply button.

Merge request workflow

Every change should follow this loop:

  1. Create a new branch: git checkout -b feature/add-db.
  2. Edit files, commit, push.
  3. Open a Merge Request in GitLab. The pipeline runs validate and plan:dev automatically — the plan output appears in the MR.
  4. A reviewer reads the plan, approves, and merges.
  5. On main, the apply jobs become available to click.

Required approvals

Under Settings → Merge requests → Merge request approvals, require at least one approval before merging. Combined with protected branches, this means no single person can ship to prod alone.

Part 12

12 Security Best Practices

These are the same habits used by mature engineering teams. Adopting them early pays off forever.

A. Identity & credentials

  • MFA everywhere. Root AWS user, IAM users, GitLab account — no exceptions.
  • OIDC over keys. Use GitLab → AWS OIDC federation so pipelines get short-lived credentials.
  • Least privilege IAM. Give roles only the actions they need (s3:PutObject, not s3:*).
  • No keys in code. Use GitLab CI/CD variables (Masked + Protected) or AWS Secrets Manager.
  • Rotate keys quarterly if you must use them.

B. Repository hygiene

  • Protect main. Block direct pushes. Force merge requests.
  • Require code review. Two pairs of eyes catch what one misses.
  • Use a .gitignore that excludes *.tfstate, .terraform/, and any *.local files.
  • Enable secret detection. GitLab Ultimate scans commits; on Free you can run git-secrets or trufflehog in your pipeline.
  • Sign commits with a GPG key for proof of authorship.

C. Pipeline safety

  • Always plan before apply. Plans are artifacts — reviewers can scrutinize them.
  • Make production apply manual with protected-environment restrictions.
  • Pin tool versions (hashicorp/terraform:1.9) so upgrades are intentional.
  • Add scanning jobs: tfsec, checkov, or terrascan catch misconfigurations.

D. Terraform code

  • Pin module & provider versions with ~>.
  • Protect state. Remote backend with locking + access controls.
  • Encrypt by default. S3 SSE, EBS encryption, RDS encryption — all on.
  • Block public S3 access with aws_s3_bucket_public_access_block on every bucket.
  • Tag everything (Environment, Owner, CostCenter) for traceability.
  • Don't hardcode IDs or ARNs — use data sources and outputs.

E. AWS account

  • Enable CloudTrail in every region.
  • Enable GuardDuty for automatic threat detection.
  • Set a billing alarm so a runaway resource doesn't surprise you.
  • Use AWS Organizations with separate accounts per environment when you grow.

What is winget?

winget is the Windows Package Manager — Microsoft's official tool for installing software from the command line. It ships with Windows 10 (1809+) and Windows 11.

Using winget means consistent installs, automatic updates, and no clicking through installer wizards.

If winget isn't found, install App Installer from the Microsoft Store.

Why MFA matters

Multi-factor authentication adds a one-time code (from your phone) to your password. Even if your password leaks, an attacker still can't get in without the device in your pocket.

For an AWS root account, this is non-negotiable — anyone with root access can delete your entire environment or rack up huge bills with crypto mining.

→ AWS MFA docs

What is EC2?

EC2 (Elastic Compute Cloud) is AWS's virtual server service. You pick an operating system, a size (CPU/RAM), and AWS gives you a Linux or Windows machine you can SSH into.

We're using t3.micro (free-tier eligible) running Amazon Linux 2023. The data source aws_ami dynamically finds the newest official image so we don't hardcode IDs.

→ EC2 documentation

Adding masked CI/CD variables

In your GitLab project: Settings → CI/CD → Variables → Add variable. For each secret:

  • Set the Key (e.g. AWS_ACCESS_KEY_ID) and paste the Value.
  • Tick Mask variable — value is hidden in pipeline logs.
  • Tick Protect variable — only available on protected branches (like main).

Terraform's AWS provider automatically reads AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, and AWS_DEFAULT_REGION — no extra code needed.

→ GitLab CI/CD variables docs

Adding a security scan to the pipeline

tfsec reads your Terraform files and warns about insecure patterns (public buckets, missing encryption, open security groups).

tfsec:
  stage: validate
  image: aquasec/tfsec:latest
