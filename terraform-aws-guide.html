Infrastructure as Code · for total beginners

Build real cloud systems by writing them down — explained like you're in middle school.

Terraform lets you describe servers, databases, and networks in a text file, then builds them for you on Amazon Web Services (AWS). This guide teaches you to read those files, run the commands in a live simulator, see the infrastructure as a diagram, and gives you copy-paste examples from a tiny website all the way to a fail-over cluster.

Section 1

What even is Terraform?

The LEGO instruction-sheet idea. Imagine you build an amazing LEGO castle and your friend wants the exact same one. You could describe every brick over the phone (slow, error-prone) — or you hand them the instruction sheet and they build an identical castle perfectly.

Terraform is the instruction sheet for computer systems on the internet. Instead of clicking around a website for hours to set up servers (and forgetting what you did), you write the steps in a file. Terraform reads it and builds everything automatically. Want it again? Run the file again.

This idea has a fancy name: . "Infrastructure" = the servers and networking. "As Code" = described in a text file like a program.

Terraform is "declarative." Big word, simple meaning: you describe what you want, not the step-by-step. Like ordering "a large pepperoni pizza" instead of explaining how to knead dough. You say "I want 3 servers and a load balancer," and Terraform figures out how to make reality match your wish.

The problem it solves

  1. 1
    Forgetting. Six months later nobody remembers which 200 buttons they clicked. A file never forgets.
  2. 2
    Mistakes. Humans clicking buttons skip steps. A computer follows the file exactly, every time.
  3. 3
    Copying is painful. Companies need the same setup for testing, staging, and real customers. One file builds all three.
Terraform keeps a memory file of everything it built, so it knows what to add, change, or remove next time.

Section 2

How to read a Terraform file

Files end in .tf. Almost everything is written in one shape called a block. Learn this one shape and you can read 90% of any Terraform file.

The shape of every block
resource "aws_instance" "web_server" {
  ami           = "ami-0abcd1234"
  instance_type = "t3.micro"

  tags = {
    Name = "MyFirstServer"
  }
}
resource— the kind of thing. "resource" means "a real thing to build."
"aws_instance"— what type: an AWS rental computer.
"web_server"— your nickname for this one, so you can point at it later.
ami / instance_type— settings: which starting software, and how big.
tags— sticky labels for organizing.
Read it out loud, in plain English

"I want a resource — an AWS rental computer — nicknamed web_server. Its starting software is ami-0abcd1234, its size is t3.micro (small and cheap), and I'm sticking a label on it that says its name is MyFirstServer."

The secret handshake: pointing at other things
vpc_id = aws_vpc.my_network.id

This means: "Put me inside the network I nicknamed my_network, using its id." Terraform fills in the ID for you and learns the build order automatically — it's called a .

Those weird network numbers (CIDR)

A like 10.0.0.0/16 describes a neighborhood of addresses. The twist: a smaller slash number means a bigger neighborhood. /16 ≈ 65,000 houses; /24 ≈ 256; /32 = a single house.

Section 5 · Copy-paste examples

From a tiny website to a fail-over cluster

Three complete examples that grow in difficulty. Each is fully commented in plain language, lists exactly what it builds, and links to the official docs. Read top to bottom and you'll watch the same ideas scale up.

Beginner

One web server on a network, reachable from the internet. The 'hello world' of Terraform.

main.tf 
# === A SIMPLE WEBSITE ON AWS ===
# This builds the smallest useful thing: one web server people can visit.

provider "aws" {
  region = "us-east-1"        # Build in Amazon's Northern Virginia data centers
}

# 1) Our own private network in the cloud.
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"   # ~65,000 addresses
  enable_dns_hostnames = true
  tags = { Name = "website-vpc" }
}

# 2) A public subnet — things here can reach the internet.
resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  map_public_ip_on_launch = true          # Give servers here a public address
  tags = { Name = "website-public" }
}

# 3) The "door" between our network and the internet.
resource "aws_internet_gateway" "gw" {
  vpc_id = aws_vpc.main.id
  tags = { Name = "website-igw" }
}

# 4) A routing rule: send internet-bound traffic through that door.
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block = "0.0.0.0/0"              # "anywhere on the internet"
    gateway_id = aws_internet_gateway.gw.id
  }
  tags = { Name = "website-routes" }
}
resource "aws_route_table_association" "public" {
  subnet_id      = aws_subnet.public.id
  route_table_id = aws_route_table.public.id
}

# 5) A firewall: allow web visitors in, allow the server to reach out.
resource "aws_security_group" "web" {
  name   = "web-sg"
  vpc_id = aws_vpc.main.id

  ingress {                               # incoming
    description = "HTTP from anyone"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {                                # outgoing
    from_port   = 0
    to_port     = 0
    protocol    = "-1"                    # all protocols
    cidr_blocks = ["0.0.0.0/0"]
  }
  tags = { Name = "web-sg" }
}

# 6) The web server itself. user_data runs once on first boot.
resource "aws_instance" "web" {
  ami                    = "ami-0abcd1234"   # an Ubuntu image id
  instance_type          = "t3.micro"        # tiny + cheap (often free-tier)
  subnet_id              = aws_subnet.public.id
  vpc_security_group_ids = [aws_security_group.web.id]

  user_data = <<-EOF
    #!/bin/bash
    apt-get update -y
    apt-get install -y nginx
    echo "<h1>Hello from Terraform!</h1>" > /var/www/html/index.html
  EOF

  tags = { Name = "website-server" }
}

# 7) Print the address to visit when we're done.
output "website_ip" {
  value = aws_instance.web.public_ip
}
What this builds
  • ✓A VPC (your private network) and one public subnet
  • ✓An internet gateway so traffic can get in and out
  • ✓A security group (firewall) allowing web traffic on port 80/443
  • ✓One EC2 server that installs a web page when it boots

Intermediate

A real small app: several API servers behind a load balancer, with a managed Postgres database.

main.tf 
# === LOAD BALANCER + API SERVERS + DATABASE ===
# A small but real backend: many API servers share the load, with a safe database.

provider "aws" {
  region = "us-east-1"
  default_tags { tags = { Project = "api-stack", ManagedBy = "terraform" } }
}

variable "db_password" {            # never hard-code secrets!
  type      = string
  sensitive = true
}

variable "ssl_cert" {               # the ARN of your TLS/HTTPS certificate
  type = string
}

# ---- Network: 2 public + 2 private subnets across two zones ----
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  tags = { Name = "api-vpc" }
}

resource "aws_subnet" "public" {
  count                   = 2
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.${count.index + 1}.0/24"
  availability_zone       = ["us-east-1a", "us-east-1b"][count.index]
  map_public_ip_on_launch = true
  tags = { Name = "public-${count.index + 1}" }
}

resource "aws_subnet" "private" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 10}.0/24"
  availability_zone = ["us-east-1a", "us-east-1b"][count.index]
  tags = { Name = "private-${count.index + 1}" }
}

# ---- Firewalls ----
resource "aws_security_group" "alb" {      # load balancer: public
  name   = "alb-sg"
  vpc_id = aws_vpc.main.id
  ingress { from_port = 443  to_port = 443  protocol = "tcp" cidr_blocks = ["0.0.0.0/0"] }
  egress  { from_port = 0    to_port = 0    protocol = "-1"  cidr_blocks = ["0.0.0.0/0"] }
}

resource "aws_security_group" "api" {       # API servers: only from the load balancer
  name   = "api-sg"
  vpc_id = aws_vpc.main.id
  ingress {
    from_port       = 8080
    to_port         = 8080
    protocol        = "tcp"
    security_groups = [aws_security_group.alb.id]   # only the ALB may talk to us
  }
  egress  { from_port = 0 to_port = 0 protocol = "-1" cidr_blocks = ["0.0.0.0/0"] }
}

resource "aws_security_group" "db" {        # database: only from API servers
  name   = "db-sg"
  vpc_id = aws_vpc.main.id
  ingress {
    from_port       = 5432
    to_port         = 5432
    protocol        = "tcp"
    security_groups = [aws_security_group.api.id]
  }
}

# ---- The load balancer (traffic cop) ----
resource "aws_lb" "main" {
  name               = "api-alb"
  load_balancer_type = "application"
  security_groups    = [aws_security_group.alb.id]
  subnets            = aws_subnet.public[*].id   # [*] = "all of them"
}

resource "aws_lb_target_group" "api" {
  name     = "api-targets"
  port     = 8080
  protocol = "HTTP"
  vpc_id   = aws_vpc.main.id
  health_check {                          # poke each server; drop the dead ones
    path                = "/health"
    interval            = 30
    healthy_threshold   = 2
    unhealthy_threshold = 3
  }
}

resource "aws_lb_listener" "https" {
  load_balancer_arn = aws_lb.main.arn
  port              = 443
  protocol          = "HTTPS"
  certificate_arn   = var.ssl_cert        # your TLS certificate
  default_action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.api.arn
  }
}

# ---- Auto Scaling Group: 2-6 API servers that grow with traffic ----
resource "aws_launch_template" "api" {
  name_prefix   = "api-"
  image_id      = "ami-0abcd1234"
  instance_type = "t3.small"
  vpc_security_group_ids = [aws_security_group.api.id]
}

resource "aws_autoscaling_group" "api" {
  desired_capacity    = 2
  min_size            = 2                  # never fewer than 2 (stay online)
  max_size            = 6                  # never more than 6 (cap the bill)
  vpc_zone_identifier = aws_subnet.public[*].id
  target_group_arns   = [aws_lb_target_group.api.arn]

  launch_template {
    id      = aws_launch_template.api.id
    version = "$Latest"
  }
}

# ---- Managed Postgres database (Amazon babysits backups + crashes) ----
resource "aws_db_subnet_group" "db" {
  name       = "db-subnets"
  subnet_ids = aws_subnet.private[*].id
}

resource "aws_db_instance" "postgres" {
  identifier              = "api-postgres"
  engine                  = "postgres"
  engine_version          = "16.3"
  instance_class          = "db.t3.medium"
  allocated_storage       = 50
  db_subnet_group_name    = aws_db_subnet_group.db.name
  vpc_security_group_ids  = [aws_security_group.db.id]
  username                = "appuser"
  password                = var.db_password
  multi_az                = true           # keep a hot backup in the other zone
  backup_retention_period = 7
  skip_final_snapshot     = false
}

output "app_url"  { value = aws_lb.main.dns_name }
What this builds
  • ✓Two public subnets (two zones) so a single data-center outage won't take you down
  • ✓An Application Load Balancer with a health check that removes crashed servers
  • ✓An Auto Scaling Group running 2–6 API servers that grows with traffic
  • ✓A managed RDS Postgres database in private subnets, hidden from the internet

Advanced

A resilient Kubernetes cluster: spread across zones, self-healing, and able to upgrade pods with zero downtime.

main.tf 
# === PRODUCTION EKS: FAIL-OVER + EASY POD UPGRADES ===
# Resilient by design: survives a data-center loss and upgrades with no downtime.

# ---- 1. Spread across THREE zones for fail-over ----
# If one Availability Zone (data center) fails, the other two keep serving.
locals {
  azs = ["us-east-1a", "us-east-1b", "us-east-1c"]
}

resource "aws_subnet" "private" {
  count             = 3
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 10}.0/24"
  availability_zone = local.azs[count.index]
  tags = {
    Name = "eks-private-${count.index + 1}"
    # These tags let Kubernetes auto-discover where to place load balancers:
    "kubernetes.io/role/internal-elb" = "1"
  }
}

# ---- 2. The cluster brain (control plane), managed by Amazon ----
resource "aws_eks_cluster" "main" {
  name     = "prod-cluster"
  role_arn = aws_iam_role.cluster.arn
  version  = "1.30"

  vpc_config {
    subnet_ids              = aws_subnet.private[*].id   # lives in all 3 zones
    endpoint_private_access = true
    endpoint_public_access  = true
  }
  depends_on = [aws_iam_role_policy_attachment.cluster_policy]
}

# ---- 3. Worker nodes: auto-scaling + safe rolling upgrades ----
resource "aws_eks_node_group" "general" {
  cluster_name    = aws_eks_cluster.main.name
  node_group_name = "general"
  node_role_arn   = aws_iam_role.nodes.arn
  subnet_ids      = aws_subnet.private[*].id   # workers in all 3 zones (fail-over)
  instance_types  = ["m5.large"]

  scaling_config {
    desired_size = 3
    min_size     = 3        # at least one healthy worker per zone
    max_size     = 12       # grow under load, then shrink back
  }

  # EASY UPGRADE: when you bump the version, AWS replaces nodes a few at a time,
  # never taking more than 1 offline at once — the app stays up the whole time.
  update_config {
    max_unavailable = 1
  }

  # Tells AWS which AMI/k8s version to run; change it and re-apply to upgrade.
  version = "1.30"

  lifecycle {
    create_before_destroy = true   # bring up the new node BEFORE removing the old
  }
}

# ====================================================================
# Below: the Kubernetes objects (managed by the kubernetes provider).
# This is where "easy pod upgrades" and "self-healing" actually happen.
# ====================================================================

# ---- 4. The app, as a Deployment with rolling updates ----
resource "kubernetes_deployment" "app" {
  metadata { name = "web-app" }

  spec {
    replicas = 4                    # keep 4 copies of the app running

    # ROLLING UPDATE: swap pods gradually so users never see downtime.
    strategy {
      type = "RollingUpdate"
      rolling_update {
        max_surge       = 1         # may add 1 extra pod during the swap
        max_unavailable = 0         # never drop below 4 healthy pods
      }
    }

    selector { match_labels = { app = "web-app" } }

    template {
      metadata { labels = { app = "web-app" } }
      spec {
        # Spread copies across zones so one zone loss can't kill them all.
        topology_spread_constraint {
          max_skew           = 1
          topology_key       = "topology.kubernetes.io/zone"
          when_unsatisfiable = "ScheduleAnyway"
          label_selector { match_labels = { app = "web-app" } }
        }

        container {
          name  = "web"
          image = "myregistry/web-app:v2"   # change the tag + apply = upgrade

          # SELF-HEALING: Kubernetes restarts a pod that stops being healthy.
          liveness_probe {                  # "are you alive?" -> restart if not
            http_get { path = "/healthz" port = 8080 }
            initial_delay_seconds = 10
            period_seconds        = 10
          }
          readiness_probe {                 # "ready for traffic?" -> hold back if not
            http_get { path = "/ready" port = 8080 }
            period_seconds = 5
          }

          resources {
            requests = { cpu = "250m", memory = "256Mi" }
            limits   = { cpu = "500m", memory = "512Mi" }
          }
        }
      }
    }
  }
}

# ---- 5. A safety net for upgrades: PodDisruptionBudget ----
# Guarantees at least 3 of the 4 pods stay up during any voluntary disruption
# (like a node upgrade) — so the app is never fully offline.
resource "kubernetes_pod_disruption_budget_v1" "app" {
  metadata { name = "web-app-pdb" }
  spec {
    min_available = 3
    selector { match_labels = { app = "web-app" } }
  }
}

# ---- 6. Auto-scale the PODS based on CPU (separate from node scaling) ----
resource "kubernetes_horizontal_pod_autoscaler_v2" "app" {
  metadata { name = "web-app-hpa" }
  spec {
    min_replicas = 4
    max_replicas = 20
    scale_target_ref {
      api_version = "apps/v1"
      kind        = "Deployment"
      name        = kubernetes_deployment.app.metadata[0].name
    }
    metric {
      type = "Resource"
      resource {
        name = "cpu"
        target { type = "Utilization" average_utilization = 60 }
      }
    }
  }
}
What this builds
  • ✓An EKS control plane (the brain), run and protected by Amazon
  • ✓Worker node groups spread across 3 zones — if one zone dies, the others carry on (fail-over)
  • ✓Auto-scaling workers (3 → 12) plus rolling updates that replace pods a few at a time
  • ✓An app Deployment with health probes + a PodDisruptionBudget so upgrades never take the app fully offline

How "fail-over" works, simply

Fail-over means: when one piece dies, another takes over so visitors never notice. We get it by spreading copies across three separate data centers (Availability Zones). If a whole building loses power, the copies in the other two keep serving. The load balancer simply stops sending people to the dead ones.

⇪

How "easy pod upgrades" work

To upgrade with zero downtime, Kubernetes does a rolling update: it starts a new pod, waits for it to report healthy, then retires one old pod — repeating a few at a time. A PodDisruptionBudget guarantees a minimum number always stay up. To upgrade, you literally just change the image tag (e.g. v2 → v3) and run apply.
