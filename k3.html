AWS EKS · Apache Kafka · Strimzi

Run Apache Kafka on Amazon's Kubernetes, the easy way.

This guide walks you through setting up Kafka on AWS EKS using the Strimzi operator. We start from zero, build the worker computers (node groups), add load balancing, make it survive failures, let it grow, and finish with logging and pretty Grafana dashboards. Plain language. Real commands. One step at a time.

~60–90 min Beginner friendly Costs real AWS money — clean up after!

/ 00

First, what are all these words?

Before we type anything, let's learn the four big words in plain English. Think of building a giant restaurant kitchen.

📨 Apache Kafka

Kafka is like a super-fast post office for computer programs. One program drops off messages, and other programs pick them up — millions per second, in order, without losing them.

📦 Kubernetes

Kubernetes (people write k8s) is a manager that runs apps inside containers. It decides which computer runs what, restarts crashed apps, and keeps everything healthy.

☁️ AWS EKS

EKS = Elastic Kubernetes Service. It's Amazon running the hard, boring parts of Kubernetes for you, so you only worry about your apps and your worker computers.

🤖 Strimzi

Strimzi is a robot helper (an "operator") that knows how to install and babysit Kafka on Kubernetes. You tell it "I want a Kafka cluster," and it builds and repairs it for you.

The kitchen picture: EKS is the building. Node groups are teams of cooks (worker computers). Kubernetes is the head chef giving orders. Strimzi is the specialist who sets up the Kafka "order conveyor belt." We'll build them in that order.

Big change to know: Old Kafka needed a sidekick called ZooKeeper. Modern Kafka (4.x) dropped it and uses KRaft instead. Strimzi removed ZooKeeper support back in version 0.46, so everything here is KRaft. That's why we make two kinds of nodes: controllers (the bookkeepers) and brokers (the message carriers).

/ 01

Step 0 — Install your tools

You need four command-line tools on your laptop. Think of them as your toolbox before any building starts.

  • aws — talks to your Amazon account.
  • eksctl — builds and changes EKS clusters with one command.
  • kubectl — sends orders to Kubernetes ("koob-control").
  • helm — installs big apps (like Strimzi) from ready-made packages.

Install on macOS (with Homebrew):

bash 
brew install awscli eksctl kubectl helm

# Check they all work
aws --version
eksctl version      # must be 0.215.0 or newer
kubectl version --client
helm version

Then connect to your AWS account:

bash 
# It will ask for your Access Key, Secret Key, region, and output format
aws configure

# Make sure it recognizes you
aws sts get-caller-identity

On Windows use choco install or download the installers; on Linux use the official scripts. The links are in the Official docs section.

Step 1 — Create the EKS control plane

Every cluster has a control plane (the brain) and nodes (the muscle). Amazon runs the brain for you. Here we create the brain only, with zero nodes, so we can add node groups carefully in the next step (that's what you asked for).

Save this as cluster.yaml:

cluster.yaml 
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: kafka-cluster
  region: us-east-1
  version: "1.33"          # use the newest EKS offers (1.30+)

# Use 3 Availability Zones so we can survive a zone outage later
availabilityZones: ["us-east-1a", "us-east-1b", "us-east-1c"]

# Start with NO worker nodes — we add node groups in Step 2
managedNodeGroups: []

# Lets pods get their own AWS permissions (needed later for the load balancer)
iam:
  withOIDC: true

Build it (takes ~15 minutes — grab a snack):

bash 
eksctl create cluster -f cluster.yaml

# When done, point kubectl at the new cluster
aws eks update-kubeconfig --name kafka-cluster --region us-east-1

# Prove it works (there are no worker nodes yet — that's expected)
kubectl get nodes
kubectl get svc

What just happened? Amazon created the brain, a private network (VPC), and all the security plumbing. You didn't pay for any worker computers yet because we asked for none.

/ 03

Step 2 — Create the node groups ⭐

A node group is a team of identical worker computers (EC2 machines) that join your cluster together. We'll use managed node groups, which means Amazon patches and replaces the machines for you.

We spread the team across 3 Availability Zones. An Availability Zone is a separate Amazon data center. If one zone has a power cut, the other two keep working — that's the foundation of failover.

Save this as nodegroup.yaml:

nodegroup.yaml 
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: kafka-cluster
  region: us-east-1

managedNodeGroups:
  - name: kafka-workers
    instanceType: m5.xlarge      # 4 CPU, 16 GB RAM — good starter size
    desiredCapacity: 3           # start with 3 machines (one per zone)
    minSize: 3                   # never go below 3
    maxSize: 6                   # allowed to grow up to 6
    volumeSize: 100              # 100 GB disk on each machine
    privateNetworking: true      # keep workers off the public internet
    availabilityZones: ["us-east-1a", "us-east-1b", "us-east-1c"]
    labels:
      role: kafka                # a name tag so Kafka pods land here

    # Roll out upgrades one machine at a time (no downtime)
    updateConfig:
      maxUnavailable: 1

    # Let AWS auto-replace a broken machine (node-level self-healing)
    nodeRepairConfig:
      enabled: true

    # Use SSM for safe shell access instead of open SSH ports
    ssh:
      enableSsm: true

Add the node group to your existing cluster:

bash 
# Create the node group from the file
eksctl create nodegroup -f nodegroup.yaml

# Watch the 3 machines show up and turn "Ready"
kubectl get nodes -o wide --watch

# See node groups Amazon knows about
eksctl get nodegroup --cluster kafka-cluster

Prefer one line instead of a file? This does the same thing:

bash 
eksctl create nodegroup \
  --cluster kafka-cluster \
  --region us-east-1 \
  --name kafka-workers \
  --node-type m5.xlarge \
  --nodes 3 --nodes-min 3 --nodes-max 6 \
  --node-volume-size 100 \
  --managed

Why "managed"? Self-managed nodes give you more knobs, but managed node groups handle patching, draining, and replacing machines automatically. For most Kafka setups, managed is the calm choice.

/ 04

Step 3 — Add more nodes & auto-heal

Three ways to get more muscle: change the size by hand, add a brand-new node group, or let the cluster grow itself automatically.

A) Scale by hand (fastest)

bash 
# Grow from 3 to 5 machines, allow up to 8
eksctl scale nodegroup \
  --cluster kafka-cluster \
  --name kafka-workers \
  --nodes 5 --nodes-min 3 --nodes-max 8 \
  --wait

B) Add a second node group

Handy when you want different machines — for example bigger boxes, or cheaper Spot machines for non-critical work.

bash 
eksctl create nodegroup \
  --cluster kafka-cluster \
  --name kafka-workers-2 \
  --node-type m5.2xlarge \
  --nodes 3 --nodes-min 3 --nodes-max 6 \
  --managed

C) Let it grow automatically

When pods can't fit, an autoscaler adds machines; when machines sit empty, it removes them to save money. The modern tool is Karpenter; the classic one is Cluster Autoscaler.

bash 
# Quick start: turn on the classic Cluster Autoscaler for our node group.
# (eksctl already tagged the node group for auto-discovery.)
kubectl apply -f https://raw.githubusercontent.com/kubernetes/autoscaler/master/cluster-autoscaler/cloudprovider/aws/examples/cluster-autoscaler-autodiscover.yaml

# Edit the deployment and set your cluster name:
#   --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/kafka-cluster
kubectl -n kube-system edit deployment cluster-autoscaler

Two kinds of self-healing: Node Repair (which we turned on in Step 2) replaces a broken machine. Kubernetes itself replaces a broken pod. Together they keep Kafka standing without you waking up at 3 a.m.

/ 05

Step 4 — Install the Strimzi operator

The operator is the robot helper. We install it once with Helm. After this, building Kafka is as simple as writing a short YAML file.

bash 
# Make a tidy room (namespace) just for Kafka
kubectl create namespace kafka

# Add the official Strimzi Helm chart and install the operator
helm repo add strimzi https://strimzi.io/charts/
helm repo update

helm install strimzi-operator strimzi/strimzi-kafka-operator \
  --namespace kafka \
  --version 1.0.1

# Wait for the operator pod to be Running
kubectl get pods -n kafka -w

The operator also installs Custom Resource Definitions — new "word types" like Kafka and KafkaNodePool that Kubernetes now understands.

/ 06

Step 5 — Deploy Kafka (KRaft + node pools)

Modern Strimzi builds Kafka from node pools. We make two pools:

  • Controllers — the bookkeepers that agree on who's in charge (KRaft). Usually 3.
  • Brokers — the workers that actually store and serve your messages. Usually 3 or more.

1. Controller pool — save as controller-pool.yaml:

controller-pool.yaml 
apiVersion: kafka.strimzi.io/v1
kind: KafkaNodePool
metadata:
  name: controller
  namespace: kafka
  labels:
    strimzi.io/cluster: my-cluster
spec:
  replicas: 3
  roles:
    - controller
  storage:
    type: jbod
    volumes:
      - id: 0
        type: persistent-claim
        size: 20Gi
        kraftMetadata: shared
        deleteClaim: false

2. Broker pool — save as broker-pool.yaml:

broker-pool.yaml 
apiVersion: kafka.strimzi.io/v1
kind: KafkaNodePool
metadata:
  name: broker
  namespace: kafka
  labels:
    strimzi.io/cluster: my-cluster
spec:
  replicas: 3
  roles:
    - broker
  storage:
    type: jbod
    volumes:
      - id: 0
        type: persistent-claim
        size: 100Gi
        deleteClaim: false

3. The Kafka cluster itself — save as kafka.yaml:

This one file also sets up failover (rack awareness + replication) and an external load balancer. We explain each part in Steps 6 and 7.

kafka.yaml 
apiVersion: kafka.strimzi.io/v1
kind: Kafka
metadata:
  name: my-cluster
  namespace: kafka
  annotations:
    strimzi.io/node-pools: enabled    # use node pools (default now)
    strimzi.io/kraft: enabled         # use KRaft, no ZooKeeper
spec:
  kafka:
    version: 4.1.0
    # metadataVersion is optional; the operator picks one that matches

    listeners:
      # Inside-the-cluster, plain text (for our test apps)
      - name: plain
        port: 9092
        type: internal
        tls: false
      # Inside-the-cluster, encrypted
      - name: tls
        port: 9093
        type: internal
        tls: true
      # Outside the cluster, via an AWS load balancer (see Step 6)
      - name: external
        port: 9094
        type: loadbalancer
        tls: true

    # --- Failover settings (see Step 7) ---
    config:
      default.replication.factor: 3
      min.insync.replicas: 2
      offsets.topic.replication.factor: 3
      transaction.state.log.replication.factor: 3
      transaction.state.log.min.isr: 2

    # Spread brokers across the 3 zones automatically
    rack:
      topologyKey: topology.kubernetes.io/zone

  # Lets us manage topics and users with YAML
  entityOperator:
    topicOperator: {}
    userOperator: {}

4. Apply all three and wait:

bash 
kubectl apply -f controller-pool.yaml
kubectl apply -f broker-pool.yaml
kubectl apply -f kafka.yaml

# Wait until Kafka reports "Ready" (a few minutes)
kubectl wait kafka/my-cluster --for=condition=Ready --timeout=600s -n kafka

# Look at your shiny new pods
kubectl get pods -n kafka

You now have Kafka running! The operator created controller pods, broker pods, persistent disks, internal addresses, and TLS certificates — all from those three small files.

Step 6 — Load balancing & outside access

A load balancer is like a receptionist at the front door: visitors arrive at one address, and the receptionist sends each one to a free broker. For Kafka on AWS we use a Network Load Balancer (NLB), not the web-style ALB.

1. Install the AWS Load Balancer Controller

This is the piece that turns a Kubernetes "LoadBalancer" request into a real AWS NLB.

bash 
# 1) Make sure OIDC is on (we set withOIDC in Step 1)
eksctl utils associate-iam-oidc-provider --cluster kafka-cluster --approve

# 2) Download the permission policy AWS publishes
curl -o iam_policy.json https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/main/docs/install/iam_policy.json

aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file://iam_policy.json

# 3) Give the controller its own identity (replace ACCOUNT_ID)
eksctl create iamserviceaccount \
  --cluster kafka-cluster \
  --namespace kube-system \
  --name aws-load-balancer-controller \
  --attach-policy-arn arn:aws:iam::ACCOUNT_ID:policy/AWSLoadBalancerControllerIAMPolicy \
  --approve

# 4) Install the controller with Helm
helm repo add eks https://aws.github.io/eks-charts
helm repo update
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=kafka-cluster \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller

2. Tell Strimzi to ask for an NLB

Add annotations to the external listener in kafka.yaml, then re-apply. Annotations are little sticky notes that tell AWS what kind of load balancer you want.

kafka.yaml (listener section) 
      - name: external
        port: 9094
        type: loadbalancer
        tls: true
        configuration:
          bootstrap:
            annotations:
              service.beta.kubernetes.io/aws-load-balancer-type: "external"
              service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
              service.beta.kubernetes.io/aws-load-balancer-scheme: "internet-facing"
          brokers:
            - broker: 0
              annotations:
                service.beta.kubernetes.io/aws-load-balancer-type: "external"
                service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
            - broker: 1
              annotations:
                service.beta.kubernetes.io/aws-load-balancer-type: "external"
                service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
            - broker: 2
              annotations:
                service.beta.kubernetes.io/aws-load-balancer-type: "external"
                service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
bash 
kubectl apply -f kafka.yaml

# Find the public address clients will connect to (port 9094)
kubectl get service my-cluster-kafka-external-bootstrap -n kafka

Why one address per broker? Kafka clients must reach every broker directly, not just one. So Strimzi creates a bootstrap address plus one address per broker. The NLB shares the load and routes traffic to healthy brokers.

Step 7 — Failover & high availability

"Failover" means: when something breaks, the system keeps working. We already added the magic words in kafka.yaml. Here's what each one does, in plain English.

default.replication.factor: 3

Every message is copied to 3 brokers. Lose one broker, your data is still safe on the other two. Like keeping three photocopies in three rooms.

min.insync.replicas: 2

A write only "counts" once 2 copies are saved. This stops you from losing data if a broker dies right after a write.

rack: topologyKey: topology.kubernetes.io/zone

Rack awareness. Strimzi puts the 3 copies in 3 different zones (data centers). One whole zone can go dark and you still have copies elsewhere. Don't put all your eggs in one basket.

Optional but smart: Cruise Control auto-balances your brokers.

Add this near the bottom of kafka.yaml (same level as entityOperator). It keeps data evenly spread, and re-spreads it after a broker is replaced.

kafka.yaml (add to spec) 
  cruiseControl: {}

Test failover safely — delete a broker pod and watch it heal:

bash 
# Kill one broker on purpose
kubectl delete pod my-cluster-broker-0 -n kafka

# Watch Kubernetes bring it right back, no data lost
kubectl get pods -n kafka -w
/ 09

Step 8 — Logging

Logs are the diary each broker writes about what it's doing. They're how you find out why something went wrong.

Read logs right now

bash 
# Live logs from one broker (Ctrl+C to stop)
kubectl logs -f my-cluster-broker-0 -n kafka

# Logs from the operator (great for troubleshooting setup)
kubectl logs -f deployment/strimzi-cluster-operator -n kafka

Set the log level (less noise, or more detail)

Add a logging block under spec.kafka in kafka.yaml. Use INFO normally, DEBUG when hunting a bug, WARN to quiet things down.

kafka.yaml (under spec.kafka) 
    logging:
      type: inline
      loggers:
        rootLogger.level: INFO
        # Quiet a chatty area as an example:
        logger.kafkacontroller.name: org.apache.kafka.controller
        logger.kafkacontroller.level: WARN

Going to production? Send all these logs to one place (CloudWatch, Loki, or OpenSearch) with a log shipper like Fluent Bit, so you can search every broker at once instead of pod by pod.

Step 9 — Grafana & Prometheus dashboards

Prometheus collects numbers (metrics) from Kafka. Grafana draws those numbers as pretty graphs. Together they're your dashboard — like the speedometer and fuel gauge in a car.

1. Turn on Kafka metrics

Grab Strimzi's example metrics file, apply its ConfigMap, then point Kafka at it.

bash 
# Strimzi ships ready-made metrics examples in its GitHub repo:
# examples/metrics/  → kafka-metrics.yaml, strimzi-pod-monitor.yaml, grafana-dashboards/
# Download kafka-metrics.yaml from there, then apply just the ConfigMap part:
kubectl apply -f kafka-metrics-configmap.yaml -n kafka

Add this under spec.kafka in kafka.yaml and re-apply:

kafka.yaml (under spec.kafka) 
    metricsConfig:
      type: jmxPrometheusExporter
      valueFrom:
        configMapKeyRef:
          name: kafka-metrics
          key: kafka-metrics-config.yml

There's also a newer option, the Strimzi Metrics Reporter, that skips JMX.

2. Install Prometheus + Grafana together

The kube-prometheus-stack Helm chart installs both at once.

bash 
kubectl create namespace monitoring

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

# The last flag lets Prometheus find Strimzi's PodMonitors automatically
helm install kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  -n monitoring \
  --set prometheus.prometheusSpec.podMonitorSelectorNilUsesHelmValues=false

3. Tell Prometheus where Kafka's metrics are

A PodMonitor is a note that says "scrape these pods." Save as kafka-podmonitor.yaml:

kafka-podmonitor.yaml 
apiVersion: monitoring.coreos.com/v1
kind: PodMonitor
metadata:
  name: kafka-resources-metrics
  namespace: monitoring
  labels:
    app: strimzi
spec:
  namespaceSelector:
    matchNames:
      - kafka
  selector:
    matchLabels:
      strimzi.io/kind: Kafka
  podMetricsEndpoints:
    - path: /metrics
      port: tcp-prometheus
      interval: 30s
bash 
kubectl apply -f kafka-podmonitor.yaml

4. Open Grafana & import Strimzi's dashboards

bash 
# Get the Grafana admin password
kubectl get secret -n monitoring kube-prometheus-stack-grafana \
  -o jsonpath="{.data.admin-password}" | base64 -d ; echo

# Open Grafana in your browser at http://localhost:3000  (user: admin)
kubectl port-forward -n monitoring svc/kube-prometheus-stack-grafana 3000:80

Inside Grafana: Dashboards → Import. Paste a Strimzi dashboard. The data source (Prometheus) is already set up for you. Good ones to start with:

  • Strimzi Kafka dashboard JSON from the Strimzi GitHub repo (examples/metrics/grafana-dashboards/).
  • Community "Strimzi Kafka" dashboards on Grafana.com (search the dashboard library).

No data on the graphs? Send a few test messages (Step 10) so there's something to measure, then refresh. Empty Kafka = empty charts.

Step 10 — Send your first messages

Let's prove it all works: make a topic, type some messages, and read them back.

1. Create a topic with YAML — save as topic.yaml:

topic.yaml 
apiVersion: kafka.strimzi.io/v1
kind: KafkaTopic
metadata:
  name: my-topic
  namespace: kafka
  labels:
    strimzi.io/cluster: my-cluster
spec:
  partitions: 3
  replicas: 3
  config:
    min.insync.replicas: 2
bash 
kubectl apply -f topic.yaml
kubectl get kafkatopics -n kafka

2. Type messages (the producer). Each line you type is one message:

bash 
kubectl -n kafka run kafka-producer -ti \
  --image=quay.io/strimzi/kafka:latest-kafka-4.1.0 \
  --rm=true --restart=Never -- \
  bin/kafka-console-producer.sh \
  --bootstrap-server my-cluster-kafka-bootstrap:9092 \
  --topic my-topic

3. In another terminal, read them back (the consumer):

bash 
kubectl -n kafka run kafka-consumer -ti \
  --image=quay.io/strimzi/kafka:latest-kafka-4.1.0 \
  --rm=true --restart=Never -- \
  bin/kafka-console-consumer.sh \
  --bootstrap-server my-cluster-kafka-bootstrap:9092 \
  --topic my-topic --from-beginning

If the consumer prints the messages you typed — congratulations, your Kafka is live! 🎉 Go check the Grafana dashboard; you'll see the traffic appear.

/ END

Clean up (so you stop paying)

EKS, EC2 machines, disks, and load balancers all cost money every hour. When you're done practising, delete everything in this order.

bash 
# 1) Remove Kafka (this also removes its load balancers)
kubectl delete kafka my-cluster -n kafka

# 2) Remove the operators and monitoring
helm uninstall strimzi-operator -n kafka
helm uninstall kube-prometheus-stack -n monitoring

# 3) Finally, delete the whole cluster, node groups and all
eksctl delete cluster --name kafka-cluster --region us-east-1

Double-check in the AWS console that no EC2 instances, EBS volumes, or load balancers are left running. A forgotten NLB or disk can quietly add to your bill.

Built as a learning guide. Versions current as of June 2026 — always confirm against the official docs above. Cloud resources cost money; remember to clean up.

Kafka stores messages in topics, and each topic is split into partitions so the work can spread across many brokers. Programs that send messages are producers; programs that read them are consumers.

The magic is that consumers can read at their own speed, replay old messages, and many consumers can read the same data. That's why Kafka is used for things like tracking clicks, moving payments, and feeding data pipelines.

A container is an app packed up with everything it needs to run. Kubernetes runs containers inside pods (the smallest unit), schedules them onto worker machines (nodes), and restarts them if they crash.

You describe the result you want in YAML ("I want 3 copies of this app"), and Kubernetes constantly works to make reality match your description. That idea is called desired state.

With EKS, Amazon runs and protects the Kubernetes control plane (the brain) across multiple data centers. You don't patch it or babysit it.

You still provide the worker nodes (the EC2 machines that run your pods) — which is exactly what node groups are for. You pay for the control plane per hour, plus whatever machines and storage you use.

Strimzi is an open-source project (part of the Cloud Native Computing Foundation). It installs an operator — a program that watches for special Kubernetes objects like Kafka and turns them into a real, running Kafka cluster.

It handles the hard parts: rolling upgrades with no downtime, TLS certificates, user and topic management, scaling, and recovery. You mostly just edit small YAML files.

Old Kafka needed a separate system called ZooKeeper to keep track of which broker was in charge and where data lived. It was one more thing to run and break.

KRaft (Kafka Raft) moves that bookkeeping inside Kafka itself, handled by special controller nodes. Kafka 4.x is KRaft-only, and Strimzi dropped ZooKeeper support in version 0.46. That's why this guide creates a controller pool and a broker pool.

Helm is the "app store" for Kubernetes. Instead of applying dozens of YAML files by hand, you install a chart — a packaged app with sensible defaults.

You can override settings with --set flags or a values file, and upgrade or uninstall the whole thing with one command.

A VPC (Virtual Private Cloud) is your own private network inside AWS. It's split into subnets, and those subnets live in different Availability Zones.

Private subnets have no direct door to the internet (safer for workers), while public subnets do (used by internet-facing load balancers). eksctl sets all of this up for you.

A managed node group is an Auto Scaling group of EC2 machines that AWS looks after for your cluster. AWS handles graceful draining when it patches or replaces a machine, so pods move safely.

It can span multiple Availability Zones for high availability, and there's no extra charge — you only pay for the EC2 machines, disks, and other resources you use.

Managed node groups: AWS patches, drains, and replaces machines; the Kubernetes version follows the control plane; simplest to run.

Self-managed node groups: you control the launch template, the AMI, and the upgrade process — more flexible, but more work. For Windows nodes or very custom setups, you may need self-managed. Most Kafka clusters are happy on managed.

Cluster Autoscaler grows and shrinks existing node groups when pods can't be placed. It's mature and simple, but tied to node group sizes.

Karpenter is newer and smarter: it launches right-sized machines on demand, picks cheaper instance types, and removes them quickly when idle. For bursty Kafka workloads, Karpenter often saves money — but it's a bit more to set up.

A Custom Resource Definition teaches Kubernetes a new kind of object. Strimzi adds objects like Kafka, KafkaNodePool, KafkaTopic, and KafkaUser.

After the CRDs are installed, you can kubectl apply a Kafka object just like a built-in one, and the operator does the rest. This guide uses the modern v1 API that Strimzi switched to in version 1.0.

An ALB (Application Load Balancer) only understands web traffic (HTTP/HTTPS). Kafka does not speak HTTP — it has its own binary protocol over plain TCP.

So Kafka needs a Network Load Balancer (NLB), which works at the TCP level (Layer 4). The annotations in this guide tell the AWS Load Balancer Controller to build an NLB and route by IP for best performance.

"ISR" means in-sync replicas — the copies that are fully caught up. min.insync.replicas: 2 means a write is only accepted once at least 2 copies have it.

With replication factor 3 and min ISR 2, you can lose 1 broker and still accept writes safely. If you set min ISR equal to the replication factor, losing even one broker stops writes — so 2-of-3 is the common, balanced choice.

Rack awareness tells Kafka which "rack" each broker is in. On AWS we map a rack to an Availability Zone using the label topology.kubernetes.io/zone.

Kafka then makes sure the copies of a partition land in different zones. If one zone fails, another zone still has a full copy, and Kafka automatically picks a new leader there. This is the heart of zone-level failover.

Reading logs pod by pod is fine for learning, but painful at scale. A log shipper like Fluent Bit runs on every node, collects all container logs, and sends them to one searchable place.

On AWS that's often CloudWatch Logs or OpenSearch; many teams use Loki, which plugs right into the same Grafana you set up in Step 9.

Prometheus regularly "scrapes" numbers from your apps (called metrics) and stores them over time. Examples for Kafka: messages per second, under-replicated partitions, JVM memory.

Grafana reads Prometheus and draws dashboards and alerts. The kube-prometheus-stack chart installs both, plus the operator that manages them, in one go.

The classic way to expose Kafka metrics is the JMX Prometheus Exporter (used in this guide) — well supported and reliable.

Newer Strimzi versions also offer the Strimzi Metrics Reporter, which exposes Prometheus metrics directly without JMX. It was announced in Strimzi 0.47 and aims to be lighter and faster. Either works; the JMX exporter is the safe default while the reporter matures.
