01

Discovery & Context Gathering

Before touching anything, build a picture of intent, ownership, and purpose.

Why this step matters: NiFi flows are often built organically over time. Without understanding the original design intent, you risk breaking critical pipelines or misidentifying issues. Talk to the original builder first.

Who designed and built this NiFi deployment?

Answers: Name, team, contact info, whether they're still available for questions.

Why it helps: You'll need someone who understands the original design decisions — especially for undocumented custom processors or unusual routing logic.

Example: "Was this built by the data engineering team? Is there a Confluence/Notion page or runbook for this environment?"

What is the primary purpose of this NiFi instance?

Answers: ETL, real-time ingestion, event routing, CDC, data transformation, API integration.

Why it helps: Sets expectations for throughput, latency requirements, and criticality. A real-time payment flow requires different review depth than a nightly batch job.

Example: "Is this instance the only data mover between our S3 data lake and Redshift, or is it one of several pipelines?"

Is there existing documentation, a runbook, or architecture diagram?

Answers: Link to wiki, Confluence space, README in Git, draw.io diagram.

Why it helps: Saves hours of reverse-engineering. Even outdated docs reveal original intent.

⚠ Common gap: Many NiFi deployments have zero documentation. If none exists, your first deliverable should be creating it as you review.

What is the current NiFi version and when was it last updated?

How to check: NiFi UI → top-right menu → "About" or run ./nifi.sh status in /opt/nifi/bin/

Why it helps: NiFi 2.x brought breaking changes (Java 21 required, templates deprecated, Variable Registry removed). Knowing the version determines which review steps apply.

✅ Target: NiFi 2.x (latest stable). NiFi 1.28.x is the last 1.x release — strongly consider upgrading. → NiFi Downloads & Release Notes

📖 Reference: NiFi 2.0 Migration Guide · Release Notes
02

AWS Infrastructure Review

Understand the AWS topology — EC2 sizing, networking, storage, IAM roles, and HA setup.

Standalone Deployment

Single EC2 node, simpler config, no ZooKeeper required in NiFi 2.x standalone mode. Suitable for dev/test or low-volume production.

Clustered Deployment

3+ EC2 nodes with ZooKeeper (embedded or external). Requires cluster protocol ports 10000–10005 open between nodes. Load balancer for UI access.

What EC2 instance type(s) are used and how many nodes?

Check via: AWS Console → EC2 → Instances, or aws ec2 describe-instances --filters "Name=tag:Name,Values=nifi*"

Why it helps: NiFi is memory and I/O intensive. Under-provisioned instances cause back-pressure and flow failures. Memory should be ≥16GB for production; dedicated EBS volumes for repositories is recommended.

Example: m5.2xlarge (8 vCPU, 32GB RAM) with dedicated GP3 EBS volumes for content_repository and flowfile_repository.

How is the VPC and networking configured?

Check via: AWS Console → VPC → Subnets, Route Tables, Security Groups

Why it helps: NiFi should never be publicly exposed without a reverse proxy or ALB. Security Groups must restrict port 8443 (HTTPS UI) to known CIDRs, and cluster ports (10000–10005) to internal VPC only.

Key questions:

  • Is NiFi in a private subnet behind an ALB?
  • Is port 8443 open to 0.0.0.0/0? (Red flag)
  • Are ZooKeeper ports 2181/2888/3888 restricted to cluster nodes only?

🔴 Critical: A public Security Group rule allowing port 8443 from anywhere is a critical misconfiguration. Lock it down immediately.

What IAM roles and policies are attached to the NiFi EC2 instances?

Check via: AWS Console → EC2 → Instance → IAM Role, or aws iam list-attached-role-policies --role-name <role-name>

Why it helps: NiFi processors that interact with AWS (S3, SQS, Kinesis, Secrets Manager) need IAM permissions. Overly permissive roles (e.g., AdministratorAccess) are a security risk.

Example of principle of least privilege:

{
  "Effect": "Allow",
  "Action": ["s3:GetObject","s3:PutObject","s3:ListBucket"],
  "Resource": ["arn:aws:s3:::my-data-bucket/*"]
}

Is there an ALB, NLB, or other load balancer in front of NiFi?

Check via: AWS Console → EC2 → Load Balancers

Why it helps: For clustered NiFi, the UI must be accessed through a single entry point. An ALB or NLB handles this and provides TLS termination. Without one, users may connect directly to individual nodes — causing cluster state confusion.

Also check: Route 53 DNS records pointing to the load balancer, and whether SSL certs are managed via ACM.

How are EBS volumes organized and sized?

Check via: df -h on each node, AWS Console → EC2 → Volumes

Why it helps: NiFi writes to three key repositories. Full disks cause flow failures and potential data loss.

03

Configuration File Audit

Review every file in /opt/nifi/conf/ — these define how NiFi behaves in every dimension.

Key config files location

/opt/nifi/conf/
├── nifi.properties          # Core configuration — ports, TLS, cluster, repositories
├── bootstrap.conf           # JVM settings, memory, startup options
├── bootstrap-notification-services.xml
├── zookeeper.properties     # Cluster ZooKeeper quorum config
├── authorizers.xml          # User/group providers and access policy engine
├── login-identity-providers.xml  # Auth methods (LDAP, OIDC, single-user)
└── logback.xml              # Log levels and appenders

Review nifi.properties — Web, Security, Cluster, and Repository sections

Key properties to verify:

# Web UI
nifi.web.https.host=0.0.0.0
nifi.web.https.port=8443
nifi.web.http.port=            # Should be EMPTY (disable HTTP)

# TLS
nifi.security.keystore=/opt/nifi/conf/keystore.p12
nifi.security.keystoreType=PKCS12
nifi.security.truststore=/opt/nifi/conf/truststore.p12
nifi.security.needClientAuth=true  # Require mutual TLS

# Cluster
nifi.cluster.is.node=true          # false = standalone
nifi.cluster.node.address=         # FQDN of this node
nifi.cluster.node.protocol.port=10000

# Provenance retention
nifi.provenance.repository.max.storage.time=30 days
nifi.provenance.repository.max.storage.size=10 GB

⚠ Check: nifi.web.http.port should be empty/commented. HTTP (unencrypted) access should never be active in production.

Review bootstrap.conf — JVM heap and startup settings

Critical settings to review:

# JVM memory — tune for your instance type
java.arg.2=-Xms4g       # Min heap
java.arg.3=-Xmx16g      # Max heap (leave ~4GB for OS)
java.arg.14=-XX:+UseG1GC

# NiFi 2.x requires Java 21
java=java  # Should resolve to Java 21+ binary

Why it matters: Incorrect heap settings cause OutOfMemory errors during high-throughput operations. For an m5.2xlarge (32GB), -Xmx16g or -Xmx24g is typical.

Review authorizers.xml — who can access NiFi and how

What to look for:

  • Is it using single-user-authorizer? (Development only — not production)
  • Is it using StandardManagedAuthorizer with LDAP/OIDC? (Correct for production)
  • Who is the initial admin identity? Is it a service account or a personal account?
  • Are node identities configured for all cluster nodes?

🔴 Red flag: Single-user mode with shared credentials in a production environment. Integrate LDAP, Kerberos, or OIDC. → Authorization docs

Check if sensitive properties in nifi.properties are encrypted

How to verify: Look for nifi.sensitive.props.key — if it's empty or plain text, sensitive values aren't encrypted. Run the NiFi Toolkit encrypt-config tool:

./bin/encrypt-config.sh \
  -n conf/nifi.properties \
  -o conf/nifi.properties \
  -k "YOUR_ENCRYPTION_KEY"

Why it matters: Processor passwords (e.g., database credentials, S3 keys stored in processor config) are stored in flow.xml.gz / flow.json.gz. Without encryption, anyone with file access can read them.

✅ Best practice (NiFi 2.x): Use Parameter Contexts backed by AWS Secrets Manager or HashiCorp Vault instead of storing credentials in flow definitions. → Parameter Providers

📖 Reference: System Properties Reference · NiFi Toolkit Guide (encrypt-config)
04

Security Audit

TLS configuration, user access policies, certificate management, and secrets handling.

Verify TLS version and cipher suites

Check via: nifi.properties entries or test with openssl s_client -connect <host>:8443

Target: TLS 1.2 minimum, TLS 1.3 preferred. Disable TLS 1.0/1.1. Ensure certificate CN/SANs match the hostnames used by all cluster nodes.

nifi.web.https.ciphersuites.include=TLS_AES_256_GCM_SHA384,TLS_AES_128_GCM_SHA256
nifi.web.https.enabled.protocols=TLSv1.2,TLSv1.3

Check certificate expiry dates

How to check:

# Check keystore cert expiry
keytool -list -v -keystore /opt/nifi/conf/keystore.p12 \
  -storetype PKCS12 | grep -E "Alias|Valid|Expired"

# Or via openssl
openssl s_client -connect <nifi-host>:8443 2>/dev/null \
  | openssl x509 -noout -dates

🔴 Action required if <90 days: Create a rotation plan. Expired certificates will take down the cluster. Set up a CloudWatch alarm or calendar reminder for 60 days before expiry.

Audit users, groups, and access policies in the NiFi UI

Check via: NiFi UI → Hamburger Menu → Users / Policies

Questions to answer:

  • Are there any users with global "Modify" access who shouldn't have it?
  • Are service accounts separate from human user accounts?
  • Is there an inactive or departed employee still in the user list?
  • Are Process Groups using component-level policies (more granular) or only global policies?

Example policy structure: data-team group → Read/Write on "Ingestion" PG; ops-team → Read-only on root PG; admin → Full access.

Are Parameter Contexts used for secrets, or are credentials hardcoded in processors?

Check via: NiFi UI → Controller Settings → Parameter Contexts, or inspect individual processor configurations

Red flags: Database passwords or API keys visible as plain text in processor properties. AWS access keys configured directly instead of using EC2 instance IAM roles.

Best practice (NiFi 2.x): Use Parameter Contexts backed by AWS Secrets Manager:

Parameter Provider Type: AWS Secrets Manager Parameter Provider
AWS Region: us-east-1
Secret Name Pattern: /nifi/prod/*
→ Parameter Contexts documentation
📖 Reference: Security Configuration Guide · AWS Secrets Manager
05

Flow Design Review

Understand what data flows exist, how they're organized, and what risks they carry.

Inventory all Process Groups and identify their purpose

Check via: NiFi UI root canvas. Use the Component Search (Ctrl/Cmd + F) to list all processor types in use.

Build an inventory table:

Process GroupPurposeSourceDestination
Ingest_S3_RawS3 → KafkaS3 bucketMSK topic
Transform_CDCDB CDCRDS DMSS3 parquet

Identify processors with un-terminated relationships ("dead ends")

Why it matters: FlowFiles that queue on un-terminated connections accumulate indefinitely, filling the content repository. NiFi will stop processing when disk is full.

How to check: Look for queued FlowFiles with no downstream processor, or processors with "failure" relationships not routed to LogAttribute or a catch handler.

⚠ Pattern to watch: failure → self (retry loop without limit), large queues on error connections, connections with no back-pressure settings.

Check back-pressure settings on connections

Check via: Right-click any connection → Configure → Back Pressure section

Default values: 10,000 FlowFiles or 1 GB data. These defaults may be too low for high-throughput flows or too high for memory-constrained ones.

# Example: High-volume ingest connection
Back Pressure Object Threshold: 50000
Back Pressure Data Size Threshold: 5 GB
FlowFile Expiration: 1 hour  # Auto-expire stale flowfiles
→ Connection configuration guide

Are flows versioned in a Flow Registry (Git-backed or NiFi Registry)?

Check via: NiFi UI → Hamburger → Controller Settings → Registry Clients

Why it matters: Without versioning, a mis-click can destroy a flow with no recovery path. NiFi 2.x supports Git-based Flow Registry Clients directly (NiFi Registry is now deprecated). Check that all production process groups are pinned to a versioned commit.

✅ NiFi 2.x best practice: Use Git-based registry client pointing to a dedicated repo with branch protection on main. → NiFi 2.x Docs

Review custom or NAR (NiFi Archive) processors — what's deployed and is it trusted?

Check via: ls /opt/nifi/lib/*.nar and ls /opt/nifi/extensions/*.nar

Why it matters: Custom NARs run inside the JVM with full access. A compromised or poorly written NAR can exfiltrate data or crash the node. Verify the source, version, and build pipeline for any non-standard NARs.

Example audit:

ls -la /opt/nifi/lib/*.nar | awk '{print $NF}' | sort
# Verify checksums against release signatures
sha256sum /opt/nifi/lib/nifi-standard-nar-2.x.x.nar
📖 Reference: NiFi User Guide · Developer Guide (Custom Processors)
06

Monitoring & Observability

How is the health of this NiFi instance being tracked? What are the alerting gaps?

Is NiFi's reporting task sending metrics to CloudWatch, Prometheus, or another system?

Check via: NiFi UI → Hamburger → Controller Settings → Reporting Tasks

Useful reporting tasks:

  • PrometheusReportingTask — exposes /metrics for Prometheus scraping
  • AmbariReportingTask / custom Reporting Task → CloudWatch
  • SiteToSiteStatusReportingTask — sends internal stats via Site-to-Site

⚠ Common gap: Many deployments have no external metrics at all — you'd only know something is wrong after users complain. Set up at minimum: disk usage alerts on repositories, JVM heap alerts, and queue depth monitors.

Review log files for errors, warnings, and anomalies

Key log locations:

/opt/nifi/logs/nifi-app.log      # Main application log
/opt/nifi/logs/nifi-user.log     # User actions (audit trail)
/opt/nifi/logs/nifi-bootstrap.log # Startup/shutdown events

# Quick health check — last 100 errors:
grep -i "ERROR\|WARN\|Exception" /opt/nifi/logs/nifi-app.log | tail -100

Are logs being shipped to: CloudWatch Logs, Splunk, Elasticsearch? If not, logs rotate and are lost. Set up a CloudWatch Agent or Fluent Bit to forward logs.

Check Provenance Repository settings and search for recent failures

Check via: NiFi UI → Global Menu → Provenance (the clock icon)

Why it helps: Data provenance records every action taken on every FlowFile. You can trace a specific record from source to destination, or identify where it dropped. It's also useful for compliance (GDPR, HIPAA) to prove data lineage.

Search for failures: Filter by Event Type: DROP in the last 24h to identify FlowFiles that were discarded unexpectedly.

📖 Reference: Reporting Tasks · CloudWatch Agent Setup

AWS Service Integrations

Review each AWS service that NiFi interacts with and validate those connections are correctly configured.

☁️ Amazon S3

• Which buckets does NiFi read from / write to?

• Are bucket policies restricting access to the NiFi IAM role only?

• Is server-side encryption (SSE-S3 or SSE-KMS) enabled?

• Is S3 VPC Endpoint configured to avoid public internet transit?

→ S3 Bucket Policy reference

📨 Amazon SQS / SNS

• Which queues does NiFi consume from (GetSQS) or publish to (PutSQS)?

• Are SQS queues configured with Dead Letter Queues (DLQ)?

• Is visibility timeout aligned with NiFi processor scheduling interval?

→ SQS DLQ configuration

🔑 AWS Secrets Manager

• Are NiFi Parameter Contexts backed by Secrets Manager?

• Is the NiFi IAM role granted secretsmanager:GetSecretValue?

• Are secrets using automatic rotation policies?

→ Secrets rotation

📊 Amazon Kinesis / MSK

• Is NiFi using ConsumeKinesisStream or Kafka processors for MSK?

• For MSK: Is mTLS or SCRAM authentication configured?

• Are consumer group offsets monitored?

→ MSK authentication

🗄️ RDS / Databases

• Which databases does NiFi connect to (ExecuteSQL, PutDatabaseRecord, etc.)?

• Are database credentials in Secrets Manager (not hardcoded)?

• Is the RDS Security Group restricted to NiFi's subnet CIDR only?

→ RDS security groups

📈 CloudWatch & Alerts

• Are there CloudWatch Alarms for EC2 disk, CPU, memory (via CW Agent)?

• Are log groups created for NiFi application and user logs?

• Is there an SNS topic for alert notifications?

→ CloudWatch Alarms

Handoff Checklist & Action Plan

Consolidate findings and create a prioritized action plan before taking full ownership.

Document the "as-found" state in a written report

Capture: NiFi version, cluster topology, AWS resources, security posture, flow inventory, monitoring gaps, and open findings. Store in your team's wiki or Git repo before making any changes. This protects you if something breaks.

Prioritize findings into P1/P2/P3 buckets

P1 — Fix Immediately

Exposed ports, expired certs, hardcoded secrets, no backups, single point of failure in prod, HTTP enabled

P2 — Fix This Sprint

No flow versioning, no external monitoring, single-user auth, unchecked queue growth, unrotated credentials

P3 — Planned Improvements

Upgrade NiFi version, consolidate IAM roles, move secrets to Secrets Manager, add Prometheus dashboards, documentation gaps

Verify backup and restore capability

Can you restore NiFi to a known-good state if something goes wrong?

  • Are EC2 AMI snapshots or EBS snapshots scheduled via AWS Backup?
  • Is flow.json.gz (or flow.xml.gz) backed up separately?
  • Is the /opt/nifi/conf/ directory in source control?
  • Have you tested a restore? (not just backup)
→ NiFi Toolkit File Manager (backup/restore)

Establish a runbook and on-call escalation path

Document: how to start/stop NiFi nodes, how to remove a node from a cluster, how to force a cluster election, what to do if a queue fills up, who to contact for upstream/downstream systems.

# Common operational commands (NiFi 2.x)
/opt/nifi/bin/nifi.sh start
/opt/nifi/bin/nifi.sh stop
/opt/nifi/bin/nifi.sh status
/opt/nifi/bin/nifi.sh diagnostics   # Captures cluster state

Real-World Example: Law Firm AI Document Pipeline

A complete ETL use case — contract review and brief preparation powered by NiFi, Claude AI, and Elasticsearch on AWS.

Scenario: A law firm receives hundreds of documents weekly in mixed formats — scanned contracts, Word briefs, email attachments, court filing images. This pipeline converts everything to searchable text, enriches it with AI, and stores it so attorneys can search, review, and build briefs quickly.

9.1 — Text Extraction Details

Documents are routed by MIME type after IdentifyMimeType inspection:

# RouteOnAttribute rules
pdf:       ${mime.type:equals('application/pdf')}
docx:      ${mime.type:equals('application/vnd.openxmlformats...')}
image:     ${mime.type:startsWith('image/')}
email:     ${mime.type:equals('message/rfc822')}
plaintext: ${mime.type:equals('text/plain')}
PDF
AWS Textract (async) for scanned PDFs. Apache Tika for text-native PDFs. Async job result retrieved via SQS notification.
DOCX
Apache POI via ExecuteStreamCommand. Preserves headings, paragraphs, tables. Also handles legacy .doc via LibreOffice conversion.
Images
AWS Textract DetectDocumentText (JPG/PNG/TIFF). AnalyzeDocument for forms with key-value pairs. Tesseract as offline fallback.

9.2 — Claude AI Enrichment

InvokeHTTP calls the Claude API with extracted text. The system prompt instructs Claude to return structured JSON only:

System: You are a legal document analyst. Return ONLY valid JSON with this schema:
{
  "document_type": "contract|brief|motion|exhibit|correspondence|order",
  "parties": ["Party A", "Party B"],
  "key_clauses": ["indemnification", "auto-renewal"],
  "risk_flags": ["unilateral modification rights"],
  "summary": "3-5 sentence plain English summary for attorneys",
  "matter_tags": ["M&A", "commercial"],
  "governing_law": "Delaware"
}
⚠ Large documents: Use SplitText to chunk contracts over 80,000 tokens into overlapping 50,000-token segments before calling Claude. Reassemble with MergeContent after enrichment.

9.3 — Elasticsearch Document Schema

{
  "document_id":   "uuid",
  "matter_id":     "2024-CORP-0041",
  "filename":      "MSA_Acme_Corp_v2.pdf",
  "document_type": "contract",
  "parties":       ["Acme Corp", "Client LLC"],
  "received_at":   "2024-05-21T14:22:00Z",
  "extracted_text":"Full plain text of the document...",
  "summary":       "3-5 sentence attorney summary",
  "key_clauses":   ["indemnification", "auto-renewal"],
  "risk_flags":    ["unilateral modification"],
  "matter_tags":   ["M&A", "commercial"],
  "governing_law": "Delaware",
  "sha256":        "a3f9...",
  "pipeline_version": "1.4.0"
}

Field mapping: extracted_text → text/standard; document_type, matter_tags, governing_law → keyword (faceted filters); received_at → date. Enable ILM: warm after 30 days, delete after 7 years.

9.4 — AWS Services Required

Compute

  • 3× EC2 m5.2xlarge — NiFi cluster
  • 2× EC2 r6i.xlarge — Elasticsearch data nodes
  • 1× EC2 t3.medium — ES master node

Storage

  • S3: intake + archive (Glacier ILM)
  • EBS GP3: 500 GB/node content_repository
  • EBS GP3: 2 TB/node Elasticsearch

AWS managed services

  • AWS Textract — PDF/image OCR
  • Secrets Manager — API keys, credentials
  • KMS — EBS + S3 encryption at rest
  • CloudWatch — logs, alarms, dashboards
  • AWS WAF — protect NiFi ALB
  • AWS Backup — daily EBS snapshots
  • MSK (optional) — durable intake queue

9.5 — Review Questions for This Pipeline

Is Textract async job polling handled correctly? (StartDocumentTextDetection → polling loop via SQS)

Is the Claude API called with chunked text for contracts over 80k tokens?

Is there a dead-letter re-processing path for failed AI enrichment records?

Are attorney PII and privilege obligations met? (Zero-retention API agreement with Anthropic confirmed?)

Are extracted texts stored in ES or referenced from S3? (Index size vs. search latency trade-off reviewed?)
